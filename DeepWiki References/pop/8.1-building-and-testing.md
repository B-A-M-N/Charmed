### Writing Tests

Pop follows standard Go testing practices. Test files should be named with a `_test.go` suffix and placed in the same package as the code they test.

For example, to test functionality in `internal/tui/form.go`, create a file named `internal/tui/form_test.go`.

Linting and Code Quality
------------------------

Pop uses `golangci-lint` for code quality checks with a specific configuration.

### Running Linters Locally

### Enabled Linters

Pop has several linters enabled to ensure code quality:

| Linter | Purpose |
| --- | --- |
| bodyclose | Checks that HTTP response bodies are closed |
| exportloopref | Checks for pointers to enclosing loop variables |
| goimports | Checks import formatting |
| gosec | Inspects source code for security problems |
| nilerr | Finds code that returns nil even if it checks that error is not nil |
| revive | Fast, configurable, extensible, flexible, and beautiful linter |
| rowserrcheck | Checks whether Rows.Err is checked after Rows.Next |
| sqlclosecheck | Checks that SQL rows and statements are properly closed |
| tparallel | Detects inappropriate usage of t.Parallel() method |
| unconvert | Removes unnecessary type conversions |
| unparam | Reports unused function parameters |
| whitespace | Tool for detection of leading and trailing whitespace |

Sources: [.golangci.yml](https://github.com/charmbracelet/pop/blob/cdaa7255/.golangci.yml)

Release Process
---------------

Pop uses GoReleaser for creating and publishing releases.

### Creating a New Release

1.  Ensure all changes are committed and pushed
2.  Create and push a new tag with semantic versioning:
    
3.  The release workflow will automatically be triggered

### Release Workflow

The release workflow creates binaries for multiple operating systems and architectures, packages them appropriately, and publishes them to various distribution channels.

Sources: [.github/workflows/goreleaser.yml](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/workflows/goreleaser.yml)

### Release Artifacts

A typical release includes:

*   Binary executables for various platforms
*   Debian and RPM packages
*   Homebrew formulas
*   Docker images
*   Arch Linux packages (via AUR)
