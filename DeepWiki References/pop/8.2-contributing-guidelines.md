Ways to Contribute
------------------

There are several ways to contribute to the Pop email client:

1.  Reporting bugs and issues
2.  Suggesting new features or enhancements
3.  Submitting code changes and improvements
4.  Improving documentation
5.  Participating in discussions

Sources: [.github/ISSUE\_TEMPLATE/bug\_report.md](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/ISSUE_TEMPLATE/bug_report.md?plain=1)
 [.github/ISSUE\_TEMPLATE/feature\_request.md](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/ISSUE_TEMPLATE/feature_request.md?plain=1)

Issue Reporting Process
-----------------------

### Bug Reports

When reporting a bug, please use the provided bug report template and ensure you include:

1.  A clear description of the bug
2.  Steps to reproduce the issue
3.  Expected behavior
4.  Screenshots (if applicable)
5.  Environment information (OS, version)
6.  Any additional context

Bug reports are managed through GitHub issues which help maintainers track and address problems efficiently.

### Feature Requests

For feature requests, use the feature request template and include:

1.  Description of the problem the feature would solve
2.  Clear explanation of your proposed solution
3.  Alternative solutions you've considered
4.  Any additional context or screenshots

Sources: [.github/ISSUE\_TEMPLATE/bug\_report.md](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/ISSUE_TEMPLATE/bug_report.md?plain=1)
 [.github/ISSUE\_TEMPLATE/feature\_request.md](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/ISSUE_TEMPLATE/feature_request.md?plain=1)

Pull Request Process
--------------------

### Setting Up Development Environment

1.  Fork the Pop repository on GitHub
2.  Clone your fork locally
3.  Set up remote to track the upstream repository
4.  Create a new branch for your changes

### Making Changes

1.  Make your changes following the code style guidelines
2.  Write tests for your changes when applicable
3.  Ensure all tests pass locally
4.  Update documentation if necessary

### Submitting Pull Requests

1.  Push your changes to your fork
2.  Create a pull request to the main Pop repository
3.  Fill in the pull request template with details about your changes
4.  Link any related issues using GitHub keywords (fixes, closes, etc.)

### Code Review Process

After submitting a pull request:

1.  Maintainers will review your code
2.  Automated tests will run via GitHub Actions
3.  You may need to make additional changes based on feedback
4.  Once approved, maintainers will merge your contribution

Sources: [.github/workflows/nightly.yml](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/workflows/nightly.yml)

Code Style and Guidelines
-------------------------

### Coding Standards

When contributing code to Pop, please adhere to these standards:

1.  Follow Go best practices and idioms
2.  Use meaningful variable and function names
3.  Write clear comments for complex logic
4.  Include documentation for public APIs
5.  Make sure your code passes formatting checks

### Testing Requirements

1.  Add appropriate tests for your changes
2.  Ensure all existing tests pass
3.  Consider edge cases in your test coverage

### Documentation

Documentation updates should be provided for:

1.  New features
2.  Changed functionality
3.  Deprecations or removals
4.  Configuration changes

Dependency Management
---------------------

Pop uses Dependabot to manage dependencies across different ecosystems:

| Ecosystem | Update Schedule | Configuration |
| --- | --- | --- |
| Go Modules | Weekly (Monday, 5:00 AM ET) | [.github/dependabot.yml4-15](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/dependabot.yml#L4-L15) |
| GitHub Actions | Weekly (Monday, 5:00 AM ET) | [.github/dependabot.yml17-28](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/dependabot.yml#L17-L28) |
| Docker | Weekly (Monday, 5:00 AM ET) | [.github/dependabot.yml30-41](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/dependabot.yml#L30-L41) |

Sources: [.github/dependabot.yml](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/dependabot.yml)
 [.github/workflows/dependabot-sync.yml](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/workflows/dependabot-sync.yml)

### Handling Dependencies

1.  Dependency updates from Dependabot should be reviewed for breaking changes before merging
2.  Manual dependency updates should follow the same process as other code changes
3.  When adding a new dependency, consider:
    *   Licensing compatibility
    *   Maintenance status
    *   Size and performance impact

Release Process and Versioning
------------------------------

Pop follows a structured release process:

1.  Nightly builds are automatically generated from the main branch
2.  Stable releases are created using semantic versioning (MAJOR.MINOR.PATCH)
3.  Release notes document notable changes, improvements, and fixes

Sources: [.github/workflows/nightly.yml](https://github.com/charmbracelet/pop/blob/cdaa7255/.github/workflows/nightly.yml)

Community Conduct
-----------------

All contributors are expected to:

1.  Be respectful and inclusive in all interactions
2.  Provide constructive feedback
3.  Focus on the technical merits of discussions
4.  Follow the project's code of conduct
