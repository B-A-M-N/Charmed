Email Delivery System
=====================

Relevant source files

*   [email.go](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go)
    
*   [main.go](https://github.com/charmbracelet/pop/blob/cdaa7255/main.go)
    

This document details the email delivery functionality in Pop, focusing on how the system processes and sends emails, including Markdown to HTML conversion and error handling. For information about the user interface components that collect email data, see [User Interface Components](https://deepwiki.com/charmbracelet/pop/3.2-user-interface-components)
.

Overview of the Email Delivery System
-------------------------------------

The Pop Email Delivery System handles the core functionality of sending emails through different providers. It supports two primary delivery methods: Resend API and SMTP servers. The system takes email data (recipients, subject, body, attachments), processes it, and attempts to deliver it.

Sources: [email.go33-60](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go#L33-L60)
 [email.go66-132](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go#L66-L132)
 [email.go134-172](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go#L134-L172)

Delivery Method Selection
-------------------------

Pop determines which delivery method to use based on available credentials in environment variables or command-line arguments.

Sources: [main.go77-88](https://github.com/charmbracelet/pop/blob/cdaa7255/main.go#L77-L88)

The decision logic is as follows:

*   If both Resend API key and SMTP credentials are provided, an error is shown due to ambiguity
*   If only Resend API key is provided, Resend API is used
*   If only SMTP credentials are provided, SMTP is used
*   If no credentials are provided, an error message prompts the user to set up a delivery method

Delivery Methods Implementation
-------------------------------

### SMTP Email Delivery

The SMTP delivery method uses the `go-simple-mail` library to connect to and send emails through SMTP servers.

Sources: [email.go66-132](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go#L66-L132)

Key features of SMTP delivery:

*   Automatic Gmail configuration detection
*   Support for different encryption types (STARTTLS, SSL, none)
*   TLS configuration with optional verification skipping
*   Fallback to plain text if Markdown conversion fails

### Resend API Delivery

The Resend API delivery method uses the Resend Go client library.

Sources: [email.go134-172](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go#L134-L172)

Key features of Resend API delivery:

*   Optional "unsafe" HTML mode that enables additional Markdown features
*   Sends both HTML and plain text versions of the email
*   Handles attachments as base64-encoded content

Markdown to HTML Conversion
---------------------------

Both delivery methods use the Goldmark library to convert Markdown content to HTML. The conversion process differs slightly between Resend and SMTP implementations:

| Feature | Resend API | SMTP |
| --- | --- | --- |
| Standard Markdown | ✅   | ✅   |
| Error handling | Ignores conversion errors | Falls back to plain text on error |
| Unsafe HTML | Optional (`--unsafe` flag) | Not supported |
| Extended features | Tables, strikethrough, linkify (in unsafe mode) | None |

Sources: [email.go115-122](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go#L115-L122)
 [email.go137-153](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go#L137-L153)

Attachment Handling
-------------------

Attachments are processed differently depending on the delivery method:

*   **SMTP**: Attachments are referenced by file path and attached directly by the mail library
*   **Resend API**: Files are read into memory and encoded as part of the API request

Error Handling
--------------

The error handling system employs a robust approach to ensure users don't lose their email content even when delivery fails.

Sources: [email.go51-58](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go#L51-L58)
 [email.go194-209](https://github.com/charmbracelet/pop/blob/cdaa7255/email.go#L194-L209)

When an email fails to send:

1.  The system attempts to save the email body to a temporary file with a timestamp-based name
2.  If successful, the error message is augmented with the path to the saved file
3.  The error is returned to the UI layer for display to the user
4.  This allows users to retrieve their content and try again or use another method

The temporary file naming follows this pattern: `pop-YYYY-MM-DD-*.txt` in the system's temporary directory.

Integration with User Interface
-------------------------------

The Email Delivery System is designed to work seamlessly with both the CLI and TUI interfaces of Pop:

*   In CLI mode, the system is called directly from the `RunE` function in `main.go`
*   In TUI mode, the system is invoked via the `sendEmailCmd` method, which returns a Bubble Tea command

This separation allows the same core delivery functionality to be used regardless of how the user interacts with Pop.
