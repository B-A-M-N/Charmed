Helper Functions
================

Relevant source files

*   [logger.go](https://github.com/charmbracelet/log/blob/cf6e8671/logger.go)
    
*   [text\_test.go](https://github.com/charmbracelet/log/blob/cf6e8671/text_test.go)
    

The Charmbracelet log library provides specialized helper functions that improve the accuracy of logging source locations. These functions allow you to mark specific functions as "helpers" so they're skipped when the logger reports the source location of log messages, resulting in cleaner and more meaningful stack traces in your logs.

Purpose and Scope
-----------------

Helper functions address a common problem in logging libraries: when wrapping a logger in utility functions, the reported source location often points to the utility function rather than where it was actually called from. By using helper functions, you can ensure that log messages show the correct caller location, making debugging easier and more intuitive.

Sources: [logger.go142-147](https://github.com/charmbracelet/log/blob/cf6e8671/logger.go#L142-L147)
 [text\_test.go212-225](https://github.com/charmbracelet/log/blob/cf6e8671/text_test.go#L212-L225)

How Helper Functions Work
-------------------------

The Charmbracelet log library implements helper functionality through the `Helper()` method on the Logger struct. When this method is called, it marks the calling function as a helper, which will then be skipped when determining the source location of log messages.

When you call `logger.Helper()` in a function, that function's name is stored in an internal map. Later, when log messages are generated, the logger examines the call stack and skips over any functions that have been marked as helpers, reporting the first non-helper function as the source of the log message.

Sources: [logger.go66-81](https://github.com/charmbracelet/log/blob/cf6e8671/logger.go#L66-L81)
 [logger.go142-156](https://github.com/charmbracelet/log/blob/cf6e8671/logger.go#L142-L156)

Helper Function Components
--------------------------

The helper function system consists of several interconnected components in the codebase:

The key components include:

1.  `Logger.Helper()` - Public method that marks the calling function as a helper
2.  `Logger.helper()` - Internal implementation that stores function names
3.  `Logger.frames()` - Captures the call stack frames for analysis
4.  `Logger.location()` - Extracts file, line, and function information from frames

Sources: [logger.go142-176](https://github.com/charmbracelet/log/blob/cf6e8671/logger.go#L142-L176)

1.  **Utility logging functions**: When creating convenience functions that wrap logging operations
2.  **Middleware**: In logging middleware that should be transparent in the stack trace
3.  **Testing helpers**: For test utilities that perform logging but shouldn't appear as the source

Technical Implementation Details
--------------------------------

### Helper Registration

When `logger.Helper()` is called, the function name of the caller is stored in a `sync.Map` within the Logger struct:

    type Logger struct {
        // other fields...
        helpers *sync.Map
        // other fields...
    }
    

The `helper()` method uses `runtime.Callers()` to capture the current call stack and registers the function name:

    func (l *Logger) helper(skip int) {
        var pcs [1]uintptr
        n := runtime.Callers(skip+2, pcs[:])
        frames := runtime.CallersFrames(pcs[:n])
        frame, _ := frames.Next()
        l.helpers.LoadOrStore(frame.Function, struct{}{})
    }
    

Sources: [logger.go24-48](https://github.com/charmbracelet/log/blob/cf6e8671/logger.go#L24-L48)
 [logger.go149-156](https://github.com/charmbracelet/log/blob/cf6e8671/logger.go#L149-L156)

### Stack Frame Processing

When determining the caller location for a log message, the logger:

1.  Captures the call stack using `frames()`
2.  Iterates through each frame
3.  Checks if the frame corresponds to a registered helper function
4.  Uses the first non-helper frame as the source location

This logic is implemented in the `Log()` method:

    frames := l.frames(l.callerOffset + 2)
    for {
        f, more := frames.Next()
        _, helper := l.helpers.Load(f.Function)
        if !helper || !more {
            // Found a frame that wasn't a helper function.
            // Or we ran out of frames to check.
            frame = f
            break
        }
    }
    

Sources: [logger.go66-81](https://github.com/charmbracelet/log/blob/cf6e8671/logger.go#L66-L81)

Best Practices
--------------

1.  **Call `Helper()` at the beginning**: Always call `Helper()` at the beginning of your function to ensure all paths are covered
2.  **Chain helpers properly**: In nested helper functions, each function should call `Helper()`
3.  **Use helpers judiciously**: Only mark true utility functions as helpers; overuse can make debugging more difficult

Sources: [text\_test.go212-225](https://github.com/charmbracelet/log/blob/cf6e8671/text_test.go#L212-L225)
 [text\_test.go77-90](https://github.com/charmbracelet/log/blob/cf6e8671/text_test.go#L77-L90)

Summary
-------

Helper functions in the Charmbracelet log library provide a clean solution for maintaining accurate caller information in logs. They allow you to create utility logging functions without sacrificing the ability to trace where log messages actually originated in your code. By properly marking helper functions, you can ensure your logs contain the most useful context for debugging.
