Introduction
------------

This document covers the logging and metrics middleware components provided by the wish SSH server framework. These middlewares enable server operators to capture information about SSH sessions, monitor connection details, and measure performance metrics such as session duration.

The logging and metrics functionality is provided by two primary packages:

*   The `logging` package for session event logging
*   The `elapsed` package for session duration measurement

These middlewares are part of the larger middleware system described in [Middleware System](https://deepwiki.com/charmbracelet/wish/3-middleware-system)
.

Middleware Architecture Context
-------------------------------

The logging and metrics middlewares work within wish's middleware chain architecture, where each middleware can perform pre-processing before the session handler and post-processing after it.

Logging Middleware
------------------

The logging middleware provides detailed information about SSH connections, capturing metadata at connection establishment and disconnection.

### Standard Logging

The standard logging middleware uses a simple string-formatted log output. It records:

*   User information
*   Remote address
*   Authentication method (public key or not)
*   Executed command
*   Terminal information (TERM setting, window dimensions)
*   Client version
*   Connection duration upon disconnect

Sources: [logging/logging.go11-19](https://github.com/charmbracelet/wish/blob/19f43236/logging/logging.go#L11-L19)
 [logging/logging.go26-55](https://github.com/charmbracelet/wish/blob/19f43236/logging/logging.go#L26-L55)

### Structured Logging

For more sophisticated logging needs, the structured logging middleware produces structured logs using the charmbracelet/log library. This approach makes logs more machine-readable and easier to parse for analysis.

The structured logging middleware captures the same information as the standard logger but in a key-value format that's better suited for log aggregation and analysis systems.

Sources: [logging/logging.go57-99](https://github.com/charmbracelet/wish/blob/19f43236/logging/logging.go#L57-L99)

Elapsed Time Middleware
-----------------------

The elapsed time middleware serves a more focused purpose: measuring and reporting the total duration of an SSH session.

### Purpose and Functionality

This middleware:

1.  Records the start time when a session begins
2.  Passes the session to the next handler in the chain
3.  Calculates and prints the elapsed time when the session completes

Sources: [elapsed/elapsed.go10-31](https://github.com/charmbracelet/wish/blob/19f43236/elapsed/elapsed.go#L10-L31)

### Implementation Details

The elapsed middleware is implemented with two key functions:

1.  `Middleware()` - Uses a default format string for displaying elapsed time
2.  `MiddlewareWithFormat(format string)` - Allows customizing the format of the elapsed time message

Important note: For accurate session duration measurement, the elapsed middleware should be placed last in the middleware chain, so it can measure the complete session time including all other middleware processing.

Sources: [elapsed/elapsed.go10-31](https://github.com/charmbracelet/wish/blob/19f43236/elapsed/elapsed.go#L10-L31)

Integration Best Practices
--------------------------

For optimal logging and metrics:

1.  Place the logging middleware early in the middleware chain to capture accurate connection information
2.  Place the elapsed middleware last in the chain to measure the complete session duration
3.  Use structured logging in production environments for better log analysis
4.  Consider using a custom logger for production deployments to redirect logs to appropriate destinations
