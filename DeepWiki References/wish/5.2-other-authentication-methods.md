Purpose and Scope
-----------------

This document covers authentication methods in `wish` beyond public key authentication. While public key authentication is generally the recommended approach for SSH (for details, see [Public Key Authentication](https://deepwiki.com/charmbracelet/wish/5.1-public-key-authentication)
), this page explains how to implement password-based authentication and keyboard-interactive authentication in your SSH applications built with `wish`.

Combining Authentication Methods
--------------------------------

You can enable multiple authentication methods simultaneously in your SSH server. When multiple methods are enabled, the SSH protocol allows clients to choose which method to use.

### Priority and Fallback

The SSH protocol defines how authentication methods are negotiated:

1.  The server advertises which authentication methods it supports
2.  The client tries methods in its preferred order
3.  If a method fails, the client can try another method
4.  Authentication succeeds when any method succeeds

This allows for flexible authentication flows, such as falling back to password authentication if public key authentication fails.

Security Considerations
-----------------------

When implementing authentication methods, consider the following security implications:

| Authentication Method | Strengths | Weaknesses | Best For |
| --- | --- | --- | --- |
| Public Key | \- No passwords to steal or crack  <br>\- Strong cryptographic security  <br>\- Can be tied to user identity  <br>\- Can be revoked individually | \- Requires key management  <br>\- Keys can be copied if not protected | Production systems, Git operations, secure access |
| Password | \- Familiar to users  <br>\- Easy to implement  <br>\- No key management | \- Subject to brute force attacks  <br>\- Can be stolen or leaked  <br>\- Can be weak if chosen by users | Development, testing, simple applications |
| Keyboard-Interactive | \- Flexible for custom flows  <br>\- Can implement 2FA  <br>\- Better UX for complex auth | \- Complex to implement correctly  <br>\- Depends on client support | Multi-factor auth, OTP systems, compliance requirements |

### Recommendations

1.  **For production systems:** Use public key authentication when possible, with strong key requirements.
2.  **For development/testing:** Password or keyboard-interactive can be acceptable if properly secured.
3.  **For multi-factor security:** Combine methods or use keyboard-interactive to implement 2FA.
4.  **Password security:** If using password authentication:
    *   Enforce strong password policies
    *   Implement rate limiting to prevent brute force attacks
    *   Consider using the rate limiting middleware (see [Rate Limiting Middleware](https://deepwiki.com/charmbracelet/wish/3.3-rate-limiting-middleware)
        )
    *   Store passwords securely (hashed and salted)

Integration with Existing Systems
---------------------------------

If you need to integrate with existing authentication systems, consider these approaches:

### Integration Examples

*   **Database integration:** Validate passwords against a user database
*   **LDAP integration:** Connect to an LDAP directory for user authentication
*   **OAuth integration:** Use keyboard-interactive to implement an OAuth flow
*   **2FA integration:** Use keyboard-interactive to add a second factor after password authentication

Sources: [options.go173-185](https://github.com/charmbracelet/wish/blob/19f43236/options.go#L173-L185)

Conclusion
----------

The `wish` library provides flexible authentication options to secure your SSH applications. By understanding the different authentication methods and their trade-offs, you can implement the most appropriate security model for your specific needs. While public key authentication remains the recommended approach for production systems, password and keyboard-interactive authentication provide alternatives for specific use cases and integration scenarios.
