Project Structure
=================

Relevant source files

*   [.gitignore](https://github.com/charmbracelet/wish/blob/19f43236/.gitignore)
    
*   [examples/go.mod](https://github.com/charmbracelet/wish/blob/19f43236/examples/go.mod)

This page documents the organization of the charmbracelet/wish codebase, intended for developers who want to understand the repository structure, key components, and relationships between files. Understanding this structure is essential for contributors looking to modify or extend wish functionality.

Overview
--------

The wish project is structured as a Go module that provides an SSH server framework with extensible capabilities through middleware. The codebase follows a modular design with core functionality in the root directory and specialized features in subdirectories.

Sources: [go.mod](https://github.com/charmbracelet/wish/blob/19f43236/go.mod)
 [.gitignore](https://github.com/charmbracelet/wish/blob/19f43236/.gitignore)

Core Components
---------------

The core SSH server implementation resides in the root directory, consisting of several key files that provide the foundation for building SSH applications.

Sources: [go.mod](https://github.com/charmbracelet/wish/blob/19f43236/go.mod)

The core components handle:

*   **Server Implementation**: The main server implementation in wish.go provides the SSH server functionality.
*   **Server Configuration**: Options for configuring the SSH server are defined in options.go.
*   **Command Execution**: Command execution handling is implemented in cmd.go.
*   **Middleware Chain**: The middleware system that enables extensibility.

Middleware Architecture
-----------------------

The middleware system is a key design pattern in wish. Middleware modules are organized in subdirectories, each focused on a specific functionality. Middleware is applied in a chain, allowing for composition of features.

### Middleware Directory Structure

Each middleware component is contained in its own directory with a consistent internal structure:

| Directory | Main Purpose | Key Files |
| --- | --- | --- |
| bubbletea/ | Terminal UI integration | tea.go |
| git/ | Git server capabilities | git.go |
| scp/ | File transfer | scp.go |
| logging/ | Session logging | logging.go |
| accesscontrol/ | Command restrictions | accesscontrol.go |
| activeterm/ | Terminal requirement | activeterm.go |
| ratelimit/ | Connection rate limiting | ratelimit.go |

### Middleware Flow

Sources: [go.mod](https://github.com/charmbracelet/wish/blob/19f43236/go.mod)

Extension Modules
-----------------

The wish library includes several extension modules that build upon the core SSH server functionality.

### BubbleTea Integration

The BubbleTea middleware enables running terminal UI applications over SSH. It connects the SSH session to the Bubble Tea TUI framework.

Sources: [go.mod](https://github.com/charmbracelet/wish/blob/19f43236/go.mod)

### Git Server Implementation

The Git middleware enables hosting Git repositories over SSH, handling Git commands like git-receive-pack and git-upload-pack.

Sources: [go.mod](https://github.com/charmbracelet/wish/blob/19f43236/go.mod)

### Other Extension Modules

*   **SCP/SFTP**: File transfer capabilities
*   **Logging**: Session logging and metrics
*   **Access Control**: Command and access restrictions
*   **Rate Limiting**: Connection rate limiting

Example Applications
--------------------

The repository includes an `examples/` directory with sample applications demonstrating different wish capabilities:

| Example Type | Purpose | Key Examples |
| --- | --- | --- |
| BubbleTea | Terminal UI over SSH | Simple TUI apps |
| Git Server | Git repository hosting | Git server implementation |
| Basic SSH | Core SSH functionality | Command execution |
| Composite | Multiple features | Combined middleware examples |

Each example in the examples directory includes a complete, runnable application that demonstrates one or more wish features.

Sources: [examples/go.mod](https://github.com/charmbracelet/wish/blob/19f43236/examples/go.mod)

Dependencies
------------

Wish relies on several key dependencies, primarily from the Charm ecosystem:

Sources: [go.mod](https://github.com/charmbracelet/wish/blob/19f43236/go.mod)

### Key Dependencies

| Dependency | Purpose |
| --- | --- |
| github.com/charmbracelet/ssh | SSH implementation |
| github.com/charmbracelet/bubbletea | Terminal UI framework |
| github.com/charmbracelet/keygen | SSH key generation |
| github.com/charmbracelet/lipgloss | Terminal styling |
| github.com/charmbracelet/log | Structured logging |
| github.com/go-git/go-git/v5 | Git implementation |
| golang.org/x/crypto | Cryptographic primitives |
| golang.org/x/sync | Synchronization utilities |
| golang.org/x/time | Rate limiting |

Development Patterns
--------------------

The codebase follows several consistent patterns:

1.  **Middleware Pattern**: Functionality is implemented as middleware that can be composed.
2.  **Options Pattern**: Server configuration is done using functional options.
3.  **Handler Functions**: SSH session handling is done through handler functions.
4.  **Hooks**: Extension points are provided through hook functions.

Repository Organization
-----------------------

The overall repository organization follows this pattern:

    charmbracelet/wish/
    ├── *.go                  # Core SSH server implementation
    ├── bubbletea/            # Terminal UI integration
    ├── git/                  # Git server implementation
    ├── scp/                  # File transfer
    ├── logging/              # Logging middleware
    ├── accesscontrol/        # Access control middleware
    ├── activeterm/           # Active terminal checking
    ├── ratelimit/            # Rate limiting middleware
    ├── examples/             # Example applications
    │   ├── bubbletea/        # Terminal UI examples
    │   ├── git/              # Git server examples
    │   └── ...               # Other examples
    ├── .github/              # GitHub-specific files
    └── go.mod                # Module dependencies
    

This structure allows for modular development and clear separation of concerns, making it easier to maintain and extend the codebase.
