Public Key Authentication
=========================

This document explains how to implement SSH key-based authentication in `wish`\-based SSH servers. It covers the configuration options available for public key authentication, including authorized\_keys files, certificate authorities, and custom authentication handlers. For password-based and other authentication methods, see [Other Authentication Methods](https://deepwiki.com/charmbracelet/wish/5.2-other-authentication-methods)

Overview of SSH Public Key Authentication
-----------------------------------------

SSH public key authentication is a secure method where clients authenticate to an SSH server using cryptographic key pairs rather than passwords. This is generally considered more secure than password authentication.

### 2\. Authorized Keys File

For compatibility with standard SSH setups, you can use an authorized\_keys file:

**Authorized Keys Authentication Flow**

The `WithAuthorizedKeys` function specifies a path to an authorized\_keys file that contains public keys for authorized users:

The format follows the standard OpenSSH authorized\_keys format, where each line contains a public key.

### 3\. Certificate Authority Authentication

For more advanced setups, `wish` supports SSH certificates signed by a Certificate Authority:

**Certificate Authority Authentication Flow**

Sources: [options.go99-134](https://github.com/charmbracelet/wish/blob/19f43236/options.go#L99-L134)

The `WithTrustedUserCAKeys` function specifies a path to a file containing public keys of trusted Certificate Authorities:

Any user presenting a valid certificate signed by one of these CAs will be authenticated. This approach is particularly useful for:

*   Large organizations managing many users
*   Environments requiring fine-grained access control through certificate principals
*   Systems needing automated key rotation through certificate expiry

Sources: [options.go99-134](https://github.com/charmbracelet/wish/blob/19f43236/options.go#L99-L134)

Implementation Details
----------------------

### How `WithAuthorizedKeys` Works

The `WithAuthorizedKeys` function implementation:

1.  Verifies the authorized\_keys file exists
2.  Creates a custom public key handler using `WithPublicKeyAuth`
3.  Inside this handler, uses the `isAuthorized` helper function to:
    *   Open and read the authorized\_keys file line by line
    *   Skip empty lines and comments
    *   Parse each line as an authorized key
    *   Compare the client's key with each key using the `ssh.KeysEqual` function
    *   Return true if any key matches, false otherwise

Sources: [options.go86-97](https://github.com/charmbracelet/wish/blob/19f43236/options.go#L86-L97)
 [options.go136-170](https://github.com/charmbracelet/wish/blob/19f43236/options.go#L136-L170)

### How `WithTrustedUserCAKeys` Works

The `WithTrustedUserCAKeys` function operates similarly:

1.  Verifies the CA keys file exists
2.  Creates a custom public key handler using `WithPublicKeyAuth`
3.  Inside this handler:
    *   Checks if the client's key is a certificate (type assertion to `*gossh.Certificate`)
    *   For each CA key in the file, creates a `CertChecker` to verify:
        *   The certificate is signed by a trusted CA
        *   The certificate is valid for the current time
        *   The certificate is valid for the authenticating user
    *   Returns true only if all verifications pass

Sources: [options.go99-134](https://github.com/charmbracelet/wish/blob/19f43236/options.go#L99-L134)

Best Practices
--------------

| Practice | Description |
| --- | --- |
| **Key Types** | Prefer Ed25519 keys over RSA for better security and performance |
| **File Permissions** | Ensure authorized\_keys and CA keys files have proper permissions (readable only by the server) |
| **Multiple Auth Methods** | For higher security, consider combining multiple authentication methods |
| **Key Rotation** | Implement procedures for rotating CA keys and user keys |
| **Rate Limiting** | Use middleware to implement rate limiting on authentication attempts |
| **Logging** | Log authentication attempts to detect potential security issues |
