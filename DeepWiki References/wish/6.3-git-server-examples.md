### Access Control Levels

The Git middleware provides three levels of access control:

| Access Level | Description | Constant |
| --- | --- | --- |
| No Access | User cannot access the repository | `git.NoAccess` |
| Read-Only | User can clone/fetch but not push | `git.ReadOnlyAccess` |
| Read-Write | User can clone/fetch and push | `git.ReadWriteAccess` |

Usage Examples
--------------

### Setting Up the Server

To run the example Git server:

1.  Create a directory for repositories:

    mkdir .repos
    

2.  Run the server:

    go run examples/git/main.go
    

The server will start on localhost:23233.

### Cloning a Repository

Once the server is running and has repositories, you can clone them with:

    git clone ssh://localhost:23233/repo_name
    

### Adding a Remote to an Existing Repository

To add the Git server as a remote to an existing repository:

    cd existing_repo
    git remote add wish_remote ssh://localhost:23233/repo_name
    git push wish_remote
    

Sources: [examples/git/main.go111-114](https://github.com/charmbracelet/wish/blob/19f43236/examples/git/main.go#L111-L114)

Conclusion
----------

This example demonstrates how to use wish to create a Git server that:

*   Handles Git protocol commands over SSH
*   Provides access control for repositories
*   Implements hooks for Git operations
*   Includes a custom user interface for repository listing

For more complex setups, you might want to implement additional features like:

*   Per-repository and per-user access control
*   Integration with existing authentication systems
*   Custom hooks for CI/CD integration
*   Web interfaces for repository management
