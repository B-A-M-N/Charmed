Rate Limiting Middleware
========================

Relevant source files

*   [ratelimiter/ratelimiter.go](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter.go)
    
*   [ratelimiter/ratelimiter\_test.go](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter_test.go)
    

The Rate Limiting Middleware for Wish provides functionality to control the frequency of connections from SSH clients to your server. This middleware helps prevent abuse, brute force attacks, and potential denial of service scenarios by limiting connections based on the client's IP address.

For information about other middleware components, see the [Middleware System](https://deepwiki.com/charmbracelet/wish/3-middleware-system)
 page. For specific information about logging, see [Logging and Metrics Middleware](https://deepwiki.com/charmbracelet/wish/3.2-logging-and-metrics-middleware)
.

Overview
--------

Rate limiting is implemented in the `ratelimiter` package, which provides a configurable middleware for restricting the number of connections a single source can make within a specific timeframe.

Sources: [ratelimiter/ratelimiter.go28-40](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter.go#L28-L40)

Components and Architecture
---------------------------

The rate limiting system consists of a few key components working together to track and control connection rates.

Sources: [ratelimiter/ratelimiter.go24-65](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter.go#L24-L65)

### RateLimiter Interface

The package defines a `RateLimiter` interface that all rate limiters must implement:

This interface requires a single method, `Allow`, which determines whether a session is permitted to proceed based on the rate limiting rules. If the rate limit is exceeded, it returns an error.

Sources: [ratelimiter/ratelimiter.go20-26](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter.go#L20-L26)

### Default Rate Limiter Implementation

The package provides a default implementation of the `RateLimiter` interface that uses an LRU (Least Recently Used) cache to track rate limiters for different IP addresses:

The implementation:

1.  Extracts the client's IP address from the SSH session
2.  Looks up or creates a rate limiter for that IP address
3.  Determines if the connection is allowed based on the configured rate and burst parameters
4.  Returns an error if the rate limit is exceeded

Sources: [ratelimiter/ratelimiter.go61-91](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter.go#L61-L91)

The package provides a factory function to create a new rate limiter:

The parameters are:

*   `r` (rate.Limit): The maximum rate of connections allowed (per second)
*   `burst` (int): The maximum burst of connections allowed at once
*   `maxEntries` (int): The maximum number of IP addresses to track in the cache

Implementation Details
----------------------

### IP Address Extraction

The rate limiter extracts the client's IP address from the SSH session using the `RemoteAddr()` method:

This ensures that rate limiting is applied on a per-IP basis, regardless of how many different SSH sessions or connections an IP attempts to establish.

Sources: [ratelimiter/ratelimiter.go67-74](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter.go#L67-L74)

### LRU Cache Management

The rate limiter uses a Least Recently Used (LRU) cache from the `github.com/hashicorp/golang-lru/v2` package to efficiently track and expire rate limiters for client IP addresses. This ensures that:

1.  Memory usage is bounded by limiting the number of tracked IP addresses
2.  Less active clients are automatically removed from the cache when it reaches capacity
3.  Performance remains high even with many unique clients

Sources: [ratelimiter/ratelimiter.go52-58](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter.go#L52-L58)
 [ratelimiter/ratelimiter.go62](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter.go#L62-L62)

Best Practices
--------------

When configuring rate limiting for your SSH server, consider the following:

| Parameter | Description | Recommendation |
| --- | --- | --- |
| Rate | Maximum connections per second | Start conservative (e.g., 5-10/sec) and adjust based on legitimate usage patterns |
| Burst | Maximum connections allowed at once | Set to accommodate legitimate burst traffic (e.g., 3-5) |
| Cache Size | Number of unique IP addresses to track | Set based on expected number of unique clients (e.g., 100-1000) |

For highly available systems, consider setting higher limits but still maintain some protection against obvious abuse patterns.

Sources: [ratelimiter/ratelimiter.go42-59](https://github.com/charmbracelet/wish/blob/19f43236/ratelimiter/ratelimiter.go#L42-L59)
