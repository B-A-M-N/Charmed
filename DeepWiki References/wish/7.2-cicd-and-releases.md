This document describes the Continuous Integration/Continuous Deployment (CI/CD) pipeline and release processes for the Wish SSH application framework. It covers the automated workflows for building, testing, linting, dependency management, and releasing new versions of the library.

For information about the general project structure, see [Project Structure](https://deepwiki.com/charmbracelet/wish/7.1-project-structure)
.

CI/CD Pipeline Overview
-----------------------

Wish uses GitHub Actions for its CI/CD pipeline, with most workflows leveraging shared configurations from the Charm organization's meta repository.

Sources: [.github/workflows/build.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/build.yml)
 [.github/workflows/lint.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/lint.yml)

Continuous Integration
----------------------

### Building and Testing

The build workflow runs automatically on every push and pull request to ensure code quality and functionality.

Sources: [.github/workflows/build.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/build.yml)

The build workflow:

*   Uses a shared workflow from the Charm organization's meta repository
*   Runs on each push and pull request
*   Runs Go tests with race detection and generates coverage reports
*   Uploads coverage data to Codecov for analysis

### Linting

The linting workflow ensures code quality by checking for common issues using golangci-lint.

Sources: [.github/workflows/lint.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/lint.yml)
 [.golangci.yml](https://github.com/charmbracelet/wish/blob/19f43236/.golangci.yml)

The lint configuration:

*   Runs golangci-lint on each push and pull request
*   Uses a non-blocking approach (`--issues-exit-code=0`)
*   Shows only new issues when run on pull requests
*   Enforces a specific set of linters defined in `.golangci.yml`
*   Includes custom rules like preventing usage of `github.com/gliderlabs/ssh` in favor of `github.com/charmbracelet/ssh`

### Example Management

A dedicated workflow keeps the examples directory's dependencies in sync with the main module.

Sources: [.github/workflows/examples.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/examples.yml)

This workflow:

*   Runs when specific files change in the main branch
*   Automatically updates the examples directory's Go modules
*   Commits changes back to the repository

Dependency Management
---------------------

Wish uses Dependabot for automated dependency management across multiple ecosystems.

Sources: [.github/dependabot.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/dependabot.yml)
 [.github/workflows/dependabot-sync.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/dependabot-sync.yml)

Dependabot is configured to:

*   Check for updates to Go modules (both in root and examples directories)
*   Check for updates to GitHub Actions
*   Check for updates to Docker dependencies
*   Run weekly on Mondays
*   Create pull requests with appropriate labels and commit message prefixes

Additionally, a synchronization workflow ensures the Dependabot configuration stays up to date with organizational standards:

*   Runs weekly on Sundays
*   Can be triggered manually
*   Uses a shared workflow from the Charm meta repository

Release Process
---------------

The release process is triggered by pushing a tag matching the pattern `v*.*.*` (semantic versioning).

Sources: [.github/workflows/goreleaser.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/goreleaser.yml)
 [.goreleaser.yml](https://github.com/charmbracelet/wish/blob/19f43236/.goreleaser.yml)

The release workflow:

*   Uses GoReleaser for building and publishing releases
*   Leverages a shared configuration from the Charm meta repository
*   Creates GitHub releases with artifacts for multiple platforms
*   Sends notifications to Twitter, Mastodon, and Discord about the new release

The GoReleaser configuration:

*   Is defined in `.goreleaser.yml`
*   Imports shared settings from `charmbracelet/meta/main/goreleaser-lib.yaml`

Development Workflow for Contributors
-------------------------------------

For contributors to the Wish project, understanding the CI/CD pipeline helps ensure smooth integration of changes.

Sources: [.github/workflows/build.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/build.yml)
 [.github/workflows/lint.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/lint.yml)
 [.github/workflows/goreleaser.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/workflows/goreleaser.yml)

Best practices for contributors:

1.  Run tests locally before submitting PRs using `go test -race ./...`
2.  Run the linter locally using golangci-lint
3.  Keep dependencies updated
4.  Follow semantic versioning for releases
5.  Wait for all CI checks to pass before requesting review

Anatomy of GitHub Workflows
---------------------------

The repository contains several GitHub workflow configurations that define the CI/CD pipeline.

| Workflow File | Purpose | Trigger | Key Actions |
| --- | --- | --- | --- |
| `build.yml` | Building and testing | Push, Pull Request | Run tests, report coverage |
| `lint.yml` | Code quality checks | Push, Pull Request | Run golangci-lint |
| `examples.yml` | Update example dependencies | Push to main, specific files | Run go mod tidy, commit changes |
| `dependabot-sync.yml` | Sync dependabot config | Weekly, Manual | Update dependabot configuration |
| `goreleaser.yml` | Release management | Tag push (`v*.*.*`) | Build, publish, notify |

Dependency Ecosystem Management
-------------------------------

The repository uses Dependabot to manage dependencies across different ecosystems.

Sources: [.github/dependabot.yml](https://github.com/charmbracelet/wish/blob/19f43236/.github/dependabot.yml)

The Dependabot configuration manages dependencies across:

*   Go modules in the root directory
*   Go modules in the examples directory
*   GitHub Actions
*   Docker dependencies

Each ecosystem is configured with consistent settings but may use different commit message prefixes according to the type of dependency.
