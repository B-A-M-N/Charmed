Purpose and Scope
-----------------

This document describes how to use the Git server integration capabilities of the wish library to host Git repositories over SSH. The Git server middleware allows users to push to and pull from Git repositories hosted by your SSH server with proper access control and operation hooks.

This page focuses on the Git-specific functionality. For general SSH server configuration, see [Core SSH Server Framework](https://deepwiki.com/charmbracelet/wish/2-core-ssh-server-framework)
, and for authentication mechanisms, see [Authentication](https://deepwiki.com/charmbracelet/wish/5-authentication)
.

Sources: [git/git.go1-241](https://github.com/charmbracelet/wish/blob/19f43236/git/git.go#L1-L241)

Architecture Overview
---------------------

The Git server integration is implemented as a middleware that intercepts Git commands sent over SSH. When a client executes a Git command that interacts with a remote repository, the middleware processes the command, checks authorization, and performs the requested operation.

Sources: [git/git.go62-118](https://github.com/charmbracelet/wish/blob/19f43236/git/git.go#L62-L118)

### AccessLevel

The `AccessLevel` type defines the level of access that can be granted to a repository:

| Access Level | Numeric Value | Description |
| --- | --- | --- |
| NoAccess | 0   | No access to the repository |
| ReadOnlyAccess | 1   | Read-only access (fetch, clone) |
| ReadWriteAccess | 2   | Read and write access (fetch, clone, push) |
| AdminAccess | 3   | Read, write, and administrative access |

Sources: [git/git.go27-42](https://github.com/charmbracelet/wish/blob/19f43236/git/git.go#L27-L42)

### Hooks Interface

The `Hooks` interface allows for:

*   Custom authorization of Git operations
*   Notification of successful Git operations

Git Command Handling
--------------------

The Git middleware intercepts and processes three main Git commands:

### Command Types and Authorization

| Git Command | Function | Required Access Level |
| --- | --- | --- |
| git-receive-pack | For pushing to a repository | ReadWriteAccess or AdminAccess |
| git-upload-pack | For fetching from a repository | ReadOnlyAccess, ReadWriteAccess, or AdminAccess |
| git-upload-archive | For creating archives of a repository | ReadOnlyAccess, ReadWriteAccess, or AdminAccess |

When a Git command is received, the middleware:

1.  Extracts the repository name
2.  Checks authorization using the `Hooks.AuthRepo` method
3.  Executes the Git command if authorized
4.  Calls the appropriate notification hook (Push or Fetch)

Sources: [git/git.go69-117](https://github.com/charmbracelet/wish/blob/19f43236/git/git.go#L69-L117)

Security Considerations
-----------------------

When implementing a Git server, consider the following security aspects:

1.  **Authentication**: Carefully control which users can connect to your SSH server
2.  **Authorization**: Implement fine-grained control in your `AuthRepo` method
3.  **Repository Path Validation**: The middleware sanitizes repository paths, but consider additional validation for your use case
4.  **Logging**: Log all Git operations for audit purposes

Advanced Usage

### Custom Repository Structure

You can implement custom repository structures by carefully implementing the `AuthRepo` method. For example, you could:

*   Organize repositories by user
*   Apply different access controls to different repository paths
*   Implement organization or project-based repository grouping

Troubleshooting
---------------

When integrating the Git server functionality, be aware of common issues:

1.  **Repository Permissions**: Ensure the process has appropriate filesystem permissions
2.  **Repository Path Handling**: Repository paths are normalized to prevent directory traversal
3.  **Authentication Issues**: Debug by reviewing the `AuthRepo` implementation
4.  **Git Command Issues**: The middleware executes Git commands via the system, so Git must be installed
