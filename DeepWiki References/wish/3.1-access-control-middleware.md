Access Control Middleware
=========================

This document covers the access control mechanisms available in the wish SSH server framework, specifically focusing on the accesscontrol package for command restriction and the activeterm package for terminal requirements. For information about rate limiting, see [Rate Limiting Middleware](https://deepwiki.com/charmbracelet/wish/3.3-rate-limiting-middleware)
.

Overview
--------

The access control middleware components provide security mechanisms to restrict what users can do after successfully authenticating to your SSH server. These components operate as middleware in the wish middleware chain, enabling granular control over command execution and terminal requirements.

Sources: [accesscontrol/accesscontrol.go1-31](https://github.com/charmbracelet/wish/blob/19f43236/accesscontrol/accesscontrol.go#L1-L31)
 [activeterm/activeterm.go1-23](https://github.com/charmbracelet/wish/blob/19f43236/activeterm/activeterm.go#L1-L23)

Command Restriction Middleware
------------------------------

The command restriction middleware (`accesscontrol.Middleware`) enables you to control which commands are allowed to execute on your SSH server. When a command is not on the allowlist, the connection is terminated with an exit code of 1.

### How It Works

The middleware checks if the command requested by the client is in the list of allowed commands. If it's not, the connection is terminated with an appropriate error message.

Sources: [accesscontrol/accesscontrol.go11-30](https://github.com/charmbracelet/wish/blob/19f43236/accesscontrol/accesscontrol.go#L11-L30)

### Implementation Details

The implementation is concise but powerful. The middleware function takes a variadic parameter of allowed command strings:

Sources: [accesscontrol/accesscontrol.go13-30](https://github.com/charmbracelet/wish/blob/19f43236/accesscontrol/accesscontrol.go#L13-L30)

### Key Behaviors

1.  **Interactive Mode**: SSH sessions without commands (interactive login) are always allowed
2.  **Allowlist Approach**: Only explicitly allowed commands can execute
3.  **First Token Match**: Only the first token of the command is checked against the allowed list
4.  **Empty Allowlist**: If no allowed commands are provided, no commands will be permitted (only interactive sessions)

Active Terminal Middleware
--------------------------

The active terminal middleware (`activeterm.Middleware`) ensures that connections have an active PTY (pseudo-terminal). This is useful for applications that require terminal features or when you want to enforce interactive sessions.

### How It Works

The middleware checks if the SSH session has an active PTY. If not, the connection is terminated with an exit code of 1.

Common Use Cases
----------------

1.  **Git Server**: Restrict commands to only Git operations (`git-receive-pack`, `git-upload-pack`, `git-upload-archive`)
2.  **Service Control**: Allow only specific service management commands
3.  **Interactive Only**: Block all commands to force interactive sessions
4.  **Application-specific**: Allow only commands related to your application
5.  **Terminal Requirements**: Ensure proper terminal capabilities for TUI applications

Testing and Verification
------------------------

The middleware includes comprehensive test cases that verify its behavior:

| Scenario | Expected Outcome |
| --- | --- |
| No allowed commands, no command provided | Allow (interactive mode) |
| No allowed commands, command provided | Deny with error message |
| Allowed commands specified, no command provided | Allow (interactive mode) |
| Allowed commands specified, allowed command provided | Allow execution |
| Allowed commands specified, disallowed command provided | Deny with error message |
| Active terminal required, no terminal | Deny with "Requires an active PTY" message |

Sources: [accesscontrol/accesscontrol\_test.go15-78](https://github.com/charmbracelet/wish/blob/19f43236/accesscontrol/accesscontrol_test.go#L15-L78)
 [activeterm/activeterm\_test.go12-22](https://github.com/charmbracelet/wish/blob/19f43236/activeterm/activeterm_test.go#L12-L22)

Security Considerations
-----------------------

*   Access control middleware is a second layer of defense after authentication
*   Commands are matched only on the first token, so combined commands with pipes or semicolons are checked only by their first command
*   The middleware does not perform any command parameter validation
*   Consider using both middlewares when appropriate for your application's security requirements

Sources: [accesscontrol/accesscontrol.go20-22](https://github.com/charmbracelet/wish/blob/19f43236/accesscontrol/accesscontrol.go#L20-L22)
