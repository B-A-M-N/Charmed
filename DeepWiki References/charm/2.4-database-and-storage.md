Database and Storage
====================

This document details the database schema and storage mechanisms used by the Charm server. It covers the SQLite database structure, file storage system, and how server components interact with these persistence layers. For information about server configuration options, see [Server Configuration](https://deepwiki.com/charmbracelet/charm/2.3-server-configuration)
.

Overview
--------

Charm uses two primary persistence mechanisms:

1.  **SQLite Database**: Stores user accounts, SSH keys, encryption keys, and metadata
2.  **File Storage System**: Manages user files with path-based organization

These systems are accessed through well-defined interfaces that provide abstraction between the server logic and the underlying storage implementations.

Sources: [server/http.go33-40](https://github.com/charmbracelet/charm/blob/aff3071d/server/http.go#L33-L40)
 [server/db/sqlite/storage.go25-28](https://github.com/charmbracelet/charm/blob/aff3071d/server/db/sqlite/storage.go#L25-L28)

Database Schema
---------------

The Charm server uses SQLite to persist data, with a schema comprising the following tables:

| Table | Purpose | Primary Fields |
| --- | --- | --- |
| User | User accounts | id, charm\_id, name, email, bio, created\_at |
| PublicKey | SSH public keys | id, user\_id, key, created\_at |
| EncryptKey | Encryption keys | id, public\_key\_id, key, global\_id, created\_at |
| NamedSeq | Sequence counters | id, user\_id, name, seq |
| News | News/announcements | id, subject, body, created\_at |
| NewsTag | News article tags | id, news\_id, tag |
| Token | Authentication tokens | id, token, created\_at |

Database Initialization
-----------------------

The database is initialized during server startup with the following process:

1.  The SQLite database is opened at the configured path
2.  Database tables are created if they don't exist
3.  The database instance is provided to the HTTP and SSH servers

Sources: [server/db/sqlite/storage.go30-44](https://github.com/charmbracelet/charm/blob/aff3071d/server/db/sqlite/storage.go#L30-L44)

Database Operations
-------------------

The SQLite database implementation provides operations for managing users, keys, sequences, and other data through the `db.DB` interface.

### User Management

Sources: [server/db/sqlite/storage.go67-172](https://github.com/charmbracelet/charm/blob/aff3071d/server/db/sqlite/storage.go#L67-L172)

### Key Management

The database handles three types of keys:

1.  **SSH Public Keys**: Used for authentication
2.  **Encryption Keys**: Used for end-to-end encryption
3.  **Tokens**: Used for account linking and other purposes

Key operations include:

*   `LinkUserKey`: Associates a public key with a user
*   `UnlinkUserKey`: Removes a public key association
*   `KeysForUser`: Lists all public keys for a user
*   `AddEncryptKeyForPublicKey`: Adds encrypted key for a public key
*   `EncryptKeysForPublicKey`: Lists encryption keys for a public key

Sources: [server/db/sqlite/storage.go175-297](https://github.com/charmbracelet/charm/blob/aff3071d/server/db/sqlite/storage.go#L175-L297)
 [server/db/sqlite/storage.go394-413](https://github.com/charmbracelet/charm/blob/aff3071d/server/db/sqlite/storage.go#L394-L413)

### Sequence Management

Named sequences provide atomic counters for various application features:

*   `GetSeq`: Retrieves current sequence value
*   `NextSeq`: Increments and returns sequence value

These operations are used to generate unique identifiers and maintain sequence ordering across distributed clients.

Sources: [server/db/sqlite/storage.go299-331](https://github.com/charmbracelet/charm/blob/aff3071d/server/db/sqlite/storage.go#L299-L331)

### Transaction Handling

All database operations are wrapped in transactions using the `WrapTransaction` method, which provides:

*   Atomicity guarantees
*   Automatic retries for SQLITE\_BUSY errors
*   Timeout handling
*   Error propagation

Sources: [server/db/sqlite/storage.go654-680](https://github.com/charmbracelet/charm/blob/aff3071d/server/db/sqlite/storage.go#L654-L680)

File Storage System
-------------------

The file storage system manages user files with a hierarchical path structure. Each user's files are isolated by their CharmID.

### Storage Limits

The server can enforce per-user storage limits:

1.  The `UserMaxStorage` config parameter sets the maximum storage per user
2.  Before storing a file, the server checks the user's current storage usage
3.  If the new file would exceed the limit, the operation is rejected

Database and HTTP Integration
-----------------------------

The HTTP server exposes endpoints that interact with the database and file storage systems:

| Endpoint | Method | Purpose |
| --- | --- | --- |
| `/v1/id/:id` | GET | Get user by ID |
| `/v1/bio/:name` | GET | Get user by name |
| `/v1/bio` | POST | Update user information |
| `/v1/encrypt-key` | POST | Add encryption key |
| `/v1/fs/*` | GET | Retrieve a file |
| `/v1/fs/*` | POST | Store a file |
| `/v1/fs/*` | DELETE | Delete a file |
| `/v1/seq/:name` | GET | Get sequence value |
| `/v1/seq/:name` | POST | Increment sequence value |
| `/v1/news` | GET | Get news list |
| `/v1/news/:id` | GET | Get news by ID |

Security Considerations
-----------------------

### Data Isolation

Each user's data is isolated through several mechanisms:

1.  User data in the database is linked to the user's ID
2.  File storage paths include the user's CharmID
3.  The `CharmUserMiddleware` ensures users can only access their own data

### Request Limits

To prevent abuse, the server implements request size limits:

*   File operations: Limited to `MaxFSRequestSize` (1GB by default)
*   Other endpoints: Limited to 1MB

Sources: [server/middleware.go25-47](https://github.com/charmbracelet/charm/blob/aff3071d/server/middleware.go#L25-L47)

### Transaction Security

Database transactions are protected with:

*   5-second timeouts to prevent long-running transactions
*   Automatic retries for busy conditions
*   Foreign key constraints to maintain data integrity

Sources: [server/db/sqlite/storage.go654-680](https://github.com/charmbracelet/charm/blob/aff3071d/server/db/sqlite/storage.go#L654-L680)

Migration and Maintenance
-------------------------

The database supports operations for account migration and maintenance:

*   `MergeUsers`: Combines two user accounts
*   Account key migration between RSA and Ed25519 formats
*   JWT token generation and validation

Sources: [server/db/sqlite/storage.go383-392](https://github.com/charmbracelet/charm/blob/aff3071d/server/db/sqlite/storage.go#L383-L392)
 [cmd/migrate\_account.go19-87](https://github.com/charmbracelet/charm/blob/aff3071d/cmd/migrate_account.go#L19-L87)

Conclusion
----------

The Charm server's database and storage systems provide a robust foundation for user management, key storage, and file operations. The separation between database and file storage allows for efficient handling of different data types, while the transaction-based approach ensures data consistency.

For server configuration options related to database and storage, including file paths and size limits, see [Server Configuration](https://deepwiki.com/charmbracelet/charm/2.3-server-configuration)
.
