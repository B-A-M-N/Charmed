This document provides a comprehensive overview of the Charm Cloud system architecture, components, and their interactions. It explains the high-level structure, client-server communication, authentication mechanisms, and data flow. For information about specific components like the Charm Key-Value store or Filesystem, please see [Core Concepts](https://deepwiki.com/charmbracelet/charm/1.1-core-concepts)
.

System Overview
---------------

Charm Cloud consists of a client-server architecture that enables secure, encrypted, and seamless interaction between terminal applications and backend services. The system is designed with privacy and security as core principles, implementing end-to-end encryption and SSH-based authentication.

The system architecture consists of client-side components that interact with server-side services through HTTP and SSH protocols. The client provides interfaces for key-value storage, filesystem operations, and encryption, while the server manages persistent storage, authentication, and data synchronization.

Client Architecture
-------------------

The Charm client provides libraries and interfaces for applications to interact with the Charm Cloud services. It manages authentication, key handling, and encrypted communication with the server.

The client is designed to be embedded in applications and provides:

1.  **Authentication Management**: Handles SSH key authentication and JWT token acquisition
2.  **Configuration**: Configurable through environment variables or direct configuration
3.  **API Access**: Provides methods to interact with server-side resources
4.  **Key Management**: Manages encryption and SSH keys

Client components rely on SSH keys for authentication and JWT tokens for API access. By default, it connects to `cloud.charm.sh` but can be configured to point to self-hosted instances.

Server Architecture
-------------------

The Charm server provides HTTP and SSH interfaces that client applications use to access cloud services. It consists of several core components working together to provide data storage, authentication, and encryption services.

Server components include:

1.  **HTTP Server**: Handles REST API requests for file and key-value operations
2.  **SSH Server**: Manages authentication and secure command execution
3.  **Database**: SQLite database for user data, metadata, and key-value storage
4.  **File Storage**: Local file storage for user files
5.  **Stats/Metrics**: Optional Prometheus metrics collection

The server is designed to be easily self-hosted and can be configured through environment variables.

Communication Protocols
-----------------------

Charm uses two primary protocols for communication between clients and servers:

1.  **SSH Protocol**:
    
    *   Used for authentication and identity verification
    *   Provides secure commands for account operations
    *   Default port: 35353
2.  **HTTP Protocol**:
    
    *   REST API for data operations
    *   JWT-authenticated requests
    *   Default port: 35354

The server configuration includes two additional services:

*   Stats/Metrics service (Port 35355)
*   Health Check service (Port 35356)

Authentication System
---------------------

Authentication in Charm is built around SSH keys, providing secure and passwordless authentication.

Key features of the authentication system:

1.  **SSH Key-Based**: Uses SSH keys to authenticate users, eliminating passwords
2.  **Frictionless Setup**: Automatically creates keys if none exist
3.  **Multi-Device Support**: Allows linking multiple devices to one account
4.  **Secure Token Exchange**: JWT tokens for API authentication
5.  **Key Revocation**: Users can unlink devices from their account

The authentication flow begins with the client checking for existing keys, generating them if necessary, and then using those keys to authenticate with the SSH server. After successful authentication, the server issues a JWT token for API access.

Data Security & Encryption
--------------------------

Charm provides end-to-end encryption for all user data, ensuring that only authorized users can access it.

The encryption system provides:

1.  **End-to-End Encryption**: All data is encrypted on the client before transmission
2.  **Key Management**: Symmetric keys are encrypted with the user's public key
3.  **Multi-Device Support**: Keys are re-encrypted for new linked devices
4.  **Zero-Knowledge Design**: Server never has access to plaintext data or decryption keys

This architecture ensures that even if the server is compromised, user data remains secure since the server never has access to unencrypted data or the keys needed to decrypt it.

Storage Systems
---------------

Charm provides two primary storage mechanisms:

1.  **Key-Value Store (KV)**:
    
    *   Built on BadgerDB
    *   Provides encrypted, synced key-value storage
    *   Supports local-first operations with cloud synchronization
    *   Handles binary data of any type
2.  **File System (FS)**:
    
    *   Compatible with Go's `fs.FS` interface
    *   Cloud-based user filesystem
    *   Supports standard file operations (read, write, delete)
    *   Transparent encryption and authentication

Both storage systems automatically handle encryption, authentication, and synchronization, providing a seamless experience for both developers and end users.

Deployment Options
------------------

Charm is designed to be flexible in deployment, supporting both cloud-hosted and self-hosted configurations.

The server can be deployed in several ways:

1.  **Cloud-Hosted**: Default `cloud.charm.sh` service
2.  **Self-Hosted**:
    *   Direct execution: `charm serve`
    *   Docker container
    *   Systemd service

Configuration for both client and server is managed through environment variables, making it easy to customize for different deployment scenarios.

System Integration
------------------

Applications can integrate with Charm using its various libraries and components.

Charm is designed to be easily integrated into terminal applications, providing a consistent and secure backend experience. Applications can use one or more Charm components as needed.

Major projects using Charm include:

*   Glow: A markdown renderer with cloud storage capabilities
*   Skate: A personal key-value store

Configuration Reference
-----------------------

Both the client and server can be configured through environment variables:

| Client Configuration | Default | Description |
| --- | --- | --- |
| `CHARM_HOST` | cloud.charm.sh | Server hostname |
| `CHARM_SSH_PORT` | 35353 | SSH port |
| `CHARM_HTTP_PORT` | 35354 | HTTP port |
| `CHARM_DEBUG` | false | Enable debug logging |
| `CHARM_DATA_DIR` |     | Custom data directory path |
| `CHARM_KEY_TYPE` | ed25519 | SSH key type |

| Server Configuration | Default | Description |
| --- | --- | --- |
| `CHARM_SERVER_BIND_ADDRESS` | 0.0.0.0 | Network interface |
| `CHARM_SERVER_HOST` | localhost | Hostname |
| `CHARM_SERVER_SSH_PORT` | 35353 | SSH port |
| `CHARM_SERVER_HTTP_PORT` | 35354 | HTTP port |
| `CHARM_SERVER_STATS_PORT` | 35355 | Stats port |
| `CHARM_SERVER_HEALTH_PORT` | 35356 | Health check port |
| `CHARM_SERVER_DATA_DIR` | ./data | Server data directory |
| `CHARM_SERVER_USE_TLS` | false | Enable TLS |
| `CHARM_SERVER_ENABLE_METRICS` | false | Enable metrics |
| `CHARM_SERVER_USER_MAX_STORAGE` | 0 (unlimited) | Max storage per user |

This configuration system allows for flexible deployment in various environments while maintaining sensible defaults.
