Encryption
==========

This document details the encryption architecture and mechanisms used within the Charm Cloud system to secure user data. The encryption system ensures that all sensitive data is encrypted locally on the client side before transmission, providing end-to-end security. For information about the overall data storage systems, see [Data Storage](https://deepwiki.com/charmbracelet/charm/5-data-storage)
 and for specific implementations of the Key-Value store or File System, refer to [Key-Value Store (KV)](https://deepwiki.com/charmbracelet/charm/5.2-key-value-store-(kv))
 and [File System (FS)](https://deepwiki.com/charmbracelet/charm/5.1-file-system-(fs))
 respectively.

Overview of Encryption in Charm
-------------------------------

The Charm system implements end-to-end encryption throughout its architecture. All user data is encrypted on the client side before being transmitted to the server, ensuring that plain text data never leaves the user's device. The server only stores encrypted data, which means even the server operators cannot access the original content.

Sources: [client/crypt.go1-156](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L1-L156)
 [kv/kv.go1-246](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L1-L246)

Encryption Key Architecture
---------------------------

Charm's encryption system employs a hybrid approach using both symmetric and asymmetric cryptography:

1.  **Symmetric Encryption Keys**: Random 64-byte keys generated for data encryption
2.  **Asymmetric SSH Keys**: Used for authentication and for encrypting/decrypting the symmetric keys
3.  **Encrypted Storage**: Data is encrypted locally before being stored or transmitted

Sources: [client/crypt.go17-42](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L17-L42)
 [client/crypt.go94-156](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L94-L156)

Encryption Key Management
-------------------------

When a user first authenticates with Charm, the system automatically generates a symmetric encryption key if one doesn't exist. This key is then encrypted with the user's public SSH key and stored on the server.

### Key Generation and Storage

The key generation process happens in the `cryptCheck` method, which ensures that each user has at least one encryption key. If no keys exist, a new random 64-byte key is generated, encrypted with the user's public key, and stored on the server.

Sources: [client/crypt.go94-156](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L94-L156)

### Key Structure

Each encryption key in Charm has the following components:

| Component | Description |
| --- | --- |
| ID  | Unique identifier for the key (UUID) |
| PublicKey | The public SSH key used to encrypt this key |
| Key | The encrypted symmetric key (base64 encoded) |
| CreatedAt | Timestamp when the key was created |

The plaintext key is a base64-encoded 64-byte random value, but only the first 32 bytes are used for BadgerDB encryption.

Sources: [client/crypt.go17-42](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L17-L42)
 [kv/client.go155-161](https://github.com/charmbracelet/charm/blob/aff3071d/kv/client.go#L155-L161)

Multi-Device Encryption
-----------------------

One of Charm's key features is the ability to link multiple devices to the same account while maintaining end-to-end encryption. This is accomplished through the key synchronization mechanism.

### Key Synchronization Process

When a new device is linked to an existing Charm account:

1.  The link process authenticates both devices
2.  Upon successful linking, `SyncEncryptKeys()` is called
3.  This function re-encrypts all symmetric encryption keys with the public keys of all linked devices
4.  Each device can now decrypt the data using its own private key

Sources: [client/link.go12-156](https://github.com/charmbracelet/charm/blob/aff3071d/client/link.go#L12-L156)
 [client/link.go125-146](https://github.com/charmbracelet/charm/blob/aff3071d/client/link.go#L125-L146)

Data Encryption in Key-Value Store
----------------------------------

The Key-Value store (KV) leverages BadgerDB's encryption capabilities to secure data locally.

### KV Store Encryption Flow

The process works as follows:

1.  The client retrieves encryption keys using `EncryptKeys()`
2.  The first 32 bytes of the key are extracted with `encryptKeyToBadgerKey()`
3.  BadgerDB is configured with encryption using these 32 bytes
4.  When data is synced, encrypted diffs are created and sent to the server
5.  The server only sees and stores encrypted data

Sources: [kv/kv.go72-79](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L72-L79)
 [kv/client.go155-189](https://github.com/charmbracelet/charm/blob/aff3071d/kv/client.go#L155-L189)
 [kv/kv.go18-30](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L18-L30)

Encryption Implementation Details
---------------------------------

### BadgerDB Encryption Configuration

The KV store configures BadgerDB with encryption by:

1.  Retrieving the user's encrypt keys
2.  Converting the first 32 bytes for use with BadgerDB
3.  Setting up encryption options with the key and an index cache
4.  Opening a managed instance of BadgerDB with these encryption options

Sources: [kv/client.go155-189](https://github.com/charmbracelet/charm/blob/aff3071d/kv/client.go#L155-L189)
 [kv/kv.go72-79](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L72-L79)

### Key Decryption Process

When a client starts, it needs to decrypt the stored encryption keys:

1.  Client retrieves encrypted keys from the server during authentication
2.  Finds SSH identities from the local SSH key files
3.  Uses Sasquatch library to decrypt the encrypted keys with the SSH identities
4.  Stores the decrypted keys in memory for use in data encryption/decryption

Sources: [client/crypt.go94-156](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L94-L156)
 [client/crypt.go44-57](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L44-L57)

Security Considerations
-----------------------

The Charm encryption system provides strong security guarantees:

1.  **End-to-End Encryption**: Data is encrypted before leaving the client
2.  **Zero Knowledge**: The server only sees encrypted data and cannot access plaintext
3.  **Key Isolation**: Encryption keys are themselves encrypted with the user's public key
4.  **Multiple Device Support**: Each device can securely access the same data via key synchronization
5.  **Local Encryption**: Even the local database files are encrypted using BadgerDB's encryption

A potential limitation is that if a user loses access to all their devices, they will lose access to their encryption keys and thus their data. There is no key recovery mechanism by design, as this would compromise the end-to-end encryption model.
