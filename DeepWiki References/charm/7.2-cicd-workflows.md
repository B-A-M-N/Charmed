CI/CD Workflows
===============

This document describes the Continuous Integration and Continuous Deployment workflows implemented for the Charm project. It covers the automated build, test, lint, and release processes that ensure code quality and streamline releases. For information about building and testing locally, see [Building and Testing](https://deepwiki.com/charmbracelet/charm/7.1-building-and-testing)
.

Overview
--------

Charm uses GitHub Actions for all CI/CD workflows. These workflows automate various aspects of the development lifecycle, from code quality checks to releasing new versions.

Sources: [.github/workflows/build.yml](https://github.com/charmbracelet/charm/blob/aff3071d/.github/workflows/build.yml)

CI Workflows
------------

### Build Workflow

The build workflow ensures that the codebase can be compiled across different platforms and Go versions.

Sources: [.github/workflows/build.yml](https://github.com/charmbracelet/charm/blob/aff3071d/.github/workflows/build.yml)

The build workflow is defined in [.github/workflows/build.yml1-12](https://github.com/charmbracelet/charm/blob/aff3071d/.github/workflows/build.yml#L1-L12)
 and is triggered on both push and pull request events. It uses a reusable workflow from the charmbracelet/meta repository, which standardizes the build process across Charm projects.

A snapshot job is also included, which creates development builds for testing purposes. This job requires the `goreleaser_key` secret to function properly.

### Lint Workflow

The lint workflow ensures code quality by running static code analysis.

Sources: [.github/workflows/lint.yml](https://github.com/charmbracelet/charm/blob/aff3071d/.github/workflows/lint.yml)

The lint workflow is defined in [.github/workflows/lint.yml1-19](https://github.com/charmbracelet/charm/blob/aff3071d/.github/workflows/lint.yml#L1-L19)
 and uses golangci-lint to perform static code analysis. It's configured to show only new issues on pull requests and doesn't fail the build on linting issues (using `--issues-exit-code=0`), making it informative rather than blocking.

### Coverage Workflow

The coverage workflow measures and reports test coverage.

Sources: [.github/workflows/coverage.yml](https://github.com/charmbracelet/charm/blob/aff3071d/.github/workflows/coverage.yml)

The coverage workflow is defined in [.github/workflows/coverage.yml1-29](https://github.com/charmbracelet/charm/blob/aff3071d/.github/workflows/coverage.yml#L1-L29)
 and runs tests with the `-race` and `-coverprofile` flags to generate coverage data. It then uses goveralls to upload the coverage report to Coveralls, which provides visibility into test coverage over time.

This workflow runs on two different Go versions (the most recent 1.20.x and the latest stable) to ensure compatibility.

CD Workflow (Release Process)
-----------------------------

The release process is automated using goreleaser, triggered when a new version tag is pushed.

Sources: [.github/workflows/goreleaser.yml](https://github.com/charmbracelet/charm/blob/aff3071d/.github/workflows/goreleaser.yml)

The release workflow is defined in [.github/workflows/goreleaser.yml1-22](https://github.com/charmbracelet/charm/blob/aff3071d/.github/workflows/goreleaser.yml#L1-L22)
 and is triggered only when a tag matching the pattern `v*.*.*` is pushed. This ensures that only proper semantic version tags trigger releases.

The workflow uses a reusable workflow from the charmbracelet/meta repository, which standardizes the release process across Charm projects. It requires several secrets to be configured:

| Secret | Purpose |
| --- | --- |
| `DOCKERHUB_USERNAME` | Username for Docker Hub publishing |
| `DOCKERHUB_TOKEN` | Authentication token for Docker Hub |
| `PERSONAL_ACCESS_TOKEN` | GitHub PAT for creating releases |
| `GORELEASER_KEY` | Key for signing goreleaser artifacts |
| `FURY_TOKEN` | Token for publishing to package repositories |
| `NFPM_GPG_KEY` | GPG key for signing system packages |
| `NFPM_PASSPHRASE` | Passphrase for the GPG key |

The workflow includes concurrency settings to prevent multiple release jobs from running simultaneously:

This ensures that if multiple tags are pushed in quick succession, only one release job will run at a time, and any running job will be canceled if a new one is started.

Implementation Details
----------------------

### Build and Test Matrix

The coverage workflow uses a matrix strategy to test across multiple Go versions:

This ensures compatibility with Go 1.20.x and the latest stable Go version.

### Reusable Workflows

Most of the workflows in Charm use reusable workflows from the charmbracelet/meta repository:

| Workflow | Reusable Workflow |
| --- | --- |
| Build | `charmbracelet/meta/.github/workflows/build.yml@main` |
| Snapshot | `charmbracelet/meta/.github/workflows/snapshot.yml@main` |
| Goreleaser | `charmbracelet/meta/.github/workflows/goreleaser.yml@main` |

This approach standardizes CI/CD processes across all Charm projects, making maintenance easier and ensuring consistency.

### Release Naming Convention

Releases are triggered only by tags that follow semantic versioning with the format `v*.*.*` (e.g., v1.2.3). This ensures that only proper version tags trigger the release workflow.

Setting Up for Development
--------------------------

When working with the Charm codebase, the CI workflows will automatically run on your pull requests. Here's how to ensure a smooth experience:

1.  **Run Tests Locally**: Before pushing, run tests locally to catch issues early
2.  **Use Linters**: Set up your IDE to use golangci-lint for real-time feedback
3.  **Check Coverage**: Run tests with the `-cover` flag to check coverage locally

For creating releases, authorized maintainers should:

1.  Update version numbers in relevant files
2.  Create and push a tag following the semantic versioning pattern
3.  Monitor the goreleaser workflow to ensure successful artifact publishing

For more details on local development practices, see [Building and Testing](https://deepwiki.com/charmbracelet/charm/7.1-building-and-testing)
.
