Key Management
==============

This document describes how cryptographic keys are managed by the Charm client. It covers the various types of keys used within the system, how they are generated, stored, and synchronized across devices, and the underlying security model. For information about the authentication process that uses these keys, see [Authentication](https://deepwiki.com/charmbracelet/charm/3.2-authentication)
.

Key Types and Their Purpose
---------------------------

Charm uses multiple types of cryptographic keys to ensure secure authentication and data protection across all linked devices.

**Key Types in Charm:**

1.  **SSH Key Pairs**: Used for authentication with the Charm server. Each device linked to a Charm account has its own SSH key pair.
    
2.  **Encryption Keys**: Symmetric keys used for end-to-end encryption of user data. These keys are themselves encrypted using the public keys from the SSH key pairs.

SSH Key Management
------------------

SSH keys are the primary method of authentication in Charm. Each device generates its own SSH key pair and links it to the user's account.

### Key Generation and Storage

SSH keys are typically generated automatically during the setup process. The system looks for existing SSH keys in standard locations and can create new ones if needed.

### Viewing and Managing Keys

The Charm client provides a terminal UI for viewing and managing SSH keys associated with your account:

The keys UI allows users to:

*   View all SSH keys linked to their account
*   See which key is currently active
*   Delete keys to unlink devices
*   Navigate between pages if many keys are present

Encryption Key Management
-------------------------

Encryption keys are symmetric keys used to encrypt user data before it's sent to the server, ensuring end-to-end encryption.

### Key Generation

When a user sets up Charm for the first time, the system automatically generates encryption keys:

The encryption key generation process is handled by the `cryptCheck()` function, which:

1.  Checks if encryption keys already exist
2.  If none exist, generates a new 64-byte random key
3.  Encrypts this key using the user's public SSH key
4.  Stores the encrypted key on the server

Sources: [client/crypt.go94-156](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L94-L156)

### Key Storage and Access

Encryption keys are never stored in plaintext. They are:

1.  Encrypted using the public SSH key of each linked device
2.  Stored on the server in encrypted form
3.  Decrypted locally when needed using the device's private SSH key

The client maintains a cache of decrypted keys in memory during operation.

Sources: [client/crypt.go18-42](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L18-L42)
 [client/crypt.go66-92](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L66-L92)

Key Linking Process
-------------------

One of Charm's key features is the ability to link multiple devices to a single account, allowing secure access to the same data from different devices.

### Linking Workflow

The linking process involves:

1.  **Token Generation**: The existing device generates a link token
2.  **Token Transfer**: The user manually copies the token to the new device
3.  **Link Request**: The new device uses the token to request linking
4.  **Approval**: The existing device approves the link request
5.  **Key Synchronization**: Encryption keys are synchronized to the new device

Sources: [ui/linkgen/linkgen.go67-123](https://github.com/charmbracelet/charm/blob/aff3071d/ui/linkgen/linkgen.go#L67-L123)
 [client/link.go12-78](https://github.com/charmbracelet/charm/blob/aff3071d/client/link.go#L12-L78)
 [client/link.go80-123](https://github.com/charmbracelet/charm/blob/aff3071d/client/link.go#L80-L123)

### Encryption Key Synchronization

When a new device is linked, encryption keys are synchronized to ensure all devices can encrypt/decrypt the same data:

During synchronization:

1.  All existing encryption keys are retrieved
2.  All authorized public SSH keys are fetched
3.  Each encryption key is re-encrypted with each public key
4.  The re-encrypted keys are stored on the server
5.  Local caches are cleared to ensure fresh data

Sources: [client/link.go125-146](https://github.com/charmbracelet/charm/blob/aff3071d/client/link.go#L125-L146)
 [client/crypt.go67-92](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L67-L92)

Security Considerations
-----------------------

### Key Security Model

Charm implements a robust security model for key management:

1.  **No Password Authentication**: Authentication relies solely on cryptographic keys, eliminating password-related vulnerabilities.
    
2.  **End-to-End Encryption**: User data is encrypted before transmission and only decrypted on authorized devices.
    
3.  **Multi-Factor Device Linking**: Adding a new device requires both possession of the link token and approval from an existing device.
    
4.  **Key Isolation**: Each device has its own SSH key pair, so compromising one device doesn't automatically compromise others.
    
5.  **Revocation**: Keys can be removed to revoke access from specific devices.

Implementation Details
----------------------

### Encryption Key Format

Encryption keys are stored in the `EncryptKey` structure with the following properties:

| Field | Description |
| --- | --- |
| ID  | Unique identifier for the key |
| PublicKey | The public SSH key this encryption key is for |
| Key | The encrypted symmetric key data |
| CreatedAt | Timestamp when the key was created |

The encryption key is a 64-byte random value that is encrypted using the public SSH key before storage.

Sources: [client/crypt.go18-42](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L18-L42)
 [client/crypt.go67-92](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L67-L92)

### SSH Key Format

SSH keys are stored in the `PublicKey` structure:

| Field | Description |
| --- | --- |
| ID  | Numeric identifier for the key |
| UserID | ID of the user this key belongs to |
| Key | The actual public key data |
| CreatedAt | Timestamp when the key was created |

Sources: [proto/user.go20-36](https://github.com/charmbracelet/charm/blob/aff3071d/proto/user.go#L20-L36)

Terminal UI for Key Management
------------------------------

Charm provides a terminal user interface for managing keys. This UI allows users to:

1.  View all linked SSH keys
2.  See which key is currently active
3.  Delete keys to unlink devices
4.  Navigate between pages if many keys are present

The UI is implemented using the Bubble Tea framework and provides visual indicators for the active key and selected items.

Integration with Other Components
---------------------------------

The key management system integrates closely with other Charm components:

1.  **Key-Value Store**: Uses encryption keys to encrypt data before storage
2.  **File System**: Secures file contents with encryption keys
3.  **Authentication**: Uses SSH keys to authenticate with the server
4.  **Client**: Manages the lifecycle of keys and provides APIs for other components
