Metrics and Monitoring
======================

This document describes the metrics collection and monitoring capabilities of the Charm server. It covers how to enable metrics, what metrics are collected, and how to access them through Prometheus.

Overview
--------

The Charm server includes a built-in metrics system that tracks various server operations, API usage, and user statistics. These metrics are exposed via a Prometheus endpoint, making it easy to integrate with standard monitoring and alerting solutions.

Enabling and Configuring Metrics
--------------------------------

Metrics collection is disabled by default and can be enabled through server configuration. The configuration options related to metrics include:

| Configuration Field | Environment Variable | Default | Description |
| --- | --- | --- | --- |
| EnableMetrics | CHARM\_SERVER\_ENABLE\_METRICS | false | Enables or disables metrics collection |
| StatsPort | CHARM\_SERVER\_STATS\_PORT | 35355 | The port on which the metrics HTTP server listens |

To enable metrics, set the `CHARM_SERVER_ENABLE_METRICS` environment variable to `true` when starting the Charm server:

    CHARM_SERVER_ENABLE_METRICS=true charm-server
    

When metrics are enabled, the Charm server will start an HTTP server on the configured stats port (default: 35355) that exposes the metrics in Prometheus format at the `/metrics` endpoint.

Sources: [server/server.go33-40](https://github.com/charmbracelet/charm/blob/aff3071d/server/server.go#L33-L40)
 [server/server.go223-228](https://github.com/charmbracelet/charm/blob/aff3071d/server/server.go#L223-L228)

Metrics Implementation Selection
--------------------------------

The Charm server uses one of two implementations for the `Stats` interface based on the configuration:

1.  **Prometheus Implementation** (`prometheus.Stats`): Used when metrics are enabled. Collects and exposes metrics via a Prometheus endpoint.
2.  **No-op Implementation** (`noop.Stats`): Used when metrics are disabled. Implements the interface with empty methods that do nothing.

Sources: [server/server.go223-228](https://github.com/charmbracelet/charm/blob/aff3071d/server/server.go#L223-L228)

Available Metrics
-----------------

When metrics are enabled, the Charm server collects the following metrics:

### API Operation Counters

| Metric Name | Description |
| --- | --- |
| charm\_id\_api\_link\_gen\_total | Total API link generation calls |
| charm\_id\_api\_link\_request\_total | Total API link request calls |
| charm\_id\_api\_unlink\_total | Total API unlink calls |
| charm\_id\_api\_auth\_total | Total API authentication calls |
| charm\_id\_api\_keys\_total | Total API keys calls |
| charm\_id\_link\_gen\_total | Total link generation calls |
| charm\_id\_link\_request\_total | Total link request calls |
| charm\_id\_keys\_total | Total keys calls |
| charm\_id\_id\_total | Total ID calls |
| charm\_id\_jwt\_total | Total JWT calls |

### User Management Counters

| Metric Name | Description |
| --- | --- |
| charm\_bio\_get\_user\_by\_id\_total | Total bio user by ID calls |
| charm\_bio\_get\_user\_total | Total bio get user calls |
| charm\_bio\_set\_username\_total | Total bio set username calls |
| charm\_bio\_users | Total users (gauge) |
| charm\_bio\_users\_names | Total usernames (gauge) |

### News Counters

| Metric Name | Description |
| --- | --- |
| charm\_news\_get\_news\_total | Total get news calls |
| charm\_news\_post\_news\_total | Total post news calls |
| charm\_news\_get\_news\_list\_total | Total get news list calls |

### File System Metrics

| Metric Name | Description |
| --- | --- |
| charm\_fs\_bytes\_read\_total | Total bytes read (with charm\_id label) |
| charm\_fs\_bytes\_written\_total | Total bytes written (with charm\_id label) |
| charm\_fs\_files\_read\_total | Total files read (with charm\_id label) |
| charm\_fs\_files\_written\_total | Total files written (with charm\_id label) |

Prometheus Implementation Details
---------------------------------

The Prometheus implementation of the `Stats` interface (`prometheus.Stats`) does the following:

1.  Creates and registers Prometheus counters and gauges for the various metrics
2.  Starts an HTTP server on the configured port that exposes the metrics
3.  Periodically collects and updates user and username counts from the database
4.  Increments counters when the corresponding methods are called

Sources: [server/stats/prometheus/prometheus.go45-68](https://github.com/charmbracelet/charm/blob/aff3071d/server/stats/prometheus/prometheus.go#L45-L68)

Accessing and Using Metrics
---------------------------

To access the metrics collected by the Charm server:

1.  Ensure metrics are enabled by setting `CHARM_SERVER_ENABLE_METRICS=true`
2.  Configure a Prometheus server to scrape the metrics endpoint ([http://your-charm-server:35355/metrics](http://your-charm-server:35355/metrics)

Implementation Recommendations
------------------------------

When deploying Charm in a production environment, consider the following recommendations for metrics and monitoring:

1.  **Enable Metrics**: Set `CHARM_SERVER_ENABLE_METRICS=true` to collect valuable operational data.
2.  **Use a Separate Port**: Ensure the stats port (default: 35355) is not publicly accessible but can be reached by your Prometheus server.
3.  **Set Up Alerting**: Configure Prometheus alerting for abnormal conditions, such as:
    *   Sudden spikes in error counts
    *   Unusual file system activity
    *   API rate limiting thresholds
4.  **Create Dashboards**: Set up Grafana dashboards to visualize:
    *   User growth trends
    *   API usage patterns
    *   File system usage by user
    *   System health metrics

By following these recommendations, you can gain valuable insights into your Charm server's operation and user activity, helping you identify issues early and understand usage patterns.
