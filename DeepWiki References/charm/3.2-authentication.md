Authentication
==============

This document explains the authentication system used in Charm Cloud. It covers how clients authenticate with the Charm server, how JWT tokens are issued and used, and the device linking mechanism that allows users to access their account from multiple devices.

For information about how keys are managed after authentication is established, see [Key Management](https://deepwiki.com/charmbracelet/charm/3.3-key-management)
.

Authentication Overview
-----------------------

Charm uses SSH public key cryptography as its primary authentication mechanism. This passwordless approach provides strong security while maintaining a seamless user experience across devices.

Sources: [client/client.go62-115](https://github.com/charmbracelet/charm/blob/aff3071d/client/client.go#L62-L115)
 [server/auth.go40-72](https://github.com/charmbracelet/charm/blob/aff3071d/server/auth.go#L40-L72)

Client-Side Authentication
--------------------------

The client authentication process begins by locating or generating SSH keys, then using these keys to establish a connection with the Charm server.

### Key Location and Generation

When initializing, the client performs these steps:

1.  Search for existing SSH keys in the Charm data directory
2.  If no keys are found, generate a new key pair (default: Ed25519)
3.  Use the private key to authenticate with the Charm server

Sources: [client/client.go62-115](https://github.com/charmbracelet/charm/blob/aff3071d/client/client.go#L62-L115)
 [client/client.go302-324](https://github.com/charmbracelet/charm/blob/aff3071d/client/client.go#L302-L324)

The client supports the following configuration options for authentication:

| Configuration | Environment Variable | Default Value | Description |
| --- | --- | --- | --- |
| Host | CHARM\_HOST | cloud.charm.sh | Charm server hostname |
| SSHPort | CHARM\_SSH\_PORT | 35353 | SSH port for authentication |
| KeyType | CHARM\_KEY\_TYPE | ed25519 | SSH key type (ed25519 or rsa) |
| IdentityKey | CHARM\_IDENTITY\_KEY | ""  | Path to specific SSH key to use |
| DataDir | CHARM\_DATA\_DIR | ""  | Directory to store keys and data |

Sources: [client/client.go28-37](https://github.com/charmbracelet/charm/blob/aff3071d/client/client.go#L28-L37)
 [client/client.go53-59](https://github.com/charmbracelet/charm/blob/aff3071d/client/client.go#L53-L59)

Server-Side Authentication
--------------------------

The server handles authentication through its SSH server component, which processes various authentication-related commands.

### SSH Command Handlers

The SSH server implements middleware that routes commands to specific handlers:

| Command | Handler | Purpose |
| --- | --- | --- |
| `api-auth` | `handleAPIAuth` | Authenticate and return user info with JWT |
| `api-keys` | `handleAPIKeys` | List keys linked to a user's account |
| `api-link` | `handleAPILink` | Initiate or respond to linking requests |
| `api-unlink` | `handleAPIUnlink` | Remove a key from a user's account |
| `id` | `handleID` | Return the user's ID |
| `jwt` | `handleJWT` | Generate a JWT token for API access |

Sources: [server/auth.go13-38](https://github.com/charmbracelet/charm/blob/aff3071d/server/auth.go#L13-L38)

### User Authentication Flow

When a client authenticates with the server, the following steps occur:

1.  The server extracts the client's public key
2.  It looks up the user associated with this key in the database
3.  If no user exists, a new one is created
4.  The server generates a JWT token for API access
5.  Authentication information is returned to the client

Sources: [server/auth.go40-72](https://github.com/charmbracelet/charm/blob/aff3071d/server/auth.go#L40-L72)

Device Linking
--------------

A key feature of Charm's authentication system is the ability to link multiple devices to the same account. This is done through a secure token-based linking process.

### Linking Process

The linking process involves three main components:

1.  **Token generation**: The primary device requests a linking token
2.  **Token transmission**: The user manually transfers this token to the secondary device
3.  **Authentication linking**: The secondary device authenticates with the token

Sources: [server/link.go116-210](https://github.com/charmbracelet/charm/blob/aff3071d/server/link.go#L116-L210)
 [server/link.go213-256](https://github.com/charmbracelet/charm/blob/aff3071d/server/link.go#L213-L256)

### Account Merging

When linking devices, if the new device's key is already associated with a different account, the accounts are merged:

1.  The server checks if the requesting key belongs to an existing account
2.  If it does, and it's a different account, the two accounts are merged
3.  All keys, data, and files from both accounts are consolidated under one account

This ensures a seamless experience even if a user accidentally creates multiple accounts.

Sources: [server/link.go182-194](https://github.com/charmbracelet/charm/blob/aff3071d/server/link.go#L182-L194)

Security Considerations
-----------------------

The Charm authentication system provides several security benefits:

1.  **Passwordless Authentication**: Using SSH keys eliminates password-related vulnerabilities
2.  **Public Key Cryptography**: Strong cryptographic security for authentication
3.  **JWT-based API Access**: Time-limited tokens for API access
4.  **Multi-Device Support**: Secure linking of multiple devices to one account
5.  **Key Revocation**: Ability to unlink keys if a device is lost or compromised

| Data Directory | \--data-dir | Directory to store SQLite db, SSH keys and data |
