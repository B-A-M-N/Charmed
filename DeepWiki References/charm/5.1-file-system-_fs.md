Purpose and Scope
-----------------

The Charm File System (FS) provides a cloud-based encrypted filesystem implementation that adheres to Go's standard `fs.FS` interfaces. It allows for secure storage and retrieval of files with client-side encryption, ensuring data privacy even from the server operators. This page documents the FS component architecture, its implementation details, and how it integrates with other Charm components. For information about the Key-Value store functionality, see [Key-Value Store (KV)](https://deepwiki.com/charmbracelet/charm/5.2-key-value-store-(kv))
.

Architecture Overview
---------------------

The File System component implements Go's standard filesystem interfaces (`fs.FS`, `fs.ReadFileFS`, and `fs.ReadDirFS`) while adding write capabilities. It operates within the client-server architecture of Charm, with encryption/decryption happening exclusively on the client side.

Sources: [fs/fs.go23-31](https://github.com/charmbracelet/charm/blob/aff3071d/fs/fs.go#L23-L31)

Client Commands Interface
-------------------------

The Charm FS provides several CLI commands for interacting with the filesystem:

| Command | Description | Usage |
| --- | --- | --- |
| `cat` | Display file contents | `charm fs cat [charm:]PATH` |
| `cp` | Copy files | `charm fs cp [charm:]PATH [charm:]PATH` |
| `rm` | Remove files | `charm fs rm [charm:]PATH` |
| `mv` | Move files | `charm fs mv [charm:]PATH [charm:]PATH` |
| `ls` | List directory contents | `charm fs ls [charm:]PATH` |
| `tree` | Display directory tree | `charm fs tree [charm:]PATH` |

The `charm:` prefix indicates a path in the cloud filesystem. Without this prefix, the path is treated as local.

Sources: [cmd/fs.go35-92](https://github.com/charmbracelet/charm/blob/aff3071d/cmd/fs.go#L35-L92)

### Encryption System

The FS implementation uses client-side encryption for all data:

1.  **Path Encryption**: Each path component is encrypted separately to allow directory traversal
2.  **Content Encryption**: File content is encrypted before transmission
3.  **End-to-End Model**: Server only stores encrypted data and cannot access plaintext

Usage Examples
--------------

### Basic Remote File Operations

To list files in your Charm Cloud storage:

    charm fs ls charm:/
    

To create or update a file in your Charm Cloud storage:

    charm fs cp local_file.txt charm:/remote_file.txt
    

To retrieve a file from your Charm Cloud storage:

    charm fs cp charm:/remote_file.txt local_file.txt
    

### Working with Remote Directories

To create a directory structure (directories are created implicitly when writing files):

    charm fs cp local_file.txt charm:/documents/notes/file.txt
    

To recursively copy directories:

    charm fs cp -r local_directory charm:/backup
    

Sources: [cmd/fs.go225-288](https://github.com/charmbracelet/charm/blob/aff3071d/cmd/fs.go#L225-L288)

Integration with Other Charm Components
---------------------------------------

The File System component integrates with other parts of the Charm ecosystem:

1.  Uses the **Client** package for authentication and server communication
2.  Leverages the **Crypt** package for data encryption
3.  Complements the **KV** (Key-Value) store for different data storage patterns

For applications that need structured data with complex relationships, the Key-Value store (covered in [KV](https://deepwiki.com/charmbracelet/charm/5.2-key-value-store-(kv))
) might be more appropriate, while the File System is ideal for blob storage and traditional file management workflows.

Sources: [fs/fs.go23-31](https://github.com/charmbracelet/charm/blob/aff3071d/fs/fs.go#L23-L31)
