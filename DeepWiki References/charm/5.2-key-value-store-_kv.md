Key-Value Store (KV)
====================

Purpose and Scope
-----------------

The Key-Value Store (KV) provides a cloud-synced, encrypted persistent storage mechanism for Charm applications. It uses BadgerDB as the underlying storage engine while adding cloud synchronization capabilities that allow data to be seamlessly shared across multiple linked devices. This document covers the implementation, architecture, and usage of the KV system.

For information about the File System implementation, see [File System (FS)](https://deepwiki.com/charmbracelet/charm/5.1-file-system-(fs))
. For encryption details beyond what's covered here, see [Encryption](https://deepwiki.com/charmbracelet/charm/5.3-encryption)
.

Overview
--------

The Charm KV store wraps BadgerDB with additional functionality for cloud synchronization and encryption. Key features include:

*   Persistent local storage using BadgerDB
*   Transparent cloud synchronization across linked devices
*   End-to-end encryption of all data
*   Support for transactions and streaming operations
*   Automatic versioning and conflict resolution

Sources: [kv/kv.go1-30](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L1-L30)

Architecture
------------

The KV system integrates local BadgerDB storage with Charm's cloud synchronization and encryption mechanisms to provide a seamless experience across multiple devices.

The architecture consists of the following components:

1.  **KV Store**: The main interface that applications interact with
2.  **BadgerDB**: The underlying key-value database providing persistence
3.  **Encryption Layer**: Handles encryption/decryption of data
4.  **Sequence Service**: Manages version sequences for synchronization
5.  **FS Service**: Stores and retrieves diff files for synchronization

Sources: [kv/kv.go18-30](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L18-L30)
 [kv/kv.go32-70](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L32-L70)
 [kv/client.go154-189](https://github.com/charmbracelet/charm/blob/aff3071d/kv/client.go#L154-L189)

Data Model and Storage
----------------------

The KV store uses BadgerDB as its storage engine, with the following characteristics:

*   Keys and values are simple byte arrays
*   All data is stored locally in an encrypted format
*   Versioned storage using sequence numbers for synchronization
*   Diffs between versions are uploaded to the cloud for syncing

### KV Type Structure

The main `KV` type consists of:

*   `DB`: The BadgerDB instance for local storage
*   `name`: The name of the KV store
*   `cc`: A Charm client for authentication and communication
*   `fs`: A Charm FS instance for storing and retrieving diffs

Sources: [kv/kv.go25-30](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L25-L30)

Encryption Mechanism
--------------------

The KV store provides end-to-end encryption for all stored data:

1.  All data is encrypted locally before being stored in BadgerDB
2.  Encryption keys are managed by the Charm client
3.  Keys are automatically shared across linked devices
4.  Even when synced to the cloud, data remains encrypted

The encryption setup happens during database initialization in the `openDB` function, which retrieves encryption keys from the Charm client and configures BadgerDB to use them.

Sources: [kv/client.go155-189](https://github.com/charmbracelet/charm/blob/aff3071d/kv/client.go#L155-L189)
 [client/crypt.go17-43](https://github.com/charmbracelet/charm/blob/aff3071d/client/crypt.go#L17-L43)
 [kv/kv.go73-79](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L73-L79)

Synchronization Process
-----------------------

The KV store uses a sequence-based synchronization mechanism:

1.  Each write transaction receives a unique sequence number from the server
2.  Diffs between sequences are backed up to the Charm cloud
3.  During sync, missing diffs are downloaded and applied to the local DB
4.  This approach ensures eventual consistency across all devices

Key synchronization methods:

*   `Sync()`: Synchronizes the local DB with cloud updates
*   `syncFrom()`: Downloads and applies diffs newer than a specific version
*   `backupSeq()`: Uploads a diff to the cloud
*   `restoreSeq()`: Applies a downloaded diff to the local DB

Sources: [kv/kv.go107-110](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L107-L110)
 [kv/client.go68-153](https://github.com/charmbracelet/charm/blob/aff3071d/kv/client.go#L68-L153)

Transactions and Operations
---------------------------

The KV store supports BadgerDB's transaction model with additional cloud synchronization:

### Transaction Flow

During a write transaction:

1.  Get current sequence number from server
2.  Perform operations locally
3.  Sync any changes from the cloud
4.  Get next sequence number from server
5.  Commit transaction with new sequence
6.  Backup diff to the cloud

Sources: [kv/kv.go82-128](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L82-L128)

Basic Operations
----------------

The KV store provides simple methods for common operations:

*   `Set(key, value)`: Sets a value for a key
*   `Get(key)`: Retrieves a value by key
*   `Delete(key)`: Deletes a key-value pair
*   `Keys()`: Lists all keys in the store

These methods are convenience wrappers around BadgerDB transactions.

Sources: [kv/kv.go134-214](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L134-L214)

Initialization and Configuration
--------------------------------

### Opening a KV Store

Two methods are provided for opening a KV store:

1.  `Open()`: Allows full customization of BadgerDB options
2.  `OpenWithDefaults()`: Uses sensible defaults for most applications

The default configuration:

*   Stores data in the user's Charm data directory
*   Sets BadgerDB logging level to ERROR
*   Limits value log file size to 10MB
*   Disables BadgerDB's default logger

Sources: [kv/kv.go32-70](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L32-L70)

### Example KV Store Initialization

    kv, err := kv.OpenWithDefaults("my-app-data")
    if err != nil {
        // Handle error
    }
    defer kv.Close()
    

Multi-Device Support
--------------------

When a user links a new device to their Charm account:

1.  Encryption keys are shared with the new device
2.  The local KV store is reset on the new device
3.  Data is synced from the cloud to the new device

This happens automatically as part of the linking process.

Sources: [client/link.go117-160](https://github.com/charmbracelet/charm/blob/aff3071d/client/link.go#L117-L160)

Reset and Recovery
------------------

The KV store provides a `Reset()` method that:

1.  Deletes the local database
2.  Recreates it from scratch
3.  Syncs all data from the cloud

This is useful for recovering from corruption or synchronization issues.

Sources: [kv/kv.go221-245](https://github.com/charmbracelet/charm/blob/aff3071d/kv/kv.go#L221-L245)

Limitations and Considerations
------------------------------

*   BadgerDB has a maximum key size of 65,000 bytes
*   Default configuration limits values to 10MB maximum size
*   The KV store is meant for moderate data sizes, not large datasets
*   Synchronization happens only when explicitly triggered via transactions or `Sync()`

Related Systems
---------------

The KV store interacts with several other Charm systems:

*   **Authentication**: For secure communication with the server
*   **Encryption**: For securing data locally and in transit
*   **File System (FS)**: For storing and retrieving diffs
*   **Linking**: For supporting multi-device synchronization
