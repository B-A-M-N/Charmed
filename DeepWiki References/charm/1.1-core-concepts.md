Core Concepts
=============

This page provides an overview of the fundamental building blocks that make up the Charm system. Understanding these core concepts is essential for developers who want to integrate Charm into their terminal-based applications or contribute to the codebase.

Charm consists of four primary components that work together to provide a complete backend solution for CLI applications. For information about the overall architecture and how these components interact, see [Architecture Overview](https://deepwiki.com/charmbracelet/charm/1.2-architecture-overview)
.

The Four Pillars of Charm
-------------------------

Sources:

*   README.md:46-60

### Charm Accounts

Charm Accounts provides a frictionless authentication system based on SSH keys. This component handles user identity and authentication without traditional username/password flows.

Key features:

*   **SSH-based authentication**: Uses SSH keys for secure, passwordless authentication
*   **Invisible account creation**: New users get accounts automatically when they first connect
*   **Multi-device linking**: Users can link multiple devices to access their data
*   **Key management**: Users can add, list, and revoke keys as needed

When a user first interacts with a Charm-enabled application, the system checks for existing SSH keys. If none are found, it generates new ones and creates an account. For existing users, authentication happens seamlessly using their SSH keys.

Sources:

*   README.md:200-211

### Charm KV

Charm KV is an encrypted key-value store built on BadgerDB that provides cloud synchronization and multi-device access to data.

Key features:

*   **Local storage with cloud sync**: Data is stored locally with cloud backup
*   **Multi-device synchronization**: Changes sync across devices automatically
*   **End-to-end encryption**: All data is encrypted before leaving the client
*   **BadgerDB integration**: Built on a proven, performant key-value store

Sources:

*   README.md:102-140

### Charm FS

Charm FS provides a virtual filesystem that implements Go's `fs.FS` interface, allowing applications to store and retrieve files from the Charm Server.

Key features:

*   **Go fs.FS compatibility**: Works with standard Go filesystem interfaces
*   **Extended write operations**: Adds write capabilities beyond the read-only fs.FS interface
*   **Cloud-based storage**: Files are stored on the Charm Server
*   **End-to-end encryption**: All files are encrypted before upload

Sources:

*   README.md:142-179

### Charm Crypt

Charm Crypt provides the encryption foundation for all Charm components, ensuring that user data remains private and secure.

Key features:

*   **End-to-end encryption**: Data is encrypted on the client before transmission
*   **SSH key integration**: Uses the user's SSH keys for secure key management
*   **Symmetric encryption for data**: Efficient encryption of actual content
*   **Asymmetric encryption for keys**: Secure distribution of symmetric keys

The encryption process described in the diagram above follows the approach outlined in the crypt/README.md file, where a symmetric key is generated for data encryption and then that key itself is encrypted with the user's public SSH key.

*   README.md:192-198
*   crypt/README.md:20-29

Client-Server Architecture
--------------------------

The Charm system operates on a client-server model, with all sensitive operations happening on the client side to maintain security.

Key aspects of this architecture:

1.  **Client-side encryption**: All data is encrypted before it leaves the client
2.  **SSH-based authentication**: Authentication happens over SSH
3.  **HTTP data transfer**: Data is transferred via HTTP API calls
4.  **Server-side storage**: The server only stores encrypted data

The client is configured through environment variables as documented in the README:

| Environment Variable | Description | Default |
| --- | --- | --- |
| CHARM\_HOST | Server address | cloud.charm.sh |
| CHARM\_SSH\_PORT | SSH port | 35353 |
| CHARM\_HTTP\_PORT | HTTP port | 35354 |
| CHARM\_DEBUG | Enable debug logging | false |
| CHARM\_LOGFILE | Debug log file path | ""  |
| CHARM\_KEY\_TYPE | SSH key type | ed25519 |
| CHARM\_DATA\_DIR | Data directory path | ""  |
| CHARM\_IDENTITY\_KEY | Identity key path | ""  |

Data Flow and Security Model
----------------------------

Charm implements a robust security model that ensures user data remains private even from the server operators.

Key security principles:

1.  **Zero-knowledge design**: The server never has access to unencrypted data
2.  **SSH key foundation**: All security is built on the user's SSH keys
3.  **Local encryption**: Encryption and decryption only happen on the client
4.  **Secure key sharing**: When devices are linked, encryption keys are securely shared

This security model ensures that even if the server is compromised, user data remains protected by their SSH keys.

Self-Hosting Capabilities
-------------------------

A key feature of Charm is that users can self-host their own Charm server. This provides complete control over their data and infrastructure.

The Charm binary includes both client and server functionality, making it easy to deploy:

Self-hosting can be configured through environment variables:

| Environment Variable | Description | Default |
| --- | --- | --- |
| CHARM\_SERVER\_BIND\_ADDRESS | Network interface | 0.0.0.0 |
| CHARM\_SERVER\_HOST | Server hostname | localhost |
| CHARM\_SERVER\_SSH\_PORT | SSH port | 35353 |
| CHARM\_SERVER\_HTTP\_PORT | HTTP port | 35354 |
| CHARM\_SERVER\_STATS\_PORT | Statistics port | 35355 |
| CHARM\_SERVER\_HEALTH\_PORT | Health check port | 35356 |
| CHARM\_SERVER\_DATA\_DIR | Data directory | ./data |
| CHARM\_SERVER\_USE\_TLS | Enable TLS | false |
| CHARM\_SERVER\_TLS\_KEY\_FILE | TLS key file | ""  |
| CHARM\_SERVER\_TLS\_CERT\_FILE | TLS certificate file | ""  |
| CHARM\_SERVER\_PUBLIC\_URL | Public URL | ""  |
| CHARM\_SERVER\_ENABLE\_METRICS | Enable metrics | false |
| CHARM\_SERVER\_USER\_MAX\_STORAGE | Storage limit per user | 0 (unlimited) |

The server can be run via Docker or systemd for production deployments.

Integration Example
-------------------

Integrating Charm into an application typically involves using one or more of the core components. Here's a simplified example of how an application might use Charm KV:

1.  Import the KV package
2.  Open a database (or create one if it doesn't exist)
3.  Sync with the cloud
4.  Perform operations on the database

When the application interacts with Charm KV, the following happens:

1.  Data is stored locally in BadgerDB
2.  Changes are encrypted and synced to the cloud
3.  The user can access their data from any linked device
4.  All authentication and encryption happens transparently

For other components like Charm FS and Charm Crypt, the integration follows similar patterns, with all components handling authentication and encryption automatically.
