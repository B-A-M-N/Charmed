This document details the release process for Soft Serve, including versioning conventions, build automation, artifact generation, and distribution methods. For information about the build system itself, see [Build System](https://deepwiki.com/charmbracelet/soft-serve/5.2-build-system)
.

Overview
--------

Soft Serve follows a standardized release process that utilizes GitHub Actions and GoReleaser to automate the building, packaging, and distribution of software artifacts. The process includes both scheduled nightly builds and versioned releases, each serving different purposes in the software lifecycle.

Sources: [.github/workflows/goreleaser.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/goreleaser.yml)
 [.github/workflows/nightly.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/nightly.yml)
 [.goreleaser.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.goreleaser.yml)

Versioning
----------

Soft Serve follows Semantic Versioning (SemVer) principles for release versioning.

### Version Format

Versions follow the `vX.Y.Z` format where:

*   `X` is the major version (breaking changes)
*   `Y` is the minor version (new features, non-breaking)
*   `Z` is the patch version (bug fixes, non-breaking)

Sources: [.github/workflows/goreleaser.yml5-7](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/goreleaser.yml#L5-L7)

Release Preparation
-------------------

Before initiating a release, several preparation steps should be completed:

1.  Ensure all intended changes are merged to the main branch
2.  Verify that tests pass successfully
3.  Update the CHANGELOG.md if applicable
4.  Decide on the appropriate version number based on the nature of changes

Creating a Release
------------------

The release process is triggered by pushing a git tag that follows the semantic versioning pattern:

    git tag v1.2.3
    git push origin v1.2.3
    

This tag push automatically triggers the GoReleaser workflow defined in the GitHub Actions configuration.

Sources: [.github/workflows/goreleaser.yml4-7](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/goreleaser.yml#L4-L7)

Release Workflow
----------------

The release process is automated using GitHub Actions and GoReleaser, a popular tool for automating the release of Go applications.

Sources: [.github/workflows/goreleaser.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/goreleaser.yml)
 [.goreleaser.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.goreleaser.yml)

### GoReleaser Configuration

Soft Serve uses a shared GoReleaser configuration from the Charm organization's meta repository, with specific variables defined for this project:

| Variable | Value | Purpose |
| --- | --- | --- |
| main | "./cmd/soft" | Entry point for building the binary |
| binary\_name | soft | Name of the generated binary |
| description | "A tasty, self-hostable Git server for the command line🍦" | Project description used in packages |
| github\_url | "[https://github.com/charmbracelet/soft-serve](https://github.com/charmbracelet/soft-serve)<br>" | Repository URL |

Sources: [.goreleaser.yml1-14](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.goreleaser.yml#L1-L14)

Release Artifacts
-----------------

Each release generates multiple artifacts for different platforms and package formats:

### Binaries

*   Cross-platform binaries for various operating systems (Linux, macOS, Windows) and architectures
*   Signed macOS binaries with Apple notarization

### Packages

*   Linux packages in DEB and RPM formats
*   Homebrew formula for macOS users

### Container Images

*   Docker images published to Docker Hub as `charmcli/soft-serve`
*   Docker images published to GitHub Container Registry as `ghcr.io/charmbracelet/soft-serve`

Sources: [.goreleaser.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.goreleaser.yml)
 [docker.md](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/docker.md?plain=1)
 [Dockerfile](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/Dockerfile)

Distribution Channels
---------------------

Release artifacts are distributed through multiple channels:

Sources: [.github/workflows/goreleaser.yml16-28](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/goreleaser.yml#L16-L28)
 [docker.md1-8](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/docker.md?plain=1#L1-L8)

### GitHub Releases

The primary distribution channel for Soft Serve binaries is GitHub Releases. Each release creates an entry on the project's Releases page with:

*   Release notes
*   Downloadable binaries for all supported platforms
*   Checksums for verification

### Docker Hub

Docker images are published to Docker Hub as `charmcli/soft-serve` with appropriate version tags.

### GitHub Container Registry

Docker images are also published to GitHub Container Registry as `ghcr.io/charmbracelet/soft-serve`.

### Homebrew

A Homebrew formula is updated for macOS users to easily install the latest version.

Sources: [.goreleaser.yml11-12](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.goreleaser.yml#L11-L12)
 [docker.md3-7](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/docker.md?plain=1#L3-L7)

Nightly Builds
--------------

In addition to versioned releases, Soft Serve has a nightly build process that generates development builds from the latest code in the main branch.

Nightly builds:

*   Are triggered automatically when changes are pushed to the main branch
*   Include development versions of binaries with `-dev` suffix
*   Publish Docker images with the `nightly` tag
*   Are not intended for production use

Sources: [.github/workflows/nightly.yml1-20](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/nightly.yml#L1-L20)

Required Secrets
----------------

The release process requires several secrets to be configured in the GitHub repository settings:

| Secret | Purpose |
| --- | --- |
| DOCKERHUB\_USERNAME | Username for publishing Docker images to Docker Hub |
| DOCKERHUB\_TOKEN | Authentication token for Docker Hub |
| GORELEASER\_KEY | GoReleaser Pro license key |
| PERSONAL\_ACCESS\_TOKEN | GitHub Personal Access Token for creating releases |
| FURY\_TOKEN | Token for publishing packages to Package Cloud/Fury.io |
| NFPM\_GPG\_KEY | GPG key for signing Linux packages |
| NFPM\_PASSPHRASE | Passphrase for the GPG key |
| MACOS\_SIGN\_P12 | Base64-encoded certificate for signing macOS binaries |
| MACOS\_SIGN\_PASSWORD | Password for the signing certificate |
| MACOS\_NOTARY\_\* | Credentials for Apple notarization service |

Sources: [.github/workflows/goreleaser.yml16-28](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/goreleaser.yml#L16-L28)
 [.github/workflows/nightly.yml11-19](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/nightly.yml#L11-L19)

Docker Image Release Process
----------------------------

The Docker image is built as part of the release process with:

*   Base image: Alpine Linux
*   Soft Serve binary copied into the image
*   Git, Bash, and OpenSSH installed as dependencies
*   Pre-configured environment variables
*   Exposed ports: 23231 (SSH), 23232 (HTTP), 23233 (Stats), 9418 (Git)

Users can pull the latest release using:

    docker pull charmcli/soft-serve:latest
    

Or a specific version:

    docker pull charmcli/soft-serve:v1.2.3
