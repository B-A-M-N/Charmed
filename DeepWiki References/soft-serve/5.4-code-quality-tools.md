Code Quality Tools
==================

This document outlines the code quality tools used in the Soft Serve project. It describes the linting configurations, formatters, and CI integration used to maintain code quality standards across the codebase. For information about the build system and testing framework, see [Build System](https://deepwiki.com/charmbracelet/soft-serve/5.2-build-system)
 and [Testing Framework](https://deepwiki.com/charmbracelet/soft-serve/5.3-testing-framework)

Overview
--------

Soft Serve uses a combination of linting tools, code formatters, and continuous integration checks to ensure high code quality standards. These tools help identify potential bugs, enforce style conventions, and maintain consistency throughout the codebase.

Sources: [.github/workflows/lint.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/lint.yml)
 [.golangci.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.golangci.yml)

Linting Configuration
---------------------

Soft Serve uses golangci-lint as its primary linting tool for Go code. The configuration is defined in the `.golangci.yml` file, which specifies the enabled linters, exclusion rules, and formatting options.

### Enabled Linters

The project employs a comprehensive set of linters to catch various issues:

Sources: [.golangci.yml4-28](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.golangci.yml#L4-L28)

The linters are categorized by their primary focus area:

| Category | Linters | Purpose |
| --- | --- | --- |
| Code Correctness | bodyclose, nilerr, rowserrcheck, sqlclosecheck, wrapcheck | Ensure proper resource handling and error management |
| Code Style | goconst, godot, goprintffuncname, whitespace, revive | Enforce consistent code style and documentation |
| Performance | prealloc, noctx | Identify potential performance issues |
| Security | gosec | Detect security vulnerabilities |
| Code Complexity | nestif, nakedret, unparam, exhaustive | Reduce code complexity and improve maintainability |
| Documentation | misspell | Catch spelling errors in comments and strings |
| Code Quality | godox, unconvert, nolintlint | Find TODOs/FIXMEs, unnecessary type conversions, and improper linter directives |

Sources: [.golangci.yml5-28](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.golangci.yml#L5-L28)

### Linter Exclusions

The configuration excludes certain paths and generated code from linting:

Sources: [.golangci.yml29-37](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.golangci.yml#L29-L37)
 [.golangci.yml44-49](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.golangci.yml#L44-L49)

Code Formatters
---------------

In addition to linters, Soft Serve uses code formatters to ensure consistent code style:

| Formatter | Purpose |
| --- | --- |
| gofumpt | Enhanced version of `gofmt` with additional style rules |
| goimports | Formats imports and adds missing ones |

These formatters are applied automatically during the linting process and follow the same exclusion rules as the linters.

Sources: [.golangci.yml40-49](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.golangci.yml#L40-L49)

CI Integration
--------------

Soft Serve integrates linting into its CI/CD pipeline using GitHub Actions workflows.

### Lint Workflow

The primary lint workflow runs on every push and pull request, ensuring that all code changes meet the project's quality standards.

Sources: [.github/workflows/lint.yml1-8](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/lint.yml#L1-L8)

### Lint Sync Workflow

To ensure consistency in linting configurations across repositories, Soft Serve uses a scheduled workflow that synchronizes linting configurations:

Sources: [.github/workflows/lint-sync.yml1-14](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/lint-sync.yml#L1-L14)

This workflow runs weekly (on Sundays at midnight) or can be triggered manually. It has permissions to write to the repository and create pull requests when updates to linting configurations are needed.

Developer Workflow
------------------

Developers working on Soft Serve should follow this workflow to ensure code quality:

1.  Run linters locally before committing code
2.  Address any linting issues
3.  Format code using the project's formatters
4.  Commit and push changes
5.  Verify that CI linting checks pass

Sources: [.github/workflows/lint.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/lint.yml)
 [.golangci.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.golangci.yml)

Best Practices
--------------

When working with Soft Serve's code quality tools:

1.  Run linters locally using `golangci-lint run` to catch issues early
2.  Do not disable linters without a good reason
3.  If you need to ignore a specific linting issue, use `//nolint:lintername` comments sparingly and with justification
4.  Understand the purpose of each enabled linter to write better code from the start
5.  Keep the codebase clean by addressing TODOs and FIXMEs (caught by the `godox` linter)

Sources: [.golangci.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.golangci.yml)
