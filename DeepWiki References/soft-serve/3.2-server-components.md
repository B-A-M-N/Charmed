This page documents the server components that make up Soft Serve, their configuration options, and how they work together. These components provide the infrastructure for serving Git repositories over multiple protocols and interfaces.

For information about the overall system architecture, see [System Architecture](https://deepwiki.com/charmbracelet/soft-serve/1.3-system-architecture)
.

Core Server Components
----------------------

Soft Serve implements four main server components that provide different ways for users to interact with Git repositories:

1.  **SSH Server**: Provides SSH access to the terminal UI, Git operations, and administrative commands
2.  **Git Daemon**: Enables native Git protocol access for faster cloning operations
3.  **HTTP Server**: Provides HTTP/HTTPS access to repositories and Git LFS support
4.  **Stats Server**: Exposes metrics and monitoring endpoints

These components are instantiated and managed by a central server orchestrator.

Configuration System
--------------------

All server components share a unified configuration system that reads settings from multiple sources in the following order:

1.  Default configuration (hardcoded values)
2.  Configuration file (YAML format)
3.  Environment variables (prefixed with `SOFT_SERVE_`)

The configuration for each server component is defined in its own section of the config file.

Server Orchestrator
-------------------

The Server struct in `cmd/soft/serve/server.go` serves as the central orchestrator that manages all server components. It handles:

1.  Initialization of server components
2.  Starting each server in its own goroutine
3.  Graceful shutdown of all components
4.  Error handling and propagation

SSH Server
----------

The SSH Server is the primary interface for user interaction with Soft Serve. It handles:

1.  Authentication via SSH keys
2.  Interactive terminal UI for browsing repositories
3.  Git operations over SSH (clone, push, pull)
4.  Administrative commands for managing the server

### SSH Server Configuration Options

| Option | Description | Default |
| --- | --- | --- |
| `enabled` | Enable/disable the SSH server | `true` |
| `listen_addr` | Address to listen on | `:23231` |
| `public_url` | Public URL for SSH access | `ssh://localhost:23231` |
| `key_path` | Path to SSH server's host key | `ssh/soft_serve_host_ed25519` |
| `client_key_path` | Path to client key for remote operations | `ssh/soft_serve_client_ed25519` |
| `max_timeout` | Maximum connection time in seconds | `0` (no timeout) |
| `idle_timeout` | Idle timeout in seconds | `600` (10 minutes) |

Git Daemon
----------

The Git Daemon provides read-only access to repositories using the native Git protocol. This protocol is faster than HTTP or SSH for cloning operations because it has lower overhead.

The daemon handles two types of Git services:

*   `git-upload-pack`: Used for fetching objects (clone, fetch)
*   `git-upload-archive`: Used for creating repository archives

### Git Daemon Implementation

The Git Daemon is implemented in the `pkg/daemon` package. It:

1.  Listens for TCP connections on the configured port
2.  Parses Git protocol requests
3.  Authenticates and authorizes access
4.  Dispatches to the appropriate Git service handler
5.  Manages connection lifecycle (timeouts, max connections)

### Git Daemon Configuration Options

| Option | Description | Default |
| --- | --- | --- |
| `enabled` | Enable/disable the Git daemon | `true` |
| `listen_addr` | Address to listen on | `:9418` |
| `public_url` | Public URL for Git access | `git://localhost` |
| `max_timeout` | Maximum connection time in seconds | `0` (no timeout) |
| `idle_timeout` | Idle timeout in seconds | `3` |
| `max_connections` | Maximum concurrent connections | `32` |

HTTP Server
-----------

The HTTP Server provides:

1.  Repository access over HTTP/HTTPS (clone, fetch)
2.  Git LFS (Large File Storage) support
3.  Webhook functionality

The server uses a router to dispatch requests to appropriate controller handlers:

*   Git controller for regular Git operations
*   LFS controller for handling large files
*   Webhook handlers for integrations

### HTTP Server Configuration Options

| Option | Description | Default |
| --- | --- | --- |
| `enabled` | Enable/disable the HTTP server | `true` |
| `listen_addr` | Address to listen on | `:23232` |
| `public_url` | Public URL for HTTP access | `http://localhost:23232` |
| `tls_key_path` | Path to TLS private key | `""` (empty) |
| `tls_cert_path` | Path to TLS certificate | `""` (empty) |

Enabling and Disabling Servers
------------------------------

Each server component can be independently enabled or disabled through configuration. This provides flexibility in deployment scenarios:

*   For maximum security, you might disable HTTP and Git protocol servers
*   For public repositories, you might enable all protocols
*   In resource-constrained environments, you might disable stats collection

To disable a server component, set its `enabled` configuration option to `false` in the config file:

Or using environment variables:

    SOFT_SERVE_SSH_ENABLED=true
    SOFT_SERVE_GIT_ENABLED=false
    SOFT_SERVE_HTTP_ENABLED=true
    SOFT_SERVE_STATS_ENABLED=false
    

The server orchestrator checks these settings when starting up and only initializes enabled components.

Server Component Interactions
-----------------------------

All server components interact with the common Backend service that provides:

*   Repository access and management
*   User authentication and authorization
*   Access control
*   Git operations and hooks

This shared backend ensures consistent behavior across all protocols.

Connection Handling and Timeouts
--------------------------------

Each server component implements specific timeout and connection handling logic to ensure resource efficiency:

1.  **SSH Server**: Uses longer timeouts (10 minutes by default) to accommodate interactive sessions
2.  **Git Daemon**: Uses short timeouts (3 seconds) and limits maximum concurrent connections (32 by default)
3.  **HTTP Server**: Standard HTTP timeout handling with keep-alive support
4.  **Stats Server**: Optimized for quick metrics queries

These timeouts can be configured to match your specific workloads and resource constraints.
