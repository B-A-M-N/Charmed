The HTTP Server component in Soft Serve provides an HTTP interface for Git operations, primarily serving Git clones and Git LFS (Large File Storage) functionality. It complements the SSH Server ([3.2.1](https://deepwiki.com/charmbracelet/soft-serve/3.2.1-ssh-server)
) and Git Daemon ([3.2.2](https://deepwiki.com/charmbracelet/soft-serve/3.2.2-git-daemon)
) by offering an alternative protocol for repository access and specialized support for large binary files through Git LFS.

Configuration
-------------

The HTTP Server's behavior is controlled through the `http` section in the Soft Serve configuration file or through environment variables. These settings determine the listen address, TLS configuration, and public URL used for repository cloning.

### HTTP Server Configuration Options

| Option | Description | Default | Environment Variable |
| --- | --- | --- | --- |
| `enabled` | Toggle the HTTP server on/off | `true` | `SOFT_SERVE_HTTP_ENABLED` |
| `listen_addr` | Address on which the HTTP server will listen | `:23232` | `SOFT_SERVE_HTTP_LISTEN_ADDR` |
| `tls_key_path` | Path to the TLS private key | `""` | `SOFT_SERVE_HTTP_TLS_KEY_PATH` |
| `tls_cert_path` | Path to the TLS certificate | `""` | `SOFT_SERVE_HTTP_TLS_CERT_PATH` |
| `public_url` | Public URL of the HTTP server | `http://localhost:23232` | `SOFT_SERVE_HTTP_PUBLIC_URL` |

Sources: [pkg/config/config.go64-80](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/config/config.go#L64-L80)
 [pkg/config/config.go354-358](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/config/config.go#L354-L358)

### TLS Configuration

For secure HTTPS connections, you must provide both a TLS key and certificate. When TLS is enabled, ensure that you also update the `public_url` to use `https://` instead of `http://`.

If the paths provided for `tls_key_path` and `tls_cert_path` are relative, they will be resolved relative to the Soft Serve data directory.

Server Architecture
-------------------

The HTTP Server in Soft Serve is implemented using Go's standard `net/http` package with the Gorilla Mux router. The server is structured with several layers of middleware and controllers for handling different types of requests.

Sources: [pkg/web/server.go1-31](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/web/server.go#L1-L31)

### Router and Middleware

The HTTP router is created using the `mux.NewRouter()` function from the Gorilla Mux package. It's wrapped with several middleware layers:

1.  **Logging Middleware**: Logs HTTP requests and responses
2.  **Context Handler**: Adds the application context to each request
3.  **Compression Handler**: Compresses responses when supported by the client
4.  **Recovery Handler**: Recovers from panics and returns 500 errors

These middleware components ensure proper request handling, logging, and error recovery.

Sources: [pkg/web/server.go12-30](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/web/server.go#L12-L30)

### Git Controller

The Git Controller handles Git-related HTTP requests, including:

1.  **Repository information**: Providing metadata about repositories
2.  **Git clone/fetch operations**: Serving Git pack files
3.  **Git LFS**: Handling Large File Storage requests

The controller authenticates requests using access tokens or checks for anonymous access permissions based on server settings.

Sources: [pkg/web/server.go18](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/web/server.go#L18-L18)
 [README.md372-390](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/README.md?plain=1#L372-L390)

Git LFS Support
---------------

Git Large File Storage (LFS) is fully supported through the HTTP server. This allows clients to store large binary files outside the Git repository structure, improving performance for repositories with large assets.

Starting the HTTP Server
------------------------

The HTTP server is started as part of the Soft Serve service initialization, alongside other server components like the SSH server and Git daemon. It's only started if the `http.enabled` configuration is set to `true`.

Sources: [cmd/soft/serve/server.go119-128](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/cmd/soft/serve/server.go#L119-L128)

Anonymous HTTP access is controlled by the server's `allow-keyless` and `anon-access` settings. By default, anonymous users have read-only access to public repositories.
