Repository Management
=====================

Repository Management is a core component of Soft Serve that handles the entire lifecycle of Git repositories on the server. This document explains how repositories are structured, created, maintained, and accessed throughout the system.

For information about Git hooks and webhooks that can be integrated with repositories, see [Git Hooks and Webhooks](https://deepwiki.com/charmbracelet/soft-serve/3.4-git-hooks-and-webhooks)
.

Repository Structure and Storage
--------------------------------

Repositories in Soft Serve consist of both filesystem storage and metadata stored in a database. The system manages the synchronization between these two storage mediums.

Sources: [pkg/backend/repo.go641-758](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L641-L758)
 [pkg/backend/repo.go632-636](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L632-L636)

### Repository Metadata

Each repository has the following properties stored in its metadata:

| Property | Type | Description |
| --- | --- | --- |
| ID  | int64 | Unique identifier for the repository |
| UserID | sql.NullInt64 | ID of the user who owns the repository (optional) |
| Name | string | Repository name, unique across the server |
| Description | string | Text description of the repository |
| ProjectName | string | Display name for the repository |
| Private | bool | Controls access restrictions |
| Hidden | bool | Controls visibility in repository listings |
| Mirror | bool | Indicates if repository is a mirror of another repo |
| CreatedAt | time.Time | Repository creation timestamp |
| UpdatedAt | time.Time | Last update timestamp |

Sources: [pkg/backend/repo.go650-737](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L650-L737)

Repository Operations
---------------------

The repository management system provides several core operations for managing repositories throughout their lifecycle.

### Repository Operations Flow

Sources: [pkg/backend/repo.go31-367](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L31-L367)

### Creating Repositories

Repositories can be created in two ways:

1.  Through the API via `CreateRepository` method
2.  On-demand via Git push to a non-existent repository

When creating a repository, the following steps occur:

1.  The repository name is sanitized and validated
2.  A transaction begins in the database
3.  Repository metadata is stored in the database
4.  A bare Git repository is initialized on disk
5.  Description and export files are written
6.  Git hooks are generated
7.  The transaction is committed

Sources: [pkg/backend/repo.go31-89](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L31-L89)
 [pkg/utils/utils.go42-55](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/utils/utils.go#L42-L55)

### Importing Repositories

Repositories can be imported from external Git sources:

1.  The system creates a background task for the import operation
2.  The remote repository is cloned to the local filesystem
3.  Repository metadata is created in the database
4.  LFS objects are processed and stored if needed
5.  Webhook configurations are established

Import operations can also create mirror repositories that can be periodically updated from their source.

Sources: [pkg/backend/repo.go93-208](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L93-L208)

### Deleting Repositories

When a repository is deleted:

1.  Repository metadata is removed from the database
2.  Associated LFS objects are deleted
3.  The repository directory is removed from the filesystem
4.  A webhook event is triggered to notify any subscribed services

Sources: [pkg/backend/repo.go210-280](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L210-L280)

### Renaming Repositories

Repository renaming involves:

1.  Validating the new name
2.  Updating the repository name in the database
3.  Moving the repository directory on disk
4.  Triggering a webhook event for the rename operation

Sources: [pkg/backend/repo.go309-367](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L309-L367)

Repository Configuration
------------------------

Repositories can be configured with various settings that affect their behavior and visibility.

### Repository Visibility Controls

Soft Serve provides two main visibility controls:

1.  **Private flag**: Controls access restrictions
    
    *   When `true`: Only collaborators and admins can access
    *   When `false`: Public access is allowed (based on server settings)
    *   Controls the presence of `git-daemon-export-ok` file
2.  **Hidden flag**: Controls visibility in listings
    
    *   When `true`: Repository is excluded from listings
    *   When `false`: Repository appears in listings
    *   Doesn't affect direct access when repository name is known

Sources: [pkg/backend/repo.go528-613](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L528-L613)

### Repository Description and Metadata

Repositories have configurable metadata including:

*   **Description**: A text description shown in the UI and exported for Git
*   **Project Name**: A display name for the repository that can differ from its technical name

Sources: [pkg/backend/repo.go542-560](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L542-L560)
 [pkg/backend/repo.go615-629](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L615-L629)

Repository Access Control
-------------------------

The repository access control system determines who can view, modify, and administer repositories.

Sources: [pkg/ssh/cmd/cmd.go112-198](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/ssh/cmd/cmd.go#L112-L198)

### Access Level Checks

The system uses several helper functions to check access:

1.  `checkIfReadable`: Verifies the user has at least read-only access
2.  `checkIfCollab`: Verifies the user has at least read-write access
3.  `checkIfAdmin`: Verifies the user has administrative access

These functions are used by SSH command handlers to enforce proper access control on repository operations.

Sources: [pkg/ssh/cmd/cmd.go112-197](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/ssh/cmd/cmd.go#L112-L197)

Git Integration and Implementation
----------------------------------

Soft Serve implements repository management by wrapping Git functionality through a custom Git package.

Sources: [git/repo.go20-190](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/git/repo.go#L20-L190)
 [pkg/backend/repo.go696-698](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L696-L698)

### Repository Operations

The Git integration layer provides methods for:

*   Initializing bare repositories
*   Cloning from remote sources
*   Managing references and branches
*   Working with trees and blobs
*   Handling diffs and patches
*   Accessing commit history

These operations are wrapped in a type-safe API that integrates with the rest of Soft Serve.

Sources: [git/repo.go30-190](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/git/repo.go#L30-L190)

Conclusion
----------

Repository Management is a foundational component of Soft Serve that enables the creation, configuration, and manipulation of Git repositories through a structured API. The system maintains repository data across both filesystem storage and database records while providing a consistent access control layer for security.

Understanding this component is essential for working with Soft Serve, as repositories are the primary resource managed by the system.
