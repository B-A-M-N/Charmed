Git Hooks and Webhooks
======================

This page documents Soft Serve's implementation of Git hooks and webhooks, which enable custom behavior and integrations with external systems. Git hooks are scripts that run at specific points in Git's execution, while webhooks allow sending HTTP notifications to external services when certain events occur in a repository.

Git Hooks
---------

Soft Serve implements standard Git server-side hooks to customize Git server behavior. These hooks are automatically generated for each repository and allow for custom behavior during Git operations.

### Supported Hooks

Soft Serve supports the following Git server-side hooks:

| Hook | Description | Arguments |
| --- | --- | --- |
| `pre-receive` | Runs before references are updated | None |
| `update` | Runs during reference updates | `$1 $2 $3` (reference name, old revision, new revision) |
| `post-receive` | Runs after all references are updated | None |
| `post-update` | Runs after all updates are completed | `$@` (all references that were updated) |

### Hook Generation Process

When a repository is created, Soft Serve automatically generates Git hook scripts for the repository using the `hooks.GenerateHooks()` function. For each supported hook type:

1.  A primary hook script is created in the repository's hooks directory
2.  A `.d` directory is created for the hook (e.g., `pre-receive.d/`)
3.  A Soft Serve-specific hook script is placed in the `.d` directory

The primary hook script reads from stdin, executes each executable in the hook's `.d` directory, and exits with the first non-zero exit code encountered. This allows multiple hook implementations to run for a single Git event.

The Soft Serve-specific hook script invokes the `soft hook` command with the appropriate hook name and arguments, connecting the Git hook system to Soft Serve's hook handling logic.

Webhooks
--------

Webhooks allow repository events to be sent to external services via HTTP requests. Soft Serve provides a comprehensive webhook system with support for various events, content types, and delivery management.

### Webhook Model

Sources: [pkg/backend/webhooks.go1-279](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/webhooks.go#L1-L279)
 [pkg/ssh/cmd/webhooks.go1-399](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/ssh/cmd/webhooks.go#L1-L399)

### Webhook Events

Soft Serve supports multiple webhook events that can trigger webhook deliveries:

| Event | Description |
| --- | --- |
| Repository create | Triggered when a repository is created |
| Repository delete | Triggered when a repository is deleted |
| Repository rename | Triggered when a repository is renamed |
| Repository visibility change | Triggered when a repository's visibility changes |
| Push | Triggered when changes are pushed to a repository |

Webhooks can be configured to trigger on specific events or multiple events.

Sources: [pkg/backend/repo.go224-228](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L224-L228)

### Webhook Content Types

Webhooks support two content types for payload delivery:

1.  `json` - JSON-formatted payload (default)
2.  `form` - Form-encoded payload

The content type can be specified when creating or updating a webhook.

### Managing Webhooks

Soft Serve provides a command-line interface for managing webhooks. The following commands are available:

Sources: [pkg/ssh/cmd/webhooks.go16-32](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/ssh/cmd/webhooks.go#L16-L32)
 [pkg/ssh/cmd/webhooks.go254-268](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/ssh/cmd/webhooks.go#L254-L268)

#### Creating a Webhook

To create a webhook, use the `webhook create` command with the following arguments and flags:

    soft webhook create REPOSITORY URL --events=EVENT1,EVENT2 --secret=SECRET --active=true|false --content-type=json|form
    

Required arguments:

*   `REPOSITORY` - Name of the repository
*   `URL` - The URL to send webhook payloads to

Optional flags:

*   `--events, -e` - Events to trigger the webhook
*   `--secret, -s` - Secret to sign the webhook payload
*   `--active, -a` - Whether the webhook is active (default: true)
*   `--content-type, -c` - Content type of the webhook payload (default: json)

Sources: [pkg/ssh/cmd/webhooks.go87-135](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/ssh/cmd/webhooks.go#L87-L135)

#### Webhook Delivery and Redelivery

When a webhook event occurs, Soft Serve sends an HTTP request to the webhook URL with the event payload. The delivery details, including request and response information, are stored in the database.

Webhook deliveries can be:

*   Listed using the `webhook deliveries list` command
*   Viewed in detail using the `webhook deliveries get` command
*   Redelivered using the `webhook deliveries redeliver` command

### Repository Creation

When a repository is created using `Backend.CreateRepository()`, the system generates Git hooks for the repository using `hooks.GenerateHooks()`. This ensures that all new repositories have the necessary hook infrastructure.

Sources: [pkg/backend/repo.go76-77](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L76-L77)

### Repository Deletion

When a repository is deleted using `Backend.DeleteRepository()`, the system creates a webhook event with the action `webhook.RepositoryEventActionDelete`. After the repository is deleted from the database and filesystem, the webhook event is sent using `webhook.SendEvent()`.

Sources: [pkg/backend/repo.go224-228](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L224-L228)
 [pkg/backend/repo.go279](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L279-L279)

### Repository Rename

When a repository is renamed using `Backend.RenameRepository()`, the system creates a webhook event with the action `webhook.RepositoryEventActionRename`. After the rename operation is completed, the webhook event is sent.

Sources: [pkg/backend/repo.go361-366](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L361-L366)

### Repository Visibility Change

When a repository's visibility is changed using `Backend.SetPrivate()`, the system checks if the visibility has actually changed. If it has, a webhook event with the action `webhook.RepositoryEventActionVisibilityChange` is created and sent.

Sources: [pkg/backend/repo.go601-611](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/repo.go#L601-L611)

Webhook Implementation Details
------------------------------

The webhook system is primarily implemented in the `webhook` package, with integration points in the `backend` package. The `Backend` struct provides methods for creating, updating, deleting, and querying webhooks and webhook deliveries.

Webhook data is stored in the database, with models defined in the `models` package. The webhook delivery system handles HTTP requests, payload generation, and delivery tracking.

Sources: [pkg/backend/webhooks.go1-279](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/pkg/backend/webhooks.go#L1-L279)

Conclusion
----------

Soft Serve provides a comprehensive Git hooks and webhooks system that allows for custom behavior and integration with external systems. Git hooks are automatically generated for each repository and provide hooks for various points in the Git workflow. The webhook system allows repository events to be sent to external services via HTTP requests, with support for various events, content types, and delivery management.
