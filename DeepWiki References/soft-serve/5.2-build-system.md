This document explains the build system for Soft Serve, covering how the project is built, tested, packaged, and released. The build system employs a combination of GoReleaser for releases, GitHub Actions for CI/CD pipelines, and Docker for containerization.

Overview
--------

Soft Serve's build system is designed to provide a consistent and automated way to build, test, and release the software across multiple platforms. It supports various output formats including binaries for different operating systems, Docker images, and package manager distributions.

Sources: [.github/workflows/goreleaser.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/goreleaser.yml)
 [.github/workflows/build.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/build.yml)
 [.github/workflows/nightly.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/nightly.yml)
 [.goreleaser.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.goreleaser.yml)

GoReleaser Configuration
------------------------

Soft Serve uses GoReleaser to manage the release process, which simplifies building and publishing Go applications.

### Configuration Structure

The GoReleaser configuration is defined in `.goreleaser.yml` and uses a template-based approach from the Charmbracelet organization's common configuration.

Sources: [.goreleaser.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.goreleaser.yml)

The main GoReleaser configuration imports a template from the Charmbracelet organization's central repository and customizes it with specific variables for Soft Serve:

*   Main entry point: `./cmd/soft`
*   Binary name: `soft`
*   Description: "A tasty, self-hostable Git server for the command line🍦"
*   GitHub URL and maintainer information

This approach ensures consistency across Charmbracelet projects while allowing for project-specific customization.

CI/CD Workflows
---------------

Soft Serve uses GitHub Actions for continuous integration and deployment. There are several key workflows:

### Build Workflow

The build workflow runs on pull requests and pushes to the main branch, handling compilation and basic testing.

Sources: [.github/workflows/build.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/build.yml)

### Release Workflow

The release workflow is triggered when a version tag is pushed (e.g., v1.2.3), and uses GoReleaser to build and publish releases.

Sources: [.github/workflows/goreleaser.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/goreleaser.yml)

### Nightly Builds

The nightly workflow runs on pushes to the main branch, creating development/nightly releases.

Sources: [.github/workflows/nightly.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/nightly.yml)

### Coverage Workflow

A dedicated workflow for measuring and reporting test coverage.

Sources: [.github/workflows/coverage.yml](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/.github/workflows/coverage.yml)

Docker Build Process
--------------------

Soft Serve provides official Docker images for easy deployment. The Docker build process is defined in the `Dockerfile` and integrated with GoReleaser for automated builds.

Sources: [Dockerfile](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/Dockerfile)
 [docker.md](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/docker.md?plain=1)

The Docker image:

*   Uses Alpine Linux as the base image
*   Installs Git and SSH
*   Configures a volume at `/soft-serve` for data persistence
*   Exposes ports for SSH (23231), HTTP (23232), Stats (23233), and Git (9418)
*   Sets environment variables including `SOFT_SERVE_DATA_PATH`
*   Uses the compiled `soft` binary with the `serve` command as the entrypoint

For developers working on Soft Serve, the standard Go build tooling is used for local development.

Release Process
---------------

The release process is automated through GitHub Actions and GoReleaser:

1.  A developer creates and pushes a version tag (e.g., `v1.2.3`)
2.  The goreleaser workflow is triggered
3.  GoReleaser:
    *   Builds binaries for multiple platforms
    *   Creates Docker images
    *   Generates packages for distribution channels
    *   Signs and notarizes macOS binaries
    *   Creates GitHub release with all artifacts
    *   Publishes to package repositories (Homebrew, etc.)

This provides a consistent and reliable way to release new versions of Soft Serve across all supported platforms.

Summary
-------

Soft Serve's build system provides a comprehensive approach to building, testing, and releasing the software. By leveraging GoReleaser, GitHub Actions, and Docker, it ensures consistent builds across different platforms and streamlines the release process. The system is designed to be maintainable, with shared configurations from the Charmbracelet organization providing consistency across projects.

For information about testing the project, see [Testing Framework](https://deepwiki.com/charmbracelet/soft-serve/5.3-testing-framework)
