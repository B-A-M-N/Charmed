Testing Framework
=================

This document describes the testing framework used in Soft Serve, focusing on its architecture, components, and usage patterns. The framework is designed to validate the behavior of Soft Serve through integration tests that simulate real-world user interactions with the system.

Overview
--------

The Soft Serve testing framework uses the `testscript` package from `github.com/rogpeppe/go-internal/testscript`, which provides a powerful way to write test scripts in a declarative format. These scripts define sequences of commands and their expected outputs, enabling comprehensive testing of the entire system.

Sources: [testscript/script\_test.go32-68](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/script_test.go#L32-L68)

Test Environment Setup
----------------------

Each test run creates a complete, isolated environment including:

1.  A temporary Soft Serve binary compiled with test coverage instrumentation
2.  Random network ports for each service (SSH, Git, HTTP, and Stats)
3.  Unique SSH key pairs for different test users
4.  A temporary data directory for repositories and configuration
5.  Custom environment variables for test scenarios

The `TestMain` function coordinates this setup:

Test Script Format
------------------

Test scripts use a format defined by the `testscript` package. Each script contains:

*   Commands to execute (with no prefix)
*   Expected stdout content (with `stdout` prefix)
*   Expected stderr content (with `stderr` prefix)
*   Files to create (in blocks at the end of the script)
*   Conditional execution based on OS (e.g., `[windows]`)

Example of a basic test script structure:

    # Comment describing the test
    exec some-command arg1 arg2
    stdout 'expected output'
    stderr 'expected error'
    
    # Negative assertion (command should fail)
    ! some-command --invalid-flag
    stderr 'error message'
    
    # Create files needed for the test
    -- filename.txt --
    Content of the file

Testing Specific Functionality
------------------------------

### Repository Management Testing

The framework tests repository management operations including:

*   Creating repositories with various options
*   Setting repository properties (hidden, private)
*   Creating, listing, and deleting branches and tags
*   Repository renaming and deletion
*   Repository imports and mirroring

Tests ensure that file system operations work correctly and repository metadata is properly stored.

Sources: [testscript/testdata/repo-create.txtar](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/testdata/repo-create.txtar)
 [testscript/testdata/repo-import.txtar](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/testdata/repo-import.txtar)

### Authentication and Authorization Testing

Tests for authentication and authorization validate:

*   SSH key-based authentication
*   Access token authentication for HTTP operations
*   Repository access permissions
*   User role-based permissions (admin vs. regular users)

Sources: [testscript/testdata/http.txtar14-29](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/testdata/http.txtar#L14-L29)
 [testscript/testdata/ssh-lfs.txtar14-46](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/testdata/ssh-lfs.txtar#L14-L46)

### HTTP API Testing

HTTP API tests verify:

*   Git HTTP protocol (smart and dumb)
*   LFS API endpoints
*   Go package discovery protocol
*   Authentication for private repositories
*   Error responses for invalid requests

Sources: [testscript/testdata/http.txtar46-140](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/testdata/http.txtar#L46-L140)

### Git LFS Testing

Git LFS tests validate:

*   SSH Git LFS transfer protocol
*   HTTP LFS API operations
*   Authorization for LFS operations
*   LFS file pushing and pulling

Sources: [testscript/testdata/ssh-lfs.txtar28-67](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/testdata/ssh-lfs.txtar#L28-L67)

Database Testing
----------------

The framework supports testing with different database backends:

*   SQLite (default)
*   PostgreSQL (when configured)

For PostgreSQL testing, the framework:

*   Creates a temporary test database with a unique name
*   Configures the server to use this database
*   Cleans up the database after tests complete

Sources: [testscript/script\_test.go544-618](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/script_test.go#L544-L618)

Running the Tests
-----------------

To run the Soft Serve tests:

1.  The framework builds the Soft Serve binary with test instrumentation
2.  Test scripts from the `testdata` directory are executed
3.  Commands and their outputs are verified against expectations

Additional options include:

*   `update` flag to update expected outputs in test scripts
*   Environment variables to control test behavior

Sources: [testscript/script\_test.go70-190](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/script_test.go#L70-L190)

Test Implementation Details
---------------------------

### SSH Command Simulation

The framework simulates SSH commands by:

1.  Establishing an SSH connection to the server
2.  Creating a new session
3.  Running the command in the session
4.  Capturing and verifying stdout/stderr

Sources: [testscript/script\_test.go192-215](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/script_test.go#L192-L215)

### Terminal UI Testing

The framework tests the terminal UI by:

1.  Opening an SSH session with a PTY
2.  Sending simulated keystrokes
3.  Capturing the terminal output
4.  Verifying the rendered UI

This allows testing of interactive terminal features without requiring actual user input.

Sources: [testscript/script\_test.go217-275](https://github.com/charmbracelet/soft-serve/blob/767bdd5a/testscript/script_test.go#L217-L275)

Conclusion
----------

The Soft Serve testing framework provides comprehensive integration testing through scripted tests. It ensures that all components work together correctly and that user interactions behave as expected. By simulating real-world usage patterns across different protocols and interfaces, the framework helps maintain system reliability and detect regressions early.
