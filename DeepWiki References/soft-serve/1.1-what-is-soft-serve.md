What is Soft Serve?
===================

Soft Serve is a self-hostable Git server designed specifically for the command line. It provides a modern, user-friendly way to host and manage Git repositories with a focus on terminal-based workflows and simplicity.

This page introduces the core concepts of Soft Serve, its purpose, and primary features. For detailed installation instructions, see [Installation and Configuration](https://deepwiki.com/charmbracelet/soft-serve/2-installation-and-configuration)
, and for in-depth technical details about the architecture, see [System Architecture](https://deepwiki.com/charmbracelet/soft-serve/1.3-system-architecture)
.

Overview
--------

Soft Serve is a lightweight Git server that allows you to host your own repositories with an SSH-accessible Terminal User Interface (TUI). It supports multiple protocols for repository operations, including SSH, HTTP, and the Git protocol, making it versatile for different workflows and environments.

Core Purpose and Features
-------------------------

Soft Serve was created to provide a lightweight, self-hostable Git server with a focus on terminal-based workflows. It offers:

1.  **Terminal UI over SSH**: Browse repositories, files, and commits through an SSH-accessible interface
2.  **Multiple Protocol Support**: Clone repositories via SSH, HTTP, or Git protocol
3.  **Git LFS Support**: Work with large files using both HTTP and SSH backends
4.  **Simple Repository Management**: Create, import, and manage repositories directly via SSH
5.  **Flexible Access Control**: Control repository access using SSH key authentication
6.  **User Management**: Create users and manage their permissions
7.  **Webhooks and Hooks**: Support for Git hooks and webhooks for automation

Use Cases
---------

Soft Serve is designed for individuals and teams who prefer command-line-centric workflows and want to:

1.  **Host Private Repositories**: Set up a personal or team Git server
2.  **Provide a Clean UI for Repository Browsing**: The TUI makes navigation intuitive
3.  **Create an Alternative to Web-Based Git Hosting**: For those who prefer terminal-based workflows
4.  **Implement Simple Access Control**: Manage repository access with SSH keys
5.  **Run a Lightweight Git Server**: Minimal resource requirements

Getting Started
---------------

To use Soft Serve, you can:

1.  **Connect to a Soft Serve Server**: `ssh git.example.com`
2.  **Clone Repositories**: `git clone ssh://git.example.com/repo`
3.  **Browse Repository Contents**: Use the TUI via SSH
4.  **Manage Repositories**: Create, import, and configure repositories via SSH commands

Server Architecture
-------------------

At its core, Soft Serve consists of several server components that work together:

| Component | Default Port | Purpose |
| --- | --- | --- |
| SSH Server | 23231 | Provides authentication, TUI, and Git operations over SSH |
| Git Daemon | 9418 | Enables Git protocol operations (read-only by default) |
| HTTP Server | 23232 | Supports HTTP Git operations and Git LFS |
| Stats Server | 23233 | Provides server statistics (optional) |

The server components are configured through a central configuration system that can read settings from a config file, environment variables, or defaults.

Repositories and Data Storage
-----------------------------

Soft Serve stores:

1.  **Git Repositories**: The actual Git data for hosted repositories
2.  **Configuration**: Server settings in a `config.yaml` file
3.  **Database**: User information, repository metadata, etc. (SQLite by default, PostgreSQL supported)
4.  **SSH Keys**: Host and client keys for authentication
5.  **LFS Objects**: Large files when Git LFS is enabled

By default, all data is stored in a directory named `data` in the current working directory, but this can be configured.

Summary
-------

Soft Serve provides a complete Git server solution with a focus on command-line workflows and simplicity. It combines the power of Git with a user-friendly terminal interface, making it an ideal solution for developers who prefer working in the terminal.

To learn more about specific features and how to use them, refer to the other sections of this documentation.
