This document provides a comprehensive overview of the Soft Serve architecture and explains how its components interact to deliver a self-hostable Git server for the command line. For specific installation instructions and configuration options, see [Installation and Configuration](https://deepwiki.com/charmbracelet/soft-serve/2-installation-and-configuration)
.

Purpose and Scope
-----------------

Soft Serve is designed as a modular system with multiple server components that handle different protocols (SSH, Git, HTTP) and aspects of Git repository management. The architecture focuses on simplicity, easy self-hosting, and providing a rich terminal-based user experience. This document explains:

*   The high-level architecture and component organization
*   Core server components and their responsibilities
*   Configuration system
*   Storage model
*   User interaction flows
*   Authentication and authorization mechanisms
*   Terminal UI architecture
*   Repository management

High-Level Architecture
-----------------------

Soft Serve is structured in a layered architecture with four main components:

### High-Level Architecture Diagram

The architecture consists of:

1.  **User Interfaces Layer**: Entry points for users to interact with Soft Serve
2.  **Core Components Layer**: Server components handling different protocols
3.  **Business Logic Layer**: Core application logic managing repositories and access
4.  **Storage Layer**: Persistence for user data, repositories, and LFS objects

Core Server Components
----------------------

Soft Serve consists of four primary server components, each handling different aspects of Git repository serving:

### Server Components Architecture

1.  **SSH Server**:
    
    *   Listens on port 23231 (default)
    *   Handles SSH connections and authentication
    *   Provides Terminal UI for repository browsing
    *   Processes administrative commands
    *   Enables Git operations over SSH
    *   Optionally supports Git LFS over SSH
2.  **Git Daemon**:
    
    *   Listens on port 9418 (default)
    *   Provides Git protocol support
    *   Enables efficient read-only access to repositories
    *   Configurable connection limits and timeouts
3.  **HTTP Server**:
    
    *   Listens on port 23232 (default)
    *   Provides HTTP Git protocol support
    *   Serves Git LFS (Large File Storage) objects
    *   Handles authentication via user tokens
    *   Optional TLS support for secure connections
4.  **Stats Server**:
    
    *   Listens on port 23233 (default)
    *   Exposes Prometheus metrics for monitoring
    *   Provides server health and status information

Each server component can be independently enabled, disabled, or configured according to deployment needs.

Configuration System
--------------------

The configuration system is a core part of Soft Serve's architecture. It manages settings through multiple sources with a clear precedence order.

### Configuration Flow

Configuration sources in order of precedence (highest to lowest):

1.  Environment variables (prefixed with `SOFT_SERVE_`)
2.  Configuration file (`config.yaml`)
3.  Default values from `DefaultConfig()` function

The configuration structure is a hierarchical model with component-specific sections:

    Config (main struct)
    ├── Name (server name)
    ├── SSH (SSH server configuration)
    ├── Git (Git daemon configuration)
    ├── HTTP (HTTP server configuration)
    ├── Stats (Stats server configuration)
    ├── Log (Logger configuration)
    ├── DB (Database configuration)
    ├── LFS (Git LFS configuration)
    ├── Jobs (Cron jobs configuration)
    └── InitialAdminKeys (Admin SSH keys)
    

This modular structure allows for targeted configuration of specific components while maintaining a coherent overall system configuration.

Data Storage Model
------------------

Soft Serve uses a combination of database storage and filesystem storage:

### Database

The system supports two database backends:

| Database | Use Case | Configuration |
| --- | --- | --- |
| SQLite (default) | Single-instance deployments | File path in data directory |
| PostgreSQL | Larger deployments, high availability | Connection string |

The database stores:

*   User information (usernames, SSH keys)
*   Repository metadata (name, description, visibility)
*   Access control settings (collaborators, permissions)
*   Webhook configurations
*   User access tokens

### Repository Storage

Git repositories are stored on the filesystem within the data directory, organized by repository name. Each repository maintains its standard Git structure, allowing direct filesystem access if needed.

### LFS Storage

When Git LFS is enabled, large files are stored separately from Git repositories in a dedicated LFS objects directory. These objects are served through the HTTP server component.

User Interaction Flows
----------------------

Soft Serve supports three primary user interaction flows:

### User Interaction Flows Diagram

1.  **SSH Terminal UI Flow**:
    
    *   User connects via SSH with public key authentication
    *   TUI presents repository browsing interface
    *   User navigates repositories, files, commits, etc.
    *   Backend retrieves data from Git repositories as needed
2.  **Git Protocol Flow**:
    
    *   User clones/fetches via Git protocol (`git://`)
    *   Git daemon handles the request, checks permissions
    *   Repository data is transferred directly via Git protocol
    *   Optimized for read operations (typically read-only)
3.  **HTTP/LFS Flow**:
    
    *   User performs Git operations over HTTP(S)
    *   Authentication via user access tokens
    *   LFS objects are served through dedicated API endpoints
    *   Supports both authentication methods and varied access controls

These flows can be used independently or in combination, depending on the user's needs and the server configuration.

Authentication and Authorization
--------------------------------

Soft Serve implements a comprehensive access control system:

### Authentication

Two primary authentication methods:

1.  **SSH Public Key Authentication**:
    
    *   Primary authentication method
    *   Each user is associated with specific SSH public keys
    *   No duplicate keys allowed across users
2.  **User Access Tokens**:
    
    *   Used for HTTP authentication
    *   Generated via SSH command line interface
    *   Can have expiration dates
    *   Format: `ss_<token>`

### Authorization

Four access levels control what operations users can perform:

| Access Level | Description | Capabilities |
| --- | --- | --- |
| `no-access` | Deny all access | None |
| `read-only` | Read public repositories | Clone public repos, browse in TUI |
| `read-write` | Read and write to repositories | Clone, push, create repositories |
| `admin-access` | Full control of server | Manage users, all repository operations |

Server-wide access settings:

*   `anon-access`: Default access level for anonymous users (default: `read-only`)
*   `allow-keyless`: Whether to allow connections without SSH keys (default: `true`)

Repository-specific access is controlled through:

*   Repository visibility (public/private)
*   Collaborator assignments with specific access levels
*   User's global access level

Repository Management
---------------------

The repository management system handles all Git repository operations:

### Repository Management Architecture

Repository operations include:

1.  **Creation**:
    
    *   Via SSH command: `ssh <host> repo create <name>`
    *   By pushing to non-existent repository: `git push origin main`
2.  **Import**:
    
    *   From external Git repositories: `ssh <host> repo import <name> <url>`
    *   Optional mirror mode for automatic updates
3.  **Management**:
    
    *   Rename: `ssh <host> repo rename <old> <new>`
    *   Delete: `ssh <host> repo delete <repo>`
    *   Set metadata (description, visibility, etc.)
4.  **Access Control**:
    
    *   Add collaborators with specific access levels
    *   Set repository visibility (public/private)
    *   Hide repositories from listings
5.  **Webhooks**:
    
    *   Create and manage webhooks for repository events
    *   Trigger on push, branch/tag creation, collaborator changes

The repository management system interacts with both the database (for metadata) and the filesystem (for Git data), providing a complete repository management solution.

Terminal UI Architecture
------------------------

The Terminal UI (TUI) is a text-based interface for browsing repositories, primarily accessed over SSH:

### Terminal UI Component Architecture

The TUI architecture is component-based:

1.  **UI Model**: Central state manager for the TUI
    
2.  **Pages**:
    
    *   Repository Selection: Lists available repositories
    *   Repository Page: Detailed view of selected repository
3.  **Repository UI Components**:
    
    *   Repository Component: Main container for repository views
    *   Tabs Navigation: Switches between different views
    *   Status Bar: Shows information and command hints
4.  **Tab Components**:
    
    *   Readme Tab: Displays formatted repository README
    *   Files Tab: File browser for repository contents
    *   Log Tab: Commit history viewer with diffs
    *   References Tab: Branch and tag explorer
    *   Stash Tab: Displays stashed changes
5.  **Content Renderers**:
    
    *   Code Component: Renders code with syntax highlighting
    *   File Item Delegate: Renders file entries in lists
    *   Log Item Delegate: Renders commit entries
    *   Stash Item Delegate: Renders stash entries

The TUI is built using Charm libraries from the Go module dependencies:

*   `bubbletea`: TUI framework
*   `bubbles`: UI components
*   `lipgloss`: Styling and layout
*   `glamour`: Markdown rendering

Hooks and Webhooks
------------------

Soft Serve implements two mechanisms for responding to repository events:

### Git Hooks

Standard Git server-side hooks supported:

*   `pre-receive`: Runs before references are updated
*   `update`: Runs during reference update
*   `post-update`: Runs after references are updated
*   `post-receive`: Runs after the entire process is completed

Hooks can be:

*   **Repository-specific**: Located in each repository's `hooks` directory
*   **Global**: Located in the data directory's `hooks` directory, applied to all repositories

### Webhooks

Webhooks send HTTP POST requests to configured URLs when events occur:

*   **Push events**: When changes are pushed to a repository
*   **Branch/tag creation**: When new branches or tags are created
*   **Collaborator changes**: When repository access changes

Webhook configurations are stored in the database and managed through the SSH command-line interface.

Conclusion
----------

Soft Serve's modular architecture enables a flexible, self-hostable Git server experience with multiple access methods while maintaining simplicity. The system balances ease of use with powerful features:

*   Multiple server components handle different protocols and responsibilities
*   Flexible configuration allows for customization to different deployment needs
*   The authentication and authorization system provides robust access control
*   Rich Terminal UI delivers a comprehensive repository browsing experience
*   Repository management enables full control over Git repositories
*   Hooks and webhooks facilitate integration with external systems

This architecture allows Soft Serve to serve as an effective self-hosted Git server for individuals and teams who prefer command-line workflows and terminal-based interfaces.

For installation and configuration details, see [Installation and Configuration](https://deepwiki.com/charmbracelet/soft-serve/2-installation-and-configuration)
