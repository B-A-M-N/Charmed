Testing and Linting

This document describes the testing and linting setup for the bubbles library. It covers the approach to testing components, the structure of tests, linting configuration, and internal utility test patterns. This information is primarily intended for developers who want to contribute to the bubbles library.

Testing Approach
----------------

The bubbles library uses Go's standard testing package for writing and running tests. Tests are organized alongside the code they test, in files with the `_test.go` suffix.

### Code-to-Test Mapping

Sources: [internal/runeutil/runeutil\_test.go8-44](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/runeutil/runeutil_test.go#L8-L44)
 [internal/memoization/memoization\_test.go31-138](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/memoization/memoization_test.go#L31-L138)
 [internal/memoization/memoization\_test.go140-220](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/memoization/memoization_test.go#L140-L220)

### Testing Patterns

Tests in the bubbles library follow these general principles:

1.  **Table-driven testing**: Used extensively for both internal utilities and UI components to test multiple scenarios concisely [internal/runeutil/runeutil\_test.go9-26](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/runeutil/runeutil_test.go#L9-L26)
     [internal/memoization/memoization\_test.go32-121](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/memoization/memoization_test.go#L32-L121)
    
2.  **Fuzzing**: Used for complex internal logic like the LRU cache to ensure robustness against edge cases [internal/memoization/memoization\_test.go140-150](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/memoization/memoization_test.go#L140-L150)
    
3.  **Golden File Testing**: While not explicitly shown in the provided internal snippets, the library uses `github.com/charmbracelet/x/exp/golden` for visual regression testing of component views.
4.  **In-place Modification Validation**: Internal utilities like `runeutil` are tested to ensure they correctly handle buffer allocations and in-place modifications [internal/runeutil/runeutil.go17-20](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/runeutil/runeutil.go#L17-L20)
     [internal/runeutil/runeutil\_test.go37-43](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/runeutil/runeutil_test.go#L37-L43)
    

### Internal Utility Testing

#### Rune Sanitizer Testing

The `runeutil` package is tested by feeding strings containing control characters, newlines, and tabs into the `Sanitizer` and verifying the output matches expected results after replacement or removal [internal/runeutil/runeutil\_test.go9-26](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/runeutil/runeutil_test.go#L9-L26)

#### Memoization Cache Testing

The `MemoCache` is tested using a sequence of `cacheAction` types (`set` or `get`) to verify LRU eviction logic and concurrent safety [internal/memoization/memoization\_test.go11-29](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/memoization/memoization_test.go#L11-L29)

Sources: [internal/memoization/memoization\_test.go18-23](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/memoization/memoization_test.go#L18-L23)

Linting Configuration
---------------------

The bubbles library uses `golangci-lint` for static code analysis. The configuration is defined in `.golangci.yml` and enforced via GitHub Actions.

### Linting System Diagram

Sources: [Taskfile.yaml6-9](https://github.com/charmbracelet/bubbles/blob/42130e89/Taskfile.yaml#L6-L9)
 [.golangci.yml4-28](https://github.com/charmbracelet/bubbles/blob/42130e89/.golangci.yml#L4-L28)
 [.github/workflows/lint.yml7-12](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/lint.yml#L7-L12)

### Enabled Linters

The library enables a strict set of linters to maintain high code quality:

| Linter | Purpose |
| --- | --- |
| `bodyclose` | Checks that HTTP response bodies are closed [.golangci.yml6](https://github.com/charmbracelet/bubbles/blob/42130e89/.golangci.yml#L6-L6) |
| `exhaustive` | Checks exhaustiveness of enum switch statements [.golangci.yml7](https://github.com/charmbracelet/bubbles/blob/42130e89/.golangci.yml#L7-L7) |
| `gosec` | Inspects source code for security problems [.golangci.yml13](https://github.com/charmbracelet/bubbles/blob/42130e89/.golangci.yml#L13-L13) |
| `revive` | Replacement for golint [.golangci.yml21](https://github.com/charmbracelet/bubbles/blob/42130e89/.golangci.yml#L21-L21) |
| `wrapcheck` | Checks that errors from external packages are wrapped [.golangci.yml28](https://github.com/charmbracelet/bubbles/blob/42130e89/.golangci.yml#L28-L28) |
| `gofumpt` | Enforces a stricter format than `gofmt` [.golangci.yml42](https://github.com/charmbracelet/bubbles/blob/42130e89/.golangci.yml#L42-L42) |

### Linting Automation

Linting is automated through two primary workflows:

1.  **lint.yml**: Runs on every push and pull request using `charmbracelet/meta` shared workflows [.github/workflows/lint.yml1-11](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/lint.yml#L1-L11)
    
2.  **lint-sync.yml**: A scheduled workflow (weekly) that ensures linting configurations remain synchronized with organization standards [.github/workflows/lint-sync.yml1-15](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/lint-sync.yml#L1-L15)
    

Task Runner (Taskfile)
----------------------

The project uses `Taskfile.yaml` to provide a consistent interface for common development commands.

| Task | Command | Description |
| --- | --- | --- |
| `lint` | `golangci-lint run` | Runs the full linting suite [.Taskfile.yaml6-9](https://github.com/charmbracelet/bubbles/blob/42130e89/.Taskfile.yaml#L6-L9) |
| `test` | `go test ./... {{.CLI_ARGS}}` | Runs all tests in the repository [.Taskfile.yaml11-14](https://github.com/charmbracelet/bubbles/blob/42130e89/.Taskfile.yaml#L11-L14) |

Conclusion
----------

The bubbles library employs a multi-layered approach to quality, combining standard unit tests, table-driven test patterns for internal utilities, fuzzing for complex logic, and a strict linting suite integrated into the CI/CD pipeline. This ensures that UI components remain visually consistent and internally robust.
