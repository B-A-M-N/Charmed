This document describes the continuous integration, continuous delivery, and release processes for the Bubbles library. These automated processes ensure code quality, facilitate dependency management, and streamline the release of new versions.

For information about testing and linting practices, see [8.1 Testing and Linting](https://github.com/charmbracelet/bubbles/blob/42130e89/8.1%20Testing%20and%20Linting)

Overview
--------

Bubbles uses GitHub Actions for its CI/CD pipeline, including automated testing, code coverage reporting, dependency updates, and releases. The project follows semantic versioning and uses GoReleaser for creating tagged releases.

Sources: [.github/workflows/build.yml1-3](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/build.yml#L1-L3)
 [.github/workflows/coverage.yml1-2](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/coverage.yml#L1-L2)
 [.github/workflows/release.yml1-6](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/release.yml#L1-L6)

Continuous Integration
----------------------

The CI pipeline automatically runs when code is pushed to the repository or when a pull request is created. It ensures the code builds successfully and passes all tests across multiple platforms and Go versions.

### Build Workflow

The build workflow checks that the code compiles and passes tests across various environments:

This workflow:

*   Runs on a matrix of `ubuntu-latest`, `macos-latest`, and `windows-latest` [.github/workflows/build.yml10-11](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/build.yml#L10-L11)
    
*   Tests on `stable` and `oldstable` Go versions [.github/workflows/build.yml9-16](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/build.yml#L9-L16)
    
*   Automatically approves and merges Dependabot pull requests that pass the `test` job [.github/workflows/build.yml30-44](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/build.yml#L30-L44)
    

Sources: [.github/workflows/build.yml1-48](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/build.yml#L1-L48)

### Coverage Workflow

The coverage workflow reports test coverage metrics:

This workflow:

*   Runs on `ubuntu-latest` with `GO111MODULE: "on"` [.github/workflows/coverage.yml9-12](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/coverage.yml#L9-L12)
    
*   Generates a coverage profile named `profile.cov` [.github/workflows/coverage.yml26](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/coverage.yml#L26-L26)
    
*   Reports results to Coveralls using the `GITHUB_TOKEN` [.github/workflows/coverage.yml24-28](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/coverage.yml#L24-L28)
    

Sources: [.github/workflows/coverage.yml1-29](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/coverage.yml#L1-L29)

Dependency Management
---------------------

Bubbles uses Dependabot to automatically keep dependencies up to date, with configuration synchronized across repositories.

### Dependabot Configuration

Dependabot monitors three ecosystems: `gomod`, `github-actions`, and `docker` [.github/dependabot.yml4-38](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/dependabot.yml#L4-L38)

| Feature | Configuration |
| --- | --- |
| **Schedule** | Weekly (Mondays at 05:00 America/New\_York) [.github/dependabot.yml7-10](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/dependabot.yml#L7-L10) |
| **Commit Prefix** | `chore` [.github/dependabot.yml14](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/dependabot.yml#L14-L14) |
| **Labels** | `dependencies` [.github/dependabot.yml11-12](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/dependabot.yml#L11-L12) |
| **Grouping** | All updates grouped into a single PR [.github/dependabot.yml16-19](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/dependabot.yml#L16-L19) |

Sources: [.github/dependabot.yml1-54](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/dependabot.yml#L1-L54)

### Dependabot Synchronization

The `dependabot-sync.yml` workflow ensures the repository's configuration matches the Charm organization standards.

*   **Trigger**: Every Sunday at midnight or manual `workflow_dispatch` [.github/workflows/dependabot-sync.yml3-5](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/dependabot-sync.yml#L3-L5)
    
*   **Implementation**: Uses the shared workflow `charmbracelet/meta/.github/workflows/dependabot-sync.yml@main` [.github/workflows/dependabot-sync.yml13](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/dependabot-sync.yml#L13-L13)
    

Sources: [.github/workflows/dependabot-sync.yml1-18](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/dependabot-sync.yml#L1-L18)

Release Process
---------------

### Release Workflow

The release process is triggered automatically when a version tag (matching `v*.*.*`) is pushed [.github/workflows/release.yml4-6](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/release.yml#L4-L6)

The workflow utilizes the `charmbracelet/meta` shared GoReleaser workflow and passes numerous secrets for distribution, including DockerHub, Twitter, Mastodon, and Discord credentials [.github/workflows/release.yml14-28](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/release.yml#L14-L28)

Sources: [.github/workflows/release.yml1-30](https://github.com/charmbracelet/bubbles/blob/42130e89/.github/workflows/release.yml#L1-L30)

### GoReleaser Configuration

The `.goreleaser.yml` file follows the GoReleaser v2 schema and includes standard library configurations from the Charm organization [.goreleaser.yml1-6](https://github.com/charmbracelet/bubbles/blob/42130e89/.goreleaser.yml#L1-L6)

Sources: [.goreleaser.yml1-6](https://github.com/charmbracelet/bubbles/blob/42130e89/.goreleaser.yml#L1-L6)
