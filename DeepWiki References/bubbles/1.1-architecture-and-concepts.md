This document explains the core architectural principles and concepts that underpin the Bubbles UI component library, and how these components integrate with the Bubble Tea framework. Bubbles provides a set of functional, stateful components that adhere to the Model-View-Update (MVU) pattern.

Core Architecture: The MVU Pattern
----------------------------------

Bubbles components are built to be used within the [Bubble Tea](https://github.com/charmbracelet/bubbles/blob/42130e89/Bubble%20Tea)
 framework. Each component is a "miniature" Bubble Tea program that implements the `tea.Model` interface.

Each Bubbles component follows this lifecycle:

1.  **Model**: A `struct` holding the component's state (e.g., `cursor.Model` [cursor/cursor.go64-97](https://github.com/charmbracelet/bubbles/blob/42130e89/cursor/cursor.go#L64-L97)
    ).
2.  **Update**: A function that takes a `tea.Msg` and returns a new model and a `tea.Cmd` [cursor/cursor.go114-158](https://github.com/charmbracelet/bubbles/blob/42130e89/cursor/cursor.go#L114-L158)
    
3.  **View**: A function that returns a `string` representation of the component [cursor/cursor.go236-241](https://github.com/charmbracelet/bubbles/blob/42130e89/cursor/cursor.go#L236-L241)

Key Binding System (`key` package)
----------------------------------

The `key` package is a non-visual utility used across almost all Bubbles components to manage keyboard shortcuts and generate help text.

### Implementation Details

*   **`Binding`**: Wraps a set of keys (e.g., "q", "esc") with descriptive help text [key/key.go43-47](https://github.com/charmbracelet/bubbles/blob/42130e89/key/key.go#L43-L47)
    
*   **`Matches`**: A generic function used in `Update` loops to check if a `tea.KeyPressMsg` matches a specific `Binding` [key/key.go130-140](https://github.com/charmbracelet/bubbles/blob/42130e89/key/key.go#L130-L140)
    
*   **Decoupling**: By using `key.Binding`, components allow developers to remap keys without changing the component's internal logic [key/key.go1-33](https://github.com/charmbracelet/bubbles/blob/42130e89/key/key.go#L1-L33)

Shared Primitives: The Cursor
-----------------------------

The `cursor` package provides a virtual cursor used by input components like `textinput` and `textarea` [cursor/cursor.go1-3](https://github.com/charmbracelet/bubbles/blob/42130e89/cursor/cursor.go#L1-L3)
 It manages blinking states and focus independently of the terminal's native cursor.

### Multi-Instance Safety (ID/Tag Pattern)

To prevent "message leakage"—where a timer or blink message intended for one component is accidentally processed by another—Bubbles uses an internal ID system.

*   **`nextID()`**: Generates a unique integer for every new model instance [cursor/cursor.go20-22](https://github.com/charmbracelet/bubbles/blob/42130e89/cursor/cursor.go#L20-L22)
    
*   **`BlinkMsg`**: Carries an `id` and a `tag`. The `Update` function ignores messages if the `id` or `tag` doesn't match the current state [cursor/cursor.go29-32](https://github.com/charmbracelet/bubbles/blob/42130e89/cursor/cursor.go#L29-L32)
     [cursor/cursor.go143-145](https://github.com/charmbracelet/bubbles/blob/42130e89/cursor/cursor.go#L143-L145)

### Memoization (`memoization`)

Used primarily in high-performance components like `textarea`, this package implements an LRU (Least Recently Used) cache [internal/memoization/memoization.go28-34](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/memoization/memoization.go#L28-L34)

*   **`Hasher`**: An interface for objects that can be turned into a cache key [internal/memoization/memoization.go14-16](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/memoization/memoization.go#L14-L16)
    
*   **`MemoCache`**: Uses a `map` for O(1) lookups and a `container/list` to track eviction order [internal/memoization/memoization.go38-45](https://github.com/charmbracelet/bubbles/blob/42130e89/internal/memoization/memoization.go#L38-L45)

Component Composition Patterns
------------------------------

Bubbles components are designed to be composed. There are two primary patterns:

1.  **Direct Embedding**: A parent model includes a Bubbles model as a field.
    *   _Example_: `list` includes a `paginator.Model` and optionally a `spinner.Model` [README.md102-114](https://github.com/charmbracelet/bubbles/blob/42130e89/README.md?plain=1#L102-L114)
        
2.  **Internal Dependency**: A component uses another component as its core engine.
    *   _Example_: `textarea` uses `viewport` for scrolling logic and `cursor` for text insertion points [README.md42-52](https://github.com/charmbracelet/bubbles/blob/42130e89/README.md?plain=1#L42-L52)
        

| Feature | Entity | Implementation |
| --- | --- | --- |
| **Key Handling** | `key.Binding` | `key.Matches(msg, binding)` [key/key.go130](https://github.com/charmbracelet/bubbles/blob/42130e89/key/key.go#L130-L130) |
| **Animation** | `tea.Tick` | Managed via `Blink()` or `Tick()` commands |
| **State Tracking** | `Model` struct | Component-specific state (e.g., `viewport.Model`) |
| **Styling** | `lipgloss.Style` | Integrated directly into `View()` methods |
