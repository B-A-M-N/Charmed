SSH Server
==========

This page documents the SSH server functionality in VHS, which allows remote execution of tape files via SSH. Users can pipe tape files to a VHS SSH server and receive the generated GIF or video output without installing VHS locally.

For information about publishing your recordings online, see [Publishing](https://deepwiki.com/charmbracelet/vhs/4.3-publishing)
. For details about recording mode, see [Recording Mode](https://deepwiki.com/charmbracelet/vhs/4.1-recording-mode)
.

Overview
--------

The SSH server feature enables users to run VHS tape files remotely by sending them over an SSH connection. This is useful for:

*   Creating terminal recordings on remote servers
*   Automating GIF generation in CI/CD pipelines
*   Sharing the VHS service with others without requiring local installation

The server accepts tape files via stdin and returns the resulting GIF or video output directly to the client.

Sources: [serve.go38-165](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L38-L165)
 [main.go275](https://github.com/charmbracelet/vhs/blob/f28239f3/main.go#L275-L275)

Server Configuration
--------------------

The VHS SSH server can be configured using environment variables with the `VHS_` prefix:

| Environment Variable | Default | Description |
| --- | --- | --- |
| `VHS_PORT` | 1976 | The port to listen on |
| `VHS_HOST` | localhost | The host address to bind to |
| `VHS_GID` | 0   | Group ID to drop privileges to (Unix only) |
| `VHS_UID` | 0   | User ID to drop privileges to (Unix only) |
| `VHS_KEY_PATH` | `.ssh/vhs_ed25519` | Path to SSH host key |
| `VHS_AUTHORIZED_KEYS_PATH` | (empty) | Path to authorized keys file |

Sources: [serve.go28-35](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L28-L35)

Starting the Server
-------------------

The SSH server is started using the `serve` command:

    vhs serve
    

This command:

1.  Parses the environment configuration
2.  Creates an SSH server instance using the Charm SSH framework
3.  Sets up middleware for handling tape execution
4.  Listens for incoming connections on the configured address

Sources: [serve.go38-165](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L38-L165)
 [main.go275](https://github.com/charmbracelet/vhs/blob/f28239f3/main.go#L275-L275)

Authentication
--------------

The VHS SSH server uses key-based authentication:

1.  Server identity: The server uses an ED25519 key (default path: `.ssh/vhs_ed25519`)
2.  Client authentication: Optionally uses an authorized keys file to authenticate clients

If no authorized keys path is specified, the server will accept connections from any client (though they still need to authenticate with the SSH protocol).

Sources: [serve.go42-61](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L42-L61)

Handling Client Connections
---------------------------

When a client connects to the SSH server, the connection is processed as follows:

The server explicitly rejects PTY (interactive terminal) requests, as it's designed to operate in a non-interactive mode where the tape file is provided upfront.

Sources: [serve.go63-124](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L63-L124)

Execution Process
-----------------

1.  The server creates a temporary file for the output (with a random identifier)
2.  It reads the tape content from stdin
3.  It executes the tape using the `Evaluate` function, capturing any errors
4.  It reads the resulting output file and sends it back to the client
5.  The temporary file is cleaned up after the session ends

Sources: [serve.go82-123](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L82-L123)

Security Considerations
-----------------------

The VHS SSH server includes several security features:

1.  **Privilege Dropping**: On Unix-like systems, the server can drop privileges to a specified UID/GID after binding to its port
2.  **Key-Based Authentication**: Server uses SSH keys rather than passwords
3.  **Temporary Files**: Output files are created with random names in the system temporary directory
4.  **No PTY**: The server rejects PTY requests, limiting the attack surface

The platform-specific implementations handle privilege dropping differently:

*   Unix systems use `syscall.Setuid` and `syscall.Setgid`
*   Windows skips this step as it doesn't support the same permission model

Sources: [serve.go140-146](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L140-L146)
 [serve\_unix.go1-19](https://github.com/charmbracelet/vhs/blob/f28239f3/serve_unix.go#L1-L19)
 [serve\_windows.go1-10](https://github.com/charmbracelet/vhs/blob/f28239f3/serve_windows.go#L1-L10)

Client Usage
------------

To use the VHS SSH server, clients can pipe a tape file to the server via SSH:

    ssh vhs.charm.sh < demo.tape > output.gif
    

The server processes the tape and returns the resulting GIF or video as binary data, which can be redirected to a file.

For authentication, clients should use their SSH key as normal:

    ssh -i ~/.ssh/id_ed25519 vhs.charm.sh < demo.tape > output.gif
    

Sources: [serve.go83-84](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L83-L84)

Integration with the Charm Ecosystem
------------------------------------

The VHS SSH server leverages several components from the Charm ecosystem:

*   **Wish**: A library for building SSH apps and servers
*   **Keygen**: For SSH key generation and management

The server implementation uses the Charm `wish` framework to handle SSH connections and middleware, making it consistent with other Charm SSH applications.

----------------------

Charm operates a public VHS SSH service at `vhs.charm.sh` that allows anyone to generate GIFs without installing VHS locally. This service accepts tape files via stdin and returns the generated GIF.

Example usage:

Sources: [serve.go83-84](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L83-L84)

Server Implementation Details
-----------------------------

The server implementation is centered around the `serveCmd` variable in `serve.go`, which defines the command and its behavior. The core functionality is implemented as SSH middleware that processes each incoming connection.

The middleware:

1.  Checks if a PTY was requested (and rejects if so)
2.  Reads the tape content from stdin
3.  Creates a temporary file path for the output
4.  Evaluates the tape and captures the output
5.  Returns either the generated file or error messages

The server gracefully handles shutdown when it receives a termination signal, with a timeout for closing connections.

Sources: [serve.go38-165](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L38-L165)

Error Handling
--------------

Errors that occur during tape execution are formatted and returned to the client. The server uses the same error formatting as the local VHS command to ensure consistency in the user experience.

If errors occur, the server:

1.  Formats the errors using `printErrors`
2.  Sends them to stderr
3.  Sets a non-zero exit code to indicate failure

Sources: [serve.go114-117](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L114-L117)

Related Components
------------------

The SSH server functionality relies on several other VHS components:

1.  **Tape Evaluation**: Uses the core `Evaluate` function to process tapes
2.  **Error Formatting**: Uses the same error formatting as the local command
3.  **Output Generation**: Uses the same output generation pipeline

This ensures consistency between local and remote execution of tape files.

Sources: [serve.go96-112](https://github.com/charmbracelet/vhs/blob/f28239f3/serve.go#L96-L112)
