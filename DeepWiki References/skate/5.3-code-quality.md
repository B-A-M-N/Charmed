Code Quality
============

Relevant source files

*   [.github/workflows/lint.yml](https://github.com/charmbracelet/skate/blob/09860417/.github/workflows/lint.yml)
    
*   [.golangci.yml](https://github.com/charmbracelet/skate/blob/09860417/.golangci.yml)
    

This document outlines the code quality tools, practices, and standards used in the Skate project. It covers the automated linting processes, the specific linters employed, and how they contribute to maintaining high code quality throughout the codebase. For information about building from source, see [Build Process](https://deepwiki.com/charmbracelet/skate/5.1-build-process)
, and for details on releasing Skate, see [Release Process](https://deepwiki.com/charmbracelet/skate/5.2-release-process)
.

Linting Infrastructure
----------------------

The Skate project employs automated linting that runs on every code push and pull request. The primary tool used is golangci-lint, a Go linter aggregator that runs multiple specialized linters in parallel.

Sources: [.github/workflows/lint.yml1-19](https://github.com/charmbracelet/skate/blob/09860417/.github/workflows/lint.yml#L1-L19)

When a developer pushes code or creates a pull request, GitHub Actions automatically triggers the linting workflow defined in `lint.yml`. This workflow runs golangci-lint with the project's custom configuration, checking the code against all enabled linters.

Lint Configuration
------------------

The Skate project uses a custom configuration for golangci-lint defined in `.golangci.yml`. This configuration specifies which linters are enabled and how they should behave.

Sources: [.golangci.yml1-32](https://github.com/charmbracelet/skate/blob/09860417/.golangci.yml#L1-L32)

Enabled Linters
---------------

The following table details the linters enabled in Skate and their purpose:

| Linter | Purpose |
| --- | --- |
| bodyclose | Checks if HTTP response bodies are properly closed |
| dupl | Finds code clones to reduce duplication |
| goconst | Identifies repeated strings that could be constants |
| godot | Ensures comments end with a period for consistency |
| goimports | Formats code and organizes imports |
| goprintffuncname | Validates printf-like function naming conventions |
| gosec | Scans code for security vulnerabilities |
| misspell | Catches commonly misspelled English words |
| prealloc | Suggests preallocation for slices to improve performance |
| revive | A comprehensive linter with various checks |
| rowserrcheck | Ensures SQL rows errors are properly checked |
| sqlclosecheck | Verifies SQL resources are properly closed |
| unconvert | Removes unnecessary type conversions |
| unparam | Identifies unused function parameters |
| whitespace | Enforces proper whitespace usage |

Sources: [.golangci.yml15-31](https://github.com/charmbracelet/skate/blob/09860417/.golangci.yml#L15-L31)

CI Integration
--------------

The linting process is integrated into the CI pipeline using GitHub Actions. The workflow is defined in `.github/workflows/lint.yml` and runs on every push and pull request.

The CI configuration has two notable features:

*   Issues don't cause the workflow to fail (`--issues-exit-code=0`), allowing the CI process to continue even if linting issues are found
*   For pull requests, only new issues are reported (`only-new-issues: true`), focusing developer attention on issues introduced by their changes

Sources: [.github/workflows/lint.yml11-18](https://github.com/charmbracelet/skate/blob/09860417/.github/workflows/lint.yml#L11-L18)

Running Linting Locally
-----------------------

Developers can run the same linting process locally to catch issues before pushing code:

1.  Install golangci-lint:
    
        go install github.com/golangci/golangci-lint/cmd/golangci-lint@latest
        
    
2.  Run linting from the project root:
    
        golangci-lint run
        
    

This local execution will use the same configuration defined in `.golangci.yml`, ensuring consistency between local development and CI.

Sources: [.golangci.yml1-32](https://github.com/charmbracelet/skate/blob/09860417/.golangci.yml#L1-L32)

Impact on Skate Components
--------------------------

The linting setup impacts various components of the Skate architecture:

| Skate Component | Linters with Highest Impact | Benefits |
| --- | --- | --- |
| CLI Layer (Cobra Commands) | goimports, whitespace, revive | Consistent command definitions, clear API |
| Application Logic | dupl, goconst, unparam, gosec | Efficient code, no redundancy, secure operations |
| Storage Layer (BadgerDB) | bodyclose, rowserrcheck, sqlclosecheck | Proper resource management, no leaks |
| Terminal UI Layer | godot, misspell | Clear and consistent user messaging |

By maintaining high code quality standards across these components, the Skate project ensures a reliable, efficient, and maintainable key-value store application.
