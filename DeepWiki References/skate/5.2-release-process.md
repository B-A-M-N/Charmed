Release Process
===============

Relevant source files

*   [.github/workflows/goreleaser.yml](https://github.com/charmbracelet/skate/blob/09860417/.github/workflows/goreleaser.yml)
    
*   [.goreleaser.yml](https://github.com/charmbracelet/skate/blob/09860417/.goreleaser.yml)
    

This document outlines the release process for Skate, detailing how new versions are tagged, built, and distributed to users. The release process is automated using GoReleaser and GitHub Actions. For information about building from source, see [Build Process](https://deepwiki.com/charmbracelet/skate/5.1-build-process)
.

Release Workflow Overview
-------------------------

The Skate project follows a streamlined release process that utilizes GoReleaser to automate builds and distribution. When a new version is ready for release, a Git tag is created, which triggers the automated release pipeline.

### Release Pipeline Diagram

Sources: [.github/workflows/goreleaser.yml1-23](https://github.com/charmbracelet/skate/blob/09860417/.github/workflows/goreleaser.yml#L1-L23)
 [.goreleaser.yml1-16](https://github.com/charmbracelet/skate/blob/09860417/.goreleaser.yml#L1-L16)

Release Triggers and Automation
-------------------------------

Releases are automatically triggered when a tag matching the pattern `v*.*.*` is pushed to the repository. This follows the Semantic Versioning format (e.g., v1.2.3).

### Workflow Configuration

The release process is defined in the GitHub Actions workflow file, which uses a reusable workflow from the charmbracelet/meta repository.

Sources: [.github/workflows/goreleaser.yml3-11](https://github.com/charmbracelet/skate/blob/09860417/.github/workflows/goreleaser.yml#L3-L11)

### Secrets Required for Release

The release workflow requires several secrets to be configured in the GitHub repository:

| Secret | Purpose |
| --- | --- |
| `DOCKERHUB_USERNAME` | Username for Docker Hub publishing |
| `DOCKERHUB_TOKEN` | Authentication token for Docker Hub |
| `PERSONAL_ACCESS_TOKEN` | GitHub token with repo access |
| `GORELEASER_KEY` | Key for signing GoReleaser artifacts |
| `FURY_TOKEN` | Token for Gemfury package repository |
| `NFPM_GPG_KEY` | GPG key for signing packages |
| `NFPM_PASSPHRASE` | Passphrase for GPG key |

Sources: [.github/workflows/goreleaser.yml15-22](https://github.com/charmbracelet/skate/blob/09860417/.github/workflows/goreleaser.yml#L15-L22)

GoReleaser Configuration
------------------------

The release process utilizes GoReleaser, a release automation tool for Go projects. Skate's GoReleaser configuration is minimal, as it inherits most settings from a shared configuration.

### Configuration Details

The GoReleaser configuration includes project-specific variables that define metadata used throughout the release process.

| Variable | Value | Purpose |
| --- | --- | --- |
| `binary_name` | skate | Name of the executable |
| `description` | "A personal key value store 🛼" | Project description |
| `github_url` | "[https://github.com/charmbracelet/skate](https://github.com/charmbracelet/skate)<br>" | Repository URL |
| `maintainer` | "Ayman Bagabas [ayman@charm.sh](mailto:ayman@charm.sh)<br>" | Primary maintainer |
| `brew_commit_author_name` | "Ayman Bagabas" | Name for Homebrew commits |
| `brew_commit_author_email` | "[ayman@charm.sh](mailto:ayman@charm.sh)<br>" | Email for Homebrew commits |

Skate inherits its primary GoReleaser configuration from the Charm shared configuration file: `charmbracelet/meta/main/goreleaser-simple.yaml`

Sources: [.goreleaser.yml1-16](https://github.com/charmbracelet/skate/blob/09860417/.goreleaser.yml#L1-L16)

Creating a New Release
----------------------

To initiate a new release, a project maintainer follows these steps:

1.  Ensure all desired changes are merged to the main branch
2.  Determine the new version number following Semantic Versioning principles:
    *   Increment MAJOR version for incompatible API changes
    *   Increment MINOR version for backward-compatible new functionality
    *   Increment PATCH version for backward-compatible bug fixes
3.  Create and push a new tag with the version:

4.  The GitHub Actions workflow will automatically start the release process
5.  Monitor the progress in the Actions tab of the GitHub repository

Sources: [.github/workflows/goreleaser.yml3-6](https://github.com/charmbracelet/skate/blob/09860417/.github/workflows/goreleaser.yml#L3-L6)

Post-Release Verification
-------------------------

After a release completes, it's good practice to verify that:

1.  All artifacts are correctly published to GitHub Releases
2.  The Docker image is available on Docker Hub
3.  The Homebrew formula has been updated
4.  Sample installation commands work as expected

If issues are found with a release, the tag can be deleted and recreated once the problems are fixed. However, this should be avoided if possible as it can cause confusion for users.

Sources: [.github/workflows/goreleaser.yml12-23](https://github.com/charmbracelet/skate/blob/09860417/.github/workflows/goreleaser.yml#L12-L23)

Release Frequency and Planning
------------------------------

The Skate project doesn't follow a fixed release schedule. Instead, releases are created as needed when:

1.  Significant new features are implemented
2.  Important bug fixes are ready
3.  Security issues need to be addressed
4.  Dependencies require updating

For larger changes, release candidates may be published with the `-rc` suffix (e.g., `v1.2.0-rc1`) before the final release.
