Data Storage
============

Relevant source files

*   [go.mod](https://github.com/charmbracelet/skate/blob/09860417/go.mod)
    
*   [main.go](https://github.com/charmbracelet/skate/blob/09860417/main.go)
    

This document explains how Skate stores and manages key-value data, focusing on the underlying storage engine, file structure, and database operations. For information about how commands are processed from the CLI to execution, see [Command Processing](https://deepwiki.com/charmbracelet/skate/4.1-command-processing)
.

Storage Architecture Overview
-----------------------------

Skate uses BadgerDB, an embedded key-value database written in Go, as its storage engine. All data is stored locally in the user's application data directory, with support for multiple databases through the `@DB` syntax in commands.

Storage Location
----------------

Skate stores all databases in a standardized location within the user's home directory. The path is determined using the `go-app-paths` library, which ensures consistency across different operating systems.

Diagram: Database File Structure

Sources: [main.go225-236](https://github.com/charmbracelet/skate/blob/09860417/main.go#L225-L236)

The storage path is constructed in the `getFilePath` function:

The specific location of the data directory varies by operating system:

*   Linux: `~/.local/share/charm/kv/`
*   macOS: `~/Library/Application Support/charm/kv/`
*   Windows: `%APPDATA%\charm\kv\`

### Database Initialization

Databases are opened using the `openKV` function:

Key points about database initialization:

*   If no database name is specified, it defaults to "default"
*   BadgerDB logging level is set to ERROR to minimize console output
*   Each database corresponds to a separate directory on disk
*   Databases are created automatically when accessed for the first time

### Transaction Management

All database operations in Skate use the `wrap` function to handle transactions:

This function:

1.  Creates a new transaction (read-only or read-write)
2.  Executes the provided function within the transaction
3.  Commits the transaction if successful, or discards it if an error occurs

### Key and Database Name Parsing

Before any database operation, the key and database name are parsed using the `keyParser` function:

This function:

*   Splits the input string by the "@" symbol
*   Extracts the key (always required) and database name (optional)
*   Converts both to lowercase for consistency
*   Returns the key as a byte slice, the database name as a string, and any error

Sources: [main.go394-407](https://github.com/charmbracelet/skate/blob/09860417/main.go#L394-L407)

### Setting Values

Values are stored using the `set` function:

The function supports two input methods:

1.  Direct value as a command argument
2.  Reading value from standard input (for larger data or piping)

Sources: [main.go123-145](https://github.com/charmbracelet/skate/blob/09860417/main.go#L123-L145)

### Getting Values

Values are retrieved using the `get` function:

This function:

1.  Parses the key and database name
2.  Opens the appropriate database
3.  Creates a read-only transaction
4.  Retrieves and copies the value data
5.  Prints the value to standard output

Sources: [main.go147-170](https://github.com/charmbracelet/skate/blob/09860417/main.go#L147-L170)

### Deleting Values

Values are deleted using the `del` function:

The function simply:

1.  Parses the key and database name
2.  Opens the appropriate database
3.  Creates a read-write transaction
4.  Deletes the key-value pair

Sources: [main.go172-186](https://github.com/charmbracelet/skate/blob/09860417/main.go#L172-L186)

### Listing Key-Value Pairs

The `list` function provides iteration through all key-value pairs in a database:

The `list` function supports several options:

*   `--reverse/-r`: List in reverse lexicographic order
*   `--keys-only/-k`: Only display keys (more efficient)
*   `--values-only/-v`: Only display values
*   `--delimiter/-d`: Specify a delimiter between keys and values
*   `--show-binary/-b`: Show binary data (otherwise omitted)

Data Handling

### Text and Binary Data

Skate can store and retrieve both text and binary data. When displaying values, it handles binary data appropriately:

Key points:

*   By default, binary data is displayed as "(omitted binary data)"
*   The `--show-binary/-b` flag forces the display of binary data
*   UTF-8 validation is used to determine if data is binary
*   Terminal detection is used to adjust output formatting

Summary
-------

Skate's data storage system uses BadgerDB to provide a robust, efficient key-value store with the following characteristics:

*   Local file-based storage in a standard user data directory
*   Support for multiple databases via the `@DB` syntax
*   Simple but powerful operations: set, get, delete, list
*   Automatic database creation and management
*   Support for both text and binary data
*   ACID transaction compliance

This implementation balances simplicity with performance, making it suitable for a wide range of personal key-value storage needs.
