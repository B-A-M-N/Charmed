Purpose and Scope
-----------------

This page documents the `renderer` interface contract and the `View` structure that defines the visual state of a Bubble Tea application. The renderer interface provides an abstraction layer between the core framework and terminal output, while the `View` struct encapsulates all visual and terminal configuration properties needed to render a frame.

For implementation details of the primary renderer, see [Cursed Renderer Implementation](https://deepwiki.com/charmbracelet/bubbletea/4.2-cursed-renderer-implementation)
. For terminal feature configuration and capabilities detection, see [Terminal Features and Capabilities](https://deepwiki.com/charmbracelet/bubbletea/4.4-terminal-features-and-capabilities)
.

Renderer Interface Contract
---------------------------

The `renderer` interface defines the contract that all Bubble Tea renderers must implement. It provides lifecycle management, rendering operations, and terminal state control.

**Sources:** [renderer.go17-57](https://github.com/charmbracelet/bubbletea/blob/f25595a8/renderer.go#L17-L57)
 [cursed\_renderer.go18-36](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L18-L36)
 [nil\_renderer.go8-53](https://github.com/charmbracelet/bubbletea/blob/f25595a8/nil_renderer.go#L8-L53)

### Core Rendering Methods

| Method | Signature | Purpose |
| --- | --- | --- |
| `start` | `start()` | Initialize or restart the renderer after being stopped |
| `render` | `render(View)` | Queue a view for rendering (stores view, does not write output) |
| `flush` | `flush(closing bool) error` | Write queued changes to output; `closing` indicates final flush |
| `close` | `close() error` | Clean up terminal state and flush remaining output |
| `reset` | `reset()` | Reset renderer state to initial configuration |

**Sources:** [renderer.go19-32](https://github.com/charmbracelet/bubbletea/blob/f25595a8/renderer.go#L19-L32)

### State Management Methods

| Method | Signature | Purpose |
| --- | --- | --- |
| `resize` | `resize(w, h int)` | Notify renderer of terminal dimension changes |
| `clearScreen` | `clearScreen()` | Clear the screen and move cursor to top-left |
| `setColorProfile` | `setColorProfile(colorprofile.Profile)` | Configure color output capabilities |
| `setSyncdUpdates` | `setSyncdUpdates(bool)` | Enable/disable synchronized output mode |
| `setWidthMethod` | `setWidthMethod(ansi.Method)` | Set method for calculating character widths |

**Sources:** [renderer.go37-50](https://github.com/charmbracelet/bubbletea/blob/f25595a8/renderer.go#L37-L50)

### I/O Methods

| Method | Signature | Purpose |
| --- | --- | --- |
| `writeString` | `writeString(string) (int, error)` | Write a string directly to renderer output |
| `insertAbove` | `insertAbove(string) error` | Insert unmanaged lines above the current frame |
| `onMouse` | `onMouse(MouseMsg) Cmd` | Handle mouse event (delegates to `View.OnMouse`) |

**Sources:** [renderer.go34-56](https://github.com/charmbracelet/bubbletea/blob/f25595a8/renderer.go#L34-L56)

--------------

Renderer Implementations
------------------------

The framework provides two implementations of the `renderer` interface:

**Sources:** [cursed\_renderer.go18-49](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L18-L49)
 [nil\_renderer.go8-53](https://github.com/charmbracelet/bubbletea/blob/f25595a8/nil_renderer.go#L8-L53)

### cursedRenderer

The primary renderer implementation that handles actual terminal output. It uses the `ultraviolet` library for screen buffering and diff-based rendering.

**Key characteristics:**

*   Maintains internal screen buffer (`uv.ScreenBuffer`)
*   Tracks last rendered view for diffing (`lastView`)
*   Uses `uv.TerminalRenderer` for ANSI sequence generation
*   Thread-safe with mutex protection

**Sources:** [cursed\_renderer.go18-36](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L18-L36)
 [cursed\_renderer.go40-49](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L40-L49)

### nilRenderer

A no-op implementation used for testing and headless operation. All methods are empty implementations that perform no actions.

**Use cases:**

*   Testing models without terminal output
*   Running programs in contexts where terminal output is not desired
*   Using `WithoutRenderer()` program option

**Sources:** [nil\_renderer.go8-53](https://github.com/charmbracelet/bubbletea/blob/f25595a8/nil_renderer.go#L8-L53)

Rendering Contract and Lifecycle
--------------------------------

The rendering process follows a two-phase pattern: **render** (queue) and **flush** (write).

**Sources:** [cursed\_renderer.go575-581](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L575-L581)
 [cursed\_renderer.go256-573](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L256-L573)

### The render() Method

Stores the provided `View` for later flushing. This method does not perform any I/O or terminal operations.

**Behavior:**

1.  Acquires mutex lock
2.  Stores `View` in internal field
3.  Releases lock
4.  Returns immediately

**Sources:** [cursed\_renderer.go575-581](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L575-L581)

### The flush() Method

Performs the actual rendering by comparing the stored view with the previously rendered view and writing only the differences to the terminal.

**Key steps:**

1.  Calculate frame area based on content and screen mode
2.  Compare current view with `lastView` using `viewEquals()`
3.  If unchanged (and not starting/closing), skip rendering
4.  Update screen buffer with new content
5.  Apply terminal mode changes (alt screen, mouse mode, etc.)
6.  Generate ANSI sequences for differences
7.  Handle cursor visibility with synchronized output or hide/show sequences
8.  Write output to terminal
9.  Update `lastView` for next comparison

**Sources:** [cursed\_renderer.go256-573](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L256-L573)

Thread Safety
-------------

The `cursedRenderer` implementation is thread-safe:

*   All public methods acquire a mutex lock (`s.mu.Lock()`)
*   State modifications are protected
*   Safe for concurrent access from multiple goroutines

The `nilRenderer` has no state and is inherently thread-safe.

**Sources:** [cursed\_renderer.go27](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L27-L27)
 [cursed\_renderer.go72-73](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L72-L73)

Integration with Program Lifecycle
----------------------------------

The renderer integrates with `tea.Program` lifecycle events:

| Program Event | Renderer Action |
| --- | --- |
| Program start | `start()` - initialize or restore state |
| Window resize | `resize(w, h)` - update dimensions |
| Model update | `render(View)` - queue new view |
| Frame tick | `flush(false)` - write changes |
| Suspend (Unix) | `close()` - clean up terminal |
| Resume (Unix) | `start()` - restore terminal state |
| Program exit | `close()` - final cleanup and flush |

**Sources:** [cursed\_renderer.go70-139](https://github.com/charmbracelet/bubbletea/blob/f25595a8/cursed_renderer.go#L70-L139)
