Purpose and Scope
-----------------

This document describes the Continuous Integration and Continuous Deployment (CI/CD) pipeline implemented for the Bubbletea repository using GitHub Actions. The pipeline automates build verification, code quality checks, test coverage reporting, and dependency maintenance across the main codebase, examples, and tutorials.

For information about local development setup and testing patterns, see [Development Setup](https://deepwiki.com/charmbracelet/bubbletea/7.1-development-setup)
 and [Testing Guidelines](https://deepwiki.com/charmbracelet/bubbletea/7.2-testing-guidelines)
. For code quality standards and expectations, see [Code Quality Standards](https://deepwiki.com/charmbracelet/bubbletea/7.3-code-quality-standards)
.

* * *

Workflow Overview
-----------------

The CI/CD pipeline consists of four GitHub Actions workflows located in [.github/workflows/](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/)
:

| Workflow File | Purpose | Trigger Events | Key Features |
| --- | --- | --- | --- |
| `build.yml` | Verify compilation across Go versions and module configurations | Push, Pull Request | Tests main module, go.mod version, and examples separately |
| `lint.yml` | Enforce code quality standards | Push, Pull Request | Uses golangci-lint with custom configuration |
| `coverage.yml` | Generate and report test coverage | Push, Pull Request | Race detection, Codecov integration |
| `examples.yml` | Maintain example dependencies | Push to master (specific paths), Manual dispatch | Automated `go mod tidy` with auto-commit |

Build Workflow
--------------

The `build.yml` workflow verifies that the codebase compiles successfully across different configurations. It reuses the centralized build workflow from `charmbracelet/meta` to maintain consistency across the Charmbracelet ecosystem.

### Job Structure

**Three Independent Build Jobs:**

1.  **`build` job**: Compiles with the latest stable Go version [.github/workflows/build.yml5-6](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml#L5-L6)
    
2.  **`build-go-mod` job**: Compiles using the Go version specified in `./go.mod` [.github/workflows/build.yml8-12](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml#L8-L12)
    
3.  **`build-examples` job**: Compiles examples using the version from `./examples/go.mod` [.github/workflows/build.yml14-19](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml#L14-L19)
    

**Configuration Parameters:**

| Job | `go-version` | `go-version-file` | `working-directory` |
| --- | --- | --- | --- |
| `build` | (default) | (none) | (root) |
| `build-go-mod` | `""` (disabled) | `./go.mod` | (root) |
| `build-examples` | `""` (disabled) | `./examples/go.mod` | `./examples` |

The empty string `""` for `go-version` disables the default version, forcing the workflow to read from `go-version-file` instead [.github/workflows/build.yml11-17](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml#L11-L17)

**Build Job Execution Flow**

### Reusable Workflow Integration

All build jobs call `charmbracelet/meta/.github/workflows/build.yml@main` [.github/workflows/build.yml6-15](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml#L6-L15)
 which implements the standard Charmbracelet build process including:

*   Go module verification
*   Compilation checks
*   Unit test execution
*   Cross-platform validation (handled by the reusable workflow)

Sources: [.github/workflows/build.yml1-20](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml#L1-L20)

Lint Workflow
-------------

The `lint.yml` workflow enforces code quality standards using `golangci-lint`. It triggers on every push and pull request to ensure consistent code style and catch common issues.

### Configuration

**Workflow Parameters:**

| Parameter | Value | Purpose |
| --- | --- | --- |
| `golangci_path` | `.golangci.yml` | Path to linter configuration file |
| `golangci_version` | `v2.9` | Specific golangci-lint version |
| `timeout` | `10m` | Maximum execution time |

The workflow delegates to `charmbracelet/meta/.github/workflows/lint.yml@main` [.github/workflows/lint.yml8](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L8-L8)
 which executes golangci-lint with the specified configuration.

**Lint Workflow Execution**

### Linter Configuration

The linter behavior is controlled by `.golangci.yml` (not shown in provided files but referenced at [.github/workflows/lint.yml10](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L10-L10)
). This configuration file defines:

*   Enabled linter rules
*   Excluded files or patterns
*   Severity thresholds
*   Custom rule parameters

The 10-minute timeout [.github/workflows/lint.yml12](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L12-L12)
 ensures the workflow doesn't hang indefinitely on large codebases or slow runners.

Sources: [.github/workflows/lint.yml1-13](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L1-L13)

Coverage Workflow
-----------------

The `coverage.yml` workflow generates test coverage reports and uploads them to Codecov for analysis and tracking. It runs on both push and pull request events, providing immediate feedback on test coverage changes.

### Workflow Configuration

**Execution Environment:**

| Setting | Value | Purpose |
| --- | --- | --- |
| Go Version | `^1` (latest 1.x) | Use latest Go 1.x release |
| OS  | `ubuntu-latest` | Linux environment for testing |
| `GO111MODULE` | `"on"` | Enable Go modules (explicit) |

The workflow uses a matrix strategy [.github/workflows/coverage.yml6-10](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L6-L10)
 that allows easy expansion to multiple Go versions or operating systems in the future.

### Coverage Generation Process

**Step-by-Step Execution:**

1.  **Install Go** [.github/workflows/coverage.yml14-17](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L14-L17)
    : Set up Go toolchain using `actions/setup-go@v6`
    
2.  **Checkout Code** [.github/workflows/coverage.yml19-20](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L19-L20)
    : Clone repository using `actions/checkout@v6`
    
3.  **Run Tests with Coverage** [.github/workflows/coverage.yml22-24](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L22-L24)
    :
    
    *   `-race`: Enable race detector to catch concurrency issues
    *   `-covermode=atomic`: Use atomic coverage mode (required for race detection)
    *   `-coverprofile=coverage.txt`: Output coverage data to file
    *   `./...`: Test all packages recursively
4.  **Upload to Codecov** [.github/workflows/coverage.yml26-29](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L26-L29)
    : Send coverage report using `codecov/codecov-action@v5`
    

**Coverage Workflow Data Flow**

### Codecov Integration

The workflow uses a secret `CODECOV_TOKEN` [.github/workflows/coverage.yml29](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L29-L29)
 for authenticated uploads. This token:

*   Authorizes the upload to the Codecov service
*   Associates coverage data with the correct repository
*   Enables private repository support

The coverage report is used to:

*   Track coverage trends over time
*   Generate coverage badges for README
*   Provide feedback on pull requests with coverage changes
*   Identify untested code paths

Sources: [.github/workflows/coverage.yml1-30](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L1-L30)

Examples Maintenance Workflow
-----------------------------

The `examples.yml` workflow automates dependency management for the `examples/` and `tutorials/` directories. It runs `go mod tidy` and automatically commits changes back to the repository, ensuring that example dependencies stay synchronized with the main module.

### Trigger Conditions

The workflow runs when:

1.  **Push to `master` branch** with changes to specific files [.github/workflows/examples.yml3-14](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L3-L14)
    :
    
    *   `.github/workflows/examples.yml` (the workflow itself)
    *   `./examples/go.mod` or `./examples/go.sum`
    *   `./tutorials/go.mod` or `./tutorials/go.sum`
    *   `./go.mod` or `./go.sum` (main module changes)
2.  **Manual dispatch** via `workflow_dispatch` [.github/workflows/examples.yml15](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L15-L15)
    

This selective triggering prevents unnecessary runs while ensuring examples are updated when dependencies change.

### Automated Maintenance Process

**Execution Steps:**

1.  **Checkout Repository** [.github/workflows/examples.yml23](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L23-L23)
    : Clone code with write permissions
2.  **Setup Go** [.github/workflows/examples.yml24-27](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L24-L27)
    : Install Go `^1` with caching enabled
3.  **Tidy Examples** [.github/workflows/examples.yml28-31](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L28-L31)
    :
    
4.  **Auto-commit Changes** [.github/workflows/examples.yml32-37](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L32-L37)
    : Commit modifications using `stefanzweifel/git-auto-commit-action@v7`

**Examples Workflow Automation Flow**

### Auto-commit Configuration

The `git-auto-commit-action` is configured with [.github/workflows/examples.yml32-37](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L32-L37)
:

| Parameter | Value | Purpose |
| --- | --- | --- |
| `commit_message` | `"chore: go mod tidy tutorials and examples"` | Descriptive commit message |
| `branch` | `master` | Target branch for commit |
| `commit_user_name` | `actions-user` | Git user for automated commits |
| `commit_user_email` | `actions@github.com` | Git email for automated commits |

The job requires `contents: write` permissions [.github/workflows/examples.yml19-20](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L19-L20)
 to push changes back to the repository.

### Why This Workflow Exists

The examples and tutorials use `replace` directives in their `go.mod` files to reference the local bubbletea source [see Development Setup](https://deepwiki.com/charmbracelet/bubbletea/7.1-development-setup)
. When the main module's dependencies change, the examples' `go.mod` files may require updates to maintain consistency. This workflow automates that synchronization, preventing dependency drift and ensuring examples always compile with the latest main module.

Sources: [.github/workflows/examples.yml1-38](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L1-L38)

Workflow Dependencies and Integration
-------------------------------------

### Execution Triggers Summary

| Event Type | Workflows Triggered | Conditions |
| --- | --- | --- |
| Push (any branch) | `build`, `lint`, `coverage` | Always |
| Pull Request | `build`, `lint`, `coverage` | Always |
| Push to `master` | `examples` | Only if go.mod/go.sum files changed |
| Manual Dispatch | `examples` | On-demand execution |

### Workflow Coordination

The workflows operate independently without explicit dependencies between them. This design provides:

*   **Parallel Execution**: All workflows run concurrently for faster feedback
*   **Isolated Failures**: One workflow failure doesn't block others
*   **Flexible Retries**: Individual workflows can be re-run independently

**Combined CI/CD Pipeline Flow**

### Status Check Configuration

While not explicitly shown in the workflow files, the workflows integrate with GitHub's branch protection rules:

*   **Required Checks**: The `build`, `lint`, and `coverage` workflows typically serve as required status checks for pull requests
*   **Merge Blocking**: Pull requests cannot be merged until all required workflows pass
*   **Retry Mechanism**: Failed workflows can be re-run individually without retriggering all workflows

### Reusable Workflow Benefits

Using `charmbracelet/meta` reusable workflows [.github/workflows/build.yml6-15](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml#L6-L15)
 and [.github/workflows/lint.yml8](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L8-L8)
 provides:

*   **Consistency**: Standardized CI processes across Charmbracelet repositories
*   **Centralized Maintenance**: Updates to CI logic propagate automatically
*   **Reduced Duplication**: No need to maintain identical workflow logic in each repository
*   **Ecosystem Coherence**: All Charmbracelet projects follow the same quality standards

Workflow Maintenance and Customization
--------------------------------------

### Modifying Build Configurations

To change build behavior:

*   **Add Go Versions**: Expand the matrix in the reusable workflow (not local to this repository)
*   **Add OS Platforms**: Modify the `charmbracelet/meta` workflow to test on different operating systems
*   **Adjust Timeouts**: Add timeout parameters to workflow calls

### Updating Linter Configuration

To change linting rules:

1.  Edit `.golangci.yml` (not shown in provided files)
2.  Update `golangci_version` in [.github/workflows/lint.yml11](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L11-L11)
     if linter version needs to change
3.  Adjust `timeout` in [.github/workflows/lint.yml12](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L12-L12)
     if linting takes longer

### Modifying Coverage Requirements

To change coverage settings:

*   **Add Platforms**: Expand the matrix at [.github/workflows/coverage.yml7-9](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L7-L9)
    
*   **Change Coverage Modes**: Modify `-covermode` flag at [.github/workflows/coverage.yml24](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L24-L24)
    
*   **Disable Race Detection**: Remove `-race` flag (not recommended)

### Extending Examples Workflow

To add more directories or customize behavior:

1.  Add paths to trigger conditions at [.github/workflows/examples.yml8-14](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L8-L14)
    
2.  Add `go mod tidy` commands at [.github/workflows/examples.yml29-31](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L29-L31)
    
3.  Modify commit message at [.github/workflows/examples.yml34](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L34-L34)
