This page explains Bubbletea's concurrency model, including how goroutines are used for command execution, how message channels coordinate between threads, and the thread safety guarantees provided by the framework. This covers the internal architecture that enables responsive, concurrent I/O operations while maintaining simple, deterministic application code.

For information about commands themselves (Cmd type, Batch, Sequence, Tick, Every), see [Command System](https://deepwiki.com/charmbracelet/bubbletea/2.4-command-system)
. For testing concurrent behavior, see [Testing Bubble Tea Applications](https://deepwiki.com/charmbracelet/bubbletea/5.4-testing-bubble-tea-applications)
. For error handling in concurrent operations, see [Error Handling and Recovery](https://deepwiki.com/charmbracelet/bubbletea/5.2-error-handling-and-recovery)
.

Concurrency Model Overview
--------------------------

Bubbletea uses a **single-threaded event loop** for state updates combined with **concurrent goroutine-based command execution**. This architecture provides the benefits of concurrent I/O operations without requiring locks, mutexes, or other synchronization primitives in application code.

The framework guarantees that:

*   The `Update()` method is **never called concurrently** - only one message is processed at a time
*   Model state modifications are **always sequential and predictable**
*   Commands execute **concurrently in separate goroutines**
*   All command results flow back through a **central message channel**

The Serial Event Loop
---------------------

The `eventLoop()` function processes messages one at a time in a strict serial order. This is the core of Bubbletea's predictable state management model.

### Event Loop Operation

**Sequence: Message Processing in eventLoop()**

The event loop performs these steps for each message:

1.  **Wait for message** - Blocks on `p.msgs` channel until a message arrives
2.  **Apply filter** - If a filter is set via `WithFilter()`, apply it to intercept/modify messages
3.  **Check for special messages** - Handle internal messages like `QuitMsg`, `InterruptMsg`, `BatchMsg`, `sequenceMsg`
4.  **Call Update()** - Invoke `model.Update(msg)` to get new model state and optional command
5.  **Queue command** - Send any returned command to the command handler channel
6.  **Render view** - Call `model.View()` and send output to renderer

This serial processing ensures **no race conditions** in model state updates.

**Sources:** [tea.go382-505](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L382-L505)
 [tea.go390-398](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L390-L398)
 [tea.go494-502](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L494-L502)

### Thread Safety Guarantee

Because `Update()` is called serially, application developers **never need synchronization primitives**:

| Scenario | Thread Safe? | Reason |
| --- | --- | --- |
| Reading model fields in `Update()` | ✅ Yes | Serial execution, no concurrent access |
| Modifying model fields in `Update()` | ✅ Yes | Serial execution, one update at a time |
| Reading model fields in `View()` | ✅ Yes | Called immediately after `Update()`, no interleaving |
| Sharing data between commands via model | ✅ Yes | Commands send messages back, processed serially |
| Using mutexes in model struct | ❌ Not needed | Adds unnecessary complexity |
| Using channels in model struct | ❌ Not needed | Messages already provide communication |

### Non-Blocking Execution

Command goroutines **do not block program shutdown**. When the program terminates:

1.  The context is cancelled ([tea.go588](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L588-L588)
     [tea.go743](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L743-L743)
    )
2.  The event loop exits ([tea.go385-386](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L385-L386)
    )
3.  `handleCommands()` goroutine exits ([tea.go341-342](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L341-L342)
    )
4.  Running command goroutines are **leaked intentionally**

This design prevents shutdown hangs from long-running commands (e.g., network requests). Commands that need graceful shutdown should monitor the context.

Message Channels
----------------

Three primary channels coordinate communication between goroutines:

| Channel | Type | Purpose | Producers | Consumer |
| --- | --- | --- | --- | --- |
| `msgs` | `chan Msg` | Central message queue | Input reader, signal handler, resize handler, commands, `Send()` | Event loop |
| `cmds` | `chan Cmd` | Command dispatch queue | Event loop (from `Update()`) | Command handler |
| `errs` | `chan error` | Error signaling | Panic recovery | Event loop |

**Sources:** [tea.go161-163](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L161-L163)
 [tea.go580-581](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L580-L581)

### Message Channel Architecture

**Architecture: Message Channel and Producers/Consumers**

The `msgs` channel is **unbuffered** ([tea.go244](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L244-L244)
), which means:

*   Message sends **block** until the event loop receives them
*   This provides **backpressure** - fast message producers wait for processing
*   No messages are lost if shutdown happens

Thread Safety Guarantees
------------------------

Bubbletea provides strong thread safety guarantees without requiring application code to use synchronization primitives.

### What is Thread-Safe

| Operation | Thread-Safe? | Details |
| --- | --- | --- |
| Reading/writing model fields in `Update()` | ✅ Yes | Serial execution |
| Reading model fields in `View()` | ✅ Yes | Called on main thread after `Update()` |
| Calling `p.Send()` from any goroutine | ✅ Yes | Uses select with context |
| Calling `p.Quit()` from any goroutine | ✅ Yes | Calls `Send()` internally |
| Calling `p.Kill()` from any goroutine | ✅ Yes | Cancels context (atomic operation) |
| Multiple commands sending messages | ✅ Yes | Channel is thread-safe |
| Commands reading each other's data | ❌ No | Commands should be independent |

**Sources:** [tea.go382-505](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L382-L505)
 [tea.go774-779](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L774-L779)
 [tea.go788-790](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L788-L790)
 [tea.go795-797](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L795-L797)

### What is NOT Thread-Safe

Application code should avoid:

1.  **Shared mutable state between commands** - Commands execute concurrently and cannot safely share data
2.  **Reading/writing model from command goroutines** - Model state changes only happen in `Update()`
3.  **Calling model methods from command goroutines** - Model methods assume serial execution context

Instead, commands should:

*   Perform I/O independently
*   Return messages with results
*   Let `Update()` modify model state based on those messages

### Context Usage Patterns

| Scenario | Context Used | Effect |
| --- | --- | --- |
| User calls `p.Quit()` | None | Sends `QuitMsg`, event loop exits naturally |
| User calls `p.Kill()` | Cancels internal `ctx` | Goroutines stop, event loop exits immediately |
| External context cancelled | Cancels internal `ctx` (derived from external) | Program killed, error includes context reason |
| Command panics | Cancels internal `ctx` | Program terminated to prevent inconsistent state |
| Normal shutdown | Calls `p.cancel()` at end of `Run()` | Cleanup during shutdown |

### Shutdown Steps

The `shutdown()` method executes these steps:

1.  **Cancel context** - `p.cancel()` stops all goroutines ([tea.go811](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L811-L811)
    )
2.  **Wait for handlers** - `p.handlers.shutdown()` waits for goroutine cleanup ([tea.go814](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L814-L814)
    )
3.  **Cancel input** - `p.cancelReader.Cancel()` stops input reading ([tea.go819](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L819-L819)
    )
4.  **Wait for input loop** - `p.waitForReadLoop()` ensures input goroutine exits ([tea.go820-822](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L820-L822)
    )
5.  **Close input** - `p.cancelReader.Close()` releases input resources ([tea.go824](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L824-L824)
    )
6.  **Stop renderer** - `p.renderer.stop()` or `kill()` based on shutdown type ([tea.go827-833](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L827-L833)
    )
7.  **Restore terminal** - `p.restoreTerminalState()` returns terminal to original state ([tea.go835](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L835-L835)
    )

`Program.Send()` can be called from any goroutine at any time:

* * *

**Summary:** Bubbletea's concurrency model separates concerns - the event loop provides deterministic, serial state updates while goroutines enable concurrent I/O. This architecture eliminates the need for locks in application code while maintaining responsiveness. All communication happens through channels, and panic recovery ensures terminal state is always restored.
