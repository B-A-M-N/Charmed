This page demonstrates practical patterns for composing and coordinating multiple commands using Bubble Tea's orchestration primitives. For basic command concepts, see [Command System](https://deepwiki.com/charmbracelet/bubbletea/2.4-command-system)
. For implementation details and concurrency patterns, see [Concurrency and Goroutines](https://deepwiki.com/charmbracelet/bubbletea/5.1-concurrency-and-goroutines)
.

Overview
--------

Command orchestration allows executing multiple commands either concurrently or sequentially. The two primary functions are:

| Function | Type | Execution | Typical Use Case |
| --- | --- | --- | --- |
| `tea.Batch` | Concurrent | All commands run in parallel | Independent I/O operations, multiple timers |
| `tea.Sequence` | Sequential | Commands run one after another | Dependent operations, ordered workflows |

Additional timer commands for periodic operations:

| Function | Behavior | Typical Use Case |
| --- | --- | --- |
| `tea.Every` | Syncs with system clock | Clock displays, synchronized updates |
| `tea.Tick` | Independent timer | Animations, countdowns, polling |

Sources: [commands.go15-27](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L15-L27)
 [commands.go102-164](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L102-L164)

Every vs Tick Timers
--------------------

Bubble Tea provides two timer commands with different synchronization behaviors.

### Comparison Table

| Aspect | `tea.Every` | `tea.Tick` |
| --- | --- | --- |
| **Definition** | [commands.go102-114](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L102-L114) | [commands.go154-164](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L154-L164) |
| **Clock Sync** | Syncs with system clock | Independent timer |
| **Duration** | May be shorter than specified | Always full duration |
| **Use Case** | Clock displays, synchronized updates | Animations, countdowns, polling |
| **Example** | Update at :00 seconds | Tick every N milliseconds |

### Every - System Clock Synchronization

**Diagram: tea.Every Timing Behavior**

`tea.Every` aligns with the system clock by truncating the current time to the specified duration, then waiting until the next boundary.

**Example: Clock Display**

The clock updates precisely at second boundaries (e.g., 12:34:00, 12:34:01, 12:34:02), even if `Init` was called mid-second.

Sources: [commands.go102-114](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L102-L114)

### Tick - Independent Timer

**Diagram: tea.Tick Timing Behavior**

`tea.Tick` starts a timer for the full duration from when it's called, independent of the system clock.

**Example: Animation Frame Timer**

The animation runs at exactly 60 FPS regardless of when it started, providing consistent frame timing.

Sources: [commands.go154-164](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L154-L164)

### When to Use Each

| Scenario | Recommended | Reason |
| --- | --- | --- |
| Wall clock display | `Every` | Updates at precise second/minute boundaries |
| Multiple synchronized updates | `Every` | All components update at same clock time |
| Animation frames | `Tick` | Consistent frame timing independent of start time |
| Countdown timer | `Tick` | Precise interval from specific event |
| Polling with backoff | `Tick` | Control exact delay between polls |
| Status checks every N minutes | `Every` | Aligns with clock (e.g., :00, :15, :30, :45) |

Sources: [commands.go102-114](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L102-L114)
 [commands.go154-164](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L154-L164)

### Nested Composition

Commands can be nested arbitrarily:

The execution model handles this through recursive processing:

| Level | Type | Behavior |
| --- | --- | --- |
| Outer | `Sequence` | Waits for inner `Batch` to complete |
| Inner | `Batch` | All three fetch commands run concurrently |
| Continue | Sequential | `processData()` runs after all fetches complete |
| Final | Sequential | `updateUI()` runs after `processData()` |

**Implementation:** The recursive handling at [tea.go522-526](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L522-L526)
 (for `sequenceMsg`) and [tea.go561-565](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L561-L565)
 (for `BatchMsg`) detects nested composition messages and processes them directly without routing through the event loop.

Sources: [tea.go522-526](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L522-L526)
 [tea.go561-565](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L561-L565)

Comparison Table
----------------

| Aspect | Batch | Sequence |
| --- | --- | --- |
| **Execution** | Concurrent (parallel) | Sequential (ordered) |
| **Goroutines** | One per command | One for entire sequence |
| **Completion** | WaitGroup waits for all | Loop waits for each |
| **Message Order** | Unpredictable | Guaranteed order |
| **Use Case** | Independent operations | Dependent operations |
| **Performance** | Faster for I/O-bound | Slower but ordered |
| **Panic Handling** | Per-command isolation | Terminates sequence |
| **Implementation** | [tea.go533-573](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L533-L573) | [tea.go507-531](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L507-L531) |

Panic Recovery Behavior
-----------------------

Both orchestration mechanisms include panic recovery, but with different scopes:

### Batch Panic Recovery

*   **Scope:** Per-command isolation at [tea.go552-558](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L552-L558)
    
*   **Behavior:** If one command panics, others continue
*   **Recovery:** Calls `p.recoverFromGoPanic()` at [tea.go555](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L555-L555)
    
*   **Program State:** Program continues running unless `WithoutCatchPanics` option is set

### Sequence Panic Recovery

*   **Scope:** Entire sequence at [tea.go508-514](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L508-L514)
    
*   **Behavior:** If any command panics, sequence terminates
*   **Recovery:** Calls `p.recoverFromGoPanic()` at [tea.go511](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L511-L511)
    
*   **Program State:** Remaining commands in sequence do not execute

**Common Behavior:** Both send `ErrProgramPanic` to the errors channel and cancel the program context, triggering graceful shutdown. See [Error Handling and Recovery](https://deepwiki.com/charmbracelet/bubbletea/5.2-error-handling-and-recovery)
 for details on panic recovery mechanisms.

Sources: [tea.go508-514](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L508-L514)
 [tea.go552-558](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L552-L558)
 [tea.go852-860](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L852-L860)
