Command System
==============

This page describes the Command System in Bubble Tea, which enables asynchronous side effects and I/O operations in a functional, message-driven architecture. Commands allow programs to perform operations like HTTP requests, timers, file I/O, and external process execution without blocking the main event loop.

For information about the messages that commands produce, see [Message System](https://deepwiki.com/charmbracelet/bubbletea/2.3-message-system)
. For information about how the event loop processes messages, see [Program Lifecycle](https://deepwiki.com/charmbracelet/bubbletea/2.1-program-lifecycle)
. For executing external processes, see [External I/O and Side Effects](https://deepwiki.com/charmbracelet/bubbletea/5.5-external-io-and-side-effects)
.

Overview
--------

Commands are the primary mechanism for performing side effects in Bubble Tea. The `Cmd` type represents an asynchronous operation that will eventually produce a `Msg`. Commands execute in separate goroutines and send their results back to the main event loop, maintaining the unidirectional data flow of The Elm Architecture.

Commands are returned from the `Init` method and the `Update` method, allowing the model to trigger asynchronous operations in response to initialization or incoming messages.

The Cmd Type
------------

The `Cmd` type is defined as a function that returns a `Msg`:

When a command is nil, it represents a no-op and will be ignored by the framework. This allows `Init` and `Update` to conditionally return commands without special handling.

**Sources:** [tea.go364-371](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L364-L371)

Command Execution Architecture
------------------------------

The following diagram illustrates how commands flow through the Bubble Tea runtime:

Commands execute asynchronously according to this lifecycle:

1.  **Return**: Commands are returned from `Init()` or `Update()` methods
2.  **Queue**: Commands are sent to the `cmds` channel
3.  **Spawn**: The `handleCommands` goroutine spawns a new goroutine for each command
4.  **Execute**: The command function executes (potentially blocking)
5.  **Send**: The resulting message is sent via `p.Send()` to the `p.msgs` channel
6.  **Process**: The event loop receives the message and dispatches it to `Update()`

**Sources:** [tea.go681-720](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L681-L720)
 [tea.go702-714](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L702-L714)
 [tea.go724-864](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L724-L864)

### Goroutine-Based Execution

The `handleCommands` function manages command execution:

Commands execute in isolated goroutines that:

*   Run concurrently with the event loop
*   Do not block the UI
*   Recover from panics (unless `WithoutCatchPanics` is set)
*   Send their result message via `p.Send()`

The command goroutines are not awaited before shutdown, which means long-running commands may leak goroutines. This is by design to prevent shutdown latency.

**Sources:** [tea.go681-720](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L681-L720)

### Batch Command

`Batch` executes multiple commands concurrently with no ordering guarantees:

The `Batch` function returns a command that produces a `BatchMsg` containing the non-nil commands. When the event loop receives a `BatchMsg`, it calls `execBatchMsg()` which spawns a goroutine for each command concurrently.

**Execution Flow:**

The `execBatchMsg()` function uses a `sync.WaitGroup` to wait for all commands to complete before returning, but the messages they produce may arrive at the event loop in any order.

### Sequence Command

`Sequence` executes commands one at a time, in order:

Unlike `Batch`, `Sequence` ensures commands execute sequentially. The `execSequenceMsg()` function executes each command and sends its result before proceeding to the next.

**Execution Comparison:**

| Aspect | Batch | Sequence |
| --- | --- | --- |
| Execution | Concurrent (goroutines) | Sequential (one at a time) |
| Ordering | No guarantee | Guaranteed order |
| Performance | Faster for I/O-bound | Slower but predictable |
| Use Case | Independent operations | Dependent operations |

### Time-Based Commands

Bubble Tea provides two time-based commands for scheduling future messages:

#### Every

`Every` ticks in sync with the system clock:

The timer truncates the current time to the specified duration boundary and schedules the next tick at that boundary. For example, `Every(time.Minute, fn)` at 12:34:45 will tick at 12:35:00 (15 seconds later).

**Characteristics:**

*   Aligns with clock boundaries (seconds, minutes, hours)
*   Multiple `Every` commands with the same duration will tick at the same time
*   Useful for synchronized periodic updates

#### Tick

`Tick` produces a timer that runs for the exact specified duration:

The timer starts immediately when the command executes and fires after exactly the specified duration.

**Characteristics:**

*   Independent of system clock
*   Precise duration from command execution
*   Useful for timeouts and delays

**Important Note:** Both `Every` and `Tick` fire only once. To create repeating timers, return the timer command again from `Update()`:

Command Helpers
---------------

### compactCmds

The `compactCmds` generic function optimizes command arrays by removing nil commands and returning the most direct representation:

| Input | Output |
| --- | --- |
| All nil | `nil` |
| One non-nil command | That `Cmd` directly |
| Multiple non-nil | `BatchMsg` or `sequenceMsg` |

This optimization avoids unnecessary message wrapper overhead when there's only one command.

**Sources:** [commands.go32-54](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L32-L54)

Custom Commands
---------------

Creating custom commands follows this pattern:

**Key Principles:**

*   Commands should be pure functions that capture all needed data
*   Long-running or blocking operations are safe inside commands
*   Commands should return exactly one message
*   Use `Batch` to return multiple commands from `Init` or `Update`

**Sources:** [tea.go364-371](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L364-L371)

Error Handling and Panic Recovery
---------------------------------

Commands execute with panic recovery enabled by default. If a command panics:

1.  The panic is recovered in the command's goroutine
2.  `recoverFromGoPanic()` is called to log the panic
3.  The program continues running (unless `WithoutCatchPanics` is set)

For `BatchMsg` and `sequenceMsg` execution, each individual command and the orchestration goroutines have their own panic recovery.

**Panic Recovery Points:**

Best Practices
--------------

1.  **Avoid blocking operations in Update()** - Use commands for I/O
2.  **Return Batch for multiple operations** - Don't execute commands directly
3.  **Keep command functions pure** - Capture all needed data when creating the command
4.  **Handle errors in messages** - Return error information in the message type
5.  **Use Sequence for dependencies** - When commands depend on each other's results
6.  **Remember timers fire once** - Re-return timer commands for periodic behavior
7.  **Design for testability** - Commands should be mockable for testing
