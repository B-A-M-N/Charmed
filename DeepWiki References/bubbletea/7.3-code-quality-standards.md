Code Quality Standards
======================

Relevant source files

*   [.github/workflows/build.yml](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml)

This document explains the code quality standards and linting infrastructure used in the Bubble Tea repository. It covers the `golangci-lint` configuration, enabled linters, formatting rules, exclusion policies, and how these standards are enforced through CI/CD workflows.

For information about the CI/CD pipeline infrastructure, see [CI/CD Pipeline](https://deepwiki.com/charmbracelet/bubbletea/7.4-cicd-pipeline)
. For testing guidelines and coverage requirements, see [Testing Guidelines](https://deepwiki.com/charmbracelet/bubbletea/7.2-testing-guidelines)
.

Linting System Overview
-----------------------

Bubble Tea uses `golangci-lint` version 2 as its primary code quality tool. The linting system combines static analysis, style checking, and automatic formatting to maintain code quality across the repository. The configuration is centralized in [.golangci.yml1-45](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L1-L45)
 and enforced through GitHub Actions workflows.

**Sources:** [.golangci.yml1-45](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L1-L45)
 [.github/workflows/lint.yml1-9](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L1-L9)

Enabled Linters
---------------

The repository enables 22 specialized linters that cover various aspects of code quality, from security vulnerabilities to style consistency. These linters are explicitly enabled in the configuration file rather than using default presets.

### Linter Catalog

| Linter | Category | Purpose |
| --- | --- | --- |
| `bodyclose` | Resource Management | Ensures HTTP response bodies are closed |
| `exhaustive` | Completeness | Checks that enum switch statements are exhaustive |
| `goconst` | Optimization | Detects repeated strings that could be constants |
| `godot` | Style | Enforces proper comment punctuation |
| `gomoddirectives` | Dependencies | Checks for problematic `go.mod` directives |
| `goprintffuncname` | Correctness | Validates printf-like function names |
| `gosec` | Security | Identifies security vulnerabilities |
| `misspell` | Quality | Detects spelling mistakes |
| `nakedret` | Readability | Flags naked returns in long functions |
| `nestif` | Complexity | Detects deeply nested if statements |
| `nilerr` | Correctness | Finds cases where nil errors are returned incorrectly |
| `noctx` | Context | Ensures HTTP requests use context |
| `nolintlint` | Maintenance | Validates nolint directives |
| `prealloc` | Performance | Suggests slice preallocation |
| `revive` | Style | Fast, configurable linter (replaces golint) |
| `rowserrcheck` | SQL | Checks for unhandled errors from `rows.Err()` |
| `sqlclosecheck` | SQL | Ensures SQL rows/statements are closed |
| `tparallel` | Testing | Detects incorrect usage of `t.Parallel()` |
| `unconvert` | Cleanup | Removes unnecessary type conversions |
| `unparam` | Cleanup | Reports unused function parameters |
| `whitespace` | Style | Enforces consistent whitespace |
| `wrapcheck` | Error Handling | Ensures errors from external packages are wrapped |

Test File Exclusion
-------------------

The linting system is configured to skip test files by default, focusing on production code quality:

    run:
      tests: false
    

This configuration at [.golangci.yml2-3](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L2-L3)
 ensures that linters only analyze production code in `*.go` files, excluding `*_test.go` files. This prevents false positives from test-specific patterns like table-driven tests, mock usage, and deliberate error conditions.

**Sources:** [.golangci.yml2-3](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L2-L3)

Formatters
----------

The repository uses two complementary formatters to maintain consistent code style automatically.

### gofumpt

`gofumpt` is a stricter formatter than `gofmt`, enforcing additional style rules:

*   Consistent import grouping and spacing
*   Standardized field alignment in structs
*   Removal of unnecessary blank lines
*   Consistent receiver naming

### goimports

`goimports` manages import statements by:

*   Automatically adding missing imports
*   Removing unused imports
*   Organizing imports into standard library, external, and local groups

Both formatters are enabled in [.golangci.yml39-42](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L39-L42)
:

**Sources:** [.golangci.yml39-42](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L39-L42)

Exclusion Rules
---------------

The linting system includes two types of exclusions to prevent false positives and handle generated code appropriately.

### Specific Exclusions

The configuration excludes certain linter warnings for specific patterns:

This rule at [.golangci.yml28-32](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L28-L32)
 allows logging functions like `slog.Info()` and `log.Printf()` to be used without context parameters. The `noctx` linter normally requires context usage, but logging is a common exception where context may not be necessary or available.

### Generated Code Handling

Both linters and formatters use the `lax` policy for generated code:

This appears in [.golangci.yml33](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L33-L33)
 for linters and [.golangci.yml43-44](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L43-L44)
 for formatters. The `lax` policy means generated code is analyzed more permissively, acknowledging that code generators may not follow all style guidelines perfectly.

### Common False Positives

The configuration includes preset exclusions for common false positives:

This preset at [.golangci.yml34-35](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L34-L35)
 excludes known patterns that frequently trigger false positives, such as certain error shadowing patterns and acceptable uses of `fmt.Errorf`.

**Sources:** [.golangci.yml28-35](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L28-L35)
 [.golangci.yml43-44](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L43-L44)

Issue Reporting Configuration
-----------------------------

The linting system is configured to report all issues without artificial limits:

These settings at [.golangci.yml36-38](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L36-L38)
 ensure:

*   **`max-issues-per-linter: 0`**: No limit on issues per linter (0 means unlimited)
*   **`max-same-issues: 0`**: All instances of the same issue are reported

This strict reporting policy ensures developers see all code quality issues, promoting comprehensive fixes rather than partial corrections.

**Sources:** [.golangci.yml36-38](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L36-L38)

CI/CD Integration
-----------------

Linting is enforced through GitHub Actions workflows that run on every push and pull request.

**Sources:** [.github/workflows/lint.yml1-9](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L1-L9)

### Workflow Configuration

The lint workflow at [.github/workflows/lint.yml1-9](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L1-L9)
 is minimal, delegating to a shared workflow:

This design pattern:

1.  Centralizes linting configuration in the `charmbracelet/meta` repository
2.  Ensures consistent linting across all Charm Bracelet projects
3.  Allows lint workflow updates without modifying each repository
4.  Reads the local `.golangci.yml` for project-specific rules

**Sources:** [.github/workflows/lint.yml1-9](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L1-L9)

Relationship to Coverage and Build Workflows
--------------------------------------------

While linting focuses on code quality and style, it works alongside other quality assurance workflows:

*   **Lint Workflow**: Validates code style and static analysis ([.github/workflows/lint.yml1-9](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L1-L9)
    )
*   **Coverage Workflow**: Ensures test coverage with race detection ([.github/workflows/coverage.yml1-30](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L1-L30)
    )
*   **Build Workflow**: Verifies the code compiles across Go versions ([.github/workflows/build.yml1-20](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml#L1-L20)
    )

All three workflows must pass for a pull request to be mergeable.

**Sources:** [.github/workflows/lint.yml1-9](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/lint.yml#L1-L9)
 [.github/workflows/coverage.yml1-30](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L1-L30)

Local Development Workflow
--------------------------

Developers can run linting locally before pushing code:

The `--fix` flag automatically applies fixes from `gofumpt` and `goimports`, reformatting code to meet standards. However, most linters only report issues that require manual correction.

**Sources:** [.golangci.yml1-45](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L1-L45)

Linter Selection Rationale
--------------------------

The enabled linters balance comprehensiveness with practical utility:

| Selection Criteria | Examples |
| --- | --- |
| **Security-critical** | `gosec` for vulnerability detection |
| **Resource safety** | `bodyclose`, `sqlclosecheck` for leak prevention |
| **Correctness** | `exhaustive`, `nilerr` for logic errors |
| **Maintainability** | `revive`, `godot` for consistent style |
| **Performance** | `prealloc`, `unconvert` for optimization |

Notably absent linters include those that:

*   Generate excessive false positives in TUI applications
*   Conflict with Bubble Tea's event-driven architecture
*   Are too opinionated for the framework's flexible design

**Sources:** [.golangci.yml5-27](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.golangci.yml#L5-L27)
