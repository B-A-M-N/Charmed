Overview
--------

The Message System is the core communication mechanism in Bubbletea, implementing the "M" in the MVU (Model-View-Update) architecture. Messages carry information about events that occur during program execution—user input, terminal events, command results, and system signals—and are delivered to the `Update` function to trigger state changes.

This page documents the `tea.Msg` interface, the catalog of built-in message types, and how messages flow through the event loop. For details about the `Update` function that processes messages, see [Model Interface](https://deepwiki.com/charmbracelet/bubbletea/2.2-model-interface)
. For information about how commands produce messages asynchronously, see [Command System](https://deepwiki.com/charmbracelet/bubbletea/2.4-command-system)
. For in-depth coverage of specific input message types, see [Keyboard Input](https://deepwiki.com/charmbracelet/bubbletea/3.1-keyboard-input)
, [Mouse Input](https://deepwiki.com/charmbracelet/bubbletea/3.2-mouse-input)
, and [Terminal Events](https://deepwiki.com/charmbracelet/bubbletea/3.3-terminal-events)
.

**Sources:** [tea.go39-41](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L39-L41)
 [tea.go380-505](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L380-L505)

The Msg Interface
-----------------

The `Msg` type is defined as a type alias to `uv.Event` from the ultraviolet library:

This design provides maximum flexibility—applications can define custom message types using any Go type (structs, primitives, function types, etc.). The Update function uses type switches or type assertions to determine the specific message type and handle it accordingly.

Built-in Message Types
----------------------

Bubbletea provides a comprehensive set of built-in message types for common terminal application needs. These are organized into several categories:

### Lifecycle Control Messages

| Message Type | Purpose | Constructor Function | Source |
| --- | --- | --- | --- |
| `QuitMsg` | Signals program should exit gracefully | `Quit()` | User command or signal |
| `SuspendMsg` | Signals program should suspend (Unix only) | `Suspend()` | User command or Ctrl+Z |
| `ResumeMsg` | Signals program resumed from suspension | N/A (automatic) | SIGCONT signal |
| `InterruptMsg` | Signals interrupt (typically Ctrl+C) | `Interrupt()` | SIGINT signal |

These messages control the program lifecycle. When the event loop receives a `QuitMsg`, it exits gracefully at [tea.go746](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L746-L746)

 `SuspendMsg` triggers terminal release on Unix systems at [tea.go752-755](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L752-L755)
 and `ResumeMsg` is sent automatically when the program resumes at [tty.go21](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty.go#L21-L21)

### Input Event Messages

Bubbletea provides structured messages for all types of user input:

#### KeyPressMsg and KeyReleaseMsg

Keyboard input is represented by two distinct message types:

`KeyPressMsg` represents key press events, while `KeyReleaseMsg` represents key release events. The `Key` struct provides detailed information about the key event. The `Code` field contains the key code (e.g., `KeyEnter`, `KeyTab`) or a character rune. The `Text` field contains printable characters when available. Both message types implement the `KeyMsg` interface and provide `String()` and `Keystroke()` methods for convenient matching.

See [Keyboard Input](https://deepwiki.com/charmbracelet/bubbletea/3.1-keyboard-input)
 for comprehensive coverage.

Unknown Messages
----------------

The ultraviolet library that Bubbletea uses for input parsing may produce diagnostic message types for unrecognized input sequences. These messages typically include information about unknown ANSI sequences or malformed input and are useful for debugging terminal compatibility issues.

Applications typically don't handle these messages, but they can be logged or inspected when debugging input-related problems.
