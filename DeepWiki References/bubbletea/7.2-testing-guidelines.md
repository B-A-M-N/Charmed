It covers test organization, common testing patterns for models and programs, synchronization techniques, error handling verification, and CI/CD integration. For information about development environment setup, see [Development Setup](https://deepwiki.com/charmbracelet/bubbletea/7.1-development-setup)

Test Organization
-----------------

Bubbletea organizes tests into several categories:

| Test File | Purpose | Key Features |
| --- | --- | --- |
| `tea_test.go` | Core program lifecycle, message flow, command execution, panic handling | 16 tests using `testModel` with `atomic.Value` |
| `screen_test.go` | View rendering, terminal feature configuration, golden file testing | Uses `testViewModel` and `github.com/charmbracelet/x/exp/golden` |
| `options_test.go` | Program option configuration validation | Tests `ProgramOption` functions |
| `exec_test.go` | External process execution with `ExecProcess` | Platform-specific command tests |

The test suite uses standard Go testing conventions with `*testing.T` and follows table-driven patterns. All tests are designed to be race-detector safe and use custom I/O redirection to avoid terminal interaction.

Golden File Testing
-------------------

Bubbletea uses golden file testing to verify terminal output sequences. The `screen_test.go` file contains tests that capture actual ANSI escape sequences and compare them against reference files stored in `testdata/`.

### Golden File Test Pattern

The golden file pattern ensures that:

*   Terminal output is deterministic across test runs
*   Changes to rendering logic are explicitly visible in git diffs
*   Platform-specific behavior is consistently tested

**Sources:** [screen\_test.go61-158](https://github.com/charmbracelet/bubbletea/blob/f25595a8/screen_test.go#L61-L158)

### testViewModel Structure

The `testViewModel` extends `testModel` to support configurable View options:

**Sources:** [screen\_test.go12-59](https://github.com/charmbracelet/bubbletea/blob/f25595a8/screen_test.go#L12-L59)

### Golden File Test Cases

The test suite covers various terminal feature configurations:

| Test Name | Options | Verifies |
| --- | --- | --- |
| `altscreen` | Enable then disable alt screen | `View.AltScreen` transitions |
| `altscreen_autoexit` | Enable alt screen only | Automatic exit on quit |
| `mouse_cellmotion` | `MouseModeCellMotion` | Mouse cell motion tracking |
| `mouse_allmotion` | `MouseModeAllMotion` | Mouse all motion tracking |
| `mouse_disable` | Enable then set `MouseModeNone` | Mouse mode disabling |
| `cursor_hide` | Default (no cursor) | Cursor hidden |
| `cursor_hideshow` | Toggle `showCursor` | Cursor visibility changes |
| `bp_stop_start` | Toggle `disableBp` | Bracketed paste mode toggling |
| `kitty_stop_startreleases` | Enable `keyReleases` | Kitty keyboard protocol with releases |
| `bg_set_color` | Set `bgColor` to RGB | Background color OSC sequences |

Each test generates a `.golden` file in `testdata/TestViewModel/` containing the exact ANSI escape sequences produced.

The `TestClearMsg` function tests commands that produce specific terminal sequences:

These tests verify that built-in commands produce correct OSC (Operating System Command) sequences for terminal queries.

Test Model Pattern
------------------

The standard test model uses thread-safe state management with `atomic.Value` to handle concurrent access from multiple goroutines:

The `testModel` structure provides a reusable foundation for testing various program behaviors:

*   **Thread-safe state**: Uses `atomic.Value` for `executed` and `counter` fields to avoid race conditions
*   **Message handling**: Responds to custom test messages (`ctxImplodeMsg`, `incrementMsg`, `panicMsg`) and standard messages (`KeyMsg`)
*   **Lifecycle tracking**: The `View` method sets the `executed` flag, allowing tests to detect when rendering has occurred

**Sources:** [tea\_test.go25-61](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea_test.go#L25-L61)

Testing Program Lifecycle
-------------------------

### Basic Program Execution

The simplest test pattern creates a program with custom I/O and runs it to completion:

Key pattern elements:

*   **Custom I/O**: Use `bytes.Buffer` for both input and output via `WithInput()` and `WithOutput()`
*   **Context timeout**: Wrap tests with `context.WithTimeout()` to prevent hanging
*   **Assertions**: Verify output buffer contains expected content

**Sources:** [tea\_test.go63-79](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea_test.go#L63-L79)

### Program Termination Methods

Bubbletea programs can be terminated in three ways, each with different semantics:

| Method | Trigger | Error Returned | Use Case |
| --- | --- | --- | --- |
| `Quit()` | Call from external goroutine | `nil` | Graceful shutdown |
| `Kill()` | Call from external goroutine | `ErrProgramKilled` | Forced termination |
| Context cancellation | Cancel external context | `ErrProgramKilled` + `context.Canceled` | Timeout or external control |

Test patterns for each termination method follow a consistent structure:

1.  Create program with custom I/O
2.  Start goroutine that waits for model execution
3.  Trigger termination method
4.  Verify returned error matches expectation

**Sources:** [tea\_test.go81-100](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea_test.go#L81-L100)
 [tea\_test.go243-270](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea_test.go#L243-L270)
 [tea\_test.go272-299](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea_test.go#L272-L299)

Synchronization with Wait()
---------------------------

The `Wait()` method blocks until the program terminates, enabling coordinated multi-goroutine testing:

This pattern ensures that:

*   Multiple goroutines can safely wait for program termination
*   Termination occurs after all waiters are registered
*   All waiters unblock when the program terminates
*   The test verifies the expected error condition

**Sources:** [tea\_test.go102-150](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea_test.go#L102-L150)

### Race Detection

The race detector is critical for Bubbletea testing because the framework heavily uses goroutines:

| Component | Concurrency Pattern | Race Risk |
| --- | --- | --- |
| Command execution | Each command spawns a goroutine | Commands accessing shared state |
| Event loop | Single goroutine processing messages | External `Send()` calls |
| Input reading | Dedicated `readLoop` goroutine | TTY state during suspend/resume |
| Rendering | `listen()` goroutine with ticker | Concurrent `write()` calls |

All tests pass with `-race` enabled, demonstrating proper synchronization throughout the codebase.

**Sources:** [.github/workflows/coverage.yml24](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/coverage.yml#L24-L24)

Testing Best Practices
----------------------

Based on the patterns in the codebase:

| Practice | Implementation | Benefit |
| --- | --- | --- |
| Thread-safe state | Use `atomic.Value` in test models | Prevents races in concurrent tests |
| Custom I/O | Use `bytes.Buffer` with `WithInput`/`WithOutput` | Avoids terminal interaction, enables output inspection |
| Timeouts | Use `context.WithTimeout` | Prevents hanging tests |
| Synchronization | Use `Wait()` method and channels | Coordinates goroutines cleanly |
| Isolation | Test one behavior per test function | Makes failures easy to diagnose |
| Error wrapping | Check with `errors.Is()` | Supports wrapped error types |
| Race detection | Always run with `-race` in CI | Catches concurrency bugs early |

### Example Test Template

A well-structured test follows this pattern:

    func TestFeature(t *testing.T) {
        // 1. Setup I/O
        var buf bytes.Buffer
        var in bytes.Buffer
        
        // 2. Create model with test state
        m := &testModel{}
        
        // 3. Create program with options
        p := NewProgram(m,
            WithInput(&in),
            WithOutput(&buf),
            WithContext(contextWithTimeout))
        
        // 4. Spawn goroutines for test interactions
        go func() {
            // Wait for execution
            // Trigger behavior
            // Signal completion
        }()
        
        // 5. Run program
        _, err := p.Run()
        
        // 6. Verify results
        if err != expectedError {
            t.Fatal(...)
        }
        // Check model state
        // Check output buffer
    }
