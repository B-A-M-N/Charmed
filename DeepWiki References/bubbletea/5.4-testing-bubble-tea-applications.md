This page provides guidance on testing Terminal User Interface (TUI) applications built with Bubble Tea. It covers the framework's built-in testing utilities, common testing patterns for models and commands, and strategies for handling the unique challenges of testing interactive terminal applications.

For general error handling patterns including panic recovery, see page 5.2 Error Handling and Recovery.

* * *

Testing Challenges in TUI Applications
--------------------------------------

Testing terminal applications presents unique challenges compared to traditional Go applications:

*   **Terminal State**: TUI applications manipulate terminal state (raw mode, alternate screen buffer, cursor position) which is difficult to mock or verify in tests
*   **ANSI Output**: View methods produce ANSI escape sequences that are hard to assert against directly
*   **Asynchronous Input**: Input processing happens on background goroutines with non-deterministic timing
*   **Concurrency**: Commands execute concurrently, making test outcomes timing-dependent
*   **Side Effects**: Real terminal I/O interferes with test runner output and requires cleanup

Bubble Tea addresses these challenges through several testing-focused features:

| Challenge | Solution |
| --- | --- |
| Terminal state manipulation | `nilRenderer` that no-ops all terminal operations |
| Output verification | `WithOutput` option to capture rendered output to a buffer |
| Input control | `WithInput` option to inject programmatic input |
| Timing issues | Synchronous message sending via `Program.Send` |
| Cleanup | Proper lifecycle management via `Quit`, `Kill`, and `Wait` |

Testing with Atomic Operations
------------------------------

When testing concurrent operations, use atomic operations for thread-safe state access and to avoid data races.

The testModel type in the core tests uses `atomic.Value` for thread-safe access:

[tea\_test.go25-28](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea_test.go#L25-L28)

Pattern for atomic-based testing:

1.  Use `atomic.Value` for test state that's accessed from multiple goroutines
2.  Store values with `store()`: `m.counter.Store(1)`
3.  Load values with `Load()`: `i := m.counter.Load()`
4.  Check for nil before type asserting: `if i != nil { val := i.(int) }`
5.  Use `atomic.LoadUint32` and `atomic.AddUint32` for counters

Example usage: [tea\_test.go40-46](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea_test.go#L40-L46)

This pattern avoids:

*   Race conditions when checking model state
*   Data races detected by `go test -race`
*   Mutex deadlocks from improper locking

2.  Commit golden files to version control
3.  Subsequent test runs compare output against golden files
4.  Update golden files when intentional changes occur

Best Practices
--------------

### Test Organization

**Unit Tests vs Integration Tests:**

| Test Type | Scope | Example |
| --- | --- | --- |
| Unit | Individual model methods | `TestModelUpdate`, `TestModelView` |
| Integration | Full program lifecycle | `TestTeaQuit`, `TestTeaContext` |
| Command | Command execution and messages | `TestTeaBatchMsg`, `TestSequenceMsg` |

**Test File Structure:**

*   Place tests in `_test.go` files alongside implementation
*   Use `testModel` structs with atomic fields for thread-safe state
*   Group related tests with `t.Run()` subtests

### Message Design for Testing

Design custom messages to be:

*   **Comparable**: Use simple types that can be compared with `==` or type assertions
*   **Synchronous when possible**: For deterministic testing
*   **Carrying minimal data**: Easier to assert against

### Timing and Synchronization

**Avoid:**

*   Fixed `time.Sleep` durations (brittle, slow)
*   Assuming specific execution order for concurrent operations

**Prefer:**

*   Polling with short sleep intervals: `for { time.Sleep(time.Millisecond); ... }`
*   Atomic operations for state checks
*   Channel-based synchronization when needed

### Program Lifecycle Management

**Always:**

*   Call either `Run()` or `Start()` followed by `Wait()`
*   Handle errors returned from these methods
*   Use `Quit()` for clean shutdown in tests
*   Use `Kill()` only when testing termination behavior

| Filter testing | Message interception | `WithFilter(fn)` |
| Panic testing | Error recovery | Verify `errors.Is(err, ErrProgramPanic)` |
| Batch/Sequence | Command orchestration | Send `BatchMsg`, `sequenceMsg` |
| Wait/Quit/Kill | Lifecycle testing | `p.Wait()`, `p.Quit()`, `p.Kill()` |
