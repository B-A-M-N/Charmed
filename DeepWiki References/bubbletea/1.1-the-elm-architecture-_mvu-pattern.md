Purpose and Scope
-----------------

This page explains how Bubble Tea implements The Elm Architecture (also known as the Model-View-Update or MVU pattern), which forms the foundational design paradigm of the framework. It covers the three core methods (`Init`, `Update`, `View`), the unidirectional data flow, and how messages and commands orchestrate application behavior.

For information about the `tea.Program` lifecycle and runtime execution, see [Program Lifecycle](https://deepwiki.com/charmbracelet/bubbletea/2.1-program-lifecycle)
. For details on message types and the message passing system, see [Message System](https://deepwiki.com/charmbracelet/bubbletea/2.3-message-system)
. For command execution patterns and side effects, see [Command System](https://deepwiki.com/charmbracelet/bubbletea/2.4-command-system)
.

Overview of The Elm Architecture
--------------------------------

The Elm Architecture is a functional programming pattern that structures applications around three key concepts:

1.  **Model**: The application's state
2.  **Update**: A pure function that transforms state in response to messages
3.  **View**: A pure function that renders UI from state

Bubble Tea adapts this pattern for terminal applications, with the `tea.Program` runtime managing the event loop and orchestrating the flow of data through these components.

**Key Characteristics of The Elm Architecture in Bubble Tea**

| Characteristic | Description | Benefit |
| --- | --- | --- |
| Unidirectional data flow | Messages flow from events → Update → Model → View | Predictable state changes |
| Pure functions | Update and View are deterministic | Easy to test and reason about |
| Side effects isolation | I/O operations isolated in Commands | Clean separation of concerns |
| Single source of truth | Model holds all application state | No synchronization issues |
| Immutable updates | Update returns new Model instances | No hidden mutations |

The Model Interface
-------------------

The `Model` interface defines the contract that all Bubble Tea applications must implement. It consists of three methods that form the core of the MVU pattern.

**Interface Definition**

The `Model` interface is defined in [tea.go52-65](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L52-L65)
:

| Method | Signature | Purpose |
| --- | --- | --- |
| `Init` | `Init() Cmd` | Returns the initial command to run on startup |
| `Update` | `Update(Msg) (Model, Cmd)` | Handles messages and returns updated state + optional command |
| `View` | `View() View` | Renders the UI based on current model state |

The model itself can be any Go type that implements this interface, though a struct is most common for storing multiple state fields.

The Init Method
---------------

The `Init` method is the first function called when a Bubble Tea program starts. It returns an optional initial `Cmd` that can perform I/O operations such as loading data, starting timers, or making network requests.

**Signature and Behavior**

    Init() Cmd
    

*   Called once at program startup, before the first render
*   Returns a `Cmd` (function that performs I/O) or `nil` for no initial action
*   The returned command executes asynchronously in a goroutine
*   The command's resulting message is sent to `Update` when complete

**Common Init Patterns**

| Pattern | Example Use Case | Return Value |
| --- | --- | --- |
| No initial I/O | Static UI, forms | `return nil` |
| Start timer | Animations, periodic updates | `return tea.Every(...)` |
| Load data | Fetch initial data | `return loadDataCmd` |
| Multiple commands | Complex initialization | `return tea.Batch(cmd1, cmd2)` |

The Update Method
-----------------

The `Update` method is the heart of The Elm Architecture. It receives messages (events), updates the model state, and optionally returns a command to perform side effects.

**Signature and Behavior**

    Update(Msg) (Model, Cmd)
    

*   Called whenever a message arrives (keyboard input, mouse events, command results, etc.)
*   Receives a `Msg` which can be any type
*   Returns the updated `Model` (usually a modified copy of the receiver)
*   Returns an optional `Cmd` for side effects, or `nil`
*   Must be a pure function for the model transformation (no I/O in Update itself)

The View Method
---------------

The `View` method renders the UI by returning a `tea.View` struct that describes what should be displayed and which terminal features should be enabled.

**Signature and Behavior**

    View() View
    

*   Called after every `Update` to render the current state
*   Must be a pure function (deterministic based on model state)
*   Returns a `tea.View` struct, not a string (though View contains a string)
*   The View struct includes content, cursor position, terminal modes, and more
*   The renderer handles diff-based updates automatically

Unidirectional Data Flow
------------------------

The Elm Architecture enforces unidirectional data flow, where data moves through the system in a single, predictable direction. This eliminates race conditions and makes state changes easy to trace.

**The Complete MVU Cycle**

**Data Flow Properties**

| Property | Description | Implementation |
| --- | --- | --- |
| Single direction | Data flows Model → View → Screen, Events → Update → Model | Enforced by method signatures |
| No callbacks | No event handlers attached to UI elements | Messages sent to central Update |
| Pure transformations | Update and View are deterministic | No I/O in these methods |
| Async isolation | Side effects run in Commands, not Update | Commands execute in goroutines |
| Centralized state | All state in Model, no scattered variables | Single Model instance |

**Message Flow Through tea.Program**

The event loop [tea.go722-863](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L722-L863)
 orchestrates the unidirectional flow:

1.  Message arrives in `p.msgs` channel
2.  Optional filter applied [tea.go737-742](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L737-L742)
    
3.  Message sent to `Update` [tea.go853](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L853-L853)
    
4.  Update returns new model and optional command
5.  Command sent to command channel [tea.go858](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L858-L858)
    
6.  Model rendered to terminal [tea.go861](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L861-L861)
    
7.  Command executes asynchronously, produces new message [tea.go702-714](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L702-L714)

Messages and Commands
---------------------

Messages and Commands are the two fundamental types for handling events and side effects in The Elm Architecture.

**Messages (Msg)**

Messages represent events that have occurred and trigger state updates.

The `Msg` type is defined as an alias [tea.go48-50](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L48-L50)

**Commands (Cmd)**

Commands represent side effects that should be performed. They are functions that execute I/O operations and return a message with the result.

The `Cmd` type is defined [tea.go364-371](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L364-L371)
:

**Command Execution**

Commands are executed asynchronously by the command handler [tea.go679-720](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L679-L720)
:

1.  Command received from Update or Init
2.  Command executed in a new goroutine [tea.go702-714](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L702-L714)
    
3.  Goroutine calls the command function (may block for I/O)
4.  Result message sent via `p.Send(msg)` [tea.go713](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L713-L713)
    
5.  Message enters event loop for processing

**Command Examples**

| Command Type | Code Pattern | Use Case |
| --- | --- | --- |
| No-op | `return nil` | Pure state update |
| Quit | `return tea.Quit` | Exit program |
| HTTP request | `return checkServer` | Network I/O |
| Timer | `return tea.Every(time.Second, tick)` | Periodic updates |
| Batch | `return tea.Batch(cmd1, cmd2)` | Concurrent operations |
| Sequence | `return tea.Sequence(cmd1, cmd2)` | Ordered operations |

**Event Loop Implementation**

The event loop [tea.go722-863](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L722-L863)
 is the core of the MVU runtime:

1.  Selects on multiple channels: `ctx.Done()`, `errs`, `msgs`
2.  Receives message from `p.msgs` channel [tea.go733](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L733-L733)
    
3.  Applies optional filter [tea.go737-742](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L737-L742)
    
4.  Handles special internal messages [tea.go744-850](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L744-L850)
    
5.  Calls `model.Update(msg)` [tea.go853](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L853-L853)
    
6.  Sends command to command channel [tea.go858](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L858-L858)
    
7.  Calls `p.render(model)` [tea.go861](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L861-L861)
    
8.  Repeats until quit or error

Complete Example: MVU in Action
-------------------------------

This section traces a complete message flow through the MVU cycle using the shopping list tutorial as an example.

**Application Structure**

From [tutorials/basics/main.go10-26](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/basics/main.go#L10-L26)

**MVU Flow: User Presses Down Arrow**

**Step-by-Step Code Flow**

1.  **Key Press Event**: Terminal input captured by input reader [tea.go84-94](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L84-L94)
    
2.  **Message Sent**: KeyPressMsg sent to `p.msgs` channel
    
3.  **Update Called**: Event loop receives message and calls Update [tea.go853](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L853-L853)
    
4.  **State Transition**: Update method processes the message [tutorials/basics/main.go32-56](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/basics/main.go#L32-L56)
    :
    
5.  **View Generated**: View method builds UI from updated model [tutorials/basics/main.go59-82](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/basics/main.go#L59-L82)
    :
    
6.  **Rendering**: Renderer updates only changed cells [tea.go867-871](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L867-L871)
    
7.  **Display**: Terminal shows updated UI with cursor moved
    

**Program Entry Point**

The main function [tutorials/basics/main.go84-90](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/basics/main.go#L84-L90)
 creates and runs the program:

**Summary Table**

| Stage | Component | Input | Output | Pure? |
| --- | --- | --- | --- | --- |
| Input | Input Reader | Terminal bytes | KeyPressMsg | N/A |
| Update | model.Update() | KeyPressMsg | (updated model, nil) | Yes |
| View | model.View() | Model state | tea.View | Yes |
| Render | renderer.render() | tea.View | ANSI sequences | No (I/O) |
| Display | Terminal | ANSI sequences | Visual output | N/A |

This cycle repeats for every event, maintaining the unidirectional flow and ensuring predictable state management.
