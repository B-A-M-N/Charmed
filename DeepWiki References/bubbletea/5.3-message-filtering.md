Purpose and Scope
-----------------

Message filtering allows intercepting and transforming messages before they reach the `Update` method of your Model. This feature enables conditional message handling, message transformation, and validation logic to be implemented at the program level rather than within the Model itself.

For information about the general message system in Bubble Tea, see [Message System](https://deepwiki.com/charmbracelet/bubbletea/2.3-message-system)
. For program configuration options, see [Program Options](https://deepwiki.com/charmbracelet/bubbletea/2.5-program-options)
.

Overview
--------

Message filtering is configured using the `WithFilter` program option. A filter function receives every message before it reaches the Model's `Update` method and can:

*   Pass the message through unchanged
*   Transform the message into a different message
*   Block the message entirely by returning `nil`

The filter has access to both the current `Model` state and the incoming `Msg`, allowing context-aware filtering decisions.

**Sources:** [options.go197-230](https://github.com/charmbracelet/bubbletea/blob/f25595a8/options.go#L197-L230)
 [tea.go190](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L190-L190)

Filter Function Signature
-------------------------

The filter function takes two parameters:

*   `Model`: The current model state at the time the message is received
*   `Msg`: The incoming message to be filtered

It returns:

*   `Msg`: The message to process (can be the original or a transformed message)
*   `nil`: Blocks the message from reaching `Update`

**Sources:** [options.go226](https://github.com/charmbracelet/bubbletea/blob/f25595a8/options.go#L226-L226)
 [tea.go190](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L190-L190)

Implementation Details
----------------------

### Event Loop Processing

The filter is called in the main event loop at [tea.go392-398](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L392-L398)
:

    case msg := <-p.msgs:
        // Filter messages.
        if p.filter != nil {
            msg = p.filter(model, msg)
        }
        if msg == nil {
            continue
        }
    

Key behaviors:

1.  The filter is only called if it's not `nil` (i.e., if `WithFilter` was used)
2.  If the filter returns `nil`, the loop continues without processing the message
3.  The filtered/transformed message is used for all subsequent processing
4.  Internal special messages (e.g., `QuitMsg`, `SuspendMsg`) are handled after filtering

**Sources:** [tea.go391-398](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L391-L398)

### Filter Execution Order

Important: The filter executes **before** internal message handling. This means:

*   A filter can block `QuitMsg` to prevent shutdown [tea.go402](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L402-L402)
    
*   A filter can block `SuspendMsg` to prevent suspension [tea.go408](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L408-L408)
    
*   A filter can transform any message type, including internal ones

**Sources:** [tea.go391-486](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L391-L486)

Performance Considerations
--------------------------

### Filter Execution Frequency

The filter function is called **for every message** that enters the event loop:

*   Keyboard input messages
*   Mouse input messages
*   Window resize messages
*   Custom messages from commands
*   Internal messages (Batch, Sequence, etc.)

Therefore, filters should be efficient and avoid expensive operations.

### Message Processing vs. Model Logic

Consider whether logic belongs in the filter or in `Update`:

| Use Filter When | Use Update When |
| --- | --- |
| Logic applies to all messages regardless of model type | Logic is model-specific |
| You need to block messages entirely | You need to update state |
| You want to transform messages before processing | You want to handle message-specific behavior |
| Decision is based on message type/content | Decision requires complex state updates |

**Sources:** [tea.go391-502](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L391-L502)

Interaction with Special Messages
---------------------------------

### Internal Message Handling

The filter runs **before** special internal message handling. This means you can intercept and modify even internal Bubble Tea messages:

| Message Type | Default Behavior | Filter Can |
| --- | --- | --- |
| `QuitMsg` | Returns from event loop | Block or transform |
| `InterruptMsg` | Returns `ErrInterrupted` | Block or transform |
| `SuspendMsg` | Suspends program | Block or transform |
| `BatchMsg` | Executes commands in batch | Block or transform |
| `sequenceMsg` | Executes commands in sequence | Block or transform |

**Sources:** [tea.go391-486](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L391-L486)

### Filter Cannot Prevent

While filters are powerful, they cannot prevent:

*   Context cancellation (`p.ctx.Done()`)
*   Panic recovery (unless `WithoutCatchPanics` is used)
*   Program termination via `Program.Kill()`

These mechanisms operate at a higher level than the message filtering system.

**Sources:** [tea.go384-406](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L384-L406)

Limitations and Caveats
-----------------------

### No Asynchronous Operations

Filter functions should not perform asynchronous operations or start goroutines. They execute synchronously in the event loop and will block message processing.

### No Direct State Mutation

Filters receive a read-only view of the Model. They cannot and should not mutate the model state directly. State changes must occur through the normal Update cycle.

### Order of Operations

The filter sees the model state **before** the current message has been processed. If you need to make decisions based on post-update state, that logic belongs in `Update` itself.

**Sources:** [tea.go391-502](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L391-L502)

Summary
-------

Message filtering provides a powerful mechanism for cross-cutting concerns in Bubble Tea applications. The key characteristics are:

*   **Location**: [options.go226](https://github.com/charmbracelet/bubbletea/blob/f25595a8/options.go#L226-L226)
     for setup, [tea.go392-398](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L392-L398)
     for execution
*   **Signature**: `func(Model, Msg) Msg`
*   **Capabilities**: Transform, pass-through, or block any message
*   **Timing**: Executes before all message handling, including internal messages
*   **Access**: Has read-only access to current model state

Message filtering is ideal for validation, conditional blocking, and message transformation that applies uniformly across your application, while the Update method remains focused on state transitions and business logic.
