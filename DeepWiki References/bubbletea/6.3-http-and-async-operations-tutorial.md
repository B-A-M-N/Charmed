Purpose and Scope
-----------------

This page provides a comprehensive walkthrough of the commands tutorial located in [tutorials/commands](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands)
 which demonstrates how to perform asynchronous I/O operations in Bubble Tea applications. The tutorial covers HTTP requests as the primary example, but the patterns apply to any I/O operation (file access, database queries, network calls, etc.).

For information about the command system architecture and command composition (Batch, Sequence), see [Command System](https://deepwiki.com/charmbracelet/bubbletea/2.4-command-system)
. For general async patterns and goroutine management, see [Concurrency and Goroutines](https://deepwiki.com/charmbracelet/bubbletea/5.1-concurrency-and-goroutines)
.

**Sources:** [tutorials/commands/README.md1-11](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/README.md?plain=1#L1-L11)

* * *

Tutorial Overview
-----------------

The tutorial implements a simple HTTP status checker that:

1.  Makes a GET request to a URL on startup
2.  Reports the HTTP status code when the response arrives
3.  Handles errors gracefully
4.  Exits after displaying the result

The program demonstrates the core pattern for all async operations in Bubble Tea: execute I/O in a `Cmd`, return results as `Msg` types, and handle them in `Update`.

**Sources:** [tutorials/commands/README.md15-16](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/README.md?plain=1#L15-L16)
 [tutorials/commands/main.go1-79](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/main.go#L1-L79)

Model Structure
---------------

The model for this tutorial is minimal, storing only the data needed to render the result:

    type model struct {
        status int    // HTTP status code (0 if not yet received)
        err    error  // Error if request failed
    }
    

The model does not store the URL or any request state because the request happens entirely within a command. The model only tracks the outcome.

The Command Pattern for Async Operations
----------------------------------------

### Command Function Definition

Commands in Bubble Tea are defined as `type Cmd func() Msg`. The `checkServer` function demonstrates the standard pattern for async I/O:

**Key aspects of the `checkServer` command:**

1.  **No arguments**: The function signature is `func() tea.Msg`, taking no parameters
2.  **Blocking I/O**: The HTTP request blocks until complete (timeout or response)
3.  **Error handling**: Errors are converted to `errMsg` types, not panics
4.  **Return message**: Always returns a `Msg` type that represents the result

**Sources:** [tutorials/commands/main.go19-28](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/main.go#L19-L28)
 [tutorials/commands/README.md48-86](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/README.md?plain=1#L48-L86)

* * *

Message Types for Async Results
-------------------------------

The tutorial defines two custom message types to represent the possible outcomes of the HTTP request:

    type statusMsg int           // Success case: HTTP status code
    type errMsg struct{ err error }  // Failure case: wrapped error
    

### The errMsg Pattern

The `errMsg` type implements the `error` interface, making it compatible with error handling code:

    func (e errMsg) Error() string { return e.err.Error() }
    

This pattern allows the message to be used anywhere an `error` is expected while maintaining type information for the `Update` function's type switch.

Initialization Phase
--------------------

The `Init` method returns the command without executing it:

    func (m model) Init() tea.Cmd {
        return checkServer  // Return function reference, don't call it
    }
    

**Important:** Notice that `checkServer` is returned without parentheses. This returns the function itself, not its result. The Bubble Tea runtime will execute the command asynchronously at the appropriate time.

### Command Execution by Runtime

When the `Init` method returns `checkServer`, the Bubble Tea runtime:

1.  Spawns a goroutine via `handleCommands` in [tea.go](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go)
    
2.  Executes the command: `msg := checkServer()`
3.  Sends the resulting `Msg` to the `Update` function via the `msgs` channel
4.  Handles any panics that occur during execution

**Sources:** [tutorials/commands/main.go38-40](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/main.go#L38-L40)
 [tutorials/commands/README.md88-98](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/README.md?plain=1#L88-L98)

Handling Async Results in Update
--------------------------------

The `Update` function uses a type switch to handle the different message types returned by the command:

### Success Case (statusMsg)

When the HTTP request succeeds:

1.  The type assertion `msg := msg.(type)` extracts the `statusMsg`
2.  The status code is stored: `m.status = int(msg)`
3.  The program quits: `return m, tea.Quit`

### Error Case (errMsg)

When the HTTP request fails:

1.  The `errMsg` is assigned to `m.err`
2.  The program quits: `return m, tea.Quit`
3.  The `View` function will render the error

### Quit Command

Both success and error cases return `tea.Quit` as the command because this program has no further work to do. In a long-running application, you would return `nil` or another command to continue operation.

View Function Logic
-------------------

The view renders based on the current model state:

### View States

1.  **Error state** (`m.err != nil`): Display error message
2.  **In-progress state** (`m.status == 0`): Display "Checking..." message
3.  **Complete state** (`m.status > 0`): Display status code and description

The view is called multiple times during program execution:

*   Initially when the program starts (showing "Checking...")
*   After the command returns (showing status or error)
*   After quit is triggered (final render before exit)

Commands with Arguments Pattern
-------------------------------

### Basic Pattern

While `checkServer` takes no arguments, most real-world commands need parameters. The pattern for parameterized commands uses a factory function:

    func checkSomeUrl(url string) tea.Cmd {
        return func() tea.Msg {
            c := &http.Client{Timeout: 10 * time.Second}
            res, err := c.Get(url)
            if err != nil {
                return errMsg{err}
            }
            return statusMsg(res.StatusCode)
        }
    }
    

### How It Works

### Key Points

1.  **Outer function** (`checkSomeUrl`): Takes arguments and returns a `tea.Cmd`
2.  **Inner function** (anonymous): Has signature `func() Msg` and captures arguments via closure
3.  **Closure capture**: The inner function can access variables from the outer function's scope
4.  **I/O in innermost function**: All blocking operations happen in the inner function, which runs asynchronously

**Important:** Do as much work as possible in the innermost function (the one with signature `func() Msg`) because that's the function that runs asynchronously in a goroutine.

**Sources:** [tutorials/commands/README.md183-214](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/README.md?plain=1#L183-L214)

Command Lifecycle and Goroutine Management
------------------------------------------

### Error and Panic Handling

Commands execute in goroutines spawned by `handleCommands`. Each goroutine has panic recovery:

1.  **Panic recovery**: `defer recover()` catches panics in command execution
2.  **Error conversion**: Panics are converted to `ErrProgramPanic` and sent to the errors channel
3.  **Graceful shutdown**: The program can still exit cleanly even if a command panics

This means you should:

*   Return errors as messages (like `errMsg`), not panic
*   Let the framework catch unexpected panics
*   Handle error messages in `Update` like any other message

**Sources:** [commands.go1-223](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands.go#L1-L223)
 [tutorials/commands/README.md100-106](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/README.md?plain=1#L100-L106)

Best Practices for Async Operations
-----------------------------------

### 1\. Always Return Messages for Results

Never try to modify the model directly from a command. Commands run in separate goroutines and have no access to the model. Always return a message that `Update` will handle:

    // ❌ Wrong: Cannot access model
    func badCommand(m *model) tea.Cmd {
        return func() tea.Msg {
            m.status = 200  // Race condition! Model is in another goroutine
            return nil
        }
    }
    
    // ✅ Correct: Return message
    func goodCommand() tea.Cmd {
        return func() tea.Msg {
            // ... do work ...
            return statusMsg(200)
        }
    }
    

### 2\. Handle All Possible Outcomes

Create message types for every possible result:

*   Success messages (data returned)
*   Error messages (operation failed)
*   Partial success messages (if applicable)

### 3\. Keep Commands Focused

Each command should do one thing:

*   One HTTP request
*   One file operation
*   One database query

Use [Batch](https://deepwiki.com/charmbracelet/bubbletea/2.4-command-system)
 to run multiple commands concurrently or [Sequence](https://deepwiki.com/charmbracelet/bubbletea/2.4-command-system)
 to run them in order.

### 4\. Set Appropriate Timeouts

Always set timeouts for I/O operations to prevent the application from hanging:

    c := &http.Client{Timeout: 10 * time.Second}
    

### 5\. Provide User Feedback

The view should indicate when async operations are in progress:

    if m.status == 0 {
        return "Checking..."  // Operation in progress
    }
    return "Status: " + fmt.Sprint(m.status)  // Operation complete
    

### 6\. Allow User Cancellation

Always handle `KeyCtrlC` to let users exit:

    case tea.KeyMsg:
        if msg.Type == tea.KeyCtrlC {
            return m, tea.Quit
        }
    

**Sources:** [tutorials/commands/README.md48-137](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/README.md?plain=1#L48-L137)

Comparison with Standard Go Patterns
------------------------------------

### Traditional Go Approach

    // Traditional goroutine pattern
    func traditionalHTTP() {
        result := make(chan int)
        errChan := make(chan error)
        
        go func() {
            resp, err := http.Get(url)
            if err != nil {
                errChan <- err
                return
            }
            result <- resp.StatusCode
        }()
        
        select {
        case status := <-result:
            // Handle success
        case err := <-errChan:
            // Handle error
        }
    }
    

### Bubble Tea Command Pattern

    // Bubble Tea pattern
    func checkServer() tea.Msg {
        resp, err := http.Get(url)
        if err != nil {
            return errMsg{err}
        }
        return statusMsg(resp.StatusCode)
    }
    
    func (m model) Update(msg tea.Msg) (tea.Model, tea.Cmd) {
        switch msg := msg.(type) {
        case statusMsg:
            // Handle success
        case errMsg:
            // Handle error
        }
    }
    

### Advantages of the Bubble Tea Pattern

| Aspect | Traditional Go | Bubble Tea Commands |
| --- | --- | --- |
| **Channel management** | Manual channel creation and cleanup | Handled by framework |
| **Error handling** | Separate error channel | Unified message type system |
| **Synchronization** | Manual select statements | Automatic via Update function |
| **State updates** | Manual locking if shared state | Single-threaded Update function |
| **Testability** | Complex mocking of goroutines | Simple command function testing |
| **Composability** | Manual coordination of multiple operations | Built-in Batch and Sequence |

**Sources:** [tutorials/commands/README.md48-137](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tutorials/commands/README.md?plain=1#L48-L137)

Testing Async Commands
----------------------

### Testing Command Execution

Commands are regular functions that return messages, making them easy to test:

    // Test from commands_test.go
    func TestTick(t *testing.T) {
        expected := "tick"
        msg := Tick(time.Millisecond, func(t time.Time) Msg {
            return expected
        })()  // Execute the command synchronously
        if expected != msg {
            t.Fatalf("expected a msg %v but got %v", expected, msg)
        }
    }
    

For HTTP commands, use standard Go testing patterns:

*   Mock HTTP responses with `httptest`
*   Test error cases by forcing failures
*   Verify correct message types are returned

### Testing Update Logic

Test the `Update` function with different message types:

    func TestUpdateWithStatus(t *testing.T) {
        m := model{}
        m, cmd := m.Update(statusMsg(200))
        
        if m.status != 200 {
            t.Error("status not set correctly")
        }
        // Verify cmd is tea.Quit
    }
    

For complete testing patterns, see [Testing Bubble Tea Applications](https://deepwiki.com/charmbracelet/bubbletea/5.4-testing-bubble-tea-applications)
.

**Sources:** [commands\_test.go19-27](https://github.com/charmbracelet/bubbletea/blob/f25595a8/commands_test.go#L19-L27)

Summary
-------

The commands tutorial demonstrates the fundamental pattern for all async operations in Bubble Tea:

1.  **Define the model** to store operation results
2.  **Create a command function** (`func() Msg`) that performs I/O
3.  **Define message types** for all possible outcomes
4.  **Return the command** from `Init` or `Update`
5.  **Handle messages** in `Update` with a type switch
6.  **Render state** in `View` based on whether operation is complete

This pattern works for any I/O operation: HTTP requests, file access, database queries, timers, or external process communication. The framework handles all goroutine management, panic recovery, and message delivery automatically.
