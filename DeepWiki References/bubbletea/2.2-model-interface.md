Purpose and Scope
-----------------

The `Model` interface is the core abstraction in Bubble Tea that defines how applications manage state and respond to events. This document explains the interface contract, its three required methods (`Init`, `Update`, `View`), and how to implement application state according to The Elm Architecture pattern.

For information about the lifecycle that executes these methods, see [Program Lifecycle](https://deepwiki.com/charmbracelet/bubbletea/2.1-program-lifecycle)
. For details about the message types passed to `Update`, see [Message System](https://deepwiki.com/charmbracelet/bubbletea/2.3-message-system)
. For information about the commands returned by `Init` and `Update`, see [Command System](https://deepwiki.com/charmbracelet/bubbletea/2.4-command-system)
.

The Model Interface Contract
----------------------------

The `Model` interface defines three methods that every Bubble Tea application must implement. These methods form a complete cycle: initialization, state updates in response to events, and rendering.

**Diagram: Model Interface Method Signatures**

The interface definition in the codebase:

| Method | Signature | Purpose |
| --- | --- | --- |
| `Init` | `Init() Cmd` | First function called; returns optional initial command |
| `Update` | `Update(Msg) (Model, Cmd)` | Handles incoming messages; updates state and returns optional command |
| `View` | `View() View` | Renders UI based on current state; called after every Update |

**Sources:** [tea.go52-65](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L52-L65)

Model Lifecycle Integration
---------------------------

The Model interface integrates with the Program lifecycle to create the complete event-driven application flow.

**Diagram: Model Methods in Program Lifecycle**

### Execution Flow

1.  **Initialization Phase** ([tea.go1099-1115](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L1099-L1115)
    )
    
    *   `NewProgram` creates Program with initial model
    *   `Run` initializes terminal
    *   `model.Init()` is called
    *   Initial command executes (if not nil)
    *   `model.View()` renders initial UI
2.  **Event Loop Phase** ([tea.go724-863](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L724-L863)
    )
    
    *   Program waits for messages
    *   Optional filter processes message
    *   `model.Update(msg)` handles message
    *   Returned command executes asynchronously
    *   `model.View()` renders updated UI
    *   Renderer diffs and outputs to terminal
    *   Loop continues
3.  **Termination Phase** ([tea.go1131-1154](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tea.go#L1131-L1154)
    )
    
    *   `QuitMsg` or context cancellation
    *   Final `model.View()` render
    *   Terminal state restored

### State Flow Through Model

**Diagram: State Transitions Through Update Method**

### Immutability Pattern

While Go doesn't enforce immutability, the pattern encourages treating the model as immutable:

*   `Update` receives the current model
*   `Update` modifies the model (typically in-place for structs)
*   `Update` returns the modified model
*   The runtime uses the returned model for subsequent operations

This pattern ensures clear state ownership and predictable behavior.

Key Implementation Principles
-----------------------------

### 1\. Model Owns State

The model struct contains all application state. External code cannot directly modify it; only the `Update` method changes state.

### 2\. Pure Rendering

The `View` method should be pure: it reads the model's state and returns a view representation without side effects. All I/O happens through commands.

### 3\. Message-Driven Updates

State changes occur only in response to messages. The `Update` method is the single point where state transitions happen.

### 4\. Command-Based Side Effects

Side effects (I/O, timers, async operations) are triggered by returning commands from `Init` and `Update`, not executed directly.

### 5\. Declarative Views

The `View` method declares what the UI should look like based on current state. The renderer handles the imperative details of updating the terminal.
