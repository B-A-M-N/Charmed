Purpose and Scope
-----------------

This document covers platform-specific implementations in Bubble Tea, focusing on how the framework handles differences between Unix-like systems (Linux, macOS, BSD variants) and Windows. The primary areas of divergence are terminal initialization, signal handling, and console API usage.

For general terminal management concepts, see [Terminal Management](https://deepwiki.com/charmbracelet/bubbletea/4.3-terminal-management)
. For cross-platform input handling patterns, see [Cross-platform Input](https://deepwiki.com/charmbracelet/bubbletea/3.4-cross-platform-input)
.

* * *

Platform Abstraction Strategy
-----------------------------

Bubble Tea uses Go build tags to conditionally compile platform-specific code while maintaining a common interface. The framework splits terminal functionality into three files:

| File | Scope | Build Tags |
| --- | --- | --- |
| `tty.go` | Platform-agnostic terminal operations | None (always compiled) |
| `tty_unix.go` | Unix-specific implementations | `darwin`, `linux`, `freebsd`, `netbsd`, `openbsd`, `solaris`, `aix`, `zos`, `dragonfly` |
| `tty_windows.go` | Windows-specific implementations | `windows` |

The common file [\`tty.go\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60tty.go%60)
() defines the high-level terminal lifecycle methods that all platforms must support:

*   `suspend()` - Process suspension handling
*   `initTerminal()` - Terminal initialization coordination
*   `restoreTerminalState()` - Terminal state restoration
*   `initCancelReader()` - Input reader setup
*   `checkResize()` - Window size detection

Platform-specific files implement:

*   `initInput()` - Console mode configuration
*   `openInputTTY()` - TTY device access
*   `suspendProcess()` - Process suspension mechanism
*   `suspendSupported` - Capability flag

**Sources:** [tty.go1-142](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty.go#L1-L142)
 [tty\_unix.go1-50](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty_unix.go#L1-L50)
 [tty\_windows.go1-69](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty_windows.go#L1-L69)

Console Mode Initialization
---------------------------

### Unix Systems

On Unix-like systems, terminal initialization focuses on setting raw mode to disable line buffering and canonical input processing. The [\`initInput()\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60initInput()%60)
() function performs the following steps:

1.  **Input Terminal Detection** - Checks if `p.input` implements `term.File` and is a terminal using `term.IsTerminal()`
2.  **Raw Mode Configuration** - Calls `term.MakeRaw()` to disable line buffering, echo, and signal generation
3.  **State Preservation** - Stores the previous terminal state in `p.previousTtyInputState` for restoration on exit
4.  **Output Terminal Detection** - Similarly checks if `p.output` is a terminal and stores reference in `p.ttyOutput`

The raw mode changes specific terminal attributes:

*   **ICANON** - Disables canonical input (line buffering)
*   **ECHO** - Disables character echo
*   **ISIG** - Disables signal generation for Ctrl+C and Ctrl+Z
*   **IEXTEN** - Disables extended input processing

**Sources:** [tty\_unix.go15-30](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty_unix.go#L15-L30)

### Windows Systems

Windows terminal initialization requires enabling Virtual Terminal (VT) processing on both input and output handles. The [\`initInput()\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60initInput()%60)
() implementation is more complex due to Windows Console API requirements:

1.  **Input Handle Configuration**
    
    *   Gets current console mode using `windows.GetConsoleMode()`
    *   Enables `ENABLE_VIRTUAL_TERMINAL_INPUT` flag to support ANSI escape sequences
    *   Stores previous state for restoration
2.  **Output Handle Configuration**
    
    *   Gets current output console mode
    *   Enables `ENABLE_VIRTUAL_TERMINAL_PROCESSING` flag to process ANSI output sequences
    *   Stores previous state separately

| Console Mode Flag | Purpose | Platform |
| --- | --- | --- |
| `ENABLE_VIRTUAL_TERMINAL_INPUT` | Enables ANSI escape sequence parsing on input | Windows |
| `ENABLE_VIRTUAL_TERMINAL_PROCESSING` | Enables ANSI escape sequence processing on output | Windows |
| Raw mode (via `tcsetattr`) | Disables canonical mode and echo | Unix |

**Sources:** [tty\_windows.go14-55](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty_windows.go#L14-L55)

* * *

Signal Handling
---------------

### Unix Process Suspension

Unix systems support process suspension via `SIGTSTP` (Ctrl+Z) and resumption via `SIGCONT`. The [\`suspendProcess()\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60suspendProcess()%60)
() function implements the suspension mechanism:

1.  **Signal Channel Setup** - Creates a channel to receive `SIGCONT` signals
2.  **Notify Registration** - Registers for `SIGCONT` notifications using `signal.Notify()`
3.  **Group Suspension** - Sends `SIGTSTP` to the entire process group (PID 0) using `syscall.Kill(0, syscall.SIGTSTP)`
4.  **Blocking Wait** - Blocks on the channel until the process receives `SIGCONT`

The suspension is initiated by the [\`suspend()\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60suspend()%60)
() method in the common code:

**Flow:**

1.  Release terminal (disable mouse mode, show cursor, exit alt screen, restore terminal state)
2.  Suspend process group (blocks until `SIGCONT`)
3.  Restore terminal state
4.  Send `ResumeMsg` to the program's event loop

**Sources:** [tty\_unix.go40-49](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty_unix.go#L40-L49)
 [tty.go13-23](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty.go#L13-L23)

### Windows Limitation

Windows does not support the Unix signal model for process suspension. The [\`suspendProcess()\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60suspendProcess()%60)
() function is a no-op, and the constant [\`suspendSupported\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60suspendSupported%60)
() is set to `false`.

| Feature | Unix | Windows | Constant |
| --- | --- | --- | --- |
| Process Suspension | Supported via `SIGTSTP`/`SIGCONT` | Not supported | `suspendSupported = true/false` |
| Suspend Signal | `syscall.SIGTSTP` | N/A | N/A |
| Resume Signal | `syscall.SIGCONT` | N/A | N/A |

**Sources:** [tty\_windows.go66-68](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty_windows.go#L66-L68)

TTY Device Access
-----------------

### Unix TTY Device

On Unix systems, the TTY device is accessed via the special file `/dev/tty`, which always refers to the controlling terminal of the current process. The [\`openInputTTY()\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60openInputTTY()%60)
() function:

This is used when the program needs direct terminal access, regardless of stdin/stdout redirection.

**Sources:** [tty\_unix.go32-38](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty_unix.go#L32-L38)

### Windows Console Device

Windows uses the special device name `CONIN$` to access the console input buffer directly. The [\`openInputTTY()\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60openInputTTY()%60)
() function:

| Platform | Device Name | Open Mode | Purpose |
| --- | --- | --- | --- |
| Unix | `/dev/tty` | `os.Open` (read-only) | Access controlling terminal |
| Windows | `CONIN$` | `os.OpenFile` (read-write) | Access console input buffer |

**Sources:** [tty\_windows.go58-64](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty_windows.go#L58-L64)

Terminal State Restoration
--------------------------

Both platforms must restore terminal state when the program exits. The common [\`restoreInput()\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60restoreInput()%60)
() method handles this by calling `term.Restore()` on saved file descriptors:

The restoration process:

1.  **Input State** - Restores the terminal input state saved during initialization (removes raw mode on Unix, removes VT input on Windows)
2.  **Output State** - Restores the output state (removes VT processing on Windows)

The [\`restoreTerminalState()\`](https://github.com/charmbracelet/bubbletea/blob/f25595a8/%60restoreTerminalState()%60)
() method coordinates higher-level restoration:

*   Disables bracketed paste mode
*   Shows cursor
*   Disables mouse tracking
*   Disables focus reporting
*   Exits alternate screen buffer (with 10ms delay to allow terminal to catch up)
*   Calls `restoreInput()` to restore console modes

**Sources:** [tty.go41-75](https://github.com/charmbracelet/bubbletea/blob/f25595a8/tty.go#L41-L75)

Summary of Platform Differences
-------------------------------

| Feature | Unix Implementation | Windows Implementation |
| --- | --- | --- |
| **Raw Mode** | `term.MakeRaw()` via `tcsetattr` | `term.MakeRaw()` + `SetConsoleMode` with VT flags |
| **Input Processing** | Disable canonical mode, echo, signals | Enable `ENABLE_VIRTUAL_TERMINAL_INPUT` |
| **Output Processing** | Standard TTY output | Enable `ENABLE_VIRTUAL_TERMINAL_PROCESSING` |
| **TTY Device** | `/dev/tty` | `CONIN$` |
| **Process Suspension** | `SIGTSTP`/`SIGCONT` signals | Not supported |
| **State Storage** | Single `previousTtyInputState` | Separate input and output states |
| **Build Tags** | `darwin`, `linux`, `freebsd`, etc. | `windows` |
