Repository Structure
--------------------

Bubbletea uses a **three-module architecture** to separate core framework code from examples and tutorials. This separation keeps the main module lightweight while allowing examples to depend on richer UI libraries.

### Module Organization

**Module Diagram: Repository Structure**

**Sources:** [go.mod1-27](https://github.com/charmbracelet/bubbletea/blob/f25595a8/go.mod#L1-L27)
 [.github/workflows/build.yml14-19](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/build.yml#L14-L19)

The main module at [go.mod1](https://github.com/charmbracelet/bubbletea/blob/f25595a8/go.mod#L1-L1)
 declares `github.com/charmbracelet/bubbletea` with Go 1.24.0 as the minimum version. It includes 16 direct and indirect dependencies, focusing on terminal control (`charmbracelet/x/ansi`, `charmbracelet/x/term`), styling (`charmbracelet/lipgloss`), and input handling (`muesli/cancelreader`).

Examples and tutorials use **replace directives** in their `go.mod` files to reference the local version of the main module during development. This allows contributors to test changes across all modules without publishing intermediate versions.

### Key Dependencies

| Dependency | Purpose | Module |
| --- | --- | --- |
| `charmbracelet/lipgloss` | Styling and layout | Main |
| `charmbracelet/x/ansi` | ANSI escape sequence generation | Main |
| `charmbracelet/x/term` | Terminal state management | Main |
| `muesli/cancelreader` | Graceful input cancellation | Main |
| `golang.org/x/sys` | Low-level system calls | Main |
| `charmbracelet/bubbles` | Pre-built UI components | Examples |
| `charmbracelet/glamour` | Markdown rendering | Examples |
| `charmbracelet/harmonica` | Spring animations | Examples |

**Sources:** [go.mod5-14](https://github.com/charmbracelet/bubbletea/blob/f25595a8/go.mod#L5-L14)

Contribution Workflow
---------------------

Contributors should follow these general steps when making changes to Bubbletea:

1.  **Fork and clone** the repository
2.  **Set up the development environment** with all three modules (see [Development Setup](https://deepwiki.com/charmbracelet/bubbletea/7.1-development-setup)
    )
3.  **Make changes** to the appropriate module(s)
4.  **Run tests** locally to verify correctness (see [Testing Guidelines](https://deepwiki.com/charmbracelet/bubbletea/7.2-testing-guidelines)
    )
5.  **Run linters** to ensure code quality (see [Code Quality Standards](https://deepwiki.com/charmbracelet/bubbletea/7.3-code-quality-standards)
    )
6.  **Submit a pull request** with a clear description of changes

### Local Development

When working on the repository locally, the replace directives in `examples/go.mod` and `tutorials/go.mod` ensure that examples and tutorials automatically use the local version of the main module. This means you can:

*   Edit core framework code in the root directory
*   Immediately test those changes in example applications
*   Verify backwards compatibility with tutorial code

No manual linking or version pinning is required during development.

### Automated Module Maintenance

The repository includes automation to keep module dependencies synchronized. The `examples.yml` workflow automatically runs `go mod tidy` on the examples and tutorials modules when dependencies change, committing the updated `go.mod` and `go.sum` files back to the repository.

**Sources:** [.github/workflows/examples.yml28-34](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L28-L34)

All jobs use `charmbracelet/meta/.github/workflows/build.yml@main` as a reusable workflow, providing standardized build steps across Charm projects.

### Examples Maintenance Workflow

The [.github/workflows/examples.yml1-38](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L1-L38)
 workflow automatically synchronizes module dependencies when changes are pushed to the master branch. It performs:

1.  `cd ./examples && go mod tidy` - Update example dependencies
2.  `cd ./tutorials && go mod tidy` - Update tutorial dependencies
3.  Auto-commit changes with message `"chore: go mod tidy tutorials and examples"`

The workflow requires `contents: write` permissions [.github/workflows/examples.yml19-20](https://github.com/charmbracelet/bubbletea/blob/f25595a8/.github/workflows/examples.yml#L19-L20)
 to commit directly to master. This automation ensures that examples and tutorials remain buildable after dependency updates without manual intervention.

Testing Requirements
--------------------

Contributors should ensure that:

1.  **All existing tests pass** locally before submitting a pull request
2.  **New functionality includes tests** covering the added behavior
3.  **Tests run without race conditions** (`go test -race`)
4.  **Test coverage does not decrease** significantly

The repository uses standard Go testing conventions. Tests are located in files with `_test.go` suffixes throughout the codebase. For detailed testing patterns and utilities, see [Testing Guidelines](https://deepwiki.com/charmbracelet/bubbletea/7.2-testing-guidelines)
.

Development Best Practices
--------------------------

### Module Dependency Management

When adding or updating dependencies:

1.  **Add to the appropriate module** - Core dependencies go in the main `go.mod`, example-specific dependencies go in `examples/go.mod`
2.  **Run `go mod tidy`** to clean up and update checksums
3.  **Verify compatibility** by running all tests across modules
4.  **Document breaking changes** if updating major versions

The examples workflow will automatically tidy the examples and tutorials modules after your changes are merged.

### Platform Considerations

Bubbletea supports both Unix and Windows platforms with platform-specific implementations:

*   Terminal initialization differs between `tty_unix.go` and `tty_windows.go`
*   Windows requires Virtual Terminal mode enablement
*   Unix supports suspend/resume via `SIGTSTP`/`SIGCONT`

When making platform-specific changes, ensure that both implementations remain functional. For more details, see [Platform-Specific Features](https://deepwiki.com/charmbracelet/bubbletea/5.6-platform-specific-features)
.

### Documentation

Contributors should:

1.  **Add godoc comments** for exported types, functions, and methods
2.  **Update example code** if API changes affect usage patterns
3.  **Include code examples** in documentation for new features
4.  **Update the README** if adding significant functionality
