Keyboard Input
==============

Relevant source files

*   [key.go](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go)
    
*   [mouse.go](https://github.com/charmbracelet/bubbletea/blob/f25595a8/mouse.go)
    

Purpose and Scope
-----------------

This document covers keyboard input handling in Bubbletea, including key event message types, key structures, key codes, modifiers, and keyboard protocol support. It explains how keyboard events flow through the framework and how to handle them in your application's `Update` method.

For mouse input handling, see [Mouse Input](https://deepwiki.com/charmbracelet/bubbletea/3.2-mouse-input)
. For other terminal events like window resize and focus events, see [Terminal Events](https://deepwiki.com/charmbracelet/bubbletea/3.3-terminal-events)
. For platform-specific input differences, see [Cross-platform Input](https://deepwiki.com/charmbracelet/bubbletea/3.4-cross-platform-input)
.

* * *

Keyboard Event Flow
-------------------

Keyboard input in Bubbletea flows from raw terminal bytes through parsing layers to typed messages delivered to your application:

**Keyboard Event Processing Pipeline**: Raw terminal input flows through the ultraviolet library's parser, which converts ANSI escape sequences into structured key events. Bubbletea wraps these events in message types and delivers them to the application's Update method.

Sources: [key.go1-372](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L1-L372)
 high-level architecture diagrams

* * *

Key Event Message Types
-----------------------

Bubbletea provides three types for keyboard events, defined in [key.go190-264](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L190-L264)
:

### KeyPressMsg

Represents a key press event. This is the most common keyboard message type.

**KeyPressMsg Structure**: A KeyPressMsg wraps a Key struct containing detailed information about the key press, including the text representation, modifiers, key code, and enhanced protocol fields.

Sources: [key.go190-221](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L190-L221)

### KeyReleaseMsg

Represents a key release event. Only available when using the Kitty Keyboard Protocol or Windows Console API with enhanced keyboard reporting.

Sources: [key.go223-255](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L223-L255)

### KeyMsg Interface

A generic interface that matches both `KeyPressMsg` and `KeyReleaseMsg`, useful for handling keyboard events regardless of press or release type:

Sources: [key.go257-264](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L257-L264)

The Key Structure
-----------------

The `Key` struct ([key.go302-343](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L302-L343)
) contains detailed information about a keyboard event:

| Field | Type | Description | Availability |
| --- | --- | --- | --- |
| `Text` | `string` | Actual characters received; non-empty for printable characters | Always |
| `Mod` | `KeyMod` | Modifier keys (Ctrl, Alt, Shift, etc.) | Always |
| `Code` | `rune` | Key code (special key constant or character) | Always |
| `ShiftedCode` | `rune` | Shifted key pressed by user (e.g., 'A' when shift+a) | Kitty/Windows only |
| `BaseCode` | `rune` | Key according to PC-101 layout (for international keyboards) | Kitty/Windows only |
| `IsRepeat` | `bool` | Whether key is auto-repeating from being held down | Kitty/Windows only |

### Key Methods

The `Key` type and its message wrappers provide methods for convenient key matching ([key.go345-371](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L345-L371)
):

**`String()`**: Returns the textual representation. Prefers the printable character over the keystroke representation. For example, returns `"?"` instead of `"shift+/"` on a US keyboard.

**`Keystroke()`**: Returns the canonical keystroke representation with modifiers in a consistent order: ctrl, alt, shift, meta, hyper, super. For example, `"ctrl+shift+alt+a"`.

Sources: [key.go302-371](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L302-L371)

Modifier Keys
-------------

Modifiers are represented as a bitmask in the `KeyMod` field. The specific modifier constants are defined in the ultraviolet library. When multiple modifiers are present, the `Keystroke()` method formats them in a consistent order:

1.  `ctrl`
2.  `alt`
3.  `shift`
4.  `meta`
5.  `hyper`
6.  `super`

For example, pressing Ctrl+Shift+Alt+A always produces `"ctrl+shift+alt+a"`, never `"shift+ctrl+alt+a"` or any other permutation.

Sources: [key.go203-215](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L203-L215)
 [key.go236-248](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L236-L248)

Keyboard Protocol Support
-------------------------

Bubbletea supports enhanced keyboard features through the Kitty Keyboard Protocol and Windows Console API. These protocols provide additional information not available in traditional terminal input.

**Keyboard Protocol Capabilities**: Standard terminal input provides basic key information. Enhanced protocols (Kitty and Windows Console API) add fields for shifted codes, base layout codes, repeat detection, and key release events.

### Enhanced Fields

**`ShiftedCode`** ([key.go315-324](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L315-L324)
): Contains the actual shifted key pressed by the user. When pressing Shift+A or with Caps Lock enabled, `ShiftedCode` is `'A'` while `Code` remains `'a'`. For non-Latin keyboards like Arabic, `ShiftedCode` contains the unshifted key.

**`BaseCode`** ([key.go326-335](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L326-L335)
): Contains the key according to the standard PC-101 keyboard layout. On international keyboards, this is the key that would be pressed if the keyboard were set to US PC-101 layout. For example, pressing 'q' on a French AZERTY keyboard sets `BaseCode` to `'q'`.

**`IsRepeat`** ([key.go337-342](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L337-L342)
): Indicates whether the key is being held down and sending repeated events. This allows distinguishing between a single key press and auto-repeat from holding a key.

### Key Release Events

With enhanced protocols enabled, the terminal sends both key press and key release events. Your application receives both `KeyPressMsg` and `KeyReleaseMsg` for each key interaction, enabling more sophisticated input handling like detecting long presses or chord combinations.

Sources: [key.go315-342](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L315-L342)

### Key.Text vs Key.Code

The `Text` field ([key.go303-306](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L303-L306)
) is populated only for printable characters (shifted or unshifted). It is empty for:

*   Special keys like `KeyEnter`, `KeyTab`, `KeyEscape`
*   Key combinations with modifiers (e.g., Ctrl+A)
*   Non-printable control characters

For special keys, always check `Code` rather than `Text`.

### Best Practices

1.  **Use KeyPressMsg for most cases**: Unless you specifically need key release detection, match against `KeyPressMsg` rather than the generic `KeyMsg` interface.
    
2.  **Prefer String() for simple matching**: When matching simple keys and common shortcuts, `msg.String()` provides a clean, readable syntax.
    
3.  **Use Code for special keys**: Always check `key.Code` for special keys like arrow keys, function keys, and Enter/Escape.
    
4.  **Check Text for printable characters**: When handling text input, check `key.Text` to get the actual character(s) typed.
    
5.  **Use Keystroke() for debugging**: The `Keystroke()` method provides a canonical representation useful for logging and debugging key combinations.
    

Sources: [key.go266-301](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L266-L301)

Key Event Type Relationships
----------------------------

**Key Type Hierarchy**: KeyPressMsg and KeyReleaseMsg are type aliases of Key, both implementing the KeyMsg interface. The Key struct wraps ultraviolet's uv.Key type, providing Bubbletea-specific methods for convenient key matching.

Integration with ultraviolet Library
------------------------------------

Bubbletea delegates key parsing and processing to the ultraviolet library (`github.com/charmbracelet/ultraviolet`). The key types in [key.go1-372](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L1-L372)
 are thin wrappers that re-export ultraviolet's types and constants:

*   Key constants (`KeyEnter`, `KeyF1`, etc.) are aliases to `uv.KeyEnter`, `uv.KeyF1`
*   The `Key` struct fields match `uv.Key`
*   String and Keystroke methods delegate to `uv.Key` methods

This design allows Bubbletea to benefit from ultraviolet's sophisticated ANSI sequence parsing while maintaining a clean API surface for applications.

Sources: [key.go1-372](https://github.com/charmbracelet/bubbletea/blob/f25595a8/key.go#L1-L372)
