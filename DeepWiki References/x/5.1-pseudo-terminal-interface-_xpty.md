Pseudo-Terminal Interface (xpty)
================================

The `xpty` package provides a unified, cross-platform interface for creating and managing pseudo-terminals (PTYs). It abstracts the differences between Unix PTYs (via `creack/pty`) and Windows ConPTY, enabling terminal applications to spawn and control processes with consistent terminal semantics across operating systems.

This document covers the PTY abstraction layer and platform-specific implementations. For terminal I/O control primitives, see section 5.3. For Windows Console API bindings, see section 5.2.

Package Dependencies
--------------------

The `xpty` package relies on the following key dependencies:

| Dependency | Purpose | Platform |
| --- | --- | --- |
| `github.com/creack/pty v1.1.24` | Unix PTY creation and management | Unix/Linux/macOS |
| `github.com/charmbracelet/x/conpty` | Windows ConPTY wrapper | Windows |
| `github.com/charmbracelet/x/term` | Terminal file operations | Cross-platform |
| `github.com/charmbracelet/x/termios` | Terminal I/O control | Unix/Linux/macOS |
| `golang.org/x/sys` | Low-level system calls | Cross-platform |

Sources: [xpty/go.mod1-11](https://github.com/charmbracelet/x/blob/7642919e/xpty/go.mod#L1-L11)

Core PTY Interface
------------------

The `Pty` interface defines the fundamental operations for pseudo-terminal management, combining standard I/O operations with terminal-specific functionality. It embeds `term.File` and `io.ReadWriteCloser` to provide both low-level file descriptor access and standard I/O operations.

### Pty Interface Methods

| Method | Signature | Purpose |
| --- | --- | --- |
| `Read` | `Read([]byte) (int, error)` | Read data from the PTY master |
| `Write` | `Write([]byte) (int, error)` | Write data to the PTY master |
| `Close` | `Close() error` | Close the PTY and release resources |
| `Fd` | `Fd() uintptr` | Get the file descriptor for the PTY |
| `Resize` | `Resize(width, height int) error` | Change the PTY window size |
| `Size` | `Size() (width, height int, error)` | Get the current PTY window size |
| `Name` | `Name() string` | Get the PTY device name |
| `Start` | `Start(*exec.Cmd) error` | Spawn a process attached to the PTY |

### Pty Interface Implementation Architecture

The `NewPty()` factory function automatically selects the appropriate implementation (`UnixPty` or `ConPty`) based on the runtime platform using build tags.

Platform-Specific Implementations
---------------------------------

### Unix PTY Implementation

The `UnixPty` struct manages traditional Unix pseudo-terminals using master and slave file descriptor pairs. It leverages the `creack/pty` package for PTY creation and provides additional functionality for window management and process control.

#### UnixPty Structure and Operations

| Field/Operation | Type/Implementation | Purpose |
| --- | --- | --- |
| `master` | `*os.File` | Master side of the PTY (application side) |
| `slave` | `*os.File` | Slave side of the PTY (process side) |
| `Master()` | Returns `*os.File` | Access the master file descriptor |
| `Slave()` | Returns `*os.File` | Access the slave file descriptor |
| `SlaveName()` | Returns `string` | Get the slave device path (e.g., `/dev/pts/0`) |
| `Control()` | Accepts `func(fd uintptr)` | Execute operations on the master fd |
| `SetWinsize()` | Uses `termios.SetWinsize()` | Change terminal window dimensions via `TIOCSWINSZ` |
| `Size()` | Uses `termios.GetWinsize()` | Retrieve current window dimensions via `TIOCGWINSZ` |

#### Unix PTY Data Flow

The `Control()` method provides safe access to the underlying file descriptor for system calls, allowing direct manipulation of terminal attributes through the `termios` package. The `Start()` method attaches the slave PTY to the child process's standard I/O streams.

### Windows ConPTY Implementation

The `ConPty` struct wraps the Windows Console PTY (ConPTY) API through the `conpty` package. ConPTY provides pseudo-terminal functionality on Windows 10 version 1809 (Build 17763) and later.

#### ConPty Structure

| Field/Operation | Type/Implementation | Purpose |
| --- | --- | --- |
| `pty` | `*conpty.ConPty` | Underlying ConPTY handle |
| `Name()` | Returns device name | Returns "ConPTY" identifier |
| `Start()` | Uses `conpty.Spawn()` | Spawns process with ConPTY attachment |
| `Resize()` | Calls `conpty.Resize()` | Changes console buffer size |
| Read/Write | Delegated to `conpty.ConPty` | I/O through input/output pipes |

#### Windows ConPTY Process Flow

The Windows implementation includes special process management logic because Go's standard `exec.Cmd.Start()` does not properly handle ConPTY-attached processes. The `Start()` method uses the ConPTY package's `Spawn()` function, which internally calls `CreateProcess` with the appropriate `STARTUPINFOEX` configuration, then manually assigns the resulting process handle to `cmd.Process`.

Sources: [xpty/conpty.go16-40](https://github.com/charmbracelet/x/blob/7642919e/xpty/conpty.go#L16-L40)
 [xpty/conpty\_windows.go15-37](https://github.com/charmbracelet/x/blob/7642919e/xpty/conpty_windows.go#L15-L37)

--------------------------------

### Cross-Platform Process Spawning

The `Start(*exec.Cmd)` method provides platform-appropriate process spawning with PTY attachment. The implementation differs significantly between Unix and Windows due to their underlying PTY mechanisms.

### Process Waiting and Termination

The `WaitProcess()` function addresses platform differences in process termination handling. This utility function is necessary because ConPTY-spawned processes on Windows do not work correctly with the standard `cmd.Wait()` method.

#### WaitProcess() Implementation Strategy

| Platform | Implementation Approach | Details |
| --- | --- | --- |
| Unix/Linux/macOS | `cmd.Wait()` | Uses standard Go process waiting mechanism |
| Windows | `process.Wait()` in goroutine | Manual wait on `cmd.Process` with channel signaling |
| Context Cancellation | `process.Kill()` | Forcefully terminates process when context is cancelled |
| Error Handling | Context error check | Returns `ctx.Err()` on cancellation, process exit code otherwise |

#### WaitProcess() Control Flow

The Windows-specific implementation launches a goroutine to wait on the process while the main thread blocks on either context cancellation or process completion. This pattern ensures proper cleanup and context cancellation support.

Factory and Configuration
-------------------------

### PTY Creation and Options

The `NewPty()` factory function provides compile-time platform detection (via build tags) and runtime PTY instantiation. The function accepts initial dimensions and optional configuration parameters.

#### NewPty() Factory Function

The factory function internally delegates to platform-specific constructors:

*   Unix: `NewUnixPty(width, height)`
*   Windows: `NewConPty(width, height, opts...)`

#### Options and Functional Configuration

The package uses the functional options pattern for configuration:

| Type/Function | Purpose | Platform Usage |
| --- | --- | --- |
| `Options` struct | Holds configuration parameters | Both |
| `Options.Flags uint32` | ConPTY creation flags | Windows only |
| `PtyOption` function type | Modifies `Options` | Both |
| `WithPtyOptions(flags uint32)` | Sets ConPTY flags | Windows |
