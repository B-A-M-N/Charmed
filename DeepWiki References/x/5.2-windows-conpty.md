Windows ConPTY
==============

The `conpty` package implements Windows Console Pseudo-terminal support, providing a modern pseudo-terminal interface for Windows systems. This package enables terminal applications to spawn processes with full terminal capabilities on Windows using the Windows ConPTY API introduced in Windows 10.

For the cross-platform PTY abstraction that uses this package, see page 5.1. For Windows-specific Console API bindings, see page 3.2.

Overview
--------

Windows Console Pseudo-terminal (ConPTY) is a Windows 10+ feature that provides UNIX-style pseudo-terminal functionality on Windows. The `conpty` package wraps the Windows `CreatePseudoConsole` API and associated functions to provide a Go-friendly interface for creating and managing pseudo-consoles.

**Title:** ConPTY Architecture

Sources: [conpty/conpty\_windows.go17-26](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L17-L26)

ConPty Structure
----------------

The `ConPty` struct represents a Windows Console Pseudo-terminal session and manages all the resources required for pseudo-terminal operation.

### Core Components

**Title:** ConPty Struct Components

Sources: [conpty/conpty\_windows.go19-26](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L19-L26)

| Field | Type | Purpose |
| --- | --- | --- |
| `hpc` | `*windows.Handle` | Handle to the pseudo-console created by `CreatePseudoConsole` |
| `inPipeRead` | `windows.Handle` | Input pipe read end (slave side) |
| `inPipeWrite` | `windows.Handle` | Input pipe write end (master side) |
| `outPipeRead` | `windows.Handle` | Output pipe read end (master side) |
| `outPipeWrite` | `windows.Handle` | Output pipe write end (slave side) |
| `attrList` | `*windows.ProcThreadAttributeListContainer` | Process thread attribute list for spawning |
| `size` | `windows.Coord` | Current terminal dimensions (X=columns, Y=rows) |
| `closeOnce` | `sync.Once` | Ensures cleanup happens only once |

The `ConPty` struct implements both `io.Reader` and `io.Writer` interfaces, allowing it to be used in standard Go I/O patterns.

Initialization

### Pipe Creation

The `CreatePipes` function creates the four pipe handles required for pseudo-console communication:

**Title:** Pipe Creation Flow

Sources: [conpty/conpty\_windows.go34-50](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L34-L50)

The pipes are created with inheritable handles (`InheritHandle: 1`) to allow child processes to inherit them. The function returns all four handle values as `uintptr` for flexibility.

Two constructor functions are available:

#### NewWithPipes()

Creates a new `ConPty` with existing pipe handles:

This is useful when integrating with existing I/O infrastructure or when you need more control over pipe creation.

Sources: [conpty/conpty\_windows.go78-117](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L78-L117)

**Title:** ConPty Initialization Sequence

Sources: [conpty/conpty\_windows.go55-117](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L55-L117)

The initialization process:

1.  **Pipe Creation**: Four pipes are created for bidirectional communication
2.  **Pseudo-Console Creation**: `windows.CreatePseudoConsole` creates the actual ConPTY with the slave ends of the pipes
3.  **Attribute List**: A process thread attribute list is allocated and configured with the pseudo-console handle
4.  **Handle Management**: The PTY-slave ends (input read, output write) can be closed after creation as the ConPTY duplicates them

Sources: [conpty/conpty\_windows.go97-114](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L97-L114)

I/O Operations
--------------

The `ConPty` struct implements `io.Reader` and `io.Writer` for standard Go I/O operations.

### Write Operations

Writing to the pseudo-console sends data to the spawned process's standard input:

**Title:** Write Operation Flow

Sources: [conpty/conpty\_windows.go164-168](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L164-L168)

Implementation uses `windows.WriteFile` to write to the master end of the input pipe (`inPipeWrite`), which is connected to the process's standard input through the pseudo-console.

### Read Operations

Reading from the pseudo-console retrieves data from the spawned process's standard output:

**Title:** Read Operation Flow

Sources: [conpty/conpty\_windows.go171-175](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L171-L175)

Implementation uses `windows.ReadFile` to read from the master end of the output pipe (`outPipeRead`), which receives data from the process's standard output.

### Resize Operations

The pseudo-console can be dynamically resized:

This calls `windows.ResizePseudoConsole` to update the console dimensions and stores the new size in the `ConPty.size` field. The spawned process receives a resize notification through the standard Windows console mechanisms.

----------------

The `Spawn` method creates a new process attached to the pseudo-console, handling all the complexities of Windows process creation with extended startup information.

*   **args**: Command-line arguments (including program name as first element)
*   **attr**: Process attributes (can be nil for defaults)
*   **Returns**: Process ID, process handle, and any error

### Spawn Components

#### Executable Resolution

The `lookExtensions` function resolves the executable name:

*   If `name` is a basename, it searches in `$PATH`
*   If `attr.Dir` is specified, handles relative and absolute paths correctly
*   Adds `.exe` extension if needed on Windows

Sources: [conpty/exec\_windows.go22-50](https://github.com/charmbracelet/x/blob/7642919e/conpty/exec_windows.go#L22-L50)

#### Command Line Composition

The command line is built using either:

*   `attr.Sys.CmdLine` if explicitly provided
*   `windows.ComposeCommandLine(args)` for automatic escaping and quoting

Sources: [conpty/conpty\_windows.go226-235](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L226-L235)

#### Environment Preparation

The `execEnvDefault` function prepares the environment:

*   Uses `attr.Env` if provided
*   Otherwise, calls `windows.CreateEnvironmentBlock` if `attr.Sys.Token` is set
*   Falls back to `syscall.Environ()` for current process environment
*   Critical environment variables like `SYSTEMROOT` are automatically added via `addCriticalEnv`

Sources: [conpty/conpty\_windows.go245-250](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L245-L250)

#### StartupInfoEx Configuration

The process is created with extended startup information:

The `ProcThreadAttributeList` contains the pseudo-console handle, instructing Windows to attach the new process to the ConPTY.

Sources: [conpty/conpty\_windows.go252-279](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L252-L279)

#### Process Creation

The function calls either:

*   `windows.CreateProcessAsUser` if `attr.Sys.Token` is provided (for impersonation)
*   `windows.CreateProcess` otherwise

Creation flags include:

*   `CREATE_UNICODE_ENVIRONMENT`: Use Unicode environment block
*   `EXTENDED_STARTUPINFO_PRESENT`: Enable extended startup info with attribute list
*   Additional flags from `attr.Sys.CreationFlags` if provided

Resource Cleanup
----------------

The `Close` method cleans up all resources associated with the pseudo-console:

**Title:** Cleanup Sequence

Sources: [conpty/conpty\_windows.go125-141](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L125-L141)

The cleanup process:

1.  **Thread Safety**: Uses `sync.Once` to ensure cleanup happens exactly once
2.  **Slave Handles**: Closes the PTY-slave ends of pipes (input read, output write)
3.  **Attribute List**: Deletes the process thread attribute list
4.  **Pseudo-Console**: Closes the pseudo-console handle with `ClosePseudoConsole`
5.  **Master Handles**: Closes the PTY-master ends of pipes (input write, output read)
6.  **Error Aggregation**: Uses `errors.Join` to return any errors from handle closures

The slave ends are closed first to signal EOF to any running processes, while the master ends are closed last to allow final I/O operations to complete.

Sources: [conpty/conpty\_windows.go125-141](https://github.com/charmbracelet/x/blob/7642919e/conpty/conpty_windows.go#L125-L141)

Integration Points
------------------

### Input System Integration

The Windows platform support integrates with the broader input processing system by providing the low-level event structures and constants that the input module uses for Windows-specific input handling.

### Dependency Management

Both modules maintain minimal external dependencies:

*   **windows module**: Only depends on `golang.org/x/sys/windows`
*   **conpty module**: Depends on `golang.org/x/sys/windows` and `github.com/charmbracelet/x/errors`

*   **Auto-commit**: Commits generated code changes automatically
