Monorepo Structure
------------------

The charmbracelet/x repository is organized as a **monorepo** containing 33+ independent Go modules. Each module is a self-contained package with its own `go.mod` file, version management, and dependency tree. This architecture allows packages to evolve independently while being co-located in a single repository for easier development and cross-package integration testing.

### Directory Layout

The repository follows a flat, category-based directory structure:

| Directory Pattern | Category | Examples |
| --- | --- | --- |
| `/[package]` | Core and stable packages | `ansi`, `cellbuf`, `input`, `vt`, `xpty`, `term` |
| `/exp/[package]` | Experimental packages | `exp/teatest`, `exp/golden`, `exp/higherorder` |
| `/examples` | Integration demonstrations | Examples with their own `go.mod` |
| `/.github` | CI/CD infrastructure | `workflows/`, `dependabot.yml` |
| `/scripts` | Automation tools | `scripts/dependabot` |

Each package directory contains:

*   `go.mod` - Module definition and dependencies
*   `go.sum` - Dependency checksums
*   `*.go` - Source files
*   `*_test.go` - Unit tests
*   Platform-specific files with build tags (e.g., `*_windows.go`, `*_unix.go`)

Automated Dependency Management
-------------------------------

The repository uses **GitHub Dependabot** to automate dependency updates across all 33+ modules. A custom shell script generates the Dependabot configuration dynamically by scanning for `go.mod` files, ensuring new modules are automatically included in dependency management.

### Dependabot Configuration Generation

The configuration is generated by [scripts/dependabot1-44](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L1-L44)
:

**Diagram 2: Dependabot Configuration Generation Pipeline**

The script performs the following operations:

1.  **Header Generation** - Writes a static header with GitHub Actions configuration [scripts/dependabot3-23](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L3-L23)
    
2.  **Module Discovery** - Uses `find . -type f -name go.mod` to locate all Go modules [scripts/dependabot25](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L25-L25)
    
3.  **Path Processing** - Extracts directory paths using `cut -f2- -d'/'` and sorts them [scripts/dependabot25](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L25-L25)
    
4.  **Configuration Template** - Generates a standardized update block for each module [scripts/dependabot26-42](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L26-L42)
    

Sources: [scripts/dependabot1-44](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L1-L44)
 [.github/dependabot.yml1-6](https://github.com/charmbracelet/x/blob/7642919e/.github/dependabot.yml#L1-L6)

### Dependabot Update Configuration

Each module receives identical dependency update settings:

| Parameter | Value | Purpose |
| --- | --- | --- |
| `package-ecosystem` | `"gomod"` | Manages Go module dependencies |
| `schedule.interval` | `"weekly"` | Checks for updates every Monday |
| `schedule.time` | `"05:00"` EST | Runs at 5 AM Eastern Time |
| `labels` | `["dependencies"]` | Tags PRs for filtering |
| `commit-message.prefix` | `"chore"` | Conventional commit format |
| `groups.all.patterns` | `["*"]` | Groups all updates into single PR |

The grouping strategy (`groups.all.patterns: ["*"]`) consolidates all dependency updates for a module into a single pull request, reducing PR noise while maintaining atomic updates per module.

Sources: [.github/dependabot.yml23-38](https://github.com/charmbracelet/x/blob/7642919e/.github/dependabot.yml#L23-L38)

### Workflow Components

#### Build Job

The `build` job runs on a matrix of operating systems to ensure cross-platform compatibility:

Steps executed [.github/workflows/vttest.yml23-30](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L23-L30)
:

1.  `actions/checkout@v6` - Checks out repository code
2.  `actions/setup-go@v6` - Installs Go version from module's `go.mod`
3.  `go build -v ./...` - Compiles all packages
4.  `go test -race -v ./...` - Runs tests with race detection

The `working-directory` is set to the module directory [.github/workflows/vttest.yml20-21](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L20-L21)
 ensuring tests run in the correct context.

Code Generation Pipelines
-------------------------

The repository includes automated code generation for configuration files and platform-specific code. The primary generator is the Dependabot configuration script.

### Dependabot Configuration Generator

Script: [scripts/dependabot1-44](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L1-L44)

**Input**: File system scan for `go.mod` files  
**Output**: `.github/dependabot.yml`  
**Execution**: Manual or via CI (when adding new modules)

The script ensures:

*   **Consistency** - All modules use identical update schedules and settings
*   **Maintainability** - Adding a new module automatically includes it in dependency management
*   **Documentation** - The generated file includes a warning header [.github/dependabot.yml1](https://github.com/charmbracelet/x/blob/7642919e/.github/dependabot.yml#L1-L1)
    

Key implementation details:

The script uses shell redirects to write the file incrementally, first creating the header [scripts/dependabot3-23](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L3-L23)
 then appending module configurations [scripts/dependabot25-43](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L25-L43)

Sources: [scripts/dependabot1-44](https://github.com/charmbracelet/x/blob/7642919e/scripts/dependabot#L1-L44)

Development Workflow Integration
--------------------------------

### Adding a New Module

When adding a new module to the repository:

1.  **Create module directory** with `go.mod`
2.  **Run dependency script**: `./scripts/dependabot`
3.  **Create workflow file** in `.github/workflows/[module].yml`
4.  **Update README.md** with module description and documentation link
5.  **Commit all changes** including auto-generated `dependabot.yml`

The automated systems will then:

*   Scan the module for dependency updates weekly
*   Run CI/CD on every push and PR
*   Auto-merge dependency updates after successful tests
*   Report coverage to Codecov

### Module Independence

Each module's independence is enforced through:

| Mechanism | Implementation |
| --- | --- |
| Separate `go.mod` | Each module manages its own dependency tree |
| Per-module workflows | CI runs only for changed modules via path filters |
| Individual versioning | Modules can release new versions independently |
| Separate Dependabot config | Dependencies updated per module, not monolithically |

This architecture allows packages like `cellbuf` to depend on `ansi` without coupling their release cycles or dependency trees.

### Testing Integration

The monorepo structure enables comprehensive integration testing through the `examples/` module [README.md126](https://github.com/charmbracelet/x/blob/7642919e/README.md?plain=1#L126-L126)
 which can import and test multiple packages together. The [exp/teatest](https://deepwiki.com/charmbracelet/x/7.2-testing-framework-(teatest))
 package specifically tests cross-package integration for Bubble Tea programs.

Sources: [README.md1-70](https://github.com/charmbracelet/x/blob/7642919e/README.md?plain=1#L1-L70)

Continuous Integration Best Practices
-------------------------------------

The repository's CI/CD implementation demonstrates several best practices:

### Matrix Testing

*   **Platform coverage**: Ubuntu, macOS, Windows [.github/workflows/vttest.yml15-17](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L15-L17)
    
*   **Go version compatibility**: Uses version from `go.mod` [.github/workflows/vttest.yml25-27](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L25-L27)
    
*   **Race detection**: All tests run with `-race` flag [.github/workflows/vttest.yml30](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L30-L30)
    

### Dependency Caching

*   **Go module cache**: `cache: true` with `cache-dependency-path` [.github/workflows/vttest.yml27-28](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L27-L28)
    
*   **Reduces build times**: Caches `go.mod`/`go.sum` artifacts between runs

### Workflow Optimization

*   **Path filtering**: Only runs when module files change [.github/workflows/vttest.yml9-11](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L9-L11)
    
*   **Job dependencies**: `dependabot` job requires `build` success [.github/workflows/vttest.yml33](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L33-L33)
    
*   **Conditional execution**: Dependabot job only runs for bot PRs [.github/workflows/vttest.yml38](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L38-L38)
    

### Security

*   **Minimal permissions**: Jobs request only required permissions [.github/workflows/vttest.yml35-37](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L35-L37)
    
*   **Pinned actions**: Uses versioned action references (`@v6`, `@v5`) [.github/workflows/vttest.yml23-24](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L23-L24)
    
*   **Token scoping**: Uses `GITHUB_TOKEN` with appropriate scopes [.github/workflows/vttest.yml49](https://github.com/charmbracelet/x/blob/7642919e/.github/workflows/vttest.yml#L49-L49)
