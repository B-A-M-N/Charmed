This page documents the ANSI sequence parser state machine and text manipulation utilities in the `ansi` package. It covers how ANSI escape sequences are decoded from raw byte streams, how text display width is calculated for terminal output, and how text can be wrapped, truncated, and manipulated while preserving ANSI formatting codes.

For information about generating ANSI control sequences, see [Terminal Control Sequences](https://deepwiki.com/charmbracelet/x/4.2-terminal-control-sequences)
. For styling and color handling, see [Styling and Colors](https://deepwiki.com/charmbracelet/x/4.3-styling-and-colors)
.

* * *

Purpose and Scope
-----------------

The `ansi` package provides two fundamental capabilities:

1.  **Parsing**: A state machine-based decoder that extracts ANSI escape sequences and printable characters from byte streams
2.  **Text Manipulation**: ANSI-aware functions for measuring, wrapping, and truncating text that preserve escape codes and handle wide characters correctly

All functions account for:

*   ANSI escape sequences (CSI, OSC, DCS, etc.) which have zero display width
*   Wide characters (East Asian characters, emojis) which occupy multiple cells
*   Unicode grapheme clusters (combining characters, multi-codepoint emojis)

* * *

Parser State Machine
--------------------

### Overview

The parser implements a DEC ANSI-compatible state machine that processes input byte-by-byte, transitioning between states to recognize escape sequences and printable text.

**Sources:** [ansi/parser\_decode.go10-22](https://github.com/charmbracelet/x/blob/7642919e/ansi/parser_decode.go#L10-L22)
 [ansi/parser\_decode.go124-352](https://github.com/charmbracelet/x/blob/7642919e/ansi/parser_decode.go#L124-L352)

### DecodeSequence Function

The primary decoding function processes input and returns individual sequences or grapheme clusters:

| Function | Width Method | Description |
| --- | --- | --- |
| `DecodeSequence` | GraphemeWidth | Uses grapheme clustering (Unicode UAX #29) |
| `DecodeSequenceWc` | WcWidth | Uses wcwidth-style calculation |

**Function Signature:**

**Parameters:**

*   `b`: Input data (string or \[\]byte)
*   `state`: Current parser state (starts at `NormalState` = 0)
*   `p`: Optional Parser for collecting parameters and data

**Returns:**

*   `seq`: The parsed sequence or grapheme cluster
*   `width`: Display width in cells (0 for escape sequences)
*   `n`: Number of bytes consumed
*   `newState`: New parser state for next iteration

**Sources:** [ansi/parser\_decode.go24-72](https://github.com/charmbracelet/x/blob/7642919e/ansi/parser_decode.go#L24-L72)
 [ansi/parser\_decode.go124-352](https://github.com/charmbracelet/x/blob/7642919e/ansi/parser_decode.go#L124-L352)

### Parser States

| State | Value | Purpose |
| --- | --- | --- |
| `NormalState` | 0   | Ground state, awaiting next character |
| `PrefixState` | \-  | Collecting CSI/DCS prefix characters (`<=>?`) |
| `ParamsState` | \-  | Collecting numeric parameters |
| `IntermedState` | \-  | Collecting intermediate bytes (space to `/`) |
| `EscapeState` | \-  | Processing ESC sequence |
| `StringState` | \-  | Collecting string data (OSC, DCS, APC, SOS, PM) |

**Sources:** [ansi/parser\_decode.go10-22](https://github.com/charmbracelet/x/blob/7642919e/ansi/parser_decode.go#L10-L22)

Width Calculation Methods

Text Width Measurement
----------------------

### StringWidth Functions

Calculate the display width of text, accounting for ANSI escape sequences:

| Function | Method | Description |
| --- | --- | --- |
| `StringWidth(s string) int` | GraphemeWidth | Recommended for modern terminals |
| `StringWidthWc(s string) int` | WcWidth | Legacy compatibility |
| `Strip(s string) string` | N/A | Removes all ANSI escape codes |

**Algorithm:**

**Sources:** [ansi/width.go60-107](https://github.com/charmbracelet/x/blob/7642919e/ansi/width.go#L60-L107)

**Implementation Details:**

The width calculation uses the ANSI parser state machine to skip over escape sequences, then measures only printable content. For UTF-8 sequences, it calls `FirstGraphemeCluster` to get the correct width.

**Sources:** [ansi/width.go78-107](https://github.com/charmbracelet/x/blob/7642919e/ansi/width.go#L78-L107)

Text Truncation
---------------

### Truncate Functions

Truncate text to a specified display width, optionally adding a tail:

| Function | Direction | Method |
| --- | --- | --- |
| `Truncate(s, length, tail)` | Right truncation | GraphemeWidth |
| `TruncateWc(s, length, tail)` | Right truncation | WcWidth |
| `TruncateLeft(s, n, prefix)` | Left truncation | GraphemeWidth |
| `TruncateLeftWc(s, n, prefix)` | Left truncation | WcWidth |
| `Cut(s, left, right)` | Extract range `[left, right)` | GraphemeWidth |\
| `CutWc(s, left, right)` | Extract range `[left, right)` | WcWidth |\
\
**Truncate Algorithm:**\
\
**Key Behavior:**\
\
1.  ANSI escape sequences are always preserved, even when truncating\
2.  The tail width is subtracted from the available length\
3.  Once the width limit is reached, only escape sequences are collected (to preserve styling)\
\
**Sources:** [ansi/truncate.go48-166](https://github.com/charmbracelet/x/blob/7642919e/ansi/truncate.go#L48-L166)\
\
### ByteToGraphemeRange\

Converts byte positions to grapheme-aware character positions:\
\
This is useful for converting byte-level ranges (from regex matches, etc.) to display positions that can be used with `Cut`, `Truncate`, or `TruncateLeft`.\

Text Wrapping\
-------------\
\
### Wrapping Functions\
\
Three wrapping strategies are available, each with GraphemeWidth and WcWidth variants:\
\
| Function | Strategy | Breakpoints |\
| --- | --- | --- |\
| `Hardwrap(s, limit, preserveSpace)` | Break anywhere | Immediate wrap at limit |\
| `Wordwrap(s, limit, breakpoints)` | Word boundaries | Space, custom breakpoints |\
| `Wrap(s, limit, breakpoints)` | Hybrid | Word boundaries + hard break fallback |\
\
**All variants:**\
\
*   Preserve ANSI escape sequences\
*   Handle wide characters correctly\
*   Respect newlines in input\
*   Support custom breakpoint characters\
\
**Sources:** [ansi/wrap.go1-473](https://github.com/charmbracelet/x/blob/7642919e/ansi/wrap.go#L1-L473)\
\
### Wrapping Strategy Comparison\
\

### Hardwrap Example\
\
When `preserveSpace` is true, leading spaces on wrapped lines are kept. Otherwise, they're trimmed.\
\
**Sources:** [ansi/wrap.go35-116](https://github.com/charmbracelet/x/blob/7642919e/ansi/wrap.go#L35-L116)\
\
### Wordwrap Example\
\
**Breakpoint characters** are treated as natural breaking points. A hyphen (`-`) is always considered a breakpoint. The breakpoints string specifies additional single-width rune characters.\
\
**Sources:** [ansi/wrap.go146-265](https://github.com/charmbracelet/x/blob/7642919e/ansi/wrap.go#L146-L265)\
\
### Wrap Example\
\
This is the most intelligent wrapping mode, preferring word boundaries but ensuring text never exceeds the limit.\
\
**Sources:** [ansi/wrap.go293-463](https://github.com/charmbracelet/x/blob/7642919e/ansi/wrap.go#L293-L463)\
\

### Command and Parameter Packing\
\
Commands are packed into a single `int` to avoid allocations:\
\
    +----------------+----------------+----------------+----------------+\
    |   (unused)     |    Prefix      | Intermediate   |     Final      |\
    |    8 bits      |    8 bits      |    8 bits      |    8 bits      |\
    +----------------+----------------+----------------+----------------+\
    \
\
Parameters use the most significant bit as `HasMoreFlag` to indicate sub-parameters:\
\
    +--------------------------------+--------------------------------+\
    |          HasMore Flag          |         Parameter Value        |\
    |           1 bit                |           31 bits              |\
    +--------------------------------+--------------------------------+\
    \
\
**Sources:** [ansi/parser\_decode.go45-51](https://github.com/charmbracelet/x/blob/7642919e/ansi/parser_decode.go#L45-L51)\

Performance Considerations\
--------------------------\
\
### Zero-Copy Operations\
\
The `DecodeSequence` function uses generic constraints (`[T string | []byte]`) to avoid unnecessary allocations when working with byte slices:\
\
**Sources:** [ansi/parser\_decode.go70-72](https://github.com/charmbracelet/x/blob/7642919e/ansi/parser_decode.go#L70-L72)\
\
### Buffering Strategy\
\
Text manipulation functions use `strings.Builder` or `bytes.Buffer` to minimize allocations:\
\
*   Pre-sized buffers when output size is known\
*   Single allocation for result string\
*   Reuse of parser state across iterations\
\
**Sources:** [ansi/truncate.go78-79](https://github.com/charmbracelet/x/blob/7642919e/ansi/truncate.go#L78-L79)\
 [ansi/wrap.go42-43](https://github.com/charmbracelet/x/blob/7642919e/ansi/wrap.go#L42-L43)\
\
### State Machine Efficiency\
\
The parser state machine uses a lookup table (`parser.Table`) for O(1) state transitions, avoiding conditional branches:\
\
**Sources:** [ansi/wrap.go56](https://github.com/charmbracelet/x/blob/7642919e/ansi/wrap.go#L56-L56)\
 [ansi/width.go36](https://github.com/charmbracelet/x/blob/7642919e/ansi/width.go#L36-L36)\
