Purpose and Scope
-----------------

The `teatest` package provides a specialized test harness for testing Bubble Tea (`tea.Model`) applications in an isolated, controlled environment. It allows developers to programmatically send messages to models, capture output, wait for specific conditions, and verify results against golden files without requiring an actual terminal.

For general terminal input event handling, see [Input Package](https://deepwiki.com/charmbracelet/x/3.1-input-package)
. For terminal rendering, see [cellbuf Package](https://deepwiki.com/charmbracelet/x/2.1-cellbuf-package)
.

* * *

Overview
--------

The `teatest` package wraps `tea.Program` instances to enable deterministic testing of interactive terminal applications. It intercepts input/output streams, provides utilities for sending messages, waiting for conditions, and comparing output against expected results.

### Key Capabilities

| Capability | Description | Primary Functions |
| --- | --- | --- |
| Model Testing | Run `tea.Model` instances in isolated environment | `NewTestModel()` |
| Message Injection | Programmatically send messages and keyboard input | `Send()`, `Type()` |
| Output Capture | Read program output without terminal | `Output()`, `FinalOutput()` |
| Condition Waiting | Poll output until specific conditions are met | `WaitFor()` |
| State Inspection | Access final model state after execution | `FinalModel()` |
| Golden Files | Compare output against reference files | `RequireEqualOutput()` |
| Multi-Program | Test communication between multiple models | Program interface |

**Sources:** [exp/teatest/teatest.go1-306](https://github.com/charmbracelet/x/blob/7642919e/exp/teatest/teatest.go#L1-L306)

Core Components
---------------

### TestModel Structure

The `TestModel` struct is the central test harness that wraps a `tea.Program` and provides controlled access to its inputs and outputs.

**Key Fields:**

*   `program`: The wrapped `tea.Program` instance running in a goroutine
*   `in`: Input buffer where messages are sent (via `Type()`)
*   `out`: Thread-safe output buffer capturing terminal sequences
*   `modelCh`: Channel receiving the final model after program completion
*   `doneCh`: Signals when the program has finished execution

**Sources:** [exp/teatest/teatest.go105-117](https://github.com/charmbracelet/x/blob/7642919e/exp/teatest/teatest.go#L105-L117)

Thread Safety
-------------

### safeReadWriter Implementation

Since `tea.Program` writes to output from its own goroutine while tests may read from it concurrently, `teatest` provides thread-safe I/O:

**Implementation:**

*   `Read()` [exp/teatest/teatest.go294-298](https://github.com/charmbracelet/x/blob/7642919e/exp/teatest/teatest.go#L294-L298)
    : Acquires read lock (`RLock()`)
*   `Write()` [exp/teatest/teatest.go301-305](https://github.com/charmbracelet/x/blob/7642919e/exp/teatest/teatest.go#L301-L305)
    : Acquires write lock (`Lock()`)
*   Created by `safe()` wrapper [exp/teatest/teatest.go283-285](https://github.com/charmbracelet/x/blob/7642919e/exp/teatest/teatest.go#L283-L285)
    
*   Applied to output buffer in `NewTestModel()` [exp/teatest/teatest.go123](https://github.com/charmbracelet/x/blob/7642919e/exp/teatest/teatest.go#L123-L123)
    

**Rationale:**

*   Program goroutine writes output continuously
*   Test goroutine may call `Output()` or `WaitFor()` at any time
*   Read/write locks prevent data races

**Sources:** [exp/teatest/teatest.go283-305](https://github.com/charmbracelet/x/blob/7642919e/exp/teatest/teatest.go#L283-L305)
 [exp/teatest/teatest.go123](https://github.com/charmbracelet/x/blob/7642919e/exp/teatest/teatest.go#L123-L123)
