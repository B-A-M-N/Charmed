Purpose and Scope
-----------------

The `input` package provides event-based terminal input handling for interactive terminal applications. It abstracts the complexities of reading and parsing keyboard input, mouse events, and window resize notifications across different platforms. The package offers a unified `Reader` interface that converts raw terminal byte streams and platform-specific console events into structured, type-safe event objects.

For information about Windows-specific console input handling, see [Windows Console Input](https://deepwiki.com/charmbracelet/x/3.2-windows-console-input)
. For terminal control and raw mode setup, see [Pseudo-Terminal Interface](https://deepwiki.com/charmbracelet/x/5.1-pseudo-terminal-interface-(xpty))
 and [Terminal I/O](https://deepwiki.com/charmbracelet/x/5.3-terminal-io-(termios-and-wcwidth))
. For ANSI sequence parsing used by this package, see [Parser and Text Manipulation](https://deepwiki.com/charmbracelet/x/4.1-parser-and-text-manipulation)
.

* * *

Package Overview
----------------

The `input` package sits at the boundary between the operating system's input facilities and application code, providing a clean abstraction that allows terminal applications to receive input events without dealing with platform-specific differences or low-level ANSI escape sequence parsing.

### Key Responsibilities

| Responsibility | Description |
| --- | --- |
| **Event Parsing** | Converts raw byte streams (Unix) or `INPUT_RECORD` structures (Windows) into typed event objects |
| **Cross-Platform Abstraction** | Provides identical API on Unix and Windows despite different underlying mechanisms |
| **Unicode Handling** | Properly handles multi-byte UTF-8 sequences and Unicode grapheme clusters |
| **Mouse Event Decoding** | Parses various mouse reporting formats (X10, normal, SGR, pixel-based) |
| **Terminal State Detection** | Recognizes window resize events, focus changes, and terminal capability queries |
| **Cancellable Reading** | Supports graceful shutdown through `cancelreader` integration |

Sources: [input/go.mod1-22](https://github.com/charmbracelet/x/blob/7642919e/input/go.mod#L1-L22)

-----------------------------

Core Event Types
----------------

### Event Type Hierarchy

**Diagram 2: Event Type Structure**

Each event type carries specific information:

*   **`KeyPressEvent`**: Includes the key (rune or special key code) and modifier keys (Ctrl, Alt, Shift, Meta)
*   **`MouseEvent`**: Contains action type (press/release/motion/wheel), button identifier, position coordinates, and modifiers
*   **`WindowSizeEvent`**: Reports new terminal dimensions in columns and rows
*   **`FocusEvent`**: Indicates whether the terminal gained or lost focus (requires focus reporting mode)
*   **`PasteEvent`**: Delivers bracketed paste content as a single event (when bracketed paste mode is enabled)
*   **`UnknownEvent`**: Captures unrecognized sequences for debugging or forward compatibility

Sources: Diagram 6 (Data Flow for Interactive Terminal Application), Diagram 2 (Terminal I/O Stack)

Reader Interface\
----------------\
\
### Primary API\
\
The `input.Reader` interface provides the main entry point for reading input events:\
\
| Method | Description |\
| --- | --- |\
| `ReadEvents()` | Blocking call that returns the next input event or an error |\
| `Close()` | Releases resources and cancels pending reads |\
\
### Typical Usage Pattern\
\
The reader must be used with the terminal in raw mode (see `term.MakeRaw`) to receive all input events properly. Without raw mode, the operating system buffers input and processes special keys (like Ctrl+C) before the application sees them.\
\
Sources: Diagram 2 (Terminal I/O Stack), Diagram 6 (Data Flow for Interactive Terminal Application)\

ANSI Sequence Parsing\
---------------------\
\
### Sequence Recognition\
\
The input package leverages the `ansi` package's parser to decode escape sequences. The parser operates as a state machine that recognizes:\
\
| Sequence Type | Example | Event |\
| --- | --- | --- |\
| **Single Character** | `a`, `\n`, `\x7f` | `KeyPressEvent` |\
| **ESC Sequences** | `\x1b[A`, `\x1bOH` | Special key press |\
| **CSI Mouse** | `\x1b[<0;10;5M` | `MouseEvent` (SGR format) |\
| **CSI Window Ops** | `\x1b[8;24;80t` | Window size report |\
| **OSC Sequences** | `\x1b]52;c;...\x07` | Clipboard operations |\
| **DCS Sequences** | `\x1bP...` | Device control strings |\
\
The parser handles partial reads gracefully, buffering incomplete sequences until more data arrives. This is critical for handling multi-byte UTF-8 characters and long escape sequences split across read boundaries.\
\
Sources: [input/go.mod6](https://github.com/charmbracelet/x/blob/7642919e/input/go.mod#L6-L6)\
 Diagram 3 (Text Processing Pipeline)\
\
### Unicode Handling\
\
For proper Unicode support, the input package uses `rivo/uniseg` to detect grapheme cluster boundaries. This ensures that combining characters (e.g., `é` as `e` + combining acute accent) are treated as single key presses:\
\
**Diagram 5: Unicode Grapheme Processing**\
\
This handling is essential for correct behavior in international input methods and emoji input.\
\
Sources: [input/go.mod9](https://github.com/charmbracelet/x/blob/7642919e/input/go.mod#L9-L9)\
 Diagram 3 (Text Processing Pipeline)\

Mouse Event Parsing\
-------------------\
\
### Mouse Reporting Formats\
\
The input package recognizes multiple mouse reporting formats that terminals may use:\
\
| Format | Sequence Example | Description |\
| --- | --- | --- |\
| **X10** | `\x1b[M !!` | Basic 3-byte format, position offset by 32 |\
| **Normal** | `\x1b[M#!!` | Includes button state in first byte |\
| **SGR** | `\x1b[<0;10;5M` | Human-readable format with explicit coordinates |\
| **SGR Pixels** | `\x1b[<0;340;120M` | Pixel-based positioning instead of character cells |\
| **URXVT** | `\x1b[0;10;5M` | Extended format used by rxvt-unicode |\
\
The SGR format is preferred for modern applications as it:\
\
*   Supports coordinates beyond 223 (X10 limitation)\
*   Clearly distinguishes press from release events (`M` vs `m`)\
*   Uses readable decimal numbers instead of offset bytes\
\
### Mouse Button Mapping\
\
**Diagram 6: Mouse Event Decoding**\
\
Additional bits in the button code indicate modifier keys (Shift=4, Alt=8, Ctrl=16) and motion events (+32 for drag).\
\
Sources: Diagram 3 (Text Processing Pipeline), Diagram 6 (Data Flow for Interactive Terminal Application)\
\

Platform-Specific Implementation\
--------------------------------\
\
### Build Tag Strategy\
\
The input package uses Go build tags to select platform-specific implementations:\
\
**Diagram 7: Platform-Specific Implementations**\
\
This design allows the same application code to work on both Unix and Windows without modification. The platform differences are completely hidden behind the `Reader` interface.\
\
Sources: Diagram 4 (Platform-Specific Subsystems), [input/go.mod7-8](https://github.com/charmbracelet/x/blob/7642919e/input/go.mod#L7-L8)\
\
### Unix Implementation Details\
\
On Unix systems:\
\
*   Opens `/dev/tty` for direct terminal access (bypassing stdin redirection)\
*   Uses `cancelreader` for interruptible I/O operations\
*   Depends on `term` package to set raw mode via `termios`\
*   Registers signal handler for `SIGWINCH` to detect resize events\
*   Falls back to polling terminal size if signals are not available\
\
### Windows Implementation Details\
\
On Windows systems:\
\
*   Opens console input handle via `windows.GetStdHandle(STD_INPUT_HANDLE)`\
*   Calls `windows.ReadConsoleInput` to receive `INPUT_RECORD` arrays\
*   Converts Windows virtual key codes to portable key representations\
*   Handles Windows-specific quirks (e.g., AltGr key combinations)\
*   Uses console screen buffer info for window size queries\
\
Sources: [input/go.mod7-8](https://github.com/charmbracelet/x/blob/7642919e/input/go.mod#L7-L8)\
 Diagram 4 (Platform-Specific Subsystems)\

Terminal Capability Detection\
-----------------------------\
\
### Terminfo Integration\
\
The input package uses `xo/terminfo` to query terminal capabilities and adjust parsing behavior:\
\
| Capability | Purpose |\
| --- | --- |\
| **kmous** | Mouse event format support |\
| **kcuu1** | Up arrow key sequence |\
| **kcub1** | Left arrow key sequence |\
| **kDC** | Delete key sequence |\
| **kf1-kf12** | Function key sequences |\
\
Different terminals use different escape sequences for the same logical keys. For example:\
\
*   VT100: Up arrow = `\x1bOA` (application mode) or `\x1b<FileRef file-url="https://github.com/charmbracelet/x/blob/7642919e/A` (normal mode)\\n- xterm#LNaN-LNaN" NaN file-path="A\` (normal mode)\\n- xterm">Hii\

Cancellable I/O\
---------------\
\
### Graceful Shutdown\
\
The input package integrates `muesli/cancelreader` to support graceful shutdown of blocking read operations:\
\
**Diagram 8: Cancellation Flow**\
\
This mechanism allows applications to cleanly shut down even when blocked waiting for input. The `cancelreader` works by:\
\
1.  Creating an internal pipe\
2.  Wrapping the actual input source\
3.  Selecting on both the pipe and input source\
4.  Writing to the pipe on cancellation to unblock the select\
\
This is particularly important for terminal applications that need to shut down on signals (SIGINT, SIGTERM) or application-level quit commands.\
\
Sources: [input/go.mod8](https://github.com/charmbracelet/x/blob/7642919e/input/go.mod#L8-L8)\
\
* * *\
\
Integration with Terminal Control\
---------------------------------\
\
### Coordination with term Package\
\
The input package works closely with the `term` package for terminal mode management:\
\
**Diagram 9: Terminal Mode Lifecycle**\
\
**Raw mode** is essential for the input package to function correctly. In raw mode:\
\
*   Input is not line-buffered (characters available immediately)\
*   Special characters (Ctrl+C, Ctrl+Z) are not interpreted by the OS\
*   Echo is disabled (characters are not automatically printed)\
*   ANSI sequences are passed through unchanged\
\
Without raw mode, the input package would only receive line-buffered input after Enter is pressed, and many special keys would be intercepted by the operating system.\
\
Sources: Diagram 2 (Terminal I/O Stack), Diagram 6 (Data Flow for Interactive Terminal Application)\
\

Performance Considerations\
--------------------------\
\
### Buffering Strategy\
\
The input package uses several buffering techniques to optimize performance:\
\
| Buffer Type | Purpose | Size |\
| --- | --- | --- |\
| **Read buffer** | Minimize syscalls by reading multiple bytes at once | 1KB-4KB typical |\
| **Parse buffer** | Accumulate incomplete escape sequences | Dynamic growth |\
| **Event queue** | Batch multiple events from single read | Single read worth |\
\
### Latency Characteristics\
\
Input latency is critical for interactive applications. The package minimizes latency by:\
\
*   **No artificial delays**: Events are dispatched immediately upon recognition\
*   **Minimal parsing overhead**: State machine parser processes sequences in O(n) time\
*   **Direct syscalls**: No additional middleware layers on Unix\
\
Typical latency from keystroke to application receipt is <1ms on modern systems, dominated by terminal emulator processing rather than the input package.\
\
Sources: Diagram 6 (Data Flow for Interactive Terminal Application)\
\
* * *\
\
Error Handling\
--------------\
\
### Common Error Conditions\
\
| Error | Cause | Recommended Action |\
| --- | --- | --- |\
| **`io.EOF`** | Input stream closed or reader cancelled | Clean shutdown |\
| **`syscall.EINTR`** | System call interrupted by signal | Retry read |\
| **`os.ErrClosed`** | Reader closed while operation pending | Exit event loop |\
| **Parse errors** | Malformed escape sequences | Log and continue with next input |\
\
The input package handles `EINTR` automatically by retrying interrupted reads. Applications typically only need to handle `EOF` (shutdown) and potentially log unexpected errors for debugging.\
\
### Graceful Degradation\
\
When the input package encounters unrecognized sequences, it:\
\
1.  Captures the raw bytes in an `UnknownEvent`\
2.  Delivers the event to the application\
3.  Continues processing subsequent input\
\
This allows applications to either ignore unknown sequences or implement custom handling for terminal-specific features not yet supported by the package.\
\
Sources: Diagram 2 (Terminal I/O Stack)\
\

Summary\
-------\
\
The `input` package provides a robust, cross-platform abstraction for terminal input handling. Its key strengths are:\
\
*   **Unified API**: Same interface works on Unix and Windows\
*   **Rich event types**: Supports keyboard, mouse, resize, focus, and paste events\
*   **ANSI-aware parsing**: Correctly handles escape sequences and Unicode\
*   **Cancellable I/O**: Supports graceful shutdown\
*   **Terminal agnostic**: Uses terminfo for capability detection\
*   **High performance**: Minimal latency and efficient buffering\
\
The package serves as the foundation for interactive terminal applications, hiding platform complexity while exposing a clean, type-safe event interface.\
\
