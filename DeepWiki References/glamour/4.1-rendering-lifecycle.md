Purpose and Scope
-----------------

This page documents the element rendering lifecycle in Glamour, which defines how individual Markdown AST nodes are converted to styled terminal output. The lifecycle follows a two-phase traversal pattern (Render and Finish) that handles nested element hierarchies. For information about specific element types like tables or code blocks, see [Tables](https://deepwiki.com/charmbracelet/glamour/4.3-tables)
 and [Code Blocks](https://deepwiki.com/charmbracelet/glamour/4.4-code-blocks)
. For information about the layout management system that underpins this lifecycle, see [BlockStack](https://deepwiki.com/charmbracelet/glamour/5.1-blockstack)
.

**Sources:** [ansi/elements.go1-480](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/elements.go#L1-L480)

* * *

Core Interfaces
---------------

The rendering lifecycle is defined by two primary interfaces that elements implement:

### ElementRenderer Interface

The `ElementRenderer` interface is invoked when **entering** a node during AST traversal:

The `Render` method is responsible for:

*   Setting up rendering state (pushing to BlockStack)
*   Rendering opening prefixes and markers
*   Writing content that appears before children

### ElementFinisher Interface

The `ElementFinisher` interface is invoked when **exiting** a node:

The `Finish` method is responsible for:

*   Applying layout (word wrapping, margins, indentation)
*   Rendering closing suffixes
*   Cleaning up state (popping from BlockStack)

### Element Structure

The `Element` struct coordinates the lifecycle and holds both interfaces:

| Field | Type | Purpose |
| --- | --- | --- |
| `Entering` | `string` | Text written before rendering children |
| `Exiting` | `string` | Text written after rendering children |
| `Renderer` | `ElementRenderer` | Handler for entering phase |
| `Finisher` | `ElementFinisher` | Handler for exiting phase |

**Sources:** [ansi/elements.go16-38](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/elements.go#L16-L38)

Two-Phase Rendering
-------------------

### Phase 1: Render (Entering)

The `Render` phase occurs when **entering** a node during tree traversal. Key responsibilities:

1.  **Push to BlockStack** - Establish a new rendering context with style and layout parameters
2.  **Write Prefixes** - Output `BlockPrefix` and `Prefix` from styles
3.  **Allocate Buffers** - Create buffer for child content if needed

Example from `HeadingElement`:

At this point:

*   A new block context is established
*   Child nodes will write to `bs.Current().Block`
*   Style cascading is applied

**Sources:** [ansi/heading.go27-59](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/heading.go#L27-L59)
 [ansi/paragraph.go18-34](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/paragraph.go#L18-L34)

### Phase 2: Finish (Exiting)

The `Finish` phase occurs when **exiting** a node after all children have been rendered. Key responsibilities:

1.  **Apply Layout** - Word wrap content to terminal width
2.  **Apply Margins** - Add indentation and spacing via `MarginWriter`
3.  **Write Suffixes** - Output `Suffix` and `BlockSuffix` from styles
4.  **Pop from BlockStack** - Remove the rendering context
5.  **Reset Buffers** - Clean up temporary buffers

Render Context
--------------

The `RenderContext` flows through all rendering operations and provides access to:

| Field | Type | Purpose |
| --- | --- | --- |
| `blockStack` | `*BlockStack` | Stack of nested block contexts for layout management |
| `options` | `Options` | Rendering options including styles, word wrap settings, color profile |
| `table` | table state | Tracking for table-specific rendering (links, images) |

The context enables:

*   **Style Cascading** - Child elements inherit and override parent styles
*   **Layout Calculation** - Available width computed from nested indentation
*   **State Management** - Shared state like link/image collections in tables

**Sources:** [ansi/elements.go16-19](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/elements.go#L16-L19)

Special Cases
-------------

### Elements Without Finish Phase

Some elements like `LinkElement` and `ImageElement` only implement `Render` because they:

*   Output content immediately without buffering
*   Don't need layout post-processing
*   Have no nested block structure

**Example:** Link rendering at [ansi/elements.go222-254](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/elements.go#L222-L254)

### Elements Without Render Phase

Some elements like `TableRowElement` and `TableHeadElement` only implement `Finish` because they:

*   Only need to perform actions after children are processed
*   Act as separators or markers in parent structures

**Example:** Table row finishing at [ansi/elements.go407-410](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/elements.go#L407-L410)

Implementation Guidelines
-------------------------

When implementing a new element renderer:

1.  **Determine Phase Requirements**
    
    *   Need buffering? ΓåÆ Implement both `Render` and `Finish`
    *   Inline output only? ΓåÆ Implement just `Render`
    *   Post-processing only? ΓåÆ Implement just `Finish`
2.  **BlockStack Usage**
    
    *   Push in `Render` if creating a new block context
    *   Always pop in `Finish` if you pushed in `Render`
    *   Use `bs.Current().Block` as the buffer for child content
3.  **Style Handling**
    
    *   Use `cascadeStyle()` to merge parent and element styles
    *   Access parent style via `bs.Parent().Style`
    *   Access current style via `bs.Current().Style`
4.  **Layout Application**
    
    *   Word wrap in `Finish` using `wordwrap.NewWriter(bs.Width(ctx))`
    *   Apply margins using `NewMarginWriter(ctx, w, style)`
    *   Always reset buffers after writing: `bs.Current().Block.Reset()`
