Architecture Overview
=====================

This page explains the high-level architecture of Glamour, detailing how Markdown text flows through the rendering pipeline to produce styled ANSI terminal output. It covers the four primary architectural layers, their interactions, and the key data structures that enable the transformation.

For information about the public API and configuration options, see [Core API](https://deepwiki.com/charmbracelet/glamour/2-core-api)
. For detailed styling system documentation, see [Styling System](https://deepwiki.com/charmbracelet/glamour/3-styling-system)
. For element-specific rendering logic, see [Element Renderers](https://deepwiki.com/charmbracelet/glamour/4-element-renderers)
.

* * *

System Overview
---------------

Glamour employs a **four-layer architecture** that separates concerns between the user-facing API, the rendering pipeline, styling configuration, and element-specific rendering logic. The system integrates the Goldmark Markdown parser as its foundation and wraps it with custom ANSI rendering capabilities.

**Sources**: [glamour.go1-323](https://github.com/charmbracelet/glamour/blob/69661fd5/glamour.go#L1-L323)

Layer 1: Public API
-------------------

The public API layer, defined in [glamour.go](https://github.com/charmbracelet/glamour/blob/69661fd5/glamour.go)
 provides the user-facing interface. The central type is `TermRenderer`, which encapsulates a Goldmark parser instance and rendering options.

| Component | Type | Purpose |
| --- | --- | --- |
| `TermRenderer` | struct | Main renderer that coordinates parsing and rendering |
| `NewTermRenderer()` | function | Constructor accepting `TermRendererOption` functions |
| `Render()` | helper | One-shot rendering with style path |
| `RenderBytes()` | helper | One-shot rendering returning bytes |
| `RenderWithEnvironmentConfig()` | helper | Rendering using `GLAMOUR_STYLE` environment variable |

The `TermRenderer` struct at [glamour.go36-41](https://github.com/charmbracelet/glamour/blob/69661fd5/glamour.go#L36-L41)
 contains:

*   `md goldmark.Markdown` - the configured Goldmark parser
*   `ansiOptions ansi.Options` - rendering configuration
*   `buf bytes.Buffer` - input accumulation buffer
*   `renderBuf bytes.Buffer` - output rendering buffer

**Configuration Options** are applied via the functional options pattern (`TermRendererOption` functions). Key options include:

*   `WithStylePath()` - load style from file or built-in name
*   `WithStyles()` - directly set `StyleConfig`
*   `WithWordWrap()` - set line wrap width
*   `WithColorProfile()` - set color support level (TrueColor/ANSI256/ANSI)
*   `WithEmoji()` - enable emoji rendering via goldmark-emoji
*   `WithChromaFormatter()` - customize code syntax highlighting

**Sources**: [glamour.go31-323](https://github.com/charmbracelet/glamour/blob/69661fd5/glamour.go#L31-L323)

Layer 2: Rendering Pipeline
---------------------------

The rendering pipeline transforms Markdown text through several stages:

### Goldmark Integration

Glamour uses the Goldmark parser ([github.com/yuin/goldmark](https://github.com/charmbracelet/glamour/blob/69661fd5/github.com/yuin/goldmark)
) configured at [glamour.go72-80](https://github.com/charmbracelet/glamour/blob/69661fd5/glamour.go#L72-L80)
 with:

*   **Extensions**: GFM (GitHub Flavored Markdown), DefinitionList
*   **Parser Options**: AutoHeadingID for automatic heading anchors
*   **Optional Extensions**: Emoji (when `WithEmoji()` is used)

The Goldmark parser converts Markdown into an Abstract Syntax Tree (AST) composed of `ast.Node` instances representing document structure (headings, paragraphs, lists, etc.).

### ANSIRenderer

The `ANSIRenderer` type at [ansi/renderer.go30-32](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go#L30-L32)
 implements Goldmark's `renderer.NodeRenderer` interface. It registers handler functions for each AST node type at [ansi/renderer.go42-91](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go#L42-L91)

The core rendering method `renderNode()` at [ansi/renderer.go93-139](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go#L93-L139)
 handles both entering and exiting each node:

1.  **Entering phase**: Creates an `Element` via `NewElement()`, renders opening content
2.  **Node children**: Processed recursively by Goldmark's walker
3.  **Exiting phase**: Calls `Element.Finisher` to render closing content

**Sources**: [glamour.go72-99](https://github.com/charmbracelet/glamour/blob/69661fd5/glamour.go#L72-L99)
 [ansi/renderer.go29-169](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go#L29-L169)

Layer 3: Styling System
-----------------------

The styling layer provides theme configuration through the `StyleConfig` structure and manages style resolution.

### StyleConfig Structure

The `StyleConfig` at [ansi/style.go](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/style.go)
 contains style definitions for all Markdown elements:

| Style Property | Type | Purpose |
| --- | --- | --- |
| `Document` | `StyleBlock` | Root document styling |
| `BlockQuote` | `StyleBlock` | Blockquote indentation/color |
| `List` | `StyleList` | List item markers, indentation |
| `Heading` | `StyleBlock` | H1-H6 styles |
| `H1` through `H6` | `StyleBlock` | Per-heading-level styles |
| `Paragraph` | `StyleBlock` | Paragraph spacing |
| `Code` | `StyleBlock` | Inline code styling |
| `CodeBlock` | `StyleCodeBlock` | Fenced code block with Chroma |
| `Table` | `StyleTable` | Table borders and cells |
| `Link` | `StylePrimitive` | Hyperlink colors |
| `Image` | `StylePrimitive` | Image text styling |

### Style Resolution

Style resolution at [glamour.go306-322](https://github.com/charmbracelet/glamour/blob/69661fd5/glamour.go#L306-L322)
 follows this priority:

1.  **Auto style**: Detects terminal background (light/dark) using `termenv.HasDarkBackground()`
2.  **Named style**: Looks up in `styles.DefaultStyles` map
3.  **File path**: Loads and unmarshals JSON file
4.  **Direct StyleConfig**: Uses programmatically provided configuration

Layer 4: Element Rendering System
---------------------------------

The element rendering layer dispatches AST nodes to specialized renderer types and manages the rendering lifecycle.

### Element Dispatch

The `NewElement()` function at [ansi/elements.go41-479](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/elements.go#L41-L479)
 implements a large switch statement that maps `ast.Node.Kind()` values to `Element` structs:

### Element Structure

The `Element` struct at [ansi/elements.go33-38](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/elements.go#L33-L38)
 contains:

*   `Entering` string - text to emit when entering the node
*   `Exiting` string - text to emit when exiting the node
*   `Renderer` ElementRenderer - interface for entering-phase rendering
*   `Finisher` ElementFinisher - interface for exiting-phase rendering

### Rendering Lifecycle

Each element follows a two-phase lifecycle:

1.  **Render Phase** (entering):
    
    *   Element's `Render(w io.Writer, ctx RenderContext)` method called
    *   Sets up layout context (indentation, margins)
    *   Emits opening markup
    *   Child nodes processed by Goldmark walker
2.  **Finish Phase** (exiting):
    
    *   Element's `Finish(w io.Writer, ctx RenderContext)` method called
    *   Emits closing markup
    *   Applies word wrapping and formatting
    *   Pops layout context

### BlockStack Management

The `BlockStack` manages nested block rendering context at [ansi/blockstack.go](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/blockstack.go)
 It maintains a stack of `BlockElement` instances, each containing:

*   `Block *bytes.Buffer` - accumulates rendered content
*   `Style StyleBlock` - cascaded style for the block
*   Indentation and margin state

During rendering at [ansi/renderer.go93-139](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go#L93-L139)
:

*   Entering a block pushes to `context.blockStack`
*   Content renders into `blockStack.Current().Block`
*   Exiting a block flushes to parent or root writer

**Sources**: [ansi/elements.go16-480](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/elements.go#L16-L480)
 [ansi/renderer.go93-139](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go#L93-L139)
 [ansi/blockstack.go](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/blockstack.go)

Data Flow Through the System
----------------------------

The complete data transformation from Markdown to ANSI output:

### Buffer Management

The rendering system uses a hierarchical buffer structure:

1.  **Input Buffer**: `TermRenderer.buf` accumulates input when using `Write()` method
2.  **Block Buffers**: Each `BlockElement` in the stack has a `Block *bytes.Buffer`
3.  **Output Buffer**: `TermRenderer.renderBuf` receives final output
4.  **Root Writer**: The final writer passed to Goldmark (often stdout or a user buffer)

Content flows "up" the block stack as elements finish rendering, with each level applying its styling and layout rules.

**Sources**: [glamour.go253-295](https://github.com/charmbracelet/glamour/blob/69661fd5/glamour.go#L253-L295)
 [ansi/renderer.go93-139](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go#L93-L139)

Key Dependencies

### Extension Architecture

Glamour extends Goldmark through:

1.  **Custom Renderer**: `ANSIRenderer` replaces Goldmark's default HTML renderer
2.  **Node Registration**: [ansi/renderer.go42-91](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go#L42-L91)
     registers handlers for each node type
3.  **Priority System**: ANSIRenderer registered at priority 1000 at [glamour.go95](https://github.com/charmbracelet/glamour/blob/69661fd5/glamour.go#L95-L95)
    

This design allows Glamour to leverage Goldmark's robust parsing while completely customizing output generation.

Context and State Management
----------------------------

The `RenderContext` structure at [ansi/renderer.go](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go)
 maintains state during rendering:

**State Flow**:

1.  Context created once at [ansi/renderer.go35-38](https://github.com/charmbracelet/glamour/blob/69661fd5/ansi/renderer.go#L35-L38)
     with initial options
2.  BlockStack grows/shrinks as nodes enter/exit
3.  Table context accumulates links/images for footer rendering
4.  Style cascading computed from current block stack position
