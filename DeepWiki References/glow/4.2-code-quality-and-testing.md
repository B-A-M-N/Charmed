Code Quality and Testing
========================

This document describes the code quality tooling and testing infrastructure used in the Glow project. It covers the automated linting system with golangci-lint, test coverage reporting workflows, and automated dependency management with Dependabot. For information about the build and release process, see [Build and Release Process](https://deepwiki.com/charmbracelet/glow/4.1-build-and-release-process)
.

Overview
--------

Glow maintains code quality through:

*   **Automated linting** via golangci-lint with 20+ enabled linters
*   **Test coverage tracking** with race detection and Coveralls integration
*   **Automated dependency updates** via Dependabot for Go modules, GitHub Actions, and Docker
*   **Continuous Integration** workflows that run on every push and pull request

All quality checks are implemented as GitHub Actions workflows that execute automatically on code changes.

Linting Configuration
---------------------

### golangci-lint Setup

The project uses golangci-lint version 2 with a comprehensive set of linters configured in [.golangci.yml](https://github.com/charmbracelet/glow/blob/752de97c/.golangci.yml)
 The linting system is configured to catch common issues while allowing code generation patterns.

**Enabled Linters**

| Linter | Purpose |
| --- | --- |
| `bodyclose` | Checks that HTTP response bodies are closed |
| `exhaustive` | Checks exhaustiveness of enum switch statements |
| `goconst` | Finds repeated strings that could be constants |
| `godot` | Checks that comments end in periods |
| `gomoddirectives` | Checks for disallowed directives in go.mod |
| `goprintffuncname` | Checks printf-like function naming |
| `gosec` | Security-focused static analysis |
| `misspell` | Finds commonly misspelled English words |
| `nakedret` | Finds naked returns in functions |
| `nestif` | Reports deeply nested if statements |
| `nilerr` | Finds code that returns nil with non-nil error |
| `noctx` | Finds HTTP requests without context |
| `nolintlint` | Reports malformed nolint directives |
| `prealloc` | Finds slice declarations that could be preallocated |
| `revive` | Fast, configurable, extensible linter |
| `rowserrcheck` | Checks that sql.Rows.Err is checked |
| `sqlclosecheck` | Checks that sql.Rows/Stmt are closed |
| `tparallel` | Detects inappropriate usage of t.Parallel() |
| `unconvert` | Removes unnecessary type conversions |
| `unparam` | Reports unused function parameters |
| `whitespace` | Detects leading/trailing whitespace |
| `wrapcheck` | Checks that errors from external packages are wrapped |

**Formatters**

The configuration also enables automatic formatting via [.golangci.yml40-44](https://github.com/charmbracelet/glow/blob/752de97c/.golangci.yml#L40-L44)
:

*   `gofumpt` - Stricter gofmt
*   `goimports` - Automatic import management

**Exclusions**

The linter excludes certain patterns [.golangci.yml28-35](https://github.com/charmbracelet/glow/blob/752de97c/.golangci.yml#L28-L35)
:

*   Logging calls (`slog` and `log` functions) are excluded from the `noctx` linter
*   Generated code is treated leniently
*   Common false positives are suppressed via presets

Sources: [.golangci.yml1-45](https://github.com/charmbracelet/glow/blob/752de97c/.golangci.yml#L1-L45)

### Linting Workflow Architecture

**Workflow Configuration Details**

The lint workflow [.github/workflows/lint.yml](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/lint.yml)
 executes with the following characteristics:

*   **Triggers**: Runs on all push events and pull requests [.github/workflows/lint.yml2-4](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/lint.yml#L2-L4)
    
*   **Exit Code**: Set to 0 via `--issues-exit-code=0` [.github/workflows/lint.yml16](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/lint.yml#L16-L16)
     making the workflow non-blocking (reports issues but doesn't fail the build)
*   **PR Mode**: When running on pull requests, only reports new issues via `only-new-issues: true` [.github/workflows/lint.yml18](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/lint.yml#L18-L18)
    
*   **Action Version**: Uses `golangci-lint-action@v9.2.0` [.github/workflows/lint.yml13](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/lint.yml#L13-L13)
    

This configuration allows developers to see linting issues without blocking merges, encouraging incremental improvements while preventing regression on pull requests.

Sources: [.github/workflows/lint.yml1-19](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/lint.yml#L1-L19)
 [.golangci.yml1-45](https://github.com/charmbracelet/glow/blob/752de97c/.golangci.yml#L1-L45)

Test Coverage System
--------------------

### Coverage Workflow

The coverage workflow [.github/workflows/coverage.yml](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/coverage.yml)
 runs tests with race detection and reports coverage to Coveralls.

**Coverage Parameters**

The test command [.github/workflows/coverage.yml22](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/coverage.yml#L22-L22)
 includes:

| Flag | Purpose |
| --- | --- |
| `-race` | Enables race detector to catch concurrent access issues |
| `-covermode atomic` | Uses atomic coverage mode, required for race detection |
| `-coverprofile=profile.cov` | Outputs coverage data to file |
| `./...` | Tests all packages recursively |

**Integration with Coveralls**

The workflow uses the `GITHUB_TOKEN` secret [.github/workflows/coverage.yml20](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/coverage.yml#L20-L20)
 to authenticate with Coveralls. The `goveralls` tool [.github/workflows/coverage.yml24](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/coverage.yml#L24-L24)
 uploads the coverage profile with GitHub service integration, allowing pull requests to show coverage changes automatically.

Sources: [.github/workflows/coverage.yml1-25](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/coverage.yml#L1-L25)

Automated Dependency Management
-------------------------------

### Dependabot Configuration

Glow uses GitHub Dependabot to automatically update three categories of dependencies: Go modules, GitHub Actions, and Docker images. The configuration in [.github/dependabot.yml](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml)
 provides consistent update scheduling and grouping.

**Update Schedule Configuration**

All dependency categories follow the same schedule [.github/dependabot.yml6-10](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L6-L10)
:

*   **Interval**: Weekly
*   **Day**: Monday
*   **Time**: 05:00 AM
*   **Timezone**: America/New\_York

**Go Module Updates**

The Go module configuration [.github/dependabot.yml3-23](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L3-L23)
:

*   Groups all Go dependency updates into a single PR via `groups.all` [.github/dependabot.yml16-19](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L16-L19)
    
*   Labels PRs with `dependencies` [.github/dependabot.yml11-12](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L11-L12)
    
*   Uses `chore` prefix for commit messages [.github/dependabot.yml13-15](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L13-L15)
    
*   Explicitly ignores `bubbletea/v2` beta versions [.github/dependabot.yml20-23](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L20-L23)
    

**GitHub Actions Updates**

The GitHub Actions ecosystem configuration [.github/dependabot.yml25-40](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L25-L40)
 follows the same pattern, automatically updating action versions in workflow files.

**Docker Updates**

The Docker ecosystem configuration [.github/dependabot.yml42-57](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L42-L57)
 keeps Dockerfile base images up to date.

Sources: [.github/dependabot.yml1-58](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L1-L58)

### Dependency Update Flow

**Dependabot Sync Workflow**

The project includes an additional automation layer via [.github/workflows/dependabot-sync.yml](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/dependabot-sync.yml)
 This workflow:

*   **Schedule**: Runs every Sunday at midnight [.github/workflows/dependabot-sync.yml4](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/dependabot-sync.yml#L4-L4)
    
*   **Manual Trigger**: Can be dispatched manually via `workflow_dispatch` [.github/workflows/dependabot-sync.yml5](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/dependabot-sync.yml#L5-L5)
    
*   **Reusable Workflow**: Delegates to `charmbracelet/meta/.github/workflows/dependabot-sync.yml@main` [.github/workflows/dependabot-sync.yml13](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/dependabot-sync.yml#L13-L13)
    
*   **Permissions**: Requires `contents: write` and `pull-requests: write` [.github/workflows/dependabot-sync.yml7-9](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/dependabot-sync.yml#L7-L9)
    
*   **Authentication**: Uses `PERSONAL_ACCESS_TOKEN` secret [.github/workflows/dependabot-sync.yml17](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/dependabot-sync.yml#L17-L17)
    

This provides centralized dependency management across Charmbracelet projects, automatically merging grouped Dependabot PRs that pass CI checks.

Sources: [.github/dependabot.yml1-58](https://github.com/charmbracelet/glow/blob/752de97c/.github/dependabot.yml#L1-L58)

CI Workflow Integration
-----------------------

### Quality Gates Summary

The following table summarizes the automated quality checks that run on every code change:

| Workflow | Trigger | Blocking | Purpose |
| --- | --- | --- | --- |
| `lint.yml` | Push, PR | No  | Report code quality issues |
| `coverage.yml` | Push, PR | No  | Track test coverage and race conditions |
| `dependabot-sync.yml` | Weekly, Manual | N/A | Auto-merge dependency updates |

**Non-Blocking Philosophy**

Both the lint and coverage workflows are configured as non-blocking [.github/workflows/lint.yml16](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/lint.yml#L16-L16)
:

*   Issues are reported but don't prevent merges
*   Encourages incremental quality improvements
*   Prevents legacy code issues from blocking new features
*   PR-specific checks (`only-new-issues: true`) prevent regression

**Coverage Insights**

The coverage workflow provides insights into:

*   **Test Coverage**: Percentage of code exercised by tests
*   **Race Conditions**: Concurrent access issues detected by `-race` flag
*   **Coverage Trends**: Historical tracking via Coveralls integration

Sources: [.github/workflows/lint.yml](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/lint.yml)
 [.github/workflows/coverage.yml](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/coverage.yml)
 [.github/workflows/dependabot-sync.yml](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/dependabot-sync.yml)

Testing Infrastructure
----------------------

While the codebase does not expose explicit test files in the provided sources, the coverage workflow [.github/workflows/coverage.yml22](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/coverage.yml#L22-L22)
 executes tests with the command:

This command:

*   Recursively tests all packages via `./...`
*   Enables Go's race detector with `-race` to catch data races
*   Uses atomic coverage mode for thread-safe coverage tracking
*   Generates a coverage profile for reporting

The atomic coverage mode is required when race detection is enabled, as it ensures thread-safe coverage counters during concurrent test execution.
