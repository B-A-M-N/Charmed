Build and Release Process
=========================

Relevant source files

Purpose and Scope
-----------------

This document describes Glow's automated build and release infrastructure, which uses GoReleaser and GitHub Actions to produce multi-platform binaries, container images, and package manager distributions. The process is triggered by git version tags and handles cross-compilation, artifact signing, and publishing to multiple distribution channels.

For information about code quality checks that run during development, see [Code Quality and Testing](https://deepwiki.com/charmbracelet/glow/4.2-code-quality-and-testing)
.

Release Trigger Mechanism
-------------------------

The release process is triggered automatically when a semantic version tag is pushed to the repository. The GitHub Actions workflow monitors for tags matching the pattern `v*.*.*` (e.g., `v1.5.0`, `v2.0.1`).

**Sources:** [.github/workflows/goreleaser.yml5-8](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/goreleaser.yml#L5-L8)

The workflow configuration specifies:

*   **Event type:** Push events only
*   **Tag pattern:** `v*.*.*` (semantic versioning)
*   **Concurrency control:** Single goreleaser job at a time, with cancellation of in-progress runs

Sources: [.github/workflows/goreleaser.yml1-26](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/goreleaser.yml#L1-L26)

GoReleaser Configuration Architecture
-------------------------------------

Glow uses a two-layer GoReleaser configuration strategy that separates project-specific metadata from shared build logic.

### Configuration Structure

**Sources:** [.goreleaser.yml1-15](https://github.com/charmbracelet/glow/blob/752de97c/.goreleaser.yml#L1-L15)

The local `.goreleaser.yml` file is minimal and declares:

*   GoReleaser version 2 format
*   An include directive pointing to `charmbracelet/meta/main/goreleaser-glow.yaml`
*   Project-specific variables used by the shared configuration

| Variable | Purpose | Value |
| --- | --- | --- |
| `description` | Package description | "Render markdown on the CLI, with pizzazz!" |
| `github_url` | Repository URL | "[https://github.com/charmbracelet/glow](https://github.com/charmbracelet/glow)<br>" |
| `maintainer` | Package maintainer | "Christian Muehlhaeuser [muesli@charm.sh](mailto:muesli@charm.sh)<br>" |
| `brew_commit_author_name` | Homebrew commits | "Christian Muehlhaeuser" |
| `brew_commit_author_email` | Homebrew commits | "[muesli@charm.sh](mailto:muesli@charm.sh)<br>" |

Sources: [.goreleaser.yml3-14](https://github.com/charmbracelet/glow/blob/752de97c/.goreleaser.yml#L3-L14)

This architecture allows the Charmbracelet organization to maintain consistent build and release configurations across multiple projects while allowing each project to specify its unique metadata.

GitHub Actions Workflow
-----------------------

The release workflow delegates to a reusable workflow maintained in the `charmbracelet/meta` repository, passing required secrets for various publishing targets.

### Workflow Execution Flow

**Sources:** [.github/workflows/goreleaser.yml14-26](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/goreleaser.yml#L14-L26)

### Secrets and Credentials

The workflow requires seven types of secrets for publishing to different distribution channels:

| Secret Name | Purpose | Used For |
| --- | --- | --- |
| `DOCKERHUB_USERNAME` | Docker Hub authentication | Container image publishing |
| `DOCKERHUB_TOKEN` | Docker Hub authentication | Container image publishing |
| `PERSONAL_ACCESS_TOKEN` | GitHub API access | Creating releases, updating repositories |
| `GORELEASER_KEY` | GoReleaser Pro license | Advanced features (if applicable) |
| `FURY_TOKEN` | Gemfury authentication | APT/YUM repository publishing |
| `NFPM_GPG_KEY` | Package signing | Debian/RPM package signatures |
| `NFPM_PASSPHRASE` | GPG key decryption | Unlocking signing key |
| `SNAPCRAFT_TOKEN` | Snap Store authentication | Snap package publishing |

Sources: [.github/workflows/goreleaser.yml18-25](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/goreleaser.yml#L18-L25)

### Concurrency Control

The workflow implements concurrency control to prevent multiple releases from running simultaneously:

This configuration ensures that if a new tag is pushed while a release is in progress, the old release is cancelled and the new one takes precedence.

Sources: [.github/workflows/goreleaser.yml10-12](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/goreleaser.yml#L10-L12)

Publishing Targets
------------------

Based on the secrets configuration and the remote GoReleaser configuration reference, Glow releases are published to multiple distribution channels:

### Distribution Matrix

**Sources:** [.github/workflows/goreleaser.yml18-25](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/goreleaser.yml#L18-L25)

### GitHub Releases

GitHub Releases serve as the primary distribution mechanism and source of truth for all release artifacts. The `gh_pat` secret enables the workflow to:

*   Create release entries with changelogs
*   Upload compiled binaries for multiple platforms
*   Generate and upload checksums
*   Attach signatures for verification

### Container Images

Docker Hub receives multi-architecture container images, authenticated using `docker_username` and `docker_token`. These images allow users to run Glow in containerized environments without local installation.

### Package Manager Integration

The release process integrates with multiple package managers:

*   **Homebrew:** Updates the formula in the Homebrew tap using the `brew_commit_author_name` and `brew_commit_author_email` variables, creating commits that reference the new release
*   **Snap Store:** Publishes to the Snap Store using `snapcraft_token`, making Glow available via `snap install glow`
*   **APT/YUM Repositories:** Publishes signed packages to Gemfury-hosted repositories using `fury_token`, `nfpm_gpg_key`, and `nfpm_passphrase`

Sources: [.goreleaser.yml10-14](https://github.com/charmbracelet/glow/blob/752de97c/.goreleaser.yml#L10-L14)
 [.github/workflows/goreleaser.yml18-25](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/goreleaser.yml#L18-L25)

Multi-Platform Build Capabilities
---------------------------------

While the specific build matrix is defined in the remote configuration file, the GoReleaser setup supports cross-compilation for multiple operating systems and architectures. The standard Charmbracelet build configuration typically includes:

### Typical Build Matrix

| Operating System | Architectures | Output Format |
| --- | --- | --- |
| Linux | amd64, arm64, arm, 386 | tar.gz, .deb, .rpm, snap |
| macOS | amd64, arm64 | tar.gz, Homebrew formula |
| Windows | amd64, 386, arm64 | zip |
| FreeBSD | amd64, arm64 | tar.gz |

The use of Go's cross-compilation capabilities means that all platforms can be built from a single Linux-based GitHub Actions runner without requiring separate build machines for each target platform.

### Artifact Verification

The inclusion of checksums and GPG signatures allows users to verify the integrity and authenticity of downloaded artifacts:

1.  **Checksum verification:** Users can verify that downloaded binaries match the published SHA-256 hashes
2.  **Signature verification:** GPG signatures prove that artifacts were produced by the official release process using the `nfpm_gpg_key`

This verification process is particularly important for system packages (.deb, .rpm) that may be installed with elevated privileges.

Sources: [.github/workflows/goreleaser.yml23-24](https://github.com/charmbracelet/glow/blob/752de97c/.github/workflows/goreleaser.yml#L23-L24)

Configuration Maintainability
-----------------------------

The separation of concerns between local and remote configuration provides several benefits:

1.  **Consistency:** All Charmbracelet projects using the shared configuration receive updates automatically
2.  **Simplicity:** Individual projects only need to maintain project-specific metadata
3.  **Security:** Secrets are managed at the organization level and passed through the reusable workflow
4.  **Flexibility:** Projects can override or extend the shared configuration if needed

The `from_url` include mechanism in `.goreleaser.yml` fetches the latest configuration from the `charmbracelet/meta` repository at build time, ensuring that improvements to the build process are automatically adopted.

Sources: [.goreleaser.yml5-7](https://github.com/charmbracelet/glow/blob/752de97c/.goreleaser.yml#L5-L7)
