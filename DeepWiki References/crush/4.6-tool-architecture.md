Purpose and Scope
-----------------

This document describes the technical architecture of the tool system in Crush. It details the `fantasy.AgentTool` interface, the registration of tools within the `SessionAgent`, the dispatch mechanism for LLM-triggered tool calls, and the safety protocols used to manage permissions.

* * *

Tool System Overview
--------------------

Crush utilizes the `fantasy.AgentTool` interface from the `charm.land/fantasy` package to provide a unified abstraction for agent capabilities. Tools bridge the gap between the LLM's natural language reasoning and the local execution environment (file system, shell, LSP, etc.).

### The AgentTool Interface

The `fantasy.AgentTool` interface defines how a tool is described to the model and how it is executed. Tools are typically constructed using `fantasy.NewAgentTool`, which encapsulates:

1.  **Name**: The identifier the LLM uses to invoke the tool (e.g., `bash`, `edit`, `view`).
2.  **Description**: A detailed prompt (often rendered from Markdown templates) that instructs the LLM on the tool's purpose and schema.
3.  **Execution Function**: A Go function that receives parsed parameters and returns a `fantasy.ToolResponse`.

In `internal/agent/tools/tools.go`, helper functions manage contextual data passed to these execution functions, such as the `SessionIDContextKey` and `ModelNameContextKey` [internal/agent/tools/tools.go20-29](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/tools.go#L20-L29)

**Sources:** [internal/agent/tools/tools.go1-29](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/tools.go#L1-L29)

Tool Registry and Initialization
--------------------------------

Tools are registered during the creation of a `SessionAgent` via `SessionAgentOptions`. This registration happens in the `Coordinator` during normal operation or in the `coderAgent` helper during testing.

Tool Execution and Dispatch
---------------------------

When the LLM generates a response containing one or more tool calls, the `SessionAgent` dispatches these calls to the corresponding tool's execution function.

Safety and Permissions
----------------------

Safety is enforced through a centralized permission system. Every tool that performs a sensitive operation (writing files, executing shell commands, making network requests) must check with the `permission.Service`.

### Permission Bypass and Safe Commands

The `bash` tool maintains a list of "safe commands" that can bypass user prompts if configured. If a tool call is denied by the user, the tool returns a specific response via `NewPermissionDeniedResponse()`, which sets `StopTurn = true` to prevent the agent from infinitely retrying the same blocked action [internal/agent/tools/tools.go66-70](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/tools.go#L66-L70)

### Dynamic Descriptions

Tool descriptions are often dynamic. For example, the `ghAvailable` check determines if the GitHub CLI is on the PATH, allowing tools to adjust their instructions to the LLM based on available local binaries [internal/agent/tools/tools.go73-79](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/tools.go#L73-L79)

Tool Result Formatting
----------------------

The agent handles the transformation of tool outputs back into a format the LLM can consume.

| Result Type | Implementation Detail |
| --- | --- |
| **Text** | Returned directly as a string in the `ToolResponse` content. |
| **Errors** | Wrapped using `fantasy.NewTextErrorResponse`. |
| **Permission Denied** | Returns "User denied permission" with `StopTurn: true` [internal/agent/tools/tools.go66-70](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/tools.go#L66-L70) |
| **Templates** | Tools use `renderToolDescription` or `renderTemplate` to ensure consistent markdown formatting for the LLM [internal/agent/tools/tools.go87-105](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/tools.go#L87-L105) |
