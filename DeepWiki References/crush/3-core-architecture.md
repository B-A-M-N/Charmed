This page describes how crush's major runtime components fit together: the central `App` struct, the services it owns, the event bus that connects the backend to the UI, and the two distinct execution paths — interactive (TUI) and non-interactive (`crush run`).

For the full application startup sequence, see [Application Lifecycle](https://deepwiki.com/charmbracelet/crush/3.1-application-lifecycle)
. For the configuration system, see [Configuration System](https://deepwiki.com/charmbracelet/crush/3.2-configuration-system)
. For the agent subsystem, see [Agent System](https://deepwiki.com/charmbracelet/crush/4-agent-system)
. For details on the server/client architecture, see [Server Mode and Backend](https://deepwiki.com/charmbracelet/crush/3.9-server-mode-and-backend)
.

Component Overview
------------------

The runtime is organized around a single `App` struct that owns all services, wires up an internal event channel, and drives either a Bubble Tea TUI program or a headless non-interactive loop. In modern versions of Crush, this can also be proxied through a `Workspace` abstraction that allows the UI to talk to either a local `App` or a remote `Server`.

The `App` Struct
----------------

`App` is the central runtime object, defined in `internal/app/app.go`. All major services are fields on this struct and are constructed once during startup via `app.New()`.

| Field | Type | Role |
| --- | --- | --- |
| `Sessions` | `session.Service` | Create, retrieve, update sessions |
| `Messages` | `message.Service` | Store and stream messages |
| `History` | `history.Service` | Track file edit history |
| `Permissions` | `permission.Service` | Gate tool execution |
| `FileTracker` | `filetracker.Service` | Record which files the agent touches |
| `AgentCoordinator` | `agent.Coordinator` | Orchestrate LLM sessions and tool execution |
| `LSPManager` | `*lsp.Manager` | Manage Language Server Protocol clients |
| `events` | `*pubsub.Broker[tea.Msg]` | Internal bus: all service events flow here |
| `globalCtx` | `context.Context` | Root context; cancellation triggers shutdown |
| `cleanupFuncs` | `[]func(context.Context) error` | Ordered teardown steps |

Sources: [internal/app/app.go54-78](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/app/app.go#L54-L78)

`app.New()` accepts a `*sql.DB` (already connected and migrated), a `*config.ConfigStore`, and a `*skills.Manager`. It constructs all services, calls `setupEvents()`, starts MCP initialization in the background, and initializes the `AgentCoordinator` via `InitCoderAgent()` if a provider is configured.

Sources: [internal/app/app.go84-152](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/app/app.go#L84-L152)

Execution Paths
---------------

Crush supports two execution paths: interactive (TUI) and non-interactive (`run`). These are now unified under the `Workspace` interface, which abstracts whether the logic is running locally or via a remote `crush server`.

**Execution Path Diagram**

Sources: [internal/cmd/root.go106-145](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/cmd/root.go#L106-L145)
 [internal/app/app.go213-350](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/app/app.go#L213-L350)
 [internal/workspace/client\_workspace.go173-175](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/workspace/client_workspace.go#L173-L175)

### Interactive Path

1.  `rootCmd.RunE` calls `setupWorkspaceWithProgressBar(cmd)`, which returns a `workspace.Workspace` (either `AppWorkspace` or `ClientWorkspace`).
2.  It constructs `ui.New(com, ...)` (the Bubble Tea model) and a `tea.NewProgram`.
3.  `ws.Subscribe(program)` is launched in a goroutine — it listens for backend events (via SSE for remote or internal broker for local) and forwards them to the TUI.
4.  `program.Run()` blocks until the user quits.

Sources: [internal/cmd/root.go106-145](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/cmd/root.go#L106-L145)
 [internal/workspace/client\_workspace.go50-61](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/workspace/client_workspace.go#L50-L61)

### Non-Interactive Path

1.  `runCmd.RunE` calls `app.RunNonInteractive(ctx, os.Stdout, prompt, ...)` on the underlying app (if local).
2.  Inside `RunNonInteractive`, it resolves the session via `app.resolveSession`.
3.  It calls `app.AgentCoordinator.Run` to execute the prompt.
4.  It sets up a subscription to the event stream to print assistant content directly to stdout as it is generated.

Sources: [internal/app/app.go213-350](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/app/app.go#L213-L350)
 [internal/cmd/root.go106-145](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/cmd/root.go#L106-L145)

Permission System
-----------------

The `permission.Service` interface intercepts every tool call. The `Request` method walks a priority chain:

1.  **YOLO mode**: if `--yolo` is set, return `true` immediately.
2.  **Allowed tools**: if the tool is in the global `allowedTools` list, return `true`.
3.  **Auto-approve sessions**: if the session is marked for auto-approval (e.g. non-interactive), return `true`.
4.  **Session permissions**: check if this specific tool/action/path was already approved in this session.
5.  **User prompt**: publish a `PermissionRequest` event and block until the UI sends a response.

Sources: [internal/app/app.go100](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/app/app.go#L100-L100)
 [internal/cmd/root.go58](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/cmd/root.go#L58-L58)

For details, see [Permission System](https://deepwiki.com/charmbracelet/crush/3.6-permission-system)
.

Shutdown Sequence
-----------------

`app.Shutdown()` follows a strict order to prevent data corruption:

1.  `AgentCoordinator.CancelAll()` — stops all in-flight LLM calls.
2.  Runs all `cleanupFuncs` concurrently (with a timeout):
    *   Closes the SQLite database connection.
    *   Closes MCP clients and transports via `mcp.Close(ctx)`.
    *   Cancels background event subscriptions.
3.  `app.LSPManager.KillAll()` — force-kills LSP server processes.
