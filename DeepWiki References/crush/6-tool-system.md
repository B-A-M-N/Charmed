The Tool System provides the mechanism by which AI agents interact with code, files, shell commands, and external services. Tools are structured functions that agents can invoke during conversation turns, enabling autonomous code editing, diagnostics retrieval, web scraping, and other operations. This page covers the core tool architecture, registration lifecycle, execution model, and permission integration. For specific tool implementations, see the child pages: [Shell Tool](https://deepwiki.com/charmbracelet/crush/6.1-shell-tool)
, [File System Tools](https://deepwiki.com/charmbracelet/crush/6.2-file-system-tools)
, [Search Tools](https://deepwiki.com/charmbracelet/crush/6.3-search-tools)

, [MCP Integration](https://deepwiki.com/charmbracelet/crush/6.5-mcp-integration)

Core Architecture
-----------------

Tools in Crush implement the `fantasy.AgentTool` interface from the Fantasy AI framework. The `coordinator` is responsible for building and filtering tools based on agent configuration, while `SessionAgent` instances receive tools during initialization and invoke them when the LLM requests tool use.

**Tool Architecture: from `coordinator` to `fantasy.AgentTool`**

Sources: [internal/agent/coordinator.go401-491](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L401-L491)
 [internal/agent/agent.go188-197](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/agent.go#L188-L197)
 [internal/agent/tools/tools.go14-23](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/tools.go#L14-L23)

The `coordinator` receives service dependencies (`permission.Service`, `lsp.Manager`, `filetracker.Service`, `history.Service`) and injects them into tool constructors. Each tool constructor returns a `fantasy.AgentTool` which exposes metadata via `Info()` and execution logic via `Execute()`. The coordinator filters tools based on the agent's `AllowedTools` configuration before passing them to the `SessionAgent`.

Tool Registration Lifecycle
---------------------------

Tool registration occurs during coordinator initialization and can be updated dynamically via `UpdateModels()`. The coordinator maintains separate tool sets for different agent types (Coder vs Task agents).

**Tool Registration Flow**

Sources: [internal/agent/coordinator.go77-116](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L77-L116)
 [internal/agent/coordinator.go401-491](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L401-L491)

The `buildTools()` method at [internal/agent/coordinator.go401-491](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L401-L491)
 constructs all available tools, then filters them against the agent configuration's `AllowedTools` slice. MCP tools are fetched via `tools.GetMCPTools()` and receive additional filtering via `AllowedMCP`, which maps server names to permitted tool names. Tools are sorted alphabetically to ensure consistent ordering across sessions.

Permission Integration
----------------------

Tools that perform potentially dangerous operations integrate with the permission system. The `permission.Service` gates tool execution based on user approval, "yolo mode", or pre-approval from hooks.

**Permission Check Logic**

Sources: [internal/permission/permission.go162-242](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L162-L242)

Tools receive the `permission.Service` during construction. When a tool requires authorization, it calls `permissions.Request()`. If the user denies the request via the UI, the tool returns a standard denied response [internal/agent/tools/tools.go66-70](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/tools.go#L66-L70)
 Pre-tool hooks can also inject approval into the context using `WithHookApproval`, which short-circuits the prompt [internal/permission/permission.go24-26](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L24-L26)

Specialized Tools
-----------------

### Sub-Agent Tools

Crush includes tools that allow the primary agent to spawn specialized sub-agents.

*   **`agent`**: Spawns a full-featured sub-agent for complex tasks.
*   **`agentic_fetch`**: Spawns a specialized agent for web searching and deep content analysis.

### Built-in Skills

The tool system supports "builtin skills" using the `crush://` URI scheme. This allows agents to read documentation and instructions stored within the Crush binary.

### Diagnostics and State

*   **`crush_info`**: Provides a comprehensive snapshot of the environment, including loaded config files, model selection, LSP/MCP health, and active skills.
*   **`crush_logs`**: Allows the agent to read the last N lines of its own execution logs to debug its internal processes.

Tool Response Types
-------------------

Tools return `fantasy.ToolResponse` which supports text, error, and media responses.

| Response Type | Constructor | Use Case |
| --- | --- | --- |
| Text | `fantasy.NewTextResponse(text)` | Standard successful output |
| Error | `fantasy.NewTextErrorResponse(text)` | Expected error conditions |
| Image | `fantasy.NewImageResponse(data, mime)` | Returning screenshots or diagrams |
| Permission Denied | `tools.NewPermissionDeniedResponse()` | User explicitly blocked the tool call [internal/agent/tools/tools.go66-70](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/tools.go#L66-L70) |
