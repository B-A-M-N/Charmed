This page covers the agent layer of crush: the `Coordinator` and `SessionAgent` abstractions, how they interact with LLM providers, and how tools extend agent capabilities. For details on individual tool implementations see [Tool System](https://deepwiki.com/charmbracelet/crush/6-tool-system)
. For provider configuration and model selection see [Provider Integration](https://deepwiki.com/charmbracelet/crush/4.2-provider-integration)
 and [Model Configuration](https://deepwiki.com/charmbracelet/crush/4.3-model-configuration)
. For message storage and session tracking see [Session Management](https://deepwiki.com/charmbracelet/crush/3.5-session-management)
.

* * *

Overview
--------

The agent system sits between the user interface and the LLM providers. It receives a prompt (and optional file attachments) from the TUI or the non-interactive CLI, assembles conversation history, dispatches tool calls, streams partial responses back through events, and persists all messages to the database.

The layer is split into two roles:

| Role | Interface | Concrete type | File |
| --- | --- | --- | --- |
| Session orchestration | `Coordinator` | `coordinator` | [internal/agent/coordinator.go73-87](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L73-L87) |
| Per-session execution | `SessionAgent` | `sessionAgent` | [internal/agent/agent.go86-100](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/agent.go#L86-L100) |

The `Coordinator` owns provider construction, tool assembly, and model resolution. The `SessionAgent` owns the actual LLM streaming loop, queuing, cancellation, and auto-summarization for a single active session.

### Construction: `NewCoordinator`

`NewCoordinator` is called once during application startup via `app.InitCoderAgent` [internal/app/app.go136](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/app/app.go#L136-L136)
 It:

1.  Discovers "skills" (custom agent instructions) available in the environment [internal/agent/coordinator.go126-132](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L126-L132)
    
2.  Reads the `coder` agent configuration from `config.Config.Agents` [internal/agent/coordinator.go150-153](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L150-L153)
    
3.  Calls `coderPrompt(...)` to build the system prompt template [internal/agent/coordinator.go156-159](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L156-L159)
    
4.  Calls `buildAgent(...)` [internal/agent/coordinator.go161](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L161-L161)
     which:
    *   Resolves the large and small `Model` via `buildAgentModels` [internal/agent/coordinator.go432-442](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L432-L442)
        
    *   Creates a `SessionAgent` via `NewSessionAgent` [internal/agent/coordinator.go444-455](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L444-L455)
        
5.  Stores the agent as `c.currentAgent` and maps it to the `AgentCoder` key [internal/agent/coordinator.go165-166](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/coordinator.go#L165-L166)

### `Model` struct

`Model` bundles three views of the same model: the runtime provider handle (`fantasy.LanguageModel`), the model metadata from `charm.land/catwalk` (context window size, capabilities), and the user's configuration (`config.SelectedModel`).

Auto-Summarization
------------------

**Auto-summarization trigger logic**

Thresholds are defined as constants:

| Constant | Value | Used when |
| --- | --- | --- |
| `largeContextWindowThreshold` | `200_000` | Determines which formula to use |
| `largeContextWindowBuffer` | `20_000` | Buffer for large context windows |
| `smallContextWindowRatio` | `0.2` | 20% of window for smaller models |

Sources: [internal/agent/agent.go52-56](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/agent.go#L52-L56)

Sub-Agent Spawning
------------------

The `AgentTool` (named `"agent"`) allows the primary coder agent to delegate a task to a child `SessionAgent`.

**Sub-agent spawning via AgentTool**

Key details:

*   The sub-agent is built using `config.AgentTask` configuration [internal/agent/agent\_tool.go27-30](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/agent_tool.go#L27-L30)
    
*   The sub-agent is created as `isSubAgent: true`, which affects UI rendering and instruction sets [internal/agent/agent\_tool.go36](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/agent_tool.go#L36-L36)
    
*   The session ID of the child is tracked to ensure costs and history are correctly associated with the parent [internal/agent/agent\_tool.go58-65](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/agent_tool.go#L58-L65)
    

Sources: [internal/agent/agent\_tool.go26-67](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/agent_tool.go#L26-L67)
