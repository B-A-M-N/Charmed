Purpose and Scope
-----------------

This document describes Crush's Language Server Protocol (LSP) integration, which provides code intelligence capabilities to AI agents. The LSP system manages connections to language servers, synchronizes file state, collects diagnostics, and exposes code analysis through tools. This integration allows agents to access the same code intelligence features developers use in their editors, including diagnostics, references, and type information.

The LSP subsystem is built around the `lsp.Manager` for lifecycle orchestration and `lsp.Client` for individual server communication.

**Sources:** [internal/lsp/manager.go1-22](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/manager.go#L1-L22)
 [internal/lsp/client.go1-30](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L1-L30)

Architecture Overview
---------------------

The LSP integration follows a manager-coordinator architecture where a `Manager` instance lazily starts and coordinates multiple LSP server clients. Each language server runs as an independent process managed by a `Client` instance. Clients communicate using the `powernap` library, which implements the LSP protocol over JSON-RPC 2.0.

Client Lifecycle
----------------

### Initialization Sequence

LSP clients are initialized lazily when files are accessed. The `Manager.Start` method detects which servers should handle a file and starts them asynchronously.

**Diagram: LSP Client State Machine**

The initialization sequence is managed by `Manager.Start` and `Manager.startServer`:

1.  **File Detection**: `Manager.Start` [internal/lsp/manager.go101-113](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/manager.go#L101-L113)
     checks if the file path is within the working directory and iterates through servers.
2.  **Command Validation**: For non-user-configured servers, the manager validates that the command exists using `exec.LookPath` [internal/lsp/manager.go175-179](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/manager.go#L175-L179)
     Generic commands like `node` or `python` are skipped for auto-start [internal/lsp/manager.go117-145](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/manager.go#L117-L145)
    
3.  **Client Creation**: `lsp.New` [internal/lsp/client.go69-95](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L69-L95)
     creates a Client instance, resolving the command via `VariableResolver` [internal/lsp/client.go166-170](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L166-L170)
    
4.  **LSP Handshake**: `Client.Initialize` [internal/lsp/client.go98-126](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L98-L126)
     calls the powernap `Initialize` method and registers standard handlers for diagnostics and workspace edits [internal/lsp/client.go206-218](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L206-L218)
    
5.  **Ready Wait**: `Client.WaitForServerReady` (referenced in [internal/lsp/manager.go247-252](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/manager.go#L247-L252)
    ) polls the server until it reports ready or the timeout expires.

**Sources:** [internal/lsp/manager.go99-255](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/manager.go#L99-L255)
 [internal/lsp/client.go69-126](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L69-L126)

### Shutdown and Force Termination

The `Manager` provides two ways to terminate clients:

*   **StopAll**: Gracefully shuts down all clients (referenced in [internal/lsp/manager.go321](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/manager.go#L321-L321)
    ).
*   **KillAll**: Forcefully terminates all client processes immediately (referenced in [internal/lsp/manager.go344](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/manager.go#L344-L344)
    ).

Individual clients implement `Close(ctx)` [internal/lsp/client.go136-160](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L136-L160)
 which attempts a graceful LSP shutdown with a 5-second timeout [internal/lsp/client.go129](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L129-L129)
 before falling back to `Kill()` [internal/lsp/client.go132](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L132-L132)

**Sources:** [internal/lsp/manager.go321-364](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/manager.go#L321-L364)
 [internal/lsp/client.go129-160](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L129-L160)

*   **OpenFile**: Ensures a file is "opened" in the LSP server (referenced in [internal/lsp/client.go212](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L212-L212)

LSP Agent Tools
---------------

Crush provides three primary tools to agents for interacting with the LSP subsystem:

### 1\. lsp\_diagnostics

Exposes code diagnostics (errors/warnings) to the agent.

*   **Implementation**: `NewDiagnosticsTool` [internal/agent/tools/diagnostics.go27-40](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/diagnostics.go#L27-L40)
     uses `getDiagnostics` [internal/agent/tools/diagnostics.go133-181](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/diagnostics.go#L133-L181)
     to aggregate results from all active clients.
*   **Data Flow**: Agent requests diagnostics -> Tool calls `notifyLSPs` [internal/agent/tools/diagnostics.go93-131](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/diagnostics.go#L93-L131)
     to ensure servers are synced -> Tool queries `lsp.Manager` for active clients -> Clients return cached `protocol.Diagnostic` arrays.
*   **Documentation**: [internal/agent/tools/diagnostics.md1](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/diagnostics.md?plain=1#L1-L1)
    

### 2\. lsp\_references

Finds all references to a symbol by name via LSP; more accurate than grep for code symbols.

*   **Implementation**: `NewReferencesTool` [internal/agent/tools/references.go36-92](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/references.go#L36-L92)
     uses `find` [internal/agent/tools/references.go98-124](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/references.go#L98-L124)
     which calls `Client.FindReferences` (referenced in [internal/agent/tools/references.go117](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/references.go#L117-L117)
    ).
*   **Process**:
    1.  Greps for the symbol to find potential locations [internal/agent/tools/references.go51-54](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/references.go#L51-L54)
        
    2.  Locates the appropriate `lsp.Client` that handles the file [internal/agent/tools/references.go104-115](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/references.go#L104-L115)
        
    3.  Sends `textDocument/references` request to the server [internal/agent/tools/references.go117-123](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/references.go#L117-L123)
        
    4.  Translates `protocol.Location` results back to the agent [internal/agent/tools/references.go174-194](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/references.go#L174-L194)
        
*   **Documentation**: [internal/agent/tools/references.md1](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/references.md?plain=1#L1-L1)
    

### 3\. lsp\_restart

Restarts a specific LSP server or all servers.

*   **Implementation**: Calls `Client.Restart()` [internal/lsp/client.go221-275](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/client.go#L221-L275)
    
*   **Process**: Closes the existing connection, clears caches, recreates the powernap client, and re-initializes.
*   **Documentation**: [internal/agent/tools/lsp\_restart.md1](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/agent/tools/lsp_restart.md?plain=1#L1-L1)

File Watcher Integration
------------------------

LSP servers can register to watch files for changes. Crush handles these registrations through `HandleRegisterCapability` [internal/lsp/handlers.go19-45](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/handlers.go#L19-L45)

1.  Server sends `client/registerCapability` with `workspace/didChangeWatchedFiles` [internal/lsp/handlers.go28](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/handlers.go#L28-L28)
    
2.  `lsp.RegisterFileWatchHandler` [internal/lsp/handlers.go72-74](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/handlers.go#L72-L74)
     allows the application to plug in a filesystem watcher.
3.  When the watcher detects changes, it notifies the handler which can then trigger updates to the LSP client.

**Sources:** [internal/lsp/handlers.go19-81](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/handlers.go#L19-L81)

---------------

     It handles UTF-8, UTF-16, and UTF-32 offset encodings [internal/lsp/util/edit.go102-120](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/util/edit.go#L102-L120)
    
*   **Document Changes**: `applyDocumentChange` [internal/lsp/util/edit.go171-234](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/lsp/util/edit.go#L171-L234)
     handles file creation, deletion, and renaming operations requested by the server.
