Crush supports a client/server architecture that allows a long-running background process to manage multiple workspaces, handle resource-intensive AI agent operations, and provide a centralized API for multiple client interfaces (TUI or CLI).

Overview
--------

The server mode is enabled by setting the `CRUSH_CLIENT_SERVER` environment variable [internal/cmd/root.go213-216](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/cmd/root.go#L213-L216)
 When active, the `crush` CLI acts as a thin client that proxies requests to a local or remote server over Unix sockets, Windows named pipes, or TCP [internal/server/server.go53-57](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/server/server.go#L53-L57)

### High-Level Architecture

The server encapsulates a `Backend` which manages the lifecycle of multiple `Workspaces`. Each workspace contains its own configuration, agent coordinator, and database services.

The Server Command
------------------

The `crush server` command (implemented via `serverCmd`) initializes the background process. It loads the global configuration and sets up a listener based on the provided host flag.

*   **Default Host:** Uses a UID-specific Unix socket on Linux/macOS (`/tmp/crush-1000.sock`) or a named pipe on Windows (`\\.\pipe\crush-1000.sock`) [internal/server/server.go47-57](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/server/server.go#L47-L57)
    
*   **Logging:** In server mode, slog is configured to output to the server's log handler, which may include file-based logging to the data directory [internal/cmd/root.go165-173](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/cmd/root.go#L165-L173)
    
*   **Graceful Shutdown:** The server provides a `/v1/control` endpoint that accepts a `shutdown` command, triggering the `backend.Shutdown()` sequence which closes all workspaces [internal/server/proto.go49-65](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/server/proto.go#L49-L65)
     [internal/backend/backend.go208-219](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/backend/backend.go#L208-L219)

Backend and Workspace Abstraction
---------------------------------

The `Backend` struct is the central orchestrator within the server. It manages a thread-safe map of active workspaces using `csync.Map` and provides a bridge to the `controllerV1` for API requests [internal/backend/backend.go40-45](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/backend/backend.go#L40-L45)

### Workspace Interface

The `workspace.Workspace` interface abstracts all operations that can be performed on a project. `ClientWorkspace` implements this by delegating to the `internal/client` SDK [internal/workspace/client\_workspace.go34-40](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/workspace/client_workspace.go#L34-L40)

| Feature | `AppWorkspace` (Local) | `ClientWorkspace` (Remote) |
| --- | --- | --- |
| **Implementation** | Calls `internal/app` services directly | Proxies via `internal/client` SDK [internal/workspace/client\_workspace.go93-131](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/workspace/client_workspace.go#L93-L131) |
| **Data Flow** | In-memory / Local SQLite | HTTP/JSON over Socket [internal/client/proto.go24-38](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/client/proto.go#L24-L38) |
| **Event Handling** | Direct channel subscription | SSE (Server-Sent Events) [internal/client/proto.go88-103](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/client/proto.go#L88-L103) |
| **Caching** | Direct access | Caches `proto.Workspace` snapshot [internal/workspace/client\_workspace.go80-84](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/workspace/client_workspace.go#L80-L84) |

### Event Streaming (SSE)

The server streams real-time updates using the `/events` endpoint. The `wrapEvent` function converts internal `pubsub.Event[T]` types (like `message.Message` or `permission.PermissionRequest`) into a `pubsub.Payload` envelope for JSON transport [internal/server/events.go24-104](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/server/events.go#L24-L104)
 The client unmarshals these based on the `PayloadType` tag [internal/client/proto.go138-178](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/client/proto.go#L138-L178)

### Skill Discovery

Workspaces discover "Skills" (standardized agent capabilities) at creation time. The `Backend.CreateWorkspace` call initializes a `skills.Manager` and snapshots the state into the `proto.Workspace` returned to the client [internal/backend/backend.go115-134](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/backend/backend.go#L115-L134)
 Subsequent skill updates flow through the SSE stream [internal/server/events.go95-99](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/server/events.go#L95-L99)

Variable Resolution
-------------------

When running in server mode, configuration variables (like API keys or paths) are resolved within the `Backend` context. The `Backend` uses the `config.ConfigStore` to manage these settings [internal/backend/backend.go42-43](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/backend/backend.go#L42-L43)

*   **Workspace Creation:** During `CreateWorkspace`, the server initializes a `ConfigStore` for the specific project path and data directory provided by the client [internal/backend/backend.go95-98](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/backend/backend.go#L95-L98)
    
*   **Skill Paths:** The `skillsDiscoveryConfig` helper adapts the workspace configuration into a format usable by the skill discovery system, including resolving environment variables via the config's resolver [internal/backend/backend.go166-183](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/backend/backend.go#L166-L183)
