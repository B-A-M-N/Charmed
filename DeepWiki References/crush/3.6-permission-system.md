Permission System
=================

The permission system is the gatekeeper between the agent's tool calls and actual execution. Every tool invocation that has side-effects â€” writing files, running shell commands, fetching URLs, executing MCP tools â€” must pass through `permission.Service` before proceeding. This page covers the `Service` interface, its internal priority chain inside `Request()`, the grant/deny lifecycle, and the YOLO mode flag.

For the configuration fields (`Permissions.AllowedTools`, `Permissions.SkipRequests`) that feed into this system, see the Configuration System page [3.2](https://github.com/charmbracelet/crush/blob/1ceeaa93/3.2)
 For how the UI renders the permission dialog and diff view when a request arrives, see the Dialogs page [5.6](https://github.com/charmbracelet/crush/blob/1ceeaa93/5.6)

### Pre-Approval Hooks

The system supports pre-approval via `PreToolUse` hooks. If a hook returns an `allow` decision, the context is stamped with the tool call ID using `WithHookApproval`. The `Request` method checks this via `hookApproved` to bypass the prompt. [internal/permission/permission.go21-36](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L21-L36)

### Allowlist Format

The `allowedTools` slice (configured in `Permissions.AllowedTools`) accepts two formats:

| Format | Example | Matches |
| --- | --- | --- |
| `toolName` | `"bash"` | Any action on `bash` |
| `toolName:action` | `"bash:execute"` | Only `bash` with action `execute` |

Grant / Deny Lifecycle
----------------------

Once a `PermissionRequest` has been published to the broker and the UI displays the dialog, the user responds via one of three methods:

**Grant/Deny sequence diagram**

Sources: [internal/permission/permission.go103-160](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L103-L160)

| Method | Publishes Notification | Sends to `respCh` | Adds to `sessionPermissions` |
| --- | --- | --- | --- |
| `Grant` | `Granted: true` | `true` | No [internal/permission/permission.go127-142](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L127-L142) |
| `GrantPersistent` | `Granted: true` | `true` | **Yes** [internal/permission/permission.go103-125](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L103-L125) |
| `Deny` | `Granted: false, Denied: true` | `false` | No [internal/permission/permission.go144-160](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L144-L160) |

`GrantPersistent` is the "Allow for this session" action â€” subsequent identical requests (same `ToolName`, `Action`, `SessionID`, and directory `Path`) are auto-approved. YOLO Mode (`--yolo` / `-y`)
---------------------------

Passing `--yolo` or `-y` to `crush` (or setting `skip_permission_requests` in config) sets the permission service to skip all requests. When `skip == true`, `Request()` returns `true` immediately at the first check. [internal/permission/permission.go163-165](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L163-L165)
 [internal/permission/permission\_test.go97-113](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission_test.go#L97-L113)

`SetSkipRequests(bool)` and `SkipRequests() bool` allow toggling this at runtime using atomic operations for thread safety. [internal/permission/permission.go275-281](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L275-L281)
 [internal/permission/permission\_test.go82-95](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission_test.go#L82-L95)

Concurrency Model
-----------------

*   **Serialization**: `requestMu` ensures only one permission request is active at a time â€” requests are serialized, not parallelized. Multiple tool calls arriving concurrently will queue up and wait for the user to respond to each in sequence. [internal/permission/permission.go185-186](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L185-L186)
    
*   **Signaling**: `pendingRequests` is a `csync.Map` (see [3.7](https://github.com/charmbracelet/crush/blob/1ceeaa93/3.7)
    ) that maps `PermissionRequest.ID -> chan bool`. This allows the UI (calling `Grant`/`Deny`) to signal the correct blocked goroutine. [internal/permission/permission.go91](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L91-L91)
     [internal/permission/permission.go108-111](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L108-L111)
    
*   **Guards**: `sessionPermissions` is a `csync.Map` providing thread-safe access to persistent grants. `autoApproveSessions` uses `sync.RWMutex` to allow concurrent reads during the priority check. [internal/permission/permission.go90-93](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L90-L93)
     [internal/permission/permission.go193-195](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/permission/permission.go#L193-L195)
