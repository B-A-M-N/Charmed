Contributor License Agreement
-----------------------------

All contributors must sign a Contributor License Agreement (CLA) before their pull request can be merged. The CLA text is in `CLA.md` [\[CLA.md:1-19\]](https://deepwiki.com/charmbracelet/crush/8.4-contributing)
.

The signing process is fully automated via the [CLA Assistant](https://github.com/charmbracelet/crush/blob/1ceeaa93/CLA%20Assistant)
 GitHub Action, configured at [.github/workflows/cla.yml](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/cla.yml)

### How the CLA Flow Works

The system uses a comment-based signing mechanism. When a contributor comments with a specific phrase, the CLA Assistant updates a JSON database of signatures stored directly in the repository.

**CLA Assistant workflow diagram:**

Sources: [.github/workflows/cla.yml](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/cla.yml)
 [.github/cla-signatures.json](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/cla-signatures.json)

### Signing Step-by-Step

1.  Open a pull request against `charmbracelet/crush`.
    
2.  The CLA Assistant bot automatically checks whether your GitHub username appears in [.github/cla-signatures.json](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/cla-signatures.json)
    
3.  If you have **not** signed, the bot posts a comment explaining what to do and the PR status check fails [\[.github/workflows/cla.yml:34-34\]](https://deepwiki.com/charmbracelet/crush/8.4-contributing)
    .
    
4.  Post the following exact comment on your PR:
    
        I have read the Contributor License Agreement (CLA) and hereby sign the CLA.
        
    
5.  The bot records your signature in [.github/cla-signatures.json](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/cla-signatures.json)
     under the `signedContributors` array [\[.github/cla-signatures.json:2-3\]](https://deepwiki.com/charmbracelet/crush/8.4-contributing)
    .
    
6.  The status check updates to passing.
    

You can also trigger a re-check of an existing PR by commenting `recheck` [\[.github/workflows/cla.yml:22-22\]](https://deepwiki.com/charmbracelet/crush/8.4-contributing)
.

Issue and PR Labeling
---------------------

Labels are applied automatically when an issue or PR is opened, driven by regex matching against the title. The labeler runs via [.github/workflows/labeler.yml](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/labeler.yml)

Code Style Requirements
-----------------------

Crush enforces code quality through linting and standard Go formatting.

### Linters and Formatting

The project uses `golangci-lint` with a specific set of enabled linters [\[.golangci.yml:2-26\]](https://deepwiki.com/charmbracelet/crush/8.4-contributing)
:

*   `bodyclose`: Checks whether HTTP response bodies are closed.
*   `gofumpt` & `goimports`: Enforced via formatters [\[.golangci.yml:44-46\]](https://deepwiki.com/charmbracelet/crush/8.4-contributing)
    .
*   `staticcheck`: Applies various static analysis checks.
*   `misspell`: Finds commonly misspelled English words in comments.

To run the linters locally, ensure you have `golangci-lint` installed and execute:

### Build Constraints

The codebase targets multiple platforms [\[.goreleaser.yml:48-55\]](https://deepwiki.com/charmbracelet/crush/8.4-contributing)
. Contributors should avoid platform-specific APIs unless wrapped in build tags. The project uses `CGO_ENABLED=0` for pure-Go builds across all targets [\[.goreleaser.yml:46-46\]](https://deepwiki.com/charmbracelet/crush/8.4-contributing)
.

Building and Testing Locally
----------------------------

The standard CI build runs on Ubuntu, macOS, and Windows. All checks that run in CI can be reproduced locally.

### Running Tests

Tests should be run with the race detector enabled to catch concurrency issues in the agent and TUI event loops.

### Manual TUI Testing

Because `crush` is a terminal application, automated tests cannot cover all visual aspects. Contributors are encouraged to build and run the binary locally to verify TUI changes:

The build process injects the version string via `ldflags` [\[.goreleaser.yml:72-73\]](https://deepwiki.com/charmbracelet/crush/8.4-contributing)
.
