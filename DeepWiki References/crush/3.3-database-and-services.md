This page covers the SQLite database setup, the migration system, the generated query layer, and the service objects that encapsulate domain logic for sessions, messages, and project history. These services are owned by `App` and are wired into the agent coordinator at startup.

* * *

Overview
--------

The storage layer is built on SQLite and follows a three-tier pattern:

1.  **Database driver** — SQLite, accessed via `database/sql`.
2.  **Query layer** — generated accessor functions in `internal/db` using `sqlc`.
3.  **Service layer** — domain-scoped structs (`session.service`, `message.service`, `history.service`) that wrap the query layer with business logic and `pubsub` event emission.

Database Backend and Migrations
-------------------------------

The database connection is initialized via `db.Connect`, which configures SQLite pragmas for performance and runs migrations using `goose`.

### Connection and Pragmas

The system uses different drivers based on the build target: `modernc.org/sqlite` for most platforms [internal/db/connect\_modernc.go10-11](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect_modernc.go#L10-L11)
 and `github.com/ncruces/go-sqlite3` for others [internal/db/connect\_ncruces.go9-10](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect_ncruces.go#L9-L10)

To prevent `SQLITE_NOTADB` errors caused by concurrent WAL/header desync, the system serializes all access through a single connection using `conn.SetMaxOpenConns(1)` [internal/db/connect.go90](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect.go#L90-L90)

| Pragma | Value | Purpose |
| --- | --- | --- |
| `foreign_keys` | `ON` | Enforce relational integrity [internal/db/connect.go18](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect.go#L18-L18) |
| `journal_mode` | `WAL` | Enable Write-Ahead Logging for concurrent access [internal/db/connect.go19](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect.go#L19-L19) |
| `synchronous` | `NORMAL` | Balance between safety and performance [internal/db/connect.go23](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect.go#L23-L23) |
| `busy_timeout` | `30000` | Wait 30s for locked tables before failing [internal/db/connect.go25](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect.go#L25-L25) |

### Migration System

Database schema versioning is managed by `github.com/pressly/goose/v3`. The migration files (located in `internal/db/migrations`) are embedded into the binary using `go:embed`.

*   **Initialization**: `goose.SetBaseFS(FS)` [internal/db/connect.go35](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect.go#L35-L35)
    
*   **Dialect**: `goose.SetDialect("sqlite3")` [internal/db/connect.go154](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect.go#L154-L154)
    
*   **Execution**: `goose.Up(conn, "migrations")` [internal/db/connect.go103](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/connect.go#L103-L103)

Query Layer — `internal/db`
---------------------------

The `internal/db` package contains code generated by `sqlc` from SQL definitions in `internal/db/sql/`. It defines a `Querier` interface and a `Queries` struct that maps SQL files to Go methods.

Key generated components:

*   `db.Queries`: The concrete implementation of database operations [internal/db/db.go20-22](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/db.go#L20-L22)
    
*   `db.Querier`: An interface containing all database methods (e.g., `CreateMessage`, `GetSessionByID`), allowing for easier testing/mocking [internal/db/querier.go11-49](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/querier.go#L11-L49)
    
*   **Prepared Statements**: `db.Prepare` initializes prepared statements for all queries to improve execution speed [internal/db/db.go24-138](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/db.go#L24-L138)
    
*   **Models**: Defines structures like `db.Session`, `db.Message`, and `db.File` that mirror the database schema [internal/db/models.go11-52](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/models.go#L11-L52)

Service Layer Implementation
----------------------------

Services wrap the database queries and handle JSON (un)marshaling of complex types (like message parts or session todos) and event broadcasting via `pubsub.Broker`.

### `session.Service`

Manages the lifecycle of conversation sessions. A session is the top-level unit of history and tracks token usage and costs.

*   **Storage**: Maps to the `sessions` table [internal/db/sql/sessions.sql1-24](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/sql/sessions.sql#L1-L24)
    
*   **Specialized Creators**:
    *   `CreateTaskSession`: Used for sub-agents or tool-driven child sessions [internal/session/session.go110-122](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L110-L122)
        
    *   `CreateTitleSession`: A session used to generate a title for the parent session [internal/session/session.go124-136](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L124-L136)
        
*   **Atomic Updates**: `UpdateTitleAndUsage` performs atomic increments of tokens and cost directly in the database to prevent lost updates [internal/session/session.go225-233](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L225-L233)
    
*   **Todo Management**: Handles the `todos` field, which is stored as a JSON string in the DB [internal/session/session.go191-221](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L191-L221)
    

Sources: [internal/session/session.go65-82](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L65-L82)
 [internal/session/session.go225-233](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L225-L233)
 [internal/db/sessions.sql.go13-37](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/sessions.sql.go#L13-L37)

### `history.Service` (File Tracker)

Manages file versions and history for sessions. This allows the agent to track changes to files over time.

*   **Versioning**: Implements an auto-incrementing version system for files. `CreateVersion` fetches the latest version for a path and increments it [internal/history/file.go65-82](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/history/file.go#L65-L82)
    
*   **Transactions**: Uses SQL transactions in `createWithVersion` to ensure uniqueness of `(path, session_id, version)` and handle retries on conflict [internal/history/file.go84-135](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/history/file.go#L84-L135)
    
*   **Retrieval**: Provides methods to list all files in a session or just the latest version of each file [internal/history/file.go156-178](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/history/file.go#L156-L178)

Usage Statistics and Heatmaps
-----------------------------

The database also supports complex queries for usage analytics, which are used by the `crush stats` command.

*   **Heatmaps**: `GetHourDayHeatmap` retrieves activity density by hour and day [internal/db/querier.go24](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/querier.go#L24-L24)
    
*   **Aggregates**: `GetTotalStats` and `GetUsageByModel` provide high-level cost and token summaries [internal/db/querier.go30-34](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/querier.go#L30-L34)
    
*   **Recent Activity**: `GetRecentActivity` lists the latest interactions across all sessions [internal/db/querier.go27](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/querier.go#L27-L27)
