This page covers the `message` package data model, how messages are created and persisted, how they flow through the agent execution pipeline, and how attachments are handled. For information about session lifecycle (session creation, summarization triggers, usage tracking), see [3.5 Session Management](https://github.com/charmbracelet/crush/blob/1ceeaa93/3.5%20Session%20Management)
 For how messages are rendered in the TUI chat view, see [5.3 Message Rendering](https://github.com/charmbracelet/crush/blob/1ceeaa93/5.3%20Message%20Rendering)

Overview
--------

The message system is the shared data layer between the agent execution engine and the TUI. Every exchange — user input, assistant text, reasoning tokens, tool invocations, and tool results — is represented as a `message.Message` persisted via `message.Service`. The TUI reads these records to reconstruct the visible chat, and the agent reads them to rebuild LLM context on every turn.

`message.Service`
-----------------

`message.Service` is the persistence interface for all message operations. It is backed by SQLite via the `db` package and `sqlc` generated queries.

`message.NewService(q)` constructs a service from a `db.Querier` [internal/message/message.go124-135](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/message/message.go#L124-L135)
 It uses an internal `pubsub.Broker` to publish lifecycle events (`CreatedEvent`, `UpdatedEvent`, `DeletedEvent`) to notify the UI and other subscribers [internal/message/message.go158-212](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/message/message.go#L158-L212)

### Buffering and Consistency

The service implements eventually consistent updates to optimize performance during high-frequency token streaming:

*   **Debouncing**: `Update` accepts state into an in-memory buffer and writes to SQLite on a debounce tick (default 33ms) [internal/message/message.go33-36](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/message/message.go#L33-L36)
    
*   **Terminal Flush**: Updates that finish a message, add a tool call, or end reasoning bypass the debounce and flush synchronously [internal/message/message.go37-39](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/message/message.go#L37-L39)
    
*   **Explicit Flush**: Callers can use `Flush` or `FlushAll` to ensure the database reflects the latest in-memory state before reading [internal/message/message.go41-45](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/message/message.go#L41-L45)

Attachment Handling
-------------------

Attachments are processed based on their MIME type and the capabilities of the target model.

*   **Text attachments**: Content is normalized and often injected into the prompt context.
*   **Binary/Media attachments**: Represented as `BinaryContent` in the message store [internal/message/content.go85-89](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/message/content.go#L85-L89)
    
*   **Media Validation**: When converting internal messages to LLM-specific `fantasy` types, `crush` validates data integrity. If media data cannot be decoded, it is substituted with a text placeholder (`mediaLoadFailedPlaceholder`) [internal/message/content.go28-30](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/message/content.go#L28-L30)
