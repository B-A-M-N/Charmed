This document details the GitHub Actions workflows used in the `crush` repository to automate testing, security scanning, configuration schema updates, and release processes. These workflows ensure that every contribution meets quality standards and that the application is securely delivered to users.

Workflow Overview
-----------------

The CI/CD pipeline is composed of several specialized workflows triggered by various events such as `push`, `pull_request`, and `schedule`.

| Workflow | Trigger | Purpose |
| --- | --- | --- |
| **build** | `push`, `pull_request` | Cross-platform matrix testing (Ubuntu, macOS, Windows). |
| **security** | `push`, `pull_request`, `schedule` | Vulnerability scanning (CodeQL, Grype, govulncheck). |
| **schema-update** | `push` (main) | Auto-generates `schema.json` from Go structs and commits changes. |
| **snapshot** | `push` (main) | Creates nightly/development builds using GoReleaser. |
| **nightly** | `schedule` | Triggers a nightly release if new commits exist. |
| **labeler** | `issues`, `pull_request_target` | Automatically labels PRs and issues based on file paths or titles. |
| **CLA Assistant** | `issue_comment`, `pull_request_target` | Manages Contributor License Agreement signatures. |
| **lint-sync** | `workflow_dispatch` | Synchronizes linting rules from the `charmbracelet/meta` repository. |

Build and Matrix Testing
------------------------

The `build` workflow is the primary CI gate. It executes a matrix build across all supported operating systems to ensure cross-platform compatibility, particularly for shell emulation and file system utilities.

**Build Pipeline Logic**

**Implementation Details:**

*   **Race Detection:** Uses the `-race` flag in both build and test steps to catch concurrent data structure issues in `csync` and `session.Service` [.github/workflows/build.yml29-30](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/build.yml#L29-L30)
    
*   **Tidy Check:** Runs `go mod tidy` followed by `git diff --exit-code` to ensure `go.mod` and `go.sum` are up-to-date and no unexpected changes were generated [.github/workflows/build.yml27-28](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/build.yml#L27-L28)
    
*   **Fail-Fast:** Uses the `-failfast` flag to stop the test suite immediately upon the first failure to save CI time [.github/workflows/build.yml30](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/build.yml#L30-L30)
    
*   **Concurrency Control:** Cancels in-progress runs for the same PR or branch to optimize runner usage [.github/workflows/build.yml10-12](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/build.yml#L10-L12)
    

**Sources:** [.github/workflows/build.yml1-31](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/build.yml#L1-L31)

Security Scanning
-----------------

The `security.yml` workflow implements a multi-layered defense strategy, scanning source code, dependencies, and GitHub Actions themselves.

### CodeQL and Static Analysis

Scans Go code and GitHub Action YAML files for common vulnerabilities and coding errors using a matrix for languages `go` and `actions` [.github/workflows/security.yml18-37](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/security.yml#L18-L37)

### Vulnerability Scanners

*   **Grype:** Scans the project directory for known vulnerabilities in dependencies using `anchore/scan-action`, failing the build on `critical` severity issues and uploading results in SARIF format [.github/workflows/security.yml39-57](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/security.yml#L39-L57)
    
*   **govulncheck:** Uses the official Go vulnerability database to find known vulnerabilities in the dependency graph that are actually reachable by the code [.github/workflows/security.yml59-78](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/security.yml#L59-L78)
    

### Dependency Review

For pull requests, this job checks for malicious dependencies and incompatible licenses [.github/workflows/security.yml80-93](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/security.yml#L80-L93)

*   **Allowed Licenses:** Includes BSD-2-Clause, BSD-3-Clause, MIT, MIT-0, Apache-2.0, MPL-2.0, ISC, and specific Google patent licenses [.github/workflows/security.yml92](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/security.yml#L92-L92)
    

**Sources:** [.github/workflows/security.yml1-93](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/security.yml#L1-L93)

Automated Schema Updates
------------------------

The `schema-update` workflow ensures that the JSON schema for `crush.json` is always synchronized with the `config.Config` Go struct.

**Data Flow: Schema Generation**

**Implementation Details:**

*   **Triggers:** Only triggers when files in `internal/config/**` or `internal/agent/hyper/**` change on the `main` branch [.github/workflows/schema-update.yml4-9](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/schema-update.yml#L4-L9)
    
*   **CLI Interaction:** Executes `go run . schema` which uses `jsonschema.Reflector` to generate the schema from `&config.Config{}` [.internal/cmd/schema.go12-26](https://github.com/charmbracelet/crush/blob/1ceeaa93/.internal/cmd/schema.go#L12-L26)
     The output is piped into `schema.json` [.github/workflows/schema-update.yml21](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/schema-update.yml#L21-L21)
    
*   **Persistence:** Uses `stefanzweifel/git-auto-commit-action` to push changes back to `main` using the `Charm` bot identity [.github/workflows/schema-update.yml23-29](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/schema-update.yml#L23-L29)
    

**Sources:** [.github/workflows/schema-update.yml1-30](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/schema-update.yml#L1-L30)

Release and Distribution
------------------------

### Snapshot Builds

The `snapshot` workflow runs on every push to `main`. It uses `goreleaser-pro` to build binaries for all supported platforms without publishing a formal release [.github/workflows/snapshot.yml1-35](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/snapshot.yml#L1-L35)

*   **High Performance:** Runs on a dedicated `releasers` runner group with more CPU and RAM [.github/workflows/snapshot.yml17-19](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/snapshot.yml#L17-L19)
    
*   **Args:** Uses `build --snapshot --clean` to ensure a fresh build environment [.github/workflows/snapshot.yml32](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/snapshot.yml#L32-L32)
    

### Nightly Releases

The `nightly` workflow runs daily at midnight [.github/workflows/nightly.yml5](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/nightly.yml#L5-L5)

1.  **Check Job:** Verifies if the current `HEAD` has already been successfully released by checking the history of `nightly.yml` runs via `gh run list` [.github/workflows/nightly.yml9-25](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/nightly.yml#L9-L25)
    
2.  **Nightly Job:** Calls the shared `charmbracelet/meta` nightly workflow using a `GORELEASER_KEY` secret if the check passes [.github/workflows/nightly.yml26-32](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/nightly.yml#L26-L32)
    

### Official Releases

The `release.yml` workflow is triggered by version tags (e.g., `v1.0.0`). It leverages the shared `charmbracelet/meta` Goreleaser workflow to handle:

*   **Signing:** macOS binaries are signed using entitlements defined in `.github/entitlements.plist` [.github/workflows/release.yml19](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/release.yml#L19-L19)
    
*   **Package Managers:** Publishes to Homebrew, AUR, Scoop, and NPM as configured in `.goreleaser.yml` [.goreleaser.yml99-213](https://github.com/charmbracelet/crush/blob/1ceeaa93/.goreleaser.yml#L99-L213)
    

**Sources:** [.github/workflows/snapshot.yml1-35](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/snapshot.yml#L1-L35)
 [.github/workflows/nightly.yml1-32](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/nightly.yml#L1-L32)

Contribution Management
-----------------------

### CLA Assistant

The `cla.yml` workflow manages the Contributor License Agreement. It allows contributors to sign the CLA by commenting on a PR [.github/workflows/cla.yml1-37](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/cla.yml#L1-L37)

*   **Triggering Comments:** Responds to "recheck" or the full signature statement "I have read the Contributor License Agreement (CLA) and hereby sign the CLA." [.github/workflows/cla.yml22-23](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/cla.yml#L22-L23)
    
*   **Signatures:** Stored in `.github/cla-signatures.json` [.github/workflows/cla.yml29](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/cla.yml#L29-L29)
    
*   **Allowlist:** Bots like `dependabot[bot]` and `Copilot` are exempt from signing [.github/workflows/cla.yml32](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/cla.yml#L32-L32)
    

### Labeler

The `labeler` workflow categorizes issues and PRs automatically based on regex matches against titles and file paths defined in `.github/labeler.yml` [.github/workflows/labeler.yml1-80](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/labeler.yml#L1-L80)

**Mapping Logic: Regex to Code Area**

**Label Categories:**

*   **Area:** Covers functional subsystems like `area: lsp`, `area: mcp`, `area: permissions`, `area: session`, and `area: tools` [.github/labeler.yml4-35](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/labeler.yml#L4-L35)
    
*   **Provider:** Specific labels for LLM backends such as `provider: anthropic claude`, `provider: azure`, or `provider: openai chatgpt` [.github/labeler.yml46-77](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/labeler.yml#L46-L77)
    
*   **OS:** Platform-specific labels for `os: windows`, `os: macos`, `os: linux`, and `os: android` [.github/labeler.yml36-43](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/labeler.yml#L36-L43)
    

**Sources:** [.github/labeler.yml1-80](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/labeler.yml#L1-L80)

Dependabot Configuration
------------------------

Dependabot is configured to provide weekly updates for both Go modules and GitHub Actions [.github/dependabot.yml1-45](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/dependabot.yml#L1-L45)

*   **Schedule:** Every Monday at 05:00 EST [.github/dependabot.yml9-35](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/dependabot.yml#L9-L35)
    
*   **Grouping:** All updates are grouped into a single PR per ecosystem (one for `gomod`, one for `github-actions`) to reduce noise [.github/dependabot.yml16-44](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/dependabot.yml#L16-L44)
    
*   **Ignores:** Specifically ignores beta versions of `bubbletea/v2` and `lipgloss/v2` to maintain stability, as well as `mvdan.cc/sh/moreinterp` [.github/dependabot.yml21-27](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/dependabot.yml#L21-L27)
    
*   **Labels:** Automatically applies the `area: dependencies` label to all generated PRs [.github/dependabot.yml11-37](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/dependabot.yml#L11-L37)
    

**Sources:** [.github/dependabot.yml1-45](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/dependabot.yml#L1-L45)
