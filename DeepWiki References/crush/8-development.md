Task Execution Flow
-------------------

The following diagram maps high-level development tasks to the `Taskfile.yaml` entities and Go commands they invoke.

**Task flow — from source to binary to release**

Sources: [Taskfile.yaml15-181](https://github.com/charmbracelet/crush/blob/1ceeaa93/Taskfile.yaml#L15-L181)

* * *

### Build

Compiles the binary to `./crush` (or `crush.exe` on Windows). If `git describe` succeeds, the version string is injected into `github.com/charmbracelet/crush/internal/version.Version` via `-ldflags`. The `-trimpath` flag is applied for reproducible builds. A race detector can be activated by creating a `race.log` file in the working directory; the presence of that file toggles `-race` in `go build`.

Sources: [Taskfile.yaml43-56](https://github.com/charmbracelet/crush/blob/1ceeaa93/Taskfile.yaml#L43-L56)
 [.goreleaser.yml72-75](https://github.com/charmbracelet/crush/blob/1ceeaa93/.goreleaser.yml#L72-L75)

* * *

### Test

Runs the full test suite with the race detector enabled using `go test -race -failfast ./...`. To re-record all VCR cassettes for the agent integration tests, use `task test:record` (aliased to `task record`), which deletes `internal/agent/testdata` and re-runs the agent tests with a one-hour timeout.

Sources: [Taskfile.yaml82-93](https://github.com/charmbracelet/crush/blob/1ceeaa93/Taskfile.yaml#L82-L93)

* * *

### Lint

Runs two checks in sequence:

1.  `lint:log`: Executes `./scripts/check_log_capitalization.sh` to enforce that all log messages start with a capital letter.
2.  `golangci-lint`: Runs `golangci-lint run --path-mode=abs --config=".golangci.yml" --timeout=5m`.

`GOEXPERIMENT` is explicitly set to `null` for linting to avoid compatibility issues. Install the linter with `task lint:install`. Auto-fix mode is available via `task lint:fix`.

Sources: [Taskfile.yaml16-42](https://github.com/charmbracelet/crush/blob/1ceeaa93/Taskfile.yaml#L16-L42)

* * *

### Profiling

Setting `CRUSH_PROFILE=true` (or running `task dev`) causes `main()` to start a `pprof` HTTP server on `localhost:6060` before executing the normal CLI flow. Three profiling tasks consume that endpoint:

| Task | pprof endpoint |
| --- | --- |
| `task profile:cpu` | `/debug/pprof/profile?seconds=10` |
| `task profile:heap` | `/debug/pprof/heap` |
| `task profile:allocs` | `/debug/pprof/allocs` |

All three open results in a browser via `go tool pprof -http :6061`.

Sources: [main.go24-31](https://github.com/charmbracelet/crush/blob/1ceeaa93/main.go#L24-L31)
 [Taskfile.yaml109-144](https://github.com/charmbracelet/crush/blob/1ceeaa93/Taskfile.yaml#L109-L144)

* * *

### Schema and Code Generation

`task schema` pipes `go run main.go schema` stdout into `schema.json`. `task hyper` triggers `go generate` on `internal/agent/hyper/`. `task sqlc` regenerates database code from SQL definitions. See [Configuration Schema](https://deepwiki.com/charmbracelet/crush/8.6-configuration-schema)
 for details.

**Release pipeline — from `task release` to published packages**

The `task release` command performs several safety checks, including verifying that the `build.yml` and `snapshot.yml` workflows have succeeded for the current commit.

### Build Matrix

The `goreleaser` build matrix produces binaries for:

| OS  | Architectures |
| --- | --- |
| `linux` | `amd64`, `arm64`, `386`, `arm` (v7) |
| `darwin` | `amd64`, `arm64` |
| `windows` | `amd64`, `arm64`, `386` |
| `freebsd`, `openbsd`, `netbsd` | `amd64`, `arm64`, `386`, `arm` (v7) |
| `android` | `arm64` |

All builds use `CGO_ENABLED=0` and `GOEXPERIMENT=greenteagc`. The `-trimpath` flag is set. macOS binaries are code-signed and notarized via `charmbracelet/meta` workflows.

Sources: [.goreleaser.yml44-76](https://github.com/charmbracelet/crush/blob/1ceeaa93/.goreleaser.yml#L44-L76)

### Archive Contents

Each archive (`.tar.gz` on Unix, `.zip` on Windows) contains:

*   The `crush` binary.
*   `README*` and `LICENSE*` files.
*   `manpages/crush.1.gz`.
*   `completions/crush.bash`, `crush.zsh`, `crush.fish`.

Shell completions and man pages are generated in the `goreleaser` `before.hooks` block by running `go run . completion <shell>` and `go run . man`.

Key Environment Variables for Developers
----------------------------------------

| Variable | Effect |
| --- | --- |
| `CRUSH_PROFILE=true` | Starts `pprof` server on `localhost:6060` via `main.go`. |
| `CRUSH_GLOBAL_DATA` | Overrides the global data directory (used in `task run:onboarding`). |

Contributing and CI
-------------------

Contributions are managed through a series of automated workflows and a Contributor License Agreement (CLA) requirement.

*   **CLA Requirement**: Contributors must sign the CLA via the `cla.yml` workflow by commenting on their Pull Request. Signatures are tracked in `.github/cla-signatures.json`.
*   **CI Verification**: Every PR triggers the `build.yml` workflow which runs tests and builds across a matrix of `ubuntu-latest`, `macos-latest`, and `windows-latest`.
*   **Security**: The `security.yml` workflow runs CodeQL, Grype vulnerability scans, and `govulncheck` on a daily schedule and on every push to `main`.
*   **Auto-Updates**: The `schema-update.yml` workflow automatically regenerates `schema.json` and Hyper provider metadata.

Sources: [.github/workflows/cla.yml1-37](https://github.com/charmbracelet/crush/blob/1ceeaa93/.github/workflows/cla.yml#L1-L37)
