Purpose and Scope
-----------------

The event system in Crush provides a decoupled communication mechanism between services, the application core, and the user interface. It is built on a generic pubsub (publish-subscribe) bus that allows backend services to broadcast state changes without direct knowledge of the TUI or the non-interactive execution engine.

This system handles critical data flow including LLM message streaming, session lifecycle updates, permission requests, and external integration events from LSP (Language Server Protocol) and MCP (Model Context Protocol).

Architecture Overview
---------------------

The architecture follows a hub-and-spoke model where the `App` struct acts as the central aggregator. Individual services maintain their own typed brokers, and the application wires these together into a unified stream for the UI.

**Event System Architecture**

**Implementation Detail**: The application core initializes its internal event channels to bridge the gap between service-specific types and the Bubble Tea `tea.Msg` interface. In the TUI mode, `ws.Subscribe(program)` is called to start the forwarding loop [internal/cmd/root.go136](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/cmd/root.go#L136-L136)

Core Broker Implementation
--------------------------

### The pubsub Package

The foundation of the event system is the `internal/pubsub` package, which provides a thread-safe, generic broker. It supports two delivery semantics: lossy best-effort and bounded-blocking "must deliver".

| Component | File Path | Role |
| --- | --- | --- |
| `Broker[T]` | [internal/pubsub/broker.go48-57](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/pubsub/broker.go#L48-L57) | Manages a map of subscriber channels and handles fan-out. |
| `Event[T]` | [internal/pubsub/events.go45-48](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/pubsub/events.go#L45-L48) | A struct containing the `EventType` and the generic `Payload`. |
| `EventType` | [internal/pubsub/events.go8-12](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/pubsub/events.go#L8-L12) | Constants: `CreatedEvent`, `UpdatedEvent`, `DeletedEvent`. |
| `Subscriber[T]` | [internal/pubsub/events.go36-38](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/pubsub/events.go#L36-L38) | Interface requiring a `Subscribe(context.Context)` method. |

**Delivery Semantics**:

*   **`Publish`**: Non-blocking and lossy. If a subscriber's channel is full, the event is dropped and a warning is logged. This is used for high-frequency updates like token streaming [internal/pubsub/broker.go6-188](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/pubsub/broker.go#L6-L188)
    
*   **`PublishMustDeliver`**: Bounded-blocking. It attempts a non-blocking send, then falls back to a blocking send with a 50ms timeout. This is used for terminal events like tool results or errors [internal/pubsub/broker.go13-236](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/pubsub/broker.go#L13-L236)

Event Flow and Wiring
---------------------

### Application Setup

During initialization, the application wires service-specific brokers into a unified stream. For example, the `message.Service` uses an internal broker to debounce streaming updates [internal/message/message.go103-104](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/message/message.go#L103-L104)

**Message Coalescing**: The `message.Service` implementation of `Update` uses the broker to handle eventually consistent state. It buffers updates and flushes them to the database and the event bus on a 33ms debounce window [internal/message/message.go16-39](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/message/message.go#L16-L39)

**Broker Lifecycle**: A broker can be gracefully shut down using the `Shutdown()` method, which closes the `done` channel and all active subscriber channels [internal/pubsub/broker.go85-102](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/pubsub/broker.go#L85-L102)

Event Categories and Typed Payloads
-----------------------------------

Events are categorized by their payload types. When running in client/server mode, these are wrapped in a `Payload` envelope for transmission over SSE (Server-Sent Events).

| Category | Payload Type | Internal Type |
| --- | --- | --- |
| **Session** | `PayloadTypeSession` | `proto.Session` |
| **Message** | `PayloadTypeMessage` | `proto.Message` |
| **Permission Request** | `PayloadTypePermissionRequest` | `proto.PermissionRequest` |
| **Permission Notify** | `PayloadTypePermissionNotification` | `proto.PermissionNotification` |
| **LSP** | `PayloadTypeLSPEvent` | `proto.LSPEvent` |
| **MCP** | `PayloadTypeMCPEvent` | `proto.MCPEvent` |
| **Agent** | `PayloadTypeAgentEvent` | `proto.AgentEvent` |

### SSE Wrapping (Server Mode)

The `wrapEvent` function in the server layer converts raw internal events into a `pubsub.Payload` envelope with proper JSON tags for the client [internal/server/events.go24-104](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/server/events.go#L24-L104)
 The client then unmarshals these based on the `PayloadType` discriminator [internal/client/proto.go138-178](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/client/proto.go#L138-L178)

TUI Integration
---------------

The `App.Subscribe()` method (or `ws.Subscribe()` in the command layer) bridges the backend event bus to the Bubble Tea program.

**Data Flow Sequence**

**Subscription Mechanism**:

*   The `Broker.Subscribe(ctx)` method creates a new buffered channel (default size 4096) for the subscriber [internal/pubsub/broker.go40-138](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/pubsub/broker.go#L40-L138)
    
*   The `SubscribeEvents` method in the client SDK handles the long-lived HTTP connection and parses the `text/event-stream` format [internal/client/proto.go88-103](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/client/proto.go#L88-L103)

Telemetry and Analytics
-----------------------

The `internal/event` package handles external telemetry via PostHog. This is separate from the internal pubsub bus used for UI updates.

**Telemetry Implementation**:

*   **Initialization**: `event.Init()` configures the PostHog client [internal/event/event.go50-61](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/event/event.go#L50-L61)
    
*   **System Context**: Every event is enriched with `GOOS`, `GOARCH`, `TERM`, and `SHELL` properties [internal/event/event.go28-36](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/event/event.go#L28-L36)
    
*   **Interactive State**: The system tracks whether the app is running in `--yolo` mode or as a non-interactive `run` command via `SetNonInteractive` [internal/event/event.go38-40](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/event/event.go#L38-L40)
    
*   **Error Tracking**: `event.Error()` captures exceptions, including the type and message, and enqueues them for remote analysis [internal/event/event.go96-116](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/event/event.go#L96-L116)
