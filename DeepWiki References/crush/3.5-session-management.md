This page covers how sessions are created, persisted, titled, and tracked at runtime in crush. It describes the `session.Service` interface, the fields that the `Session` data model carries, how usage statistics accumulate per-step, how titles are generated asynchronously, and how child sessions spawned by `AgentTool` relate to their parent.

* * *

Core Concepts
-------------

A **session** is the top-level container for a conversation. Every message belongs to a session. Every agent run is scoped to a session ID. The session record accumulates token usage across all steps, tracks its summarization checkpoint, and carries cost metadata.

The `session.Service` is owned by the `App` struct and passed into the `Coordinator` and then into each `SessionAgent`. All reads and writes to sessions in agent code go through this service.

Session Creation
----------------

Sessions are created in several primary contexts via the `session.Service`:

| Context | Code Location | Method |
| --- | --- | --- |
| Interactive TUI — new chat | `ui.New` | `session.Service.Create` |
| Non-interactive (`crush run`) | `app.resolveSession` | `session.Service.Create` |
| AgentTool sub-agent | `AgentTool` | `session.Service.CreateTaskSession` |
| Title Generation sub-agent | `agent.generateTitle` | `session.Service.CreateTitleSession` |

In non-interactive mode or when switching contexts, the application resolves the session based on flags:

*   `--session <id>`: Continues an existing session.
*   `--continue`: Resumes the most recent session via `GetLast` [internal/session/session.go181-189](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L181-L189)
    
*   Default: Creates a new session with a UUID [internal/session/session.go96-108](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L96-L108)

Session Data Model
------------------

The `Session` struct holds these runtime-relevant fields, persisted in the `sessions` table:

| Field | Purpose |
| --- | --- |
| `ID` | Unique session identifier (UUID or derived hash) |
| `ParentSessionID` | Reference to a parent session for sub-agents |
| `Title` | Human-readable label; set after first prompt |
| `PromptTokens` | Cumulative input tokens across all steps |
| `CompletionTokens` | Cumulative output tokens across all steps |
| `Cost` | Cumulative cost (used for pricing tracking) |
| `SummaryMessageID` | ID of the latest summary message; skips previous history when set |
| `Todos` | Structured todo list (`[]Todo`) persisted as JSON |
| `EstimatedUsage` | Memory-only flag for UI to show "~" marker for unconfirmed costs |

Title Generation
----------------

Title generation typically runs after the first interaction to replace the placeholder "Untitled Session".

**Title generation logic:**

1.  The agent spawns a background task via `CreateTitleSession` [internal/session/session.go124-136](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L124-L136)
    
2.  This session is linked to the parent via `ParentSessionID`.
3.  The resulting title is saved via `UpdateTitleAndUsage` for atomic updates [internal/session/session.go225-235](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L225-L235)
    
4.  Users can manually rename sessions via the `Rename` method [internal/session/session.go237-239](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L237-L239)

Usage and Cost Tracking
-----------------------

Token usage and cost are accumulated onto the `Session` record. This allows for long-term tracking of AI provider expenses.

**Usage Update Logic:**

*   `UpdateTitleAndUsage` performs an additive update on `prompt_tokens`, `completion_tokens`, and `cost` directly in the database [internal/db/sessions.sql.go246-274](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/sessions.sql.go#L246-L274)
    
*   **Estimated Usage**: The `estimatedUsage` map in the service tracks which sessions have pending/unconfirmed usage totals to avoid clearing UI markers during fetch-modify-save cycles [internal/session/session.go89-94](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L89-L94)

Child Sessions (AgentTool)
--------------------------

When the primary agent invokes a sub-agent (e.g., via `AgentTool`), a dedicated child session is created to isolate the sub-agent's context.

**Key Implementation Details:**

*   **ID Derivation**: `CreateAgentToolSessionID` generates the child session ID from the parent's message ID and the tool call ID [internal/session/session.go241-243](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L241-L243)
    
*   **Linking**: `CreateTaskSession` links the child to the parent via `ParentSessionID` [internal/session/session.go110-122](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L110-L122)
    
*   **Filtering**: Standard session listings (`ListSessions`) filter out child sessions by checking `WHERE parent_session_id is NULL` [internal/db/sessions.sql.go137-142](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/db/sessions.sql.go#L137-L142)
    
*   **Reconstruction**: `ParseAgentToolSessionID` can reverse the ID to find the originating message and tool call [internal/session/session.go245-253](https://github.com/charmbracelet/crush/blob/1ceeaa93/internal/session/session.go#L245-L253)
