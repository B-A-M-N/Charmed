Output Rendering and Formatting
===============================

Relevant source files

*   [mods.go](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go)
    
*   [stream.go](https://github.com/charmbracelet/mods/blob/07a05d5b/stream.go)
    

Purpose and Scope
-----------------

This document describes the output rendering and formatting system in mods, which handles how AI-generated content is displayed to the user. The system implements adaptive rendering that produces formatted markdown output for interactive terminals and raw text for pipes and scripts. For terminal capabilities and TTY detection, see [4.1](https://deepwiki.com/charmbracelet/mods/4.1-terminal-capabilities-and-styling)
. For loading spinner animations, see [4.2](https://deepwiki.com/charmbracelet/mods/4.2-animation-system)
.

The rendering system coordinates three main components: the `appendToOutput` function that accumulates and processes content, the Glamour markdown renderer that formats text, and the Bubble Tea viewport that enables scrolling. The system adapts its behavior based on whether output is directed to a TTY or a pipe, and whether raw output mode is enabled.

Rendering Architecture Overview
-------------------------------

The rendering system follows a streaming architecture where content chunks arrive incrementally from AI providers and are progressively rendered to the user. The system maintains two parallel output paths: a formatted path using Glamour for TTY output, and a raw path that buffers unformatted content for non-TTY destinations.

**Diagram: Output Rendering Flow**

The system branches at `appendToOutput` based on TTY detection. For interactive terminals, content flows through Glamour rendering and viewport management. For pipes and scripts, content is buffered in `m.content` and printed directly in the `View` method.

Sources: [mods.go635-661](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L635-L661)
 [mods.go221-257](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L221-L257)

Key Data Structures
-------------------

The `Mods` struct maintains multiple fields for managing output rendering:

| Field | Type | Purpose |
| --- | --- | --- |
| `Output` | `string` | Accumulated raw output text |
| `glamOutput` | `string` | Rendered markdown output from Glamour |
| `glamHeight` | `int` | Height in lines of rendered output |
| `glam` | `*glamour.TermRenderer` | Glamour markdown renderer instance |
| `glamViewport` | `viewport.Model` | Bubble Tea viewport for scrolling |
| `content` | `[]string` | Buffer for streaming raw output to non-TTY |
| `contentMutex` | `*sync.Mutex` | Protects concurrent access to content buffer |
| `width`, `height` | `int` | Terminal dimensions from WindowSizeMsg |

Sources: [mods.go52-78](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L52-L78)

The appendToOutput Function
---------------------------

The `appendToOutput` function is the central hub for all output rendering. It maintains dual output paths and coordinates viewport scrolling behavior.

### Function Signature and Core Logic

Located at [mods.go635-661](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L635-L661)
 the function signature is:

    func (m *Mods) appendToOutput(s string)
    

The function follows this execution flow:

1.  **Accumulate to Output**: Always appends to `m.Output` field for maintaining complete conversation history [mods.go636](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L636-L636)
    
2.  **Path Selection**: Checks TTY status and raw mode to determine rendering path [mods.go637](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L637-L637)
    
3.  **Raw Path**: For non-TTY or raw mode, appends to thread-safe `m.content` buffer [mods.go638-641](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L638-L641)
    
4.  **Formatted Path**: For TTY with formatted output, proceeds through Glamour rendering

### Formatted Output Path

For interactive terminal output, the function performs several operations:

**Diagram: Formatted Output Processing Sequence**

The viewport auto-scrolling logic at [mods.go655-660](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L655-L660)
 ensures that new content remains visible when the user is viewing the bottom of the output. If content is added and the user was at the bottom, the viewport automatically scrolls to maintain that position.

Sources: [mods.go643-661](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L643-L661)

Viewport Management
-------------------

The `glamViewport` field (type `viewport.Model` from Bubble Tea's bubbles package) manages scrolling for content that exceeds terminal height.

### Viewport State Management

**Diagram: Viewport State Lifecycle**

The viewport is created with zero dimensions at [mods.go91](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L91-L91)
 and positioned at the bottom via `GotoBottom()` at [mods.go92](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L92-L92)
 It receives dimension updates from `tea.WindowSizeMsg` events.

### viewportNeeded Check

The `viewportNeeded` method at [mods.go217-219](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L217-L219)
 determines whether scrolling is required:

    func (m Mods) viewportNeeded() bool {
        return m.glamHeight > m.height
    }
    

This simple check compares the rendered content height against the terminal height. When true, the viewport intercepts keyboard input for scrolling (see [mods.go207-213](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L207-L213)
).

View Method Output Logic
------------------------

The `View` method implements Bubble Tea's `tea.Model` interface and determines what gets displayed based on the current state. Located at [mods.go221-257](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L221-L257)
 it uses a state machine pattern.

### State-Based Rendering

**Diagram: View Method State Machine**

Sources: [mods.go221-257](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L221-L257)

### Response State Rendering Paths

In `responseState`, the method implements three distinct output paths:

**Path 1: Formatted with Viewport** (Interactive TTY, content exceeds height)

*   Condition: `!m.Config.Raw && isOutputTTY() && m.viewportNeeded()`
*   Returns: `m.glamViewport.View()` at [mods.go234](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L234-L234)
    
*   Displays scrollable viewport with rendered markdown

**Path 2: Formatted without Viewport** (Interactive TTY, content fits)

*   Condition: `!m.Config.Raw && isOutputTTY() && !m.viewportNeeded()`
*   Returns: `m.glamOutput` at [mods.go237](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L237-L237)
    
*   Displays rendered markdown directly

**Path 3: Raw/Buffered Output** (Non-TTY or raw mode)

*   Condition: `!isOutputTTY() || m.Config.Raw`
*   Behavior: Prints buffered content from `m.content` at [mods.go244-249](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L244-L249)
    
*   Thread-safe printing with mutex protection
*   Clears buffer after printing

Sources: [mods.go232-249](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L232-L249)

Streaming Content Buffer
------------------------

For non-TTY output, the system uses a thread-safe content buffer to handle incremental streaming. This ensures that content can be written to pipes and files as it arrives, without waiting for the entire response.

### Buffer Implementation

The `m.content` field is a `[]string` slice protected by `m.contentMutex` for concurrent access:

**Diagram: Thread-Safe Content Buffer Flow**

The mutex ensures safe concurrent access between the goroutine receiving stream chunks and the main UI thread calling `View()`. The buffer is cleared after each print cycle at [mods.go248](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L248-L248)

Adaptive Output Mode Selection
------------------------------

The rendering system automatically adapts based on runtime context. The decision tree combines TTY detection with configuration flags:

| Context | TTY Status | Raw Flag | Rendering Mode |
| --- | --- | --- | --- |
| Interactive shell | `isOutputTTY() == true` | `false` | Glamour markdown with viewport |
| Piped output | `isOutputTTY() == false` | N/A | Raw buffered output |
| `--raw` flag | `true` | `true` | Raw output (bypass Glamour) |
| `--raw` flag piped | `false` | `true` | Raw buffered output |

The `isOutputTTY()` function (documented in [4.1](https://deepwiki.com/charmbracelet/mods/4.1-terminal-capabilities-and-styling)
) performs the actual terminal detection. The `m.Config.Raw` flag allows users to explicitly disable markdown formatting even in interactive mode.

### Raw Mode Use Cases

Raw mode bypasses Glamour rendering and is useful for:

*   Capturing output to files without ANSI formatting
*   Processing output with other command-line tools
*   Debugging rendering issues
*   Performance optimization for large outputs

Sources: [mods.go232](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L232-L232)
 [mods.go237](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L237-L237)

Text Preprocessing
------------------

Before rendering, the system applies text preprocessing to normalize output:

### Whitespace Removal

The `removeWhitespace` function at [mods.go664-669](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L664-L669)
 handles empty input detection:

    func removeWhitespace(s string) string {
        if strings.TrimSpace(s) == "" {
            return ""
        }
        return s
    }
    

This converts whitespace-only input to empty strings, preventing spurious requests. Applied at [mods.go144](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L144-L144)
 when processing `completionInput`.

### Tab Expansion

Tabs are expanded to spaces using the `tabWidth` constant:

    m.glamOutput = strings.ReplaceAll(m.glamOutput, "\t", strings.Repeat(" ", tabWidth))
    

Located at [mods.go648](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L648-L648)
 this ensures consistent rendering across terminals with different tab stop settings. The tab width is 4 spaces as defined at [mods.go633](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L633-L633)

Sources: [mods.go633](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L633-L633)
 [mods.go648](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L648-L648)
 [mods.go664-669](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L664-L669)

Done State Output
-----------------

When the response stream completes, the system enters `doneState`. At this point, special handling ensures proper output termination:

*   For non-TTY output: Prints a final newline at [mods.go251-252](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L251-L252)
    
*   For TTY output: Returns empty string (content already displayed)

This ensures that piped output ends with a newline character, which is important for shell command chaining and file output.

Sources: [mods.go250-254](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L250-L254)

Integration with Message Streaming
----------------------------------

The rendering system integrates with the message streaming architecture documented in [2.4](https://deepwiki.com/charmbracelet/mods/2.4-message-stream-context)
. Content arrives through `completionOutput` messages in the `Update` method:

1.  Stream sends `completionOutput{content: chunk}` message
2.  `Update` calls `appendToOutput` at [mods.go179](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L179-L179)
    
3.  Content is processed through appropriate rendering path
4.  System schedules next stream read with `receiveCompletionStreamCmd`

This creates a continuous flow where each content chunk is immediately rendered, providing real-time streaming output to the user.

Sources: [mods.go173-185](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L173-L185)
 [mods.go492-528](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L492-L528)
