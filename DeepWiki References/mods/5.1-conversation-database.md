Conversation Database
=====================

Relevant source files

*   [db.go](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go)
    
*   [db\_test.go](https://github.com/charmbracelet/mods/blob/07a05d5b/db_test.go)
    

Purpose and Scope
-----------------

The Conversation Database system provides persistent storage of conversation metadata in a SQLite database. This page documents the `convoDB` struct, the `conversations` table schema, migration logic, and all database operations (Save, Delete, Find, List, Completions). The database stores only lightweight metadata about conversations, including ID, title, timestamps, API provider, and model used.

For information about full message history storage, see [Cache System](https://deepwiki.com/charmbracelet/mods/5.2-cache-system)
. For the broader conversation management architecture, see [Conversation Management](https://deepwiki.com/charmbracelet/mods/2.5-conversation-management)
.

* * *

Database Structure
------------------

The conversation database is implemented using SQLite with the `modernc.org/sqlite` driver and the `jmoiron/sqlx` query builder. The system centers around two main types: `convoDB` for database operations and `Conversation` for row representation.

### Core Types

| Type | Purpose | Location |
| --- | --- | --- |
| `convoDB` | Database connection wrapper with operation methods | [db.go96-98](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L96-L98) |
| `Conversation` | Represents a conversation row with metadata fields | [db.go101-107](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L101-L107) |

**Sources:** [db.go96-107](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L96-L107)

### Table Schema

The `conversations` table stores metadata with the following structure:

| Column | Type | Constraints | Purpose |
| --- | --- | --- | --- |
| `id` | string | PRIMARY KEY, NOT NULL, CHECK (id <> '') | SHA-1 conversation identifier |
| `title` | string | NOT NULL, CHECK (title <> '') | Human-readable conversation name |
| `updated_at` | datetime | NOT NULL, DEFAULT CURRENT\_TIMESTAMP | Last modification time |
| `api` | string | nullable | AI provider used (e.g., "openai", "anthropic") |
| `model` | string | nullable | Model name used (e.g., "gpt-4o", "claude-3-5-sonnet-20241022") |

The schema includes two indexes for query performance:

*   `idx_conv_id` on the `id` column [db.go56-58](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L56-L58)
    
*   `idx_conv_title` on the `title` column [db.go60-63](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L60-L63)
    

**Sources:** [db.go43-64](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L43-L64)

Database Operations
-------------------

### Save Operation

The `Save` method implements an upsert pattern: update existing conversations or insert new ones.

The operation first attempts an UPDATE [db.go114-126](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L114-L126)
 If no rows are affected (conversation doesn't exist), it performs an INSERT [db.go137-144](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L137-L144)
 The `updated_at` column is automatically set to `CURRENT_TIMESTAMP` on updates.

**Sources:** [db.go113-147](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L113-L147)
 [db\_test.go30-70](https://github.com/charmbracelet/mods/blob/07a05d5b/db_test.go#L30-L70)

### Find Operations

The database provides multiple find methods with different semantics:

#### Find by ID or Title

The `Find` method accepts either a conversation ID (full or partial SHA-1) or an exact title:

The logic distinguishes between short inputs (less than `sha1minLen`) which are treated as exact title matches, and longer inputs which are matched against both ID prefixes (using `glob`) and exact titles [db.go251-271](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L251-L271)

**Key behaviors:**

*   ID matching uses glob pattern `in+"*"` for prefix matching [db.go213](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L213-L213)
    
*   Title matching requires exact equality [db.go199](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L199-L199)
     [db.go214](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L214-L214)
    
*   Returns `errManyMatches` if multiple conversations match [db.go265](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L265-L265)
    
*   Returns `errNoMatches` if no conversations match [db.go270](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L270-L270)
    

**Sources:** [db.go192-271](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L192-L271)
 [db\_test.go101-127](https://github.com/charmbracelet/mods/blob/07a05d5b/db_test.go#L101-L127)

#### FindHEAD

The `FindHEAD` method retrieves the most recently updated conversation, analogous to Git's HEAD pointer:

This enables the `--continue` flag without arguments to resume the last conversation [db.go175-190](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L175-L190)

 This is used for cleanup operations to identify stale conversations.

### Completions

The `Completions` method generates shell completion candidates for conversation IDs and titles:

The method performs two queries combined with UNION:

1.  **ID-based completions**: Matches IDs starting with the input, formatted as `id\ttitle` [db.go224-233](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L224-L233)
    
2.  **Title-based completions**: Matches titles starting with the input, formatted as `title\tid` [db.go239-244](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L239-L244)
    

ID truncation logic:

*   If input length is less than `sha1short`, display truncated ID [db.go228-229](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L228-L229)
    
*   Otherwise, display full ID [db.go165-169](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L165-L169)
    

The tab character (`char(9)`) separates the primary match from the description for shell completion frameworks.

**Sources:** [db.go221-249](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L221-L249)
 [db\_test.go148-170](https://github.com/charmbracelet/mods/blob/07a05d5b/db_test.go#L148-L170)

Error Handling
--------------

The database system defines custom error types and wrapping functions:

| Error | Type | When Returned |
| --- | --- | --- |
| `errNoMatches` | `errors.New("no conversations found")` | Find returns no results [db.go13](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L13-L13) |
| `errManyMatches` | `errors.New("multiple conversations matched the input")` | Find returns multiple results [db.go14](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L14-L14) |

### SQLite Error Handling

The `handleSqliteErr` function unwraps SQLite-specific errors and adds human-readable error code descriptions:

This converts low-level SQLite error codes into descriptive strings using `sqlite.ErrorCodeString[sqerr.Code()]` [db.go23](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L23-L23)

Testing Strategy
----------------

The database tests use an in-memory SQLite database (`:memory:`) for isolation and speed. The test helper `testDB` creates a fresh database for each test and ensures cleanup via `tb.Cleanup` [db\_test.go11-18](https://github.com/charmbracelet/mods/blob/07a05d5b/db_test.go#L11-L18)
