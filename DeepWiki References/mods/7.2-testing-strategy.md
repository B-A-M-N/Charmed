Purpose and Scope
-----------------

This document describes the testing approach for the mods codebase, including the organization of test files, test patterns used, helper utilities, and coverage areas. The test suite focuses on unit testing core functionality such as conversation management, database operations, and flag parsing.

For information about the build system and dependencies, see [Dependencies and Build System](https://deepwiki.com/charmbracelet/mods/7.1-dependencies-and-build-system)
. For CI/CD automation including linting and automated testing workflows, see [Code Quality and CI/CD](https://deepwiki.com/charmbracelet/mods/7.3-code-quality-and-cicd)
.

Test Organization
-----------------

The test suite is organized into three primary test files that correspond to the core subsystems being tested:

**Test File Structure**

| Test File | Primary Focus | Key Functions Tested |
| --- | --- | --- |
| `mods_test.go` | Conversation cache operations | `findCacheOpsDetails()`, `removeWhitespace()`, `cutPrompt()` |
| `db_test.go` | Database operations | `Save()`, `Find()`, `Delete()`, `List()`, `FindHEAD()`, `Completions()` |
| `flag_test.go` | Flag parsing and error handling | `newFlagParseError()` |

Test Framework and Assertions
-----------------------------

All tests use the `testify/require` assertion library, which provides clear error messages and immediately halts test execution on failure. This pattern is applied consistently across all test files:

The use of `require` rather than `assert` ensures that tests fail fast when preconditions are not met, preventing cascading failures and making debugging easier.

Core Test Suites
----------------

### Conversation Cache Operations Tests

The `TestFindCacheOpsDetails` function in `mods_test.go` contains 12 sub-tests that verify the conversation cache resolution logic. This is the most comprehensive test in the suite, covering various scenarios for conversation continuation, branching, and creation.

**Key Test Scenarios**

The test creates a `Mods` instance using the `newMods` helper [mods\_test.go11-17](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L11-L17)
 and verifies:

1.  **New Conversation**: When all flags are empty, a new `WriteID` is generated [mods\_test.go19-25](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L19-L25)
    
2.  **Show by ID/Title**: The `Show` flag correctly resolves conversations [mods\_test.go27-45](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L27-L45)
    
3.  **Continue**: The `Continue` flag loads existing conversations and preserves the ID [mods\_test.go47-81](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L47-L81)
    
4.  **Conversation Branching**: Combining `Continue` with `Title` creates a fork [mods\_test.go121-149](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L121-L149)
    
5.  **Error Handling**: Invalid conversation identifiers return proper `modsError` [mods\_test.go151-158](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L151-L158)
    
6.  **Configuration Propagation**: Model and API settings are preserved [mods\_test.go160-172](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L160-L172)
    

Sources: [mods\_test.go10-173](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L10-L173)

**Test Data Flow**

**Conversation Table Operations**

| `Find()` | By ID prefix, by exact title | Single match returned [db\_test.go101-111](https://github.com/charmbracelet/mods/blob/07a05d5b/db_test.go#L101-L111) |

**Constraint Validation**

The tests verify database constraints are enforced:

*   Empty ID rejected [db\_test.go45-48](https://github.com/charmbracelet/mods/blob/07a05d5b/db_test.go#L45-L48)
    
*   Empty title rejected [db\_test.go50-53](https://github.com/charmbracelet/mods/blob/07a05d5b/db_test.go#L50-L53)
    

**Error Conditions**

Tests verify error handling for:

*   `errManyMatches`: Multiple conversations match ambiguous identifier [db\_test.go120-127](https://github.com/charmbracelet/mods/blob/07a05d5b/db_test.go#L120-L127)

### Flag Parsing Tests

The `TestFlagParseError` function validates the parsing and formatting of command-line flag errors. This uses a table-driven test pattern to verify multiple error message formats.

**Error Types Tested**

The test data structure [flag\_test.go10-45](https://github.com/charmbracelet/mods/blob/07a05d5b/flag_test.go#L10-L45)
 defines expected parsing for:

*   Unknown flags (long and short forms)
*   Missing required arguments
*   Invalid argument types (duration, int, bool)

Each test case verifies:

*   `Flag()` returns the problematic flag name
*   `ReasonFormat()` returns the formatted reason string
*   `Error()` returns the original error message

Sources: [flag\_test.go10-56](https://github.com/charmbracelet/mods/blob/07a05d5b/flag_test.go#L10-L56)
 [flag.go11-49](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L11-L49)

### Database Test Helper

The `testDB` helper function creates an isolated in-memory SQLite database for each test:

Key features:

*   Uses `:memory:` data source for fast, isolated tests
*   Accepts `testing.TB` interface, works with tests and benchmarks

Utility Function Tests
----------------------

### Text Processing Tests

**TestRemoveWhitespace**

Verifies the `removeWhitespace()` function handles:

*   Strings containing only whitespace → empty string
*   Regular text with whitespace → preserved unchanged

Sources: [mods\_test.go175-183](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L175-L183)

**TestCutPrompt**

Tests the `cutPrompt()` function for handling token limit errors from LLM APIs. Uses a table-driven approach with test cases for:

| Scenario | Input Error Message | Expected Behavior |
| --- | --- | --- |
| Bad error | Unrecognized format | Return original prompt |
| Crazy error | Token count exceeds 10x limit | Return original prompt |
| Cut prompt | Exceeds limit by few tokens | Truncate prompt appropriately |
| Mismatch | Estimation differs from API | Return original prompt |

The test uses a helper function `tokenErrMsg(length, maxLength)` to generate realistic OpenAI-style error messages [mods\_test.go212-214](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L212-L214)

Sources: [mods\_test.go185-222](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L185-L222)

Test Coverage Areas
-------------------

**Coverage Focus Areas**

The test suite emphasizes:

1.  **Conversation Lifecycle**: All operations for creating, continuing, showing, and branching conversations
2.  **Data Persistence**: CRUD operations on the SQLite database with constraint validation
3.  **User Input Parsing**: Error handling and formatting for command-line flags
4.  **Token Management**: Prompt truncation logic for API token limits
5.  **Error Conditions**: Ambiguous identifiers, missing conversations, invalid inputs

**Areas Without Tests**

Notable areas not covered by unit tests:

*   LLM provider integration (OpenAI, Anthropic, etc.)
*   Bubble Tea UI state machine transitions
*   Streaming response handling
*   Terminal rendering and styling
*   Configuration file parsing
*   Network I/O operations
