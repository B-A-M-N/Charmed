Release Process
===============

This document outlines the automated release process for the Mods CLI tool. It covers how releases are created, versioned, and deployed to various distribution channels. For information about building and testing the codebase locally, see [Build and Test](https://deepwiki.com/charmbracelet/mods/5.1-conversation-database)
.

Overview
--------

Mods uses GitHub Actions and GoReleaser to automate the release process. There are two primary types of builds:

1.  **Official releases** - Tagged releases that follow semantic versioning
2.  **Nightly builds** - Development builds created from the main branch

Sources: [.github/workflows/goreleaser.yml5-9](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/goreleaser.yml#L5-L9)
 [.github/workflows/nightly.yml3-7](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/nightly.yml#L3-L7)

Release Triggers
----------------

### Official Releases

Official releases are triggered automatically when a git tag following the semantic versioning pattern (`v*.*.*`) is pushed to the repository. This activates the GoReleaser workflow.

Sources: [.github/workflows/goreleaser.yml5-12](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/goreleaser.yml#L5-L12)

### Nightly Builds

Nightly builds are automatically created whenever changes are pushed to the main branch. This provides a continuous stream of development builds.

Sources: [.github/workflows/nightly.yml3-7](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/nightly.yml#L3-L7)

Build Pipeline
--------------

The build pipeline is managed through reusable workflows defined in the Charm organization's central repository (`charmbracelet/meta`). This ensures consistency across all Charm projects.

Sources: [.github/workflows/goreleaser.yml15-30](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/goreleaser.yml#L15-L30)
 [.github/workflows/nightly.yml9-19](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/nightly.yml#L9-L19)

Release Authentication
----------------------

Mods releases are cryptographically signed to ensure authenticity:

1.  **Linux packages** are signed using GPG
2.  **macOS binaries** are signed with Apple certificates and notarized with Apple's notary service
3.  **Windows releases** may also be signed (though this isn't explicitly evident in the provided files)

Sources: [.github/workflows/goreleaser.yml26-30](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/goreleaser.yml#L26-L30)

Distribution Channels
---------------------

Mods is distributed through multiple channels to reach users on various platforms:

| Channel | Purpose | Configuration |
| --- | --- | --- |
| GitHub Releases | Official binaries and source code | Automatic via GoReleaser |
| Docker Hub | Container images | Requires Docker credentials |
| Homebrew | macOS and Linux package manager | Part of GoReleaser pipeline |
| AUR | Arch Linux User Repository | Requires AUR SSH key |
| Gemfury | Possibly used for package hosting | Requires Fury token |
| Linux packages | deb, rpm formats | NFPM configuration |

Sources: [.github/workflows/goreleaser.yml17-25](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/goreleaser.yml#L17-L25)

Version Management
------------------

Mods follows semantic versioning (MAJOR.MINOR.PATCH). The version is derived from the git tag that triggered the release workflow:

1.  **MAJOR** version increments indicate incompatible API changes
2.  **MINOR** version increments indicate new functionality in a backward-compatible manner
3.  **PATCH** version increments indicate backward-compatible bug fixes

Manual Release Steps
--------------------

While most of the release process is automated, some manual steps might be required:

1.  Ensuring the changelog is updated
2.  Creating and pushing the version tag
3.  Monitoring the release workflow for any issues
4.  Publishing release notes on GitHub

Security Considerations
-----------------------

The release process includes several security measures:

1.  **Signed commits** - Best practice is to sign all commits, especially for release tags
2.  **Binary signing** - Binaries are cryptographically signed
3.  **Secure secrets** - Release credentials are stored as GitHub secrets
4.  **Notarization** - macOS binaries are notarized with Apple

Sources: [.github/workflows/goreleaser.yml26-30](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/goreleaser.yml#L26-L30)

Troubleshooting Releases
------------------------

If issues occur during the release process:

1.  Check the GitHub Actions logs for the specific workflow run
2.  Verify that all required secrets are properly configured
3.  Ensure the tag format follows the expected pattern (v\*._._)
4.  Check that the GoReleaser configuration is valid
