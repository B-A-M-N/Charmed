Flag Parsing and Error Handling
===============================

Relevant source files

*   [error.go](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go)
    
*   [flag.go](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go)
    
*   [flag\_test.go](https://github.com/charmbracelet/mods/blob/07a05d5b/flag_test.go)
    
*   [load.go](https://github.com/charmbracelet/mods/blob/07a05d5b/load.go)
    
*   [load\_test.go](https://github.com/charmbracelet/mods/blob/07a05d5b/load_test.go)
    

This document covers the custom error handling infrastructure in mods, focusing on two specialized systems: command-line flag parsing errors and general application errors. The flag parsing system ([flag.go](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go)
) provides structured error handling for Cobra/pflag command-line arguments, while the general error system ([error.go](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go)
) offers user-friendly error wrapping. For information about how flags are bound and initialized at application startup, see [Application Entry Point](https://deepwiki.com/charmbracelet/mods/2.1-application-entry-point)
.

Purpose and Scope
-----------------

The flag parsing and error handling systems serve three key functions:

1.  **Parse and structure flag errors** from Cobra/pflag into a queryable format with extracted flag names and formatted reason strings
2.  **Provide custom flag types** (specifically `durationFlag`) that extend pflag's built-in type system
3.  **Wrap errors with context** using `modsError` to add human-readable reason strings for user display

These systems are used throughout the application initialization phase when command-line arguments are processed, and during runtime when errors need to be presented to users with appropriate context.

Sources: [flag.go1-90](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L1-L90)
 [error.go1-24](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L1-L24)

Error Type Architecture
-----------------------

The codebase defines three custom error types that serve different purposes in the error handling pipeline:

| Error Type | Purpose | Key Methods | Used For |
| --- | --- | --- | --- |
| `flagParseError` | Structure flag parsing errors | `Flag()`, `ReasonFormat()` | Command-line argument validation |
| `modsError` | Add context to errors | `Reason()` | User-facing error messages |
| `durationFlag` | Parse duration flags | `Set()`, `String()`, `Type()` | Time duration command-line flags |

Sources: [flag.go51-89](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L51-L89)
 [error.go11-23](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L11-L23)

Flag Parse Error System
-----------------------

### flagParseError Structure

The `flagParseError` type wraps pflag/Cobra errors and extracts structured information through pattern matching:

The struct maintains three fields [flag.go51-55](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L51-L55)
:

*   `err`: The original error from pflag
*   `reason`: A format string template (e.g., `"Flag %s needs an argument."`)
*   `flag`: The extracted flag name (e.g., `"--delete"`, `"-d"`)

Sources: [flag.go51-67](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L51-L67)

### Error Pattern Matching

The `newFlagParseError` function [flag.go11-49](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L11-L49)
 uses string prefix matching and regular expressions to parse different error formats:

**Pattern 1: Missing Argument** [flag.go15-23](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L15-L23)

    Input:  "flag needs an argument: --delete"
    Output: flag = "--delete", reason = "Flag %s needs an argument."
    
    Input:  "flag needs an argument: 'd' in -d"  
    Output: flag = "-d", reason = "Flag %s needs an argument."
    

**Pattern 2: Unknown Flag** [flag.go24-26](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L24-L26)

    Input:  "unknown flag: --nope"
    Output: flag = "--nope", reason = "Flag %s is missing."
    

**Pattern 3: Unknown Shorthand** [flag.go27-33](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L27-L33)

    Regex:  `unknown shorthand flag: '.*' in (-\w)`
    Input:  "unknown shorthand flag: 'x' in -xyz"
    Output: flag = "-xyz", reason = "Short flag %s is missing."
    

**Pattern 4: Invalid Argument** [flag.go34-40](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L34-L40)

    Regex:  `invalid argument ".*" for "(.*)" flag: .*`
    Input:  `invalid argument "20dd" for "--delete-older-than" flag: time: unknown unit "dd" in duration "20dd"`
    Output: flag = "--delete-older-than", reason = "Flag %s have an invalid argument."
    

The extracted information allows error handlers to format user-friendly messages with the actual flag name interpolated into the reason template.

Sources: [flag.go11-49](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L11-L49)
 [flag\_test.go10-56](https://github.com/charmbracelet/mods/blob/07a05d5b/flag_test.go#L10-L56)

### Usage Pattern

The `flagParseError` type implements the standard `Error()` interface but extends it with additional methods:

The methods [flag.go57-67](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L57-L67)
:

*   `Error() string`: Returns the original error message for logging
*   `ReasonFormat() string`: Returns the reason template with `%s` placeholder
*   `Flag() string`: Returns the extracted flag name

Sources: [flag.go57-67](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L57-L67)

Custom Duration Flag Type
-------------------------

### durationFlag Implementation

The `durationFlag` type [flag.go69-89](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L69-L89)
 implements the `pflag.Value` interface to provide custom parsing for duration strings using the [caarlos0/duration](https://github.com/charmbracelet/mods/blob/07a05d5b/caarlos0/duration)
 package:

The type is a simple wrapper around `time.Duration` that delegates parsing to `duration.Parse()` [flag.go76-81](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L76-L81)
:

| Method | Signature | Purpose |
| --- | --- | --- |
| `Set` | `Set(s string) error` | Parses duration string and updates value |
| `String` | `String() string` | Returns duration in standard Go format |
| `Type` | `Type() string` | Returns `"duration"` for help text |

### Initialization Pattern

Flags using `durationFlag` are initialized with the `newDurationFlag` helper [flag.go69-72](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L69-L72)
:

    func newDurationFlag(val time.Duration, p *time.Duration) *durationFlag {
        *p = val                    // Set initial value
        return (*durationFlag)(p)   // Return as durationFlag pointer
    }
    

This pattern is used in the application to define duration-based flags like `--delete-older-than`, allowing users to specify durations using natural syntax like `7d`, `2w`, or `30m` instead of nanosecond integers.

Sources: [flag.go69-89](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L69-L89)

General Error Wrapper Types
---------------------------

### modsError Structure

The `modsError` type [error.go11-23](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L11-L23)
 provides a simple wrapper that adds a `reason` field to any error:

The struct fields [error.go12-14](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L12-L14)
:

*   `err error`: The wrapped error
*   `reason string`: Additional context for user display

Methods [error.go17-23](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L17-L23)
:

*   `Error() string`: Returns `err.Error()` for standard error interface
*   `Reason() string`: Returns the `reason` field with additional context

This allows error handling code to check for `modsError` types and extract both the underlying error and the user-facing reason string separately.

Sources: [error.go11-23](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L11-L23)

### newUserErrorf Helper

The `newUserErrorf` function [error.go5-9](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L5-L9)
 is a thin wrapper around `fmt.Errorf`:

The function exists primarily to satisfy linter rules that discourage capitalizing error messages. The comment [error.go5-6](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L5-L6)
 explains:

> "this function is mostly to avoid linters complain about errors starting with a capitalized letter."

By using this helper, the codebase can create user-facing errors with proper capitalization for display purposes while keeping linters satisfied that the errors are intentionally user-facing rather than programmer-facing.

Sources: [error.go5-9](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L5-L9)

Error Handling Flow
-------------------

The complete error handling flow from flag parsing through user display follows this pattern:

The flow demonstrates how different error types are handled at different stages:

1.  **Flag parsing stage**: pflag errors are caught and wrapped in `flagParseError` for structured extraction
2.  **Runtime stage**: Application errors can be wrapped in `modsError` to provide additional context
3.  **Display stage**: Error handlers can interrogate error types to format appropriate messages

Sources: [flag.go11-49](https://github.com/charmbracelet/mods/blob/07a05d5b/flag.go#L11-L49)
 [error.go11-23](https://github.com/charmbracelet/mods/blob/07a05d5b/error.go#L11-L23)

Test Coverage
-------------

The flag parsing error system includes comprehensive tests [flag\_test.go10-56](https://github.com/charmbracelet/mods/blob/07a05d5b/flag_test.go#L10-L56)
 covering all error pattern types:

| Test Case | Input Error | Expected Flag | Expected Reason |
| --- | --- | --- | --- |
| Unknown flag | `"unknown flag: --nope"` | `"--nope"` | `"Flag %s is missing."` |
| Long flag needs arg | `"flag needs an argument: --delete"` | `"--delete"` | `"Flag %s needs an argument."` |
| Short flag needs arg | `"flag needs an argument: 'd' in -d"` | `"-d"` | `"Flag %s needs an argument."` |
| Invalid duration | `invalid argument "20dd" for "--delete-older-than" flag: ...` | `"--delete-older-than"` | `"Flag %s have an invalid argument."` |
| Invalid int | `invalid argument "sdfjasdl" for "--max-tokens" flag: ...` | `"--max-tokens"` | `"Flag %s have an invalid argument."` |
| Invalid bool | `invalid argument "nope" for "-r, --raw" flag: ...` | `"-r, --raw"` | `"Flag %s have an invalid argument."` |

The test suite [flag\_test.go47-56](https://github.com/charmbracelet/mods/blob/07a05d5b/flag_test.go#L47-L56)
 verifies that:

1.  The flag name is correctly extracted via `Flag()`
2.  The reason template is correctly assigned via `ReasonFormat()`
3.  The original error message is preserved via `Error()`

This ensures that error messages can be consistently formatted across the CLI while preserving detailed error information for debugging.

Sources: [flag\_test.go10-56](https://github.com/charmbracelet/mods/blob/07a05d5b/flag_test.go#L10-L56)
