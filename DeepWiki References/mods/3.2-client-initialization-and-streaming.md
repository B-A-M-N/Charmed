Client Initialization and Streaming
===================================

Relevant source files

*   [mods.go](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go)
    
*   [stream.go](https://github.com/charmbracelet/mods/blob/07a05d5b/stream.go)
    

This page documents the client initialization and streaming response mechanisms in the mods codebase. It covers how provider-specific AI clients are instantiated, authenticated, configured, and how streaming responses are received and processed.

For information about provider configuration and API definitions, see [Provider Configuration](https://deepwiki.com/charmbracelet/mods/3.1-provider-configuration)
. For details on model resolution and fallback chains, see [Model Resolution and Fallbacks](https://deepwiki.com/charmbracelet/mods/3.3-model-resolution-and-fallbacks)
. For the message context setup that precedes client initialization, see [Message Stream Context](https://deepwiki.com/charmbracelet/mods/2.4-message-stream-context)
.

* * *

Overview
--------

The client initialization and streaming system is responsible for:

*   Resolving which provider and model to use based on configuration
*   Instantiating provider-specific clients with appropriate authentication
*   Constructing the request payload with messages, parameters, and tools
*   Managing the streaming response lifecycle
*   Handling tool calls and errors during streaming

The primary entry points are `startCompletionCmd` [mods.go276-454](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L276-L454)
 for initialization and `receiveCompletionStreamCmd` [mods.go492-528](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L492-L528)
 for streaming.

Provider-Specific Configuration
-------------------------------

The system initializes different configuration structs based on the resolved provider. Each provider has a default configuration factory and optional overrides.

**Provider Configuration Initialization Paths**

### Ollama Configuration

Ollama requires no authentication but uses a configurable base URL for local inference.

    occfg = ollama.DefaultConfig()
    if api.BaseURL != "" {
        occfg.BaseURL = api.BaseURL
    }
    

Sources: [mods.go314-318](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L314-L318)

### Anthropic Configuration

Anthropic requires an API key retrieved via `ensureKey` with fallback to `ANTHROPIC_API_KEY` environment variable.

    key, err := m.ensureKey(api, "ANTHROPIC_API_KEY", "https://console.anthropic.com/settings/keys")
    accfg = anthropic.DefaultConfig(key)
    if api.BaseURL != "" {
        accfg.BaseURL = api.BaseURL
    }
    

Sources: [mods.go319-327](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L319-L327)

### Google Configuration

Google configuration includes model name and optional thinking budget for reasoning models.

    key, err := m.ensureKey(api, "GOOGLE_API_KEY", "https://aistudio.google.com/app/apikey")
    gccfg = google.DefaultConfig(mod.Name, key)
    gccfg.ThinkingBudget = mod.ThinkingBudget
    

Sources: [mods.go328-334](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L328-L334)

### Cohere Configuration

Cohere follows a similar pattern with API key retrieval.

    key, err := m.ensureKey(api, "COHERE_API_KEY", "https://dashboard.cohere.com/api-keys")
    cccfg = cohere.DefaultConfig(key)
    if api.BaseURL != "" {
        ccfg.BaseURL = api.BaseURL
    }
    

Sources: [mods.go335-343](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L335-L343)

### Azure OpenAI Configuration

Azure supports two authentication modes: standard API key and Azure AD authentication.

    key, err := m.ensureKey(api, "AZURE_OPENAI_KEY", "https://aka.ms/oai/access")

Authentication via ensureKey
----------------------------

The `ensureKey` function [mods.go456-490](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L456-L490)
 provides a three-tier authentication resolution strategy:

| Priority | Source | Description |
| --- | --- | --- |
| 1   | `api.APIKey` | Direct key in config |
| 2   | `api.APIKeyEnv` | Environment variable name |
| 3   | `api.APIKeyCmd` | Shell command to retrieve key |
| 4   | `defaultEnv` | Fallback environment variable |

**API Key Resolution Flow**

The function uses `shellwords.Parse` [mods.go462](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L462-L462)
 to safely parse the `api-key-cmd` and executes it via `exec.Command` [mods.go466](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L466-L466)
 This enables integration with secret managers or password vaults.

Sources: [mods.go456-490](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L456-L490)

Request Construction
--------------------

After client configuration, a `proto.Request` [mods.go406-424](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L406-L424)
 is constructed with the following fields:

| Field | Source | Description |
| --- | --- | --- |
| `Messages` | `m.messages` | Context from `setupStreamContext` |
| `API` | `mod.API` | Provider name |
| `Model` | `mod.Name` | Model identifier |
| `User` | `cfg.User` | User identifier for tracking |
| `Temperature` | `cfg.Temperature` | Sampling temperature (pointer or nil) |
| `TopP` | `cfg.TopP` | Nucleus sampling parameter |
| `TopK` | `cfg.TopK` | Top-K sampling parameter |
| `Stop` | `cfg.Stop` | Stop sequences |
| `MaxTokens` | `cfg.MaxTokens` | Maximum completion tokens |
| `Tools` | `mcpTools(ctx)` | Available MCP tools |
| `ToolCaller` | `toolCall` closure | Tool execution callback |

### Special Handling for o1 Models

OpenAI's o1 models do not support the `max_tokens` parameter. The code detects o1 models and unsets this parameter [mods.go386-392](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L386-L392)
:

    if strings.HasPrefix(mod.Name, "o1") {
        cfg.MaxTokens = 0
    }
    

### Tool Integration

Tools are retrieved via `mcpTools(ctx)` [mods.go397-400](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L397-L400)
 and a `ToolCaller` function is provided [mods.go416-420](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L416-L420)
:

    ToolCaller: func(name string, data []byte) (string, error) {
        ctx, cancel := context.WithTimeout(m.ctx, config.MCPTimeout)
        m.cancelRequest = append(m.cancelRequest, cancel)
        return toolCall(ctx, name, data)
    }
    

This enables the AI to invoke external tools during streaming.

Sources: [mods.go406-424](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L406-L424)
 [mods.go397-400](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L397-L400)

Initiating the Stream
---------------------

Once the client is instantiated, the stream is initiated [mods.go446](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L446-L446)
:

    stream := client.Request(m.ctx, request)
    

The returned `stream.Stream` object is immediately passed to `receiveCompletionStreamCmd` [mods.go447-452](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L447-L452)
:

    return m.receiveCompletionStreamCmd(completionOutput{
        stream: stream,
        errh: func(err error) tea.Msg {
            return m.handleRequestError(err, mod, m.Input)
        },
    })()
    

The `errh` error handler provides model-specific error recovery logic.

Sources: [mods.go446-452](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L446-L452)

Streaming Response Handling
---------------------------

The `receiveCompletionStreamCmd` function [mods.go492-528](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L492-L528)
 manages the streaming lifecycle as a Bubble Tea command that recursively calls itself.

**receiveCompletionStreamCmd State Machine**

### Stream Iteration

The function calls `stream.Next()` [mods.go494](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L494-L494)
 to advance the stream cursor. If `true`, a chunk is available via `stream.Current()` [mods.go495](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L495-L495)

    if msg.stream.Next() {
        chunk, err := msg.stream.Current()
        if err != nil && !errors.Is(err, stream.ErrNoContent) {
            _ = msg.stream.Close()
            return msg.errh(err)
        }
        return completionOutput{
            content: chunk.Content,
            stream:  msg.stream,
            errh:    msg.errh,
        }
    }
    

The `completionOutput` message is returned to the Bubble Tea `Update` method, which will call this command again, creating a recursive streaming loop.

Sources: [mods.go494-505](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L494-L505)

### Error Handling

Errors during chunk retrieval are ignored if they're `stream.ErrNoContent` [mods.go496](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L496-L496)
 Otherwise, the stream is closed and the error handler is invoked [mods.go497-499](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L497-L499)

After streaming completes, `stream.Err()` [mods.go508](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L508-L508)
 is checked for any deferred errors:

    if err := msg.stream.Err(); err != nil {
        return msg.errh(err)
    }
    

Sources: [mods.go496-510](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L496-L510)

### Tool Call Processing

After streaming completes, the stream may contain tool calls that need execution [mods.go512-526](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L512-L526)
:

    results := msg.stream.CallTools()
    toolMsg := completionOutput{
        stream: msg.stream,
        errh:   msg.errh,
    }
    for _, call := range results {
        toolMsg.content += call.String()
    }
    if len(results) == 0 {
        m.messages = msg.stream.Messages()
        return completionOutput{
            errh: msg.errh,
        }
    }
    return toolMsg
    

If tools were called, their results are concatenated into `toolMsg.content` and returned, allowing them to be appended to the output. If no tools were called, the final message history is retrieved via `stream.Messages()` [mods.go521](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L521-L521)
 and stored in `m.messages` for caching.

Sources: [mods.go512-527](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L512-L527)

Integration with Update Loop
----------------------------

The `Mods.Update` method [mods.go125-215](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L125-L215)
 handles `completionOutput` messages [mods.go173-185](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L173-L185)
:

    case completionOutput:
        if msg.stream == nil {
            m.state = doneState
            return m, m.quit
        }
        if msg.content != "" {
            m.appendToOutput(msg.content)
            m.state = responseState
        }
        cmds = append(cmds, m.receiveCompletionStreamCmd(completionOutput{
            stream: msg.stream,
            errh:   msg.errh,
        }))
    

When `msg.stream` is `nil`, streaming has completed and the model transitions to `doneState`. Otherwise, content is appended to output and the streaming command is called again.

Sources: [mods.go173-185](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L173-L185)

Cancellation Support
--------------------

The system supports request cancellation via context cancels stored in `m.cancelRequest` [mods.go65](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L65-L65)
 [mods.go394-395](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L394-L395)
:

    ctx, cancel := context.WithTimeout(m.ctx, config.MCPTimeout)
    m.cancelRequest = append(m.cancelRequest, cancel)
    

When the user quits via `ctrl+c` or `q`, the `quit` method [mods.go259-264](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L259-L264)
 invokes all cancel functions:

    func (m *Mods) quit() tea.Msg {
        for _, cancel := range m.cancelRequest {
            cancel()
        }
        return tea.Quit()
    }
    

This ensures graceful shutdown and prevents orphaned goroutines.

Sources: [mods.go259-264](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L259-L264)
 [mods.go394-395](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L394-L395)

Summary
-------

The client initialization and streaming system provides:

*   **Provider abstraction**: All providers implement `stream.Client` interface
*   **Flexible authentication**: Three-tier key resolution (direct, env, command)
*   **Unified streaming**: Recursive `receiveCompletionStreamCmd` handles all providers
*   **Tool integration**: MCP tools can be invoked during streaming
*   **Error recovery**: Provider-specific error handlers via `handleRequestError`
*   **Cancellation**: Context-based request cancellation on user quit

The design enables seamless switching between providers while maintaining a consistent streaming API.
