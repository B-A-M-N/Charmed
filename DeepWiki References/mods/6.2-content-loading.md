Purpose and Scope
-----------------

The Content Loading subsystem provides a unified mechanism for loading content from multiple sources (plain text, local files, and remote URLs) and normalizing them into string format for use as AI model input. This system enables users to reference external content in prompts using URI-style prefixes (`file://` and `http(s)://`).

For information about how loaded content is processed into messages, see [Message Utilities](https://deepwiki.com/charmbracelet/mods/6.3-message-utilities)
. For details on how messages are assembled into stream contexts, see [Message Stream Context](https://deepwiki.com/charmbracelet/mods/2.4-message-stream-context)
.

Overview
--------

The content loading system supports three input modes:

| Input Type | Prefix | Example |
| --- | --- | --- |
| Plain text | None | `"explain this code"` |
| Local file | `file://` | `file:///path/to/code.go` |
| Remote URL | `http://` or `https://` | `https://example.com/data.txt` |

Implementation
--------------

### Core Loading Function

The `loadMsg` function in [load.go10-33](https://github.com/charmbracelet/mods/blob/07a05d5b/load.go#L10-L33)
 implements a simple pattern-matching approach to determine the content source:

    func loadMsg(msg string) (string, error)
    

The function performs sequential prefix checks:

1.  **URL Loading** [load.go11-22](https://github.com/charmbracelet/mods/blob/07a05d5b/load.go#L11-L22)
    : If the message starts with `https://` or `http://`, the function:
    
    *   Executes `http.Get(msg)` to fetch the content
    *   Reads the response body using `io.ReadAll(resp.Body)`
    *   Defers closing the response body with `defer func() { _ = resp.Body.Close() }()`
    *   Returns the content as a string
2.  **File Loading** [load.go24-30](https://github.com/charmbracelet/mods/blob/07a05d5b/load.go#L24-L30)
    : If the message starts with `file://`, the function:
    
    *   Strips the `file://` prefix using `strings.TrimPrefix(msg, "file://")`
    *   Reads the file content using `os.ReadFile`
    *   Returns the content as a string
3.  **Plain Text** [load.go32](https://github.com/charmbracelet/mods/blob/07a05d5b/load.go#L32-L32)
    : If no prefix matches, the function returns the message unchanged

### Error Propagation

All errors from underlying operations (`http.Get`, `io.ReadAll`, `os.ReadFile`) are returned directly without wrapping. The `//nolint:wrapcheck` directives [load.go14-27](https://github.com/charmbracelet/mods/blob/07a05d5b/load.go#L14-L27)
 indicate this is intentional design to avoid unnecessary error wrapping in this low-level utility function.

HTTP Content Loading
--------------------

### Network Request Details

When loading HTTP(S) URLs [load.go11-22](https://github.com/charmbracelet/mods/blob/07a05d5b/load.go#L11-L22)
:

*   Uses `http.Get` directly (no custom client or timeout configuration)
*   No context is passed (`//nolint:noctx` annotation indicates this is intentional)
*   Security warning is suppressed (`//nolint:gosec`) as the URL is user-provided
*   Response body is guaranteed to be closed via deferred function

### Security Considerations

The HTTP loading mechanism:

*   Accepts any user-provided URL without validation
*   Does not set timeouts, which may cause hangs on slow/unresponsive servers
*   Does not limit response size, potentially allowing large file downloads
*   Trusts user input (appropriate for a CLI tool where the user is in control)

File Content Loading
--------------------

### Path Handling

File path processing [load.go24-30](https://github.com/charmbracelet/mods/blob/07a05d5b/load.go#L24-L30)
:

1.  The `file://` prefix is stripped using `strings.TrimPrefix`
2.  The remaining path is passed directly to `os.ReadFile`
3.  Relative paths are resolved relative to the current working directory
4.  Absolute paths (e.g., `file:///etc/hosts`) work as expected

### Permission Handling

File reading respects system permissions:

*   If the file is unreadable, `os.ReadFile` returns a permission error
*   If the file doesn't exist, it returns a not-found error
*   These errors propagate to the caller without modification

Usage Context
-------------

The `loadMsg` function is invoked during prompt construction when building the message context for AI providers. It enables:

1.  **Documentation Reference**: Loading code files, READMEs, or documentation for analysis
2.  **API Integration**: Fetching remote data or configurations for processing
3.  **Batch Processing**: Reading input from files without shell redirection

While the function itself is stateless, it integrates with the broader message stream context system (see [Message Stream Context](https://deepwiki.com/charmbracelet/mods/2.4-message-stream-context)
) where loaded content becomes part of the message array sent to AI providers.

Design Rationale
----------------

### Simplicity Over Features

The implementation prioritizes simplicity:

*   No caching of loaded content
*   No retry logic for failed HTTP requests
*   No streaming for large files
*   No content type detection or transformation

This minimalist approach is appropriate because:

*   Content loading is typically a one-time operation per invocation
*   Users control input sources and can retry failed commands
*   The CLI context assumes reasonable file sizes
*   Content transformation is not the loading layer's responsibility

### Protocol Extensibility

The sequential `strings.HasPrefix` checks [load.go11-24](https://github.com/charmbracelet/mods/blob/07a05d5b/load.go#L11-L24)
 provide a straightforward extension point. Future protocols (e.g., `s3://`, `gs://`, `ssh://`) can be added by inserting additional prefix checks before the plain text fallback.
