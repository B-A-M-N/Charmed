Cache System
============

Relevant source files

*   [messages\_test.go](https://github.com/charmbracelet/mods/blob/07a05d5b/messages_test.go)
    
*   [mods\_test.go](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go)
    

Purpose and Scope
-----------------

The Cache System is responsible for storing and retrieving complete conversation message histories in the mods application. Unlike the Conversation Database ([5.1](https://deepwiki.com/charmbracelet/mods/5.1-conversation-database)
), which stores lightweight metadata (IDs, titles, timestamps), the cache system stores full message arrays containing all system prompts, user inputs, and assistant responses. This enables conversation continuation with complete context preservation and supports conversation branching when combined with the `--title` flag.

For information about database metadata operations, see [Conversation Database](https://deepwiki.com/charmbracelet/mods/5.1-conversation-database)
. For how messages are prepared for AI providers, see [Message Stream Context](https://deepwiki.com/charmbracelet/mods/2.4-message-stream-context)
.

* * *

Architecture Overview
---------------------

The cache system implements a **dual-storage architecture** where:

*   **SQLite Database**: Stores metadata (conversation ID, title, API, model, timestamps)
*   **Cache Directory**: Stores complete message arrays for each conversation

**Sources**: [mods\_test.go10-173](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L10-L173)
 High-Level System Architecture Diagram 5

The cache system operates through three key phases:

1.  **Operation Determination**: `findCacheOpsDetails` analyzes configuration flags to determine which conversations to read from and write to
2.  **Message Retrieval**: `cacheReadFromID` loads existing message arrays from the cache directory
3.  **Message Persistence**: `cacheWriteToID` saves updated message arrays to the cache directory and metadata to the database

* * *

Cache Operation Determination
-----------------------------

The `findCacheOpsDetails` function determines cache read/write operations based on configuration flags. It returns a `cacheDetailsMsg` containing operation parameters.

**Sources**: [mods\_test.go10-173](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L10-L173)

### Operation Scenarios

| Scenario | Config Flags | ReadID | WriteID | Title | Behavior |
| --- | --- | --- | --- | --- | --- |
| **New Conversation** | None | Empty | New UUID | Config.Title or empty | Creates fresh conversation |
| **Show Conversation** | `--show <id/title>` | Resolved ID | Empty | Existing title | Read-only display |
| **Continue (Same)** | `--continue <id/title>` + prompt | Resolved ID | Same ID | Existing title | Appends to conversation |
| **Continue (Branch)** | `--continue <id> --title "new"` | Resolved ID | New UUID | "new" | Forks conversation |
| **Continue Last** | `--continue-last` + prompt | Most recent ID | Same ID | Existing title | Continues last conversation |

**Sources**: [mods\_test.go19-149](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L19-L149)

### `cacheDetailsMsg` Structure

The `cacheDetailsMsg` type encapsulates all cache operation parameters:

**Key Fields**:

*   **ReadID**: When non-empty, the cache system loads message history from this conversation ID
*   **WriteID**: When non-empty, the cache system saves the updated conversation to this ID
*   **Title**: Used for database metadata when creating new conversations or branches
*   **Model/API**: Copied from Config for metadata storage

**Sources**: [mods\_test.go21-24](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L21-L24)
 [mods\_test.go160-172](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L160-L172)

* * *

Cache Read Flow
---------------

### Reading Message History

When `ReadID` is non-empty, the cache system loads the complete message array for that conversation:

**Sources**: TOC entry 5.2, High-Level System Architecture Diagram 2

### Message Array Structure

The cache stores messages in the `proto.Message` format:

| Field | Type | Description |
| --- | --- | --- |
| `Role` | `proto.Role` | `RoleSystem`, `RoleUser`, or `RoleAssistant` |
| `Content` | `string` | The message text content |

**Example message array**:

    [\
      {Role: proto.RoleSystem, Content: "You are a helpful assistant"},\
      {Role: proto.RoleUser, Content: "What is Go?"},\
      {Role: proto.RoleAssistant, Content: "Go is a programming language..."},\
      {Role: proto.RoleUser, Content: "Tell me more"}\
    ]
    

When continuing a conversation, the cache-loaded messages are inserted before the new user prompt, preserving the complete interaction history for the AI provider.

**Sources**: [messages\_test.go10-48](https://github.com/charmbracelet/mods/blob/07a05d5b/messages_test.go#L10-L48)
 TOC entry 2.4

Cache Write Flow
----------------

### Writing Updated Conversations

When `WriteID` is non-empty, the cache system persists the complete conversation after receiving the AI response:

**Sources**: High-Level System Architecture Diagram 2, TOC entry 5.2

### Write Operations

| Operation | ReadID | WriteID | Cache Action | Database Action |
| --- | --- | --- | --- | --- |
| **New** | Empty | New UUID | Write full array | Insert metadata |
| **Continue (Same)** | ID-A | ID-A | Overwrite array | Update timestamp |
| **Branch** | ID-A | New UUID | Write full array (from ID-A + new) | Insert metadata |
| **Show** | ID-A | Empty | No write | No update |

**Sources**: [mods\_test.go110-149](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L110-L149)

Conversation Branching
----------------------

Branching creates a new conversation from an existing one by combining the `--continue` and `--title` flags:

**Sources**: High-Level System Architecture Diagram 5

### Branching Test Case

The test demonstrates branching behavior:

    Config:
      Continue: "message 1"  (or ID)
      Title: "some title"
      Prefix: "new prompt"
    
    Result:
      ReadID: <original-conversation-id>
      WriteID: <new-conversation-id>
      Title: "some title"
    

The original conversation (referenced by `Continue`) remains unchanged. The cache system:

1.  Reads all messages from the original conversation
2.  Appends the new user prompt
3.  Receives the AI response
4.  Writes the complete array to a **new** conversation ID
5.  Saves metadata with the new title

**Sources**: [mods\_test.go121-149](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L121-L149)

Relationship with Database
--------------------------

The cache and database systems maintain consistency through coordinated operations:

**Sources**: [mods\_test.go27-45](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L27-L45)
 High-Level System Architecture Diagram 5, TOC entry 5.1

### Synchronization Points

1.  **ID Resolution**: When users specify `--continue` or `--show` with a partial ID or title, the database is queried to resolve the full conversation ID before cache operations
2.  **Write Coordination**: After cache writes complete, metadata is saved to the database with timestamp updates
3.  **Consistency**: Both systems use the same conversation ID as the primary key, ensuring referential integrity

Configuration Independence
--------------------------

The cache system respects configuration parameters independently of global settings:

The `cacheDetailsMsg` captures the current Config values for Model and API, ensuring that metadata stored in the database reflects the actual configuration used for that conversation, not any global defaults.

**Sources**: [mods\_test.go160-172](https://github.com/charmbracelet/mods/blob/07a05d5b/mods_test.go#L160-L172)
