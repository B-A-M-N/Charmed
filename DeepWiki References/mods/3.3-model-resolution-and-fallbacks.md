Model Resolution and Fallbacks
==============================

Relevant source files

*   [config\_template.yml](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml)
    
*   [mods.go](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go)
    

This page documents the model resolution system in mods, which translates user-specified model identifiers (including aliases) into concrete API configurations and handles fallback chains for graceful degradation.

For information about how provider configurations are defined and structured, see [Provider Configuration](https://deepwiki.com/charmbracelet/mods/3.1-provider-configuration)
. For details on how the resolved models are used to initialize streaming clients, see [Client Initialization and Streaming](https://deepwiki.com/charmbracelet/mods/3.2-client-initialization-and-streaming)
.

Overview
--------

The model resolution system serves three primary functions:

1.  **Alias Resolution**: Translates user-friendly model names (e.g., `4o`, `sonnet`) to canonical model identifiers (e.g., `gpt-4o`, `claude-3-7-sonnet-latest`)
2.  **API Association**: Determines which API provider should be used for a given model, either explicitly via the `--api` flag or implicitly by searching all configured providers
3.  **Input Validation**: Enforces maximum input character limits (`max-input-chars`) for each model to prevent exceeding context windows

The resolution process occurs early in the request lifecycle, before client initialization and streaming begin.

-------------------------

The core resolution logic is implemented in the `resolveModel` method, which performs a two-level search across APIs and their models.

### Code Implementation

The `resolveModel` function at [mods.go704-747](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L704-L747)
 implements the following logic:

| Step | Action | Code Reference |
| --- | --- | --- |
| 1   | Iterate through `cfg.APIs` | [mods.go705](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L705-L705) |
| 2   | Filter by API name if `cfg.API` is non-empty | [mods.go706-708](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L706-L708) |
| 3   | Iterate through `api.Models` map | [mods.go709](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L709-L709) |
| 4   | Check if model name or alias matches | [mods.go710](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L710-L710) |
| 5   | Canonicalize `cfg.Model` to actual key | [mods.go711-713](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L711-L713) |
| 6   | Return API and Model structs | [mods.go717-719](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L717-L719) |

When a match is found, the function:

*   Sets `mod.Name` to the canonical model name (the map key in `api.Models`)
*   Sets `mod.API` to the API provider name
*   Returns both the `API` struct (containing base URL and authentication) and the `Model` struct (containing model-specific settings)

Sources: [mods.go704-747](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L704-L747)

### Error Cases

The function handles two distinct error scenarios:

**API Specified but Model Not Found:**

    The API endpoint {api} does not contain the model {model}
    Available models are: {comma-separated list}
    

Triggered at [mods.go722-732](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L722-L732)
 when `cfg.API` is non-empty but the model isn't found in that API's model catalog.

**Model Not Found Anywhere:**

    Model {model} is not in the settings file.
    Please specify an API endpoint with --api or configure the model in the settings: mods --settings
    

Triggered at [mods.go736-746](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L736-L746)

Alias Resolution
----------------

Aliases provide user-friendly shortcuts for commonly used models. The alias resolution system is case-sensitive and checks both the canonical model name and the `aliases` array.

### Alias Matching Logic

The matching logic at [mods.go710](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L710-L710)
 uses `slices.Contains` to check the aliases:

This means:

*   **Exact match on canonical name** always works (e.g., `gpt-4o` matches model key `gpt-4o`)
*   **Match on any alias** triggers canonicalization (e.g., `4o` matches via the `aliases: ["4o"]` array)

When an alias matches, the canonical name is written back to `cfg.Model` at [mods.go711](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L711-L711)
 ensuring downstream code always works with canonical identifiers.

### Example Alias Configurations

The following table shows common alias patterns from the default configuration:

| Model (Canonical) | Aliases | Purpose |
| --- | --- | --- |
| `gpt-4o` | `["4o"]` | Short version identifier |
| `gpt-5` | `["5", "gpt5", "gpt-5-thinking", "gpt5-thinking"]` | Multiple naming conventions |
| `claude-opus-4-1-20250805` | `["claude-opus-4.1", "opus-4.1", "opus-4-1", "opus"]` | Versioned and shorthand forms |
| `llama-3.3-70b-versatile` | `["llama3.3", "llama3.3-70b", "llama3.3-versatile"]` | Model family variations |
| `gemini-1.5-pro-latest` | `["gmp", "gemini", "gemini-1.5-pro"]` | Abbreviations and version aliases |

Sources: [config\_template.yml78-192](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml#L78-L192)

### Multi-Provider Alias Handling

When no API is explicitly specified, the resolution system searches all configured APIs in order. This means:

1.  The same alias can exist in multiple providers
2.  The first matching provider wins
3.  Order of APIs in the configuration file determines precedence

Fallback Chains
---------------

Fallback chains provide graceful degradation when a model is unavailable or encounters errors. While fallbacks are defined in the model configuration, they are **not automatically applied** by the `resolveModel` function itself. Instead, they are used during error recovery in the request handling flow.

### Runtime Fallback Behavior

The fallback field is stored in the `Model` struct but is **not automatically invoked** during model resolution. The error handling in `handleRequestError` (referenced at [mods.go450](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L450-L450)
) could potentially implement retry logic using the fallback chain, though the current implementation focuses on retry counting rather than model fallbacks.

To implement automatic fallbacks, the system would need to:

1.  Capture the original error from the API request
2.  Check if `mod.Fallback` is non-empty
3.  Re-invoke `resolveModel` with the fallback model name
4.  Retry the request with the fallback model

Sources: [mods.go266-274](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L266-L274)
 [mods.go449-452](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L449-L452)

Maximum Input Character Enforcement
-----------------------------------

Each model defines a `max-input-chars` limit to prevent exceeding the model's context window. This limit is enforced during message preparation, not during resolution.

### Resolution-Time MaxChars Assignment

After successful model resolution, the `MaxChars` field is set at [mods.go382-384](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L382-L384)
:

This implements a two-tier system:

1.  **Model-specific limit**: If the model definition includes `max-input-chars`, it takes precedence
2.  **Global default**: If not specified, falls back to `cfg.MaxInputChars` (default: 12250)

Sources: [mods.go382-384](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L382-L384)
 [config\_template.yml65](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml#L65-L65)

### MaxChars Values Across Providers

The following table shows representative `max-input-chars` values from the configuration:

| Model | Max Input Chars | Context Window Equivalent |
| --- | --- | --- |
| `gpt-5` | 794,000 | ~200K tokens |
| `gpt-4o` | 392,000 | ~100K tokens |
| `gpt-3.5-turbo` | 12,250 | ~3K tokens |
| `claude-opus-4-1-20250805` | 680,000 | ~170K tokens |
| `gemini-2.0-flash-001` | 4,194,304 | ~1M tokens |
| `llama-3.3-70b-versatile` (Groq) | 392,000 | ~100K tokens |
| `deepseek-chat` | 384,000 | ~96K tokens |

The character limits use an approximate conversion of **1 token ≈ 4 characters**, which is applied in reverse during error recovery at [mods.go686-688](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L686-L688)

Sources: [config\_template.yml78-425](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml#L78-L425)

### Enforcement During Stream Context Setup

The actual enforcement happens in `setupStreamContext` (called at [mods.go402](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L402-L402)
), which is documented in [Message Stream Context](https://deepwiki.com/charmbracelet/mods/2.4-message-stream-context)
. The setup function:

1.  Loads existing conversation messages from cache if continuing
2.  Adds role-based system prompts
3.  Appends the user's new input
4.  Calculates total character count across all messages
5.  Truncates messages if they exceed `mod.MaxChars`

This ensures that the complete message array sent to the API respects the model's context window limitations.

Sources: [mods.go402](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L402-L402)

Error Messages and User Guidance
--------------------------------

The resolution system provides detailed error messages to guide users toward resolution:

### Model Not Found in Specified API

When using `--api openai --model sonnet`, the system reports:

    The API endpoint openai does not contain the model sonnet
    Available models are: gpt-4o-mini, gpt-5, gpt-5-mini, gpt-5-nano, ...
    

This error includes:

*   The problematic API name (styled with `InlineCode`)
*   The unfound model name (styled with `InlineCode`)
*   Complete list of available models from that API's configuration

Sources: [mods.go722-732](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L722-L732)

### Model Not Found in Any API

When using just `--model nonexistent`, the system reports:

    Model nonexistent is not in the settings file.
    Please specify an API endpoint with --api or configure the model in the settings: mods --settings
    

This error guides users to either:

*   Explicitly specify an API that contains the model
*   Add the model to their configuration via `mods --settings`

Sources: [mods.go736-746](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L736-L746)

Summary
-------

The model resolution and fallback system provides:

| Component | Function | Implementation |
| --- | --- | --- |
| **Alias Resolution** | Maps user-friendly names to canonical identifiers | [mods.go710](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L710-L710) |
| **API Association** | Determines which provider hosts the model | [mods.go706-708](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L706-L708) |
| **Canonicalization** | Updates `cfg.Model` to the map key name | [mods.go711](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L711-L711) |
| **Fallback Chains** | Defines degradation paths (configuration-only) | [config\_template.yml](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml)<br> various |
| **MaxChars Enforcement** | Sets character limits for context windows | [mods.go382-384](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L382-L384) |
| **Error Guidance** | Provides actionable error messages | [mods.go722-746](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L722-L746) |

The resolution occurs early in the request lifecycle at [mods.go291](https://github.com/charmbracelet/mods/blob/07a05d5b/mods.go#L291-L291)
 immediately after configuration loading and before client initialization, ensuring that all downstream code works with validated, canonical model identifiers.
