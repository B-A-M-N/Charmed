Dependencies and Build System
=============================

Relevant source files

*   [.github/workflows/build.yml](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml)
    
*   [go.mod](https://github.com/charmbracelet/mods/blob/07a05d5b/go.mod)
    
*   [go.sum](https://github.com/charmbracelet/mods/blob/07a05d5b/go.sum)
    

This page documents the dependency management and build system for the mods project. It covers the Go module configuration, key dependencies organized by functional category, and the build/CI workflow. For information about testing infrastructure, see [Testing Strategy](https://deepwiki.com/charmbracelet/mods/7.2-testing-strategy)
. For CI/CD quality controls, see [Code Quality and CI/CD](https://deepwiki.com/charmbracelet/mods/7.3-code-quality-and-cicd)
.

Purpose and Scope
-----------------

The mods project uses Go modules for dependency management and GitHub Actions for continuous integration. This document explains:

*   The complete dependency tree and how dependencies are organized
*   The role of each major dependency in the system architecture
*   The build process and workflow automation
*   How dependencies map to the architectural layers described in [Core Architecture](https://deepwiki.com/charmbracelet/mods/2-core-architecture)

Build System
------------

### Build Process Flow

The build process is straightforward, using standard Go tooling:

**Sources:** [.github/workflows/build.yml24-30](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L24-L30)

### Build Commands

The standard Go build commands used by the project:

1.  **Download Dependencies**: `go mod download` [.github/workflows/build.yml24](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L24-L24)
    
    *   Fetches all modules listed in go.mod
    *   Verifies checksums against go.sum
    *   Caches modules in `$GOPATH/pkg/mod`
2.  **Build**: `go build -v ./...` [.github/workflows/build.yml27](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L27-L27)
    
    *   Compiles all packages in the module
    *   `-v` flag enables verbose output showing package names
    *   `./...` pattern matches all packages recursively
3.  **Test**: `go test -v -cover -timeout=30s ./...` [.github/workflows/build.yml30](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L30-L30)
    
    *   Runs all tests with verbose output
    *   `-cover` enables coverage reporting
    *   `-timeout=30s` sets maximum test duration per package

### Cross-Platform Build Matrix

The CI system builds on three operating systems:

**Sources:** [.github/workflows/build.yml7-9](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L7-L9)

The matrix strategy ensures cross-platform compatibility without requiring platform-specific code or build scripts.

CI/CD Workflow Integration
--------------------------

### GitHub Actions Workflow

The build workflow is defined in `.github/workflows/build.yml` and consists of three jobs:

**Sources:** [.github/workflows/build.yml1-55](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L1-L55)

### Build Job Configuration

The build job runs on all three platforms with the following configuration:

*   **Go Version**: Uses `setup-go@v6` with `go-version: stable` [.github/workflows/build.yml15-17](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L15-L17)
    
*   **Module Mode**: `GO111MODULE: "on"` environment variable [.github/workflows/build.yml12](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L12-L12)
    
*   **Checkout**: Uses `actions/checkout@v5` [.github/workflows/build.yml21](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L21-L21)
    

### Snapshot Release Job

The snapshot job uses a reusable workflow from `charmbracelet/meta` [.github/workflows/build.yml33](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L33-L33)
:

*   Only runs after successful builds
*   Uses GoReleaser for creating snapshot releases
*   Requires `GORELEASER_KEY` secret for signing

### Dependabot Auto-Merge Job

The dependabot job automatically approves and merges dependency update PRs [.github/workflows/build.yml37-54](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L37-L54)
:

*   **Condition**: Only runs for `dependabot[bot]` actor on pull requests [.github/workflows/build.yml43](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L43-L43)
    
*   **Permissions**: Requires `pull-requests: write` and `contents: write` [.github/workflows/build.yml41-42](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L41-L42)
    
*   **Actions**: Approves PR and merges with squash strategy [.github/workflows/build.yml49-51](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L49-L51)
    

This automation reduces maintenance overhead for dependency updates while ensuring all builds pass before merging.

**Sources:** [.github/workflows/build.yml37-54](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L37-L54)

Dependency Version Management
-----------------------------

### Version Constraints

The project uses semantic versioning with specific version pinning:

*   **Major versions**: Most dependencies pin to specific major versions (e.g., `v1.x.x`, `v2.x.x`)
*   **Patch updates**: Accepted automatically through dependabot
*   **Pre-release versions**: Some Charm libraries use commit-based versions for latest features [go.mod13](https://github.com/charmbracelet/mods/blob/07a05d5b/go.mod#L13-L13)
     [go.mod17](https://github.com/charmbracelet/mods/blob/07a05d5b/go.mod#L17-L17)
    

### Indirect Dependencies

The project has 60+ indirect dependencies [go.mod40-98](https://github.com/charmbracelet/mods/blob/07a05d5b/go.mod#L40-L98)
 that fall into these categories:

1.  **SDK Dependencies**: Required by AI provider SDKs (Azure, AWS, etc.)
2.  **Rendering Dependencies**: Required by Glamour and Lipgloss (chroma, goldmark, bluemonday)
3.  **System Dependencies**: Required by terminal libraries (terminfo, termios, xpty)
4.  **Utility Dependencies**: Required by various packages (uuid, gjson, cast)

All indirect dependencies are tracked in go.sum for reproducible builds [go.sum1-264](https://github.com/charmbracelet/mods/blob/07a05d5b/go.sum#L1-L264)

**Sources:** [go.mod40-98](https://github.com/charmbracelet/mods/blob/07a05d5b/go.mod#L40-L98)
 [go.sum1-264](https://github.com/charmbracelet/mods/blob/07a05d5b/go.sum#L1-L264)

Development Workflow
--------------------

### Local Development

For local development, the standard Go workflow applies:

1.  **Install Dependencies**: `go mod download`
2.  **Run Tests**: `go test ./...`
3.  **Build Binary**: `go build .`
4.  **Run Linter**: See [Code Quality and CI/CD](https://deepwiki.com/charmbracelet/mods/7.3-code-quality-and-cicd)
    

### Adding New Dependencies

To add a new dependency:

1.  Import the package in Go code
2.  Run `go mod tidy` to update go.mod and go.sum
3.  Commit both files together
4.  Dependabot will track future updates automatically

### Updating Dependencies

Dependencies are updated through:

1.  **Automatic**: Dependabot creates weekly PRs for updates
2.  **Manual**: Run `go get -u ./...` to update all dependencies
3.  **Selective**: Run `go get package@version` for specific updates

**Sources:** Build process derived from [.github/workflows/build.yml1-55](https://github.com/charmbracelet/mods/blob/07a05d5b/.github/workflows/build.yml#L1-L55)
