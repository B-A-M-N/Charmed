Purpose and Scope
-----------------

The Conversation Management system handles the persistence, retrieval, and lifecycle of AI conversations in mods. It implements a **dual-storage architecture** where lightweight metadata is stored in SQLite while complete message histories reside in the cache directory. This system enables users to continue conversations across sessions, branch conversations into new threads, and search for past conversations by ID or title.

For information about how message arrays are constructed and loaded from cache, see [Message Stream Context](https://deepwiki.com/charmbracelet/mods/2.4-message-stream-context)
. For details about the cache directory structure and message serialization, see [Cache System](https://deepwiki.com/charmbracelet/mods/5.2-cache-system)
.

Dual-Storage Architecture
-------------------------

Mods separates conversation storage into two complementary layers:

| Storage Layer | Purpose | Data Stored | Access Pattern |
| --- | --- | --- | --- |
| **SQLite Database** (`convoDB`) | Fast metadata queries | ID, Title, UpdatedAt, API, Model | Listing, searching, completion |
| **Cache Directory** | Full conversation history | Complete message arrays (system + user + assistant) | Read for continuation, write after completion |

This architecture is inspired by Git's object model: the database acts as the "index" for fast lookups, while the cache stores the full "blobs" of conversation data.

Conversation Metadata Schema
----------------------------

The `convoDB` type wraps a SQLite database connection and manages the `conversations` table. The schema is created and migrated in `openDB()`:

The `Conversation` struct maps to this schema:

| Field | Type | Purpose |
| --- | --- | --- |
| `ID` | `string` | SHA-1 hash (40 chars), primary key |
| `Title` | `string` | Human-readable identifier from first prompt or `--title` flag |
| `UpdatedAt` | `time.Time` | Timestamp for sorting and finding HEAD |
| `API` | `*string` | Provider name (e.g., "openai", "anthropic"), nullable for backwards compatibility |
| `Model` | `*string` | Model identifier (e.g., "gpt-4o", "claude-3-sonnet"), nullable for backwards compatibility |

The database includes indices on `id` and `title` for fast lookups. The `api` and `model` columns were added via migration logic using `hasColumn()` checks.

Database Operations
-------------------

The `convoDB` type provides these core operations:

### Save Operation

The `Save()` method attempts an UPDATE first, then falls back to INSERT if no rows were affected. This upsert pattern ensures idempotency.

**Sources:** [db.go113-147](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L113-L147)

### Find Operation

The `Find(in string)` method uses two strategies based on input length:

| Input Length | Strategy | SQL Pattern | Use Case |
| --- | --- | --- | --- |
| `< sha1minLen` | Exact title match | `WHERE title = ?` | "my conversation" |
| `>= sha1minLen` | ID glob or exact title | `WHERE id glob ? OR title = ?` | "df31ae" or "my conversation" |

The glob pattern appends `*` to enable prefix matching for SHA-1 IDs. If multiple matches are found, `errManyMatches` is returned. If no matches are found, `errNoMatches` is returned.

**Sources:** [db.go251-271](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L251-L271)
 [db.go192-219](https://github.com/charmbracelet/mods/blob/07a05d5b/db.go#L192-L219)

### Other Operations

*   **`FindHEAD()`**: Returns most recently updated conversation using `ORDER BY updated_at DESC LIMIT 1`
*   **`Delete(id)`**: Removes conversation by exact ID match
*   **`List()`**: Returns all conversations ordered by `updated_at DESC`
*   **`ListOlderThan(t time.Duration)`**: Returns conversations older than specified duration
*   **`Completions(in string)`**: Returns shell completion candidates formatted as `<id/title>\t<title/id>`

Conversation Branching
----------------------

Branching creates a new conversation thread while preserving the original conversation's history. This occurs when `--continue` and `--title` are both specified:

Error Handling
--------------

The database layer defines two sentinel errors for find operations:

| Error | Condition | User Message |
| --- | --- | --- |
| `errNoMatches` | No conversations match input | "Could not find the conversation." |
| `errManyMatches` | Multiple conversations match ID prefix | "Multiple conversations matched the input" |

The `modsError` wrapper (see [Flag Parsing and Error Handling](https://deepwiki.com/charmbracelet/mods/6.1-flag-parsing-and-error-handling)
) formats these errors for terminal display.

SQLite errors are wrapped using `handleSqliteErr()` which extracts error codes and converts them to readable strings using `sqlite.ErrorCodeString`.

-------------

*   All `findCacheOpsDetails()` scenarios (new, show, continue, branch)
*   CRUD operations (Save, Find, Delete, List)
*   ID prefix matching with various lengths
*   Exact title matching
*   Multiple match detection
*   HEAD finding
*   Completions generation

The tests use an in-memory SQLite database (`:memory:`) for fast, isolated execution.
