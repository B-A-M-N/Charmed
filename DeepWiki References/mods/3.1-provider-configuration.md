Provider Configuration
======================

Relevant source files

*   [config.go](https://github.com/charmbracelet/mods/blob/07a05d5b/config.go)
    
*   [config\_template.yml](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml)
    

This document explains how LLM providers are configured in mods through the `API` and `Model` structs, authentication methods, and the complete provider catalog. It covers the configuration file structure, authentication options (api-key, api-key-env, api-key-cmd), model aliases and fallbacks, and the supported provider ecosystem.

For information about how these configurations are used at runtime, see Client Initialization and Streaming (page 3.2). For details about the overall configuration system, see Configuration System (page 2.2).

Overview
--------

Provider configuration in mods is declarative and defined in YAML format within the `mods.yml` configuration file. Each provider (OpenAI, Anthropic, Ollama, etc.) is configured as an `API` entry with associated models, authentication methods, and endpoint URLs. This separation allows users to easily add custom providers, configure authentication, and manage multiple API endpoints.

**Sources:** [config.go90-100](https://github.com/charmbracelet/mods/blob/07a05d5b/config.go#L90-L100)
 [config\_template.yml71-533](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml#L71-L533)

API Struct Definition
---------------------

The `API` struct represents a single provider configuration with authentication, endpoints, and model definitions. It is defined in [config.go90-100](https://github.com/charmbracelet/mods/blob/07a05d5b/config.go#L90-L100)
 and serves as the primary configuration unit for LLM providers.

### API Struct Fields

| Field | Type | YAML Key | Description |
| --- | --- | --- | --- |
| `Name` | `string` | (key in apis map) | Provider identifier (e.g., "openai", "anthropic") |
| `APIKey` | `string` | `api-key` | API key stored directly in config (not recommended) |
| `APIKeyEnv` | `string` | `api-key-env` | Name of environment variable containing API key |
| `APIKeyCmd` | `string` | `api-key-cmd` | Shell command to execute to retrieve API key |
| `BaseURL` | `string` | `base-url` | Base URL for API requests |
| `Models` | `map[string]Model` | `models` | Map of model names to Model configurations |
| `User` | `string` | `user` | Optional user identifier for certain APIs |
| `Version` | `string` | `version` | API version (currently unused) |

**Sources:** [config.go90-100](https://github.com/charmbracelet/mods/blob/07a05d5b/config.go#L90-L100)

### Authentication Methods

Mods supports three authentication methods with the following precedence order:

1.  **Direct API Key (`api-key`)**: Key stored directly in YAML (least secure)
2.  **Environment Variable (`api-key-env`)**: Reference to environment variable
3.  **Command Execution (`api-key-cmd`)**: Execute shell command to retrieve key

**Example configurations:**

The command-based approach is particularly useful for password managers like Bitwarden (`rbw`), 1Password (`op`), or custom key retrieval scripts.

**Sources:** [config.go93-95](https://github.com/charmbracelet/mods/blob/07a05d5b/config.go#L93-L95)
 [config\_template.yml74-76](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml#L74-L76)

Model Struct Definition
-----------------------

The `Model` struct defines configuration for individual LLM models within a provider. It includes metadata about capabilities, aliases for convenient access, and fallback chains for graceful degradation.

### Model Struct Fields

| Field | Type | YAML Key | Description |
| --- | --- | --- | --- |
| `Name` | `string` | (key in models map) | Canonical model identifier |
| `API` | `string` | N/A | Provider name (set during resolution) |
| `MaxChars` | `int64` | `max-input-chars` | Maximum characters allowed in input |
| `Aliases` | `[]string` | `aliases` | Alternative names for the model (e.g., "4o" → "gpt-4o") |
| `Fallback` | `string` | `fallback` | Model to use if this one fails or is unavailable |
| `ThinkingBudget` | `int` | `thinking-budget` | Budget for reasoning models (optional) |

**Example configuration:**

This configuration allows users to invoke GPT-4o by typing `mods --model 4o` instead of the full model name. If GPT-4o is unavailable, mods will automatically attempt GPT-4, and then GPT-3.5-turbo.

**Sources:** [config.go80-88](https://github.com/charmbracelet/mods/blob/07a05d5b/config.go#L80-L88)
 [config\_template.yml99-118](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml#L99-L118)

### Model Alias Resolution

Model aliases enable convenient shorthand references. The resolution process:

1.  User specifies model via `--model` flag or config default
2.  System searches all providers for exact model name match
3.  If not found, searches for alias match across all models
4.  Returns resolved model with provider information

Common alias patterns:

*   Version shortcuts: `"4o"` → `gpt-4o`, `"35t"` → `gpt-3.5-turbo`
*   Semantic names: `"sonnet"` → `claude-3-5-sonnet-20241022`, `"opus"` → `claude-opus-4-1`
*   Provider-specific: `"llama3.3"` → `llama-3.3-70b-versatile` (on Groq)

**Sources:** [config\_template.yml79-169](https://github.com/charmbracelet/mods/blob/07a05d5b/config_template.yml#L79-L169)

**Notable models:** `gpt-5`, `gpt-4o`, `gpt-4o-mini`, `o1`, `o3`, `o3-mini`, `o3-pro`

APIs Configuration Loading
--------------------------

The `APIs` type implements custom YAML unmarshaling to maintain provider order and correctly parse the nested configuration structure.

### Custom YAML Unmarshaling

The `UnmarshalYAML` method in [config.go106-116](https://github.com/charmbracelet/mods/blob/07a05d5b/config.go#L106-L116)
 handles the conversion:

1.  Iterates through YAML node pairs (key + value)
2.  Extracts API name from the key node
3.  Decodes the value node into an `API` struct
4.  Sets the `API.Name` field
5.  Appends to the `APIs` slice

This approach preserves the declaration order of providers in the config file and correctly handles the map-based YAML structure.

**Sources:** [config.go102-116](https://github.com/charmbracelet/mods/blob/07a05d5b/config.go#L102-L116)

### Configuration File Hierarchy

Provider configurations follow the standard four-tier cascade (see Configuration System, page 2.2):

1.  **Embedded Template** (`config_template.yml`): Default provider catalog
2.  **User Config File** (`~/.config/mods/mods.yml`): User customizations
3.  **Environment Variables**: API key overrides via `MODS_*` prefix
4.  **CLI Flags**: Per-invocation provider/model selection

Streaming Protocol Implementation
---------------------------------

The streaming architecture uses provider-specific streaming libraries wrapped in a common `stream.Stream` interface. Each provider maintains streaming state and handles response accumulation differently.

### Stream State Management

Both providers implement similar state management patterns:

1.  **Factory Pattern**: Each stream maintains a factory function to create new streams for tool call continuations
2.  **Done Flag**: Tracks when a stream is completed and needs to be reset for the next request
3.  **Message Accumulation**: Collects streaming chunks into complete messages for conversation history

The `Next()` method handles the core streaming loop:

*   Returns `true` while chunks are available
*   Sets `done = true` when stream ends
*   Resets stream via factory when restarting after tool calls

Provider Registration and Selection
-----------------------------------

The architecture allows for dynamic provider selection through configuration, with each provider package exposing a standardized interface for client creation. The main application selects providers based on API configuration without tight coupling to specific implementations.
