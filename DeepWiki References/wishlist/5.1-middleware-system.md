Purpose and Scope
-----------------

This document describes the middleware architecture in the Wishlist system that enables extension of SSH server functionality. The middleware system provides a way to add custom behavior to SSH servers, allowing for terminal UI applications, authentication methods, request handling, and other custom functionality. For information about the overall SSH server implementation, see [SSH Server Implementation](https://deepwiki.com/charmbracelet/wishlist/5-ssh-server-implementation)
.

Middleware Architecture Overview
--------------------------------

### What is a Middleware in Wishlist?

In Wishlist, a middleware is a function that wraps an SSH handler and can modify its behavior. Each middleware conforms to the `wish.Middleware` type and follows a standard pattern:

    func(ssh.Handler) ssh.Handler
    

This pattern allows middlewares to be composed together to form a chain of handlers that process SSH sessions sequentially.

Built-in Middlewares
--------------------

Wishlist provides several built-in middlewares to handle common functionality:

### Commands Middleware

The `cmdsMiddleware` handles SSH commands using the `-t` flag, allowing users to connect directly to specific endpoints. For example: `ssh host -t app_name`

The middleware checks if a command is provided and, if it matches an endpoint name, directly connects to that endpoint, bypassing the normal TUI flow.

Sources: [middleware.go20-47](https://github.com/charmbracelet/wishlist/blob/5fe34a29/middleware.go#L20-L47)

### Listing Middleware

The `listingMiddleware` manages the listing and handoff of endpoints, providing a terminal UI for selecting endpoints to connect to.

This middleware:

1.  Sets up multiplexed input/output to handle user interaction
2.  Creates a terminal UI for displaying and selecting endpoints
3.  Listens for events such as window resizing and endpoint updates
4.  Handles the connection to selected endpoints

Sources: [middleware.go49-88](https://github.com/charmbracelet/wishlist/blob/5fe34a29/middleware.go#L49-L88)

### Event Handling

The `listenAppEvents` function is a key part of the listing middleware that manages various events:

This function monitors and handles events during an SSH session, including terminal resizing, endpoint updates, and session termination.

Sources: [middleware.go90-128](https://github.com/charmbracelet/wishlist/blob/5fe34a29/middleware.go#L90-L128)

Middleware Integration with Endpoints
-------------------------------------

Middleware functions are attached to endpoints as part of their configuration:

When creating an SSH server, the middlewares associated with an endpoint are applied to that server, along with any global middlewares specified in the server factory function.

Middleware Execution Flow
-------------------------

When an SSH connection is established:

1.  The SSH server processes the connection
2.  Middlewares are executed in the order they were added
3.  Each middleware can perform pre-processing actions before calling the next handler
4.  The base handler is eventually called to handle the session
5.  Control flows back up the middleware chain, allowing for post-processing
6.  The final response is sent back to the client
