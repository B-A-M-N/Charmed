Purpose and Scope
-----------------

This document describes how Wishlist integrates with standard SSH configuration files (typically found at `~/.ssh/config`). Wishlist can read and parse SSH config files to populate its endpoint list, allowing users to leverage their existing SSH configurations without duplicating information. This integration provides a seamless experience for users already familiar with SSH configuration.

For information about other configuration methods, such as YAML-based configuration, see [Configuration System](https://deepwiki.com/charmbracelet/wishlist/3-configuration-system)
.

Sources: [sshconfig/parse.go1-3](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L1-L3)

SSH Config Parsing Process
--------------------------

Wishlist uses a dedicated package to parse SSH configuration files into Wishlist endpoints. The process involves reading SSH config files, parsing the SSH configuration syntax, and converting the host entries into Wishlist endpoint structures.

### Parsing Workflow

Sources: [sshconfig/parse.go30-93](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L30-L93)

### Configuration Loading

Wishlist searches for SSH config files in multiple locations, including:

1.  `.wishlist/config`
2.  User configuration directory
3.  `~/.ssh/config`
4.  `/etc/ssh/ssh_config`

When a valid SSH config file is found, it's parsed and converted into Wishlist endpoints.

SSH Config to Endpoint Mapping
------------------------------

The parsing process converts each SSH config `Host` entry into a Wishlist `Endpoint` structure. Here's how SSH config directives map to Wishlist endpoint fields:

Sources: [sshconfig/parse.go70-86](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L70-L86)

### Mapping Details

The SSH config directives are processed and mapped to Wishlist endpoint fields as follows:

| SSH Config Directive | Wishlist Endpoint Field | Description |
| --- | --- | --- |
| `Host` | `Name` | The name of the endpoint |
| `HostName` + `Port` | `Address` | Combined as `hostname:port` |
| `User` | `User` | Username for authentication |
| `IdentityFile` | `IdentityFiles` | SSH key files for authentication |
| `ForwardAgent` | `ForwardAgent` | Enable SSH agent forwarding |
| `RequestTTY` | `RequestTTY` | Request TTY allocation |
| `RemoteCommand` | `RemoteCommand` | Command to execute on connection |
| `ConnectTimeout` | `Timeout` | Connection timeout in seconds |
| `SendEnv` | `SendEnv` | Environment variables to send |
| `SetEnv` | `SetEnv` | Environment variables to set |
| `PreferredAuthentications` | `PreferredAuthentications` | Authentication methods to try |
| `ProxyJump` | `ProxyJump` | Proxy jump configuration |

Sources: [sshconfig/parse.go70-86](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L70-L86)
 [\_example/config1-50](https://github.com/charmbracelet/wishlist/blob/5fe34a29/_example/config#L1-L50)

Supported Features
------------------

Wishlist's SSH config integration supports several advanced SSH configuration features:

### Wildcard Host Patterns

Wishlist supports wildcard patterns in Host entries, such as `*.example.com`. These patterns are matched against host names during endpoint creation.

Sources: [sshconfig/parse.go47-67](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L47-L67)
 [sshconfig/parse.go273-294](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L273-L294)

### Multiple Hosts in One Entry

SSH config allows defining multiple hosts in a single entry. Wishlist processes each host separately while applying the shared configuration.

    Host server1 server2 server3
        User admin
        ForwardAgent yes
    

This creates three separate endpoints with the same user and ForwardAgent setting.

Sources: [sshconfig/testdata/good27-31](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/testdata/good#L27-L31)

### Include Directives

Wishlist supports the `Include` directive, which allows including other SSH config files. This enables modular configuration and organization of SSH hosts.

Sources: [sshconfig/parse.go235-262](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L235-L262)
 [sshconfig/parse\_test.go202-249](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse_test.go#L202-L249)

Example SSH Config
------------------

Below is an example of a typical SSH config file that demonstrates the supported features:

    Host *.example.com
        User admin
        ForwardAgent yes
    
    Host server1
        HostName server1.example.com
        Port 2222
        IdentityFile ~/.ssh/server1_key
        RequestTTY yes
        
    Host jump-host
        HostName jump.example.com
        
    Host internal-server
        HostName 192.168.1.10
        ProxyJump jump-host
        RemoteCommand tmux a
    

This configuration would create three endpoints in Wishlist:

*   `server1` with custom port and identity file
*   `jump-host` for accessing the jump server
*   `internal-server` that uses ProxyJump through the jump-host

Sources: [\_example/config1-51](https://github.com/charmbracelet/wishlist/blob/5fe34a29/_example/config#L1-L51)

Parsing Implementation Details
------------------------------

The SSH config parsing is implemented in the `sshconfig` package, primarily in `parse.go`. The main functions are:

1.  `ParseFile`: Takes a file path and parses the SSH config at that location
2.  `ParseReader`: Parses SSH config from an io.Reader
3.  `parseInternal`: Core parsing function that uses the `ssh_config` package

The parser handles:

*   Basic SSH config syntax
*   Wildcard patterns using the `gobwas/glob` package
*   Merging configurations from different sources
*   Converting boolean strings (`yes`, `true`) to actual boolean values
*   Expanding paths for identity files and includes

Sources: [sshconfig/parse.go30-93](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L30-L93)
 [sshconfig/parse.go159-370](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L159-L370)

Hosts and Wildcards Processing
------------------------------

Wishlist processes wildcard hosts separately from regular hosts, then applies matching wildcards to the specific hosts:

First, host entries are split into two categories:

1.  Regular hosts (e.g., `server1.example.com`)
2.  Wildcard hosts (e.g., `*.example.com`)

Then, for each regular host, Wishlist checks if any wildcard patterns match and applies those configurations. This allows for hierarchical configuration where more specific host entries can override more general ones.

Sources: [sshconfig/parse.go273-294](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L273-L294)
 [sshconfig/parse.go47-93](https://github.com/charmbracelet/wishlist/blob/5fe34a29/sshconfig/parse.go#L47-L93)

Integration with Overall Configuration System
---------------------------------------------

SSH config integration is just one part of Wishlist's configuration system. When Wishlist starts, it tries to load configuration from multiple sources in this order:

1.  Explicitly specified configuration file (if provided)
2.  Default configuration locations including SSH config files
3.  Discovered endpoints (from network, Tailscale, etc.)

SSH config-derived endpoints can be merged with endpoints from other sources to create the final list of available endpoints.

Sources: [cmd/wishlist/main\_test.go13-75](https://github.com/charmbracelet/wishlist/blob/5fe34a29/cmd/wishlist/main_test.go#L13-L75)
