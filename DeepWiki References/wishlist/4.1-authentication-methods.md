Authentication Methods
======================

*   [README.md](https://github.com/charmbracelet/wishlist/blob/5fe34a29/README.md?plain=1)

This page details the authentication methods supported by the Wishlist system and how they are prioritized in different operation modes. Authentication in Wishlist varies depending on whether the system is running in local mode or server mode.

--------------------------------------------

Wishlist supports the following authentication methods as defined in the config:

### Local Mode Authentication

When Wishlist runs in local mode (directly from the terminal), it follows this authentication process:

Sources: [client\_auth.go69-103](https://github.com/charmbracelet/wishlist/blob/5fe34a29/client_auth.go#L69-L103)
 [README.md113-120](https://github.com/charmbracelet/wishlist/blob/5fe34a29/README.md?plain=1#L113-L120)

In local mode, Wishlist offers multiple authentication strategies for each authentication method type:

1.  For **publickey** authentication:
    
    *   First uses identity files specified in the endpoint configuration
    *   Then tries to use an SSH agent if available
    *   Finally tries common key names in the `~/.ssh/` directory
2.  For **keyboard-interactive** and **password** authentication:
    
    *   Prompts the user for input

### Server Mode Authentication

When running in server mode (as an SSH server that other clients connect to), the authentication flow is:

Sources: [client\_auth.go24-57](https://github.com/charmbracelet/wishlist/blob/5fe34a29/client_auth.go#L24-L57)
 [README.md121-127](https://github.com/charmbracelet/wishlist/blob/5fe34a29/README.md?plain=1#L121-L127)

In server mode, Wishlist primarily:

1.  First tries to forward the current SSH agent
2.  If no agent is available, it creates or uses an existing ed25519 key in `.wishlist/client_ed25519`

Authentication Method Implementation Details
--------------------------------------------

### Public Key Authentication

Public key authentication in Wishlist is implemented with several helper functions:

| Function | Purpose |
| --- | --- |
| `tryRemoteAuthAgent` | Attempts to use an SSH agent in server mode |
| `tryNewKey` | Creates or uses a keypair in `.wishlist/client_ed25519` |
| `tryIdentityFiles` | Tries each specified identity file |
| `tryUserKeys` | Attempts to find standard key files in `~/.ssh/` |
| `parsePrivateKey` | Parses key files, handling password-protected keys |

Sources: [client\_auth.go160-305](https://github.com/charmbracelet/wishlist/blob/5fe34a29/client_auth.go#L160-L305)

For local mode, Wishlist checks for standard SSH key types:

Sources: [client\_auth.go241-249](https://github.com/charmbracelet/wishlist/blob/5fe34a29/client_auth.go#L241-L249)

### SSH Agent Integration

Wishlist integrates with SSH agents through:

1.  **Local agent access** - Connects to the agent specified in `SSH_AUTH_SOCK` environment variable
2.  **Agent forwarding** - Can forward a local agent to remote connections using the `ForwardAgent` option

Sources: [client\_auth.go124-157](https://github.com/charmbracelet/wishlist/blob/5fe34a29/client_auth.go#L124-L157)

### Host Key Verification

Wishlist verifies host keys similarly to standard SSH clients:

1.  It checks the known\_hosts file to verify the server's host key
2.  If the host is not known, it adds the host key to the known\_hosts file
3.  If the host key has changed, it warns about a potential man-in-the-middle attack

Sources: [client\_auth.go311-338](https://github.com/charmbracelet/wishlist/blob/5fe34a29/client_auth.go#L311-L338)

Agent Forwarding
----------------

Agent forwarding is supported in both Wishlist modes:

1.  In **local mode**, it forwards your local SSH agent to the remote connection if `ForwardAgent` is enabled
2.  In **server mode**, it can forward the agent that was used to connect to Wishlist to subsequent connections

Example SSH config enabling agent forwarding:

    Host wishlist
        HostName example.com
        Port 2222
        ForwardAgent yes
        UserKnownHostsFile /dev/null
    

Sources: [README.md132-152](https://github.com/charmbracelet/wishlist/blob/5fe34a29/README.md?plain=1#L132-L152)

This allows you to maintain a chain of authenticated connections without storing private keys on intermediate servers.
