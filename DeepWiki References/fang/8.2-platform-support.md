Platform Support
================

Relevant source files

This document explains how fang handles platform-specific implementations to ensure consistent terminal behavior across different operating systems. The primary focus is on Windows virtual terminal processing, which enables ANSI escape sequences and styled output on Windows systems. For information about terminal styling and color schemes, see [Styling and Themes](https://deepwiki.com/charmbracelet/fang/3.3-styling-and-themes)
.

Overview
--------

Fang uses Go's build tags to provide platform-specific implementations while maintaining a unified interface. The most significant platform difference is Windows' requirement for explicit virtual terminal processing enablement to support ANSI escape sequences, which are essential for styled terminal output.

Build Tag Architecture
----------------------

Fang implements platform-specific behavior using Go's build constraint system, with two separate implementations of the same function:

| Platform | File | Build Tag | Behavior |
| --- | --- | --- | --- |
| Windows | `fang_windows.go` | `//go:build windows` | Enables virtual terminal processing |
| Others (Unix-like) | `fang_other.go` | `//go:build !windows` | No-op implementation |

Both files implement the `enableVirtualTerminalProcessing` function with identical signatures but different implementations, allowing the Go toolchain to select the appropriate version at compile time.

**Sources:** [fang\_windows.go1-3](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L1-L3)
 [fang\_other.go1-3](https://github.com/charmbracelet/fang/blob/d89b30af/fang_other.go#L1-L3)

Windows Virtual Terminal Processing
-----------------------------------

Windows requires explicit configuration to support ANSI escape sequences, which are used by Lipgloss for styled output. Without this enablement, ANSI codes appear as literal text in the terminal.

### Implementation Details

The Windows implementation performs several operations to enable virtual terminal processing:

**Sources:** [fang\_windows.go14-30](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L14-L30)

### Function Breakdown

The `enableVirtualTerminalProcessing` function in [fang\_windows.go14-30](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L14-L30)
 implements the following logic:

1.  **Type Assertion** [fang\_windows.go15-18](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L15-L18)
    : Checks if the writer implements `term.File` interface and verifies it's connected to a terminal. Returns `nil` immediately if not, as VT processing is only relevant for interactive terminals.
    
2.  **Mode Retrieval** [fang\_windows.go19-22](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L19-L22)
    : Calls `windows.GetConsoleMode` to retrieve the current console mode flags. This preserves existing settings while adding VT support.
    
3.  **Mode Update** [fang\_windows.go24-27](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L24-L27)
    : Calls `windows.SetConsoleMode` with the original mode OR'd with `windows.ENABLE_VIRTUAL_TERMINAL_PROCESSING`. This flag enables ANSI escape sequence interpretation in the Windows console.
    

### Windows API Integration

**Sources:** [fang\_windows.go6-12](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L6-L12)
 [fang\_windows.go14-30](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L14-L30)

The implementation uses two key Windows API functions via the `golang.org/x/sys/windows` package:

| Function | Purpose | Parameters | Return |
| --- | --- | --- | --- |
| `GetConsoleMode` | Retrieves current console mode flags | Console handle, pointer to mode | Error or nil |
| `SetConsoleMode` | Updates console mode flags | Console handle, new mode | Error or nil |

The `ENABLE_VIRTUAL_TERMINAL_PROCESSING` flag is a Windows 10+ feature that enables ANSI escape sequence processing in the Windows Console.

**Sources:** [fang\_windows.go19-27](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L19-L27)

Non-Windows Platforms
---------------------

Unix-like systems (Linux, macOS, BSD) natively support ANSI escape sequences without special configuration. The non-Windows implementation is a no-op stub.

### Stub Implementation

The [fang\_other.go10](https://github.com/charmbracelet/fang/blob/d89b30af/fang_other.go#L10-L10)
 implementation:

This function:

*   Accepts the same `io.Writer` parameter as the Windows version
*   Immediately returns `nil` without performing any operations
*   Maintains interface compatibility with the Windows implementation

The parameter name is intentionally omitted since it's unused, following Go conventions for documenting that a parameter is required by the interface but not used in this implementation.

**Sources:** [fang\_other.go6-10](https://github.com/charmbracelet/fang/blob/d89b30af/fang_other.go#L6-L10)

Integration with fang.Execute
-----------------------------

The `enableVirtualTerminalProcessing` function is called early in the `fang.Execute` flow to ensure terminal capabilities are configured before any output occurs.

The function is called with `cmd.OutOrStdout()` as the writer argument, which is typically the stdout stream where all help text, version information, and styled output will be written. By configuring the terminal before any command execution or help rendering, fang ensures that all subsequent output will display correctly with ANSI styling.

**Sources:** [fang\_windows.go14](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L14-L14)
 [fang\_other.go10](https://github.com/charmbracelet/fang/blob/d89b30af/fang_other.go#L10-L10)

Error Handling
--------------

The Windows implementation can return errors in two scenarios:

| Error Source | Condition | Error Message Format |
| --- | --- | --- |
| `GetConsoleMode` | Unable to query console mode | `"error getting console mode: %w"` |
| `SetConsoleMode` | Unable to update console mode | `"error setting console mode: %w"` |

These errors are propagated up to `fang.Execute`, which will return them to the caller. In practice, these errors are rare and typically indicate:

*   The file descriptor is not a valid console handle
*   Insufficient permissions to modify console properties
*   The application is running in a context where console manipulation is restricted

The non-Windows implementation never returns errors.

**Sources:** [fang\_windows.go20-27](https://github.com/charmbracelet/fang/blob/d89b30af/fang_windows.go#L20-L27)

Dependencies
------------

The platform support system relies on the following external packages:

| Package | Usage | Platform |
| --- | --- | --- |
| `github.com/charmbracelet/x/term` | Terminal detection and file interface | All |
| `golang.org/x/sys/windows` | Windows console API bindings | Windows only |

The `term.File` interface provides a cross-platform abstraction for file descriptors, while the Windows-specific package provides typed access to Windows console functions without unsafe code or CGo.
