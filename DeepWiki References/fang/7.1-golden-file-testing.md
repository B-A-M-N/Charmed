Purpose and Scope
-----------------

This document explains the golden file testing approach used to validate the fang library's CLI output. Golden file testing is a snapshot-based testing methodology where test outputs are compared against pre-approved reference files ("golden files") to ensure consistent behavior across code changes.

This document covers the test infrastructure, golden file organization, test scenarios, and procedures for updating golden files. For information about the example application that can be run interactively, see [Example Application](https://deepwiki.com/charmbracelet/fang/7.2-example-application)
.

Overview
--------

The fang library uses golden file testing as its primary testing strategy to validate:

*   Help text rendering and formatting
*   Version output and flag handling
*   Error messages and styling
*   Manpage generation output
*   Shell completion behavior

Golden file testing is particularly appropriate for CLI tools because:

1.  **Visual Output Matters**: The styled, formatted output is a key feature of fang
2.  **Regression Prevention**: Any unintended change to output formatting is immediately detected
3.  **Human Review**: Changes to golden files require explicit review, making output changes deliberate
4.  **Terminal Width Consistency**: Tests set a fixed terminal width to ensure reproducible output

**Sources:** [fang\_test.go1-13](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L1-L13)

Test Infrastructure
-------------------

### Core Test Helpers

The test suite in [fang\_test.go](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go)
 provides several helper functions that form the foundation of the testing approach:

| Helper Function | Signature | Purpose |
| --- | --- | --- |
| `toMkroot` | `func(c *cobra.Command) func() *cobra.Command` | Converts a `*cobra.Command` into a factory function that returns fresh command instances |
| `exercise` | `func(t *testing.T, mkroot func() *cobra.Command, options ...fang.Option)` | Runs a standard test battery (error, version, help) against a command |
| `doExercise` | `func(t *testing.T, mkroot func() *cobra.Command, args []string, assert func(...), options ...fang.Option)` | Executes a single test scenario with specific arguments and options |
| `assertNoError` | `func(t *testing.T, err error, stdout, stderr bytes.Buffer)` | Validates successful execution and compares stdout against golden file via `golden.RequireEqual` |
| `assertError` | `func(t *testing.T, err error, stdout, stderr bytes.Buffer)` | Validates error execution and compares stderr against golden file via `golden.RequireEqual` |

### Helper Function Details

**Diagram: Test Infrastructure Flow**

**Sources:** [fang\_test.go302-369](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L302-L369)

#### toMkroot Function

The `toMkroot` function wraps a `*cobra.Command` in a factory function. This pattern ensures each test gets a fresh command instance, preventing state contamination between tests.

**Usage Pattern:**

**Sources:** [fang\_test.go355-359](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L355-L359)

#### exercise Function

The `exercise` function runs three standard tests against any command factory:

1.  Error handling test: calls `doExercise` with args `[]string{"--nope-nope-nope"}` and `assertError`
2.  Version flag test: calls `doExercise` with args `[]string{"--version"}` and `assertNoError`
3.  Help flag test: calls `doExercise` with args `[]string{"--help"}` and `assertNoError`

Each test runs as a subtest (using `t.Run`) named "error", "version", and "help" respectively. This pattern ensures consistent baseline testing across all command configurations.

**Sources:** [fang\_test.go302-331](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L302-L331)

#### doExercise Function

The `doExercise` function is the workhorse that orchestrates test execution:

1.  Sets `t.Setenv("__FANG_TEST_WIDTH", "45")` for consistent output width
2.  Creates a fresh command via `mkroot()`
3.  Creates `bytes.Buffer` instances for stdout and stderr
4.  Configures command via `root.SetOut(&stdout)`, `root.SetErr(&stderr)`, and `root.SetArgs(args)`
5.  Executes via `fang.Execute(t.Context(), root, options...)`
6.  Passes `err`, `stdout`, and `stderr` to the assertion function for validation

The `__FANG_TEST_WIDTH` environment variable overrides terminal width detection to ensure reproducible output regardless of the actual terminal dimensions.

**Sources:** [fang\_test.go333-353](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L333-L353)

#### Assertion Functions

Two assertion functions handle the golden file comparison:

**assertNoError**: Used for successful command execution (help, version, etc.)

*   Calls `require.NoError(t, err, stderr.String())` to verify no error occurred
*   Calls `golden.RequireEqual(t, stdout.Bytes())` to compare stdout against golden file
*   If error occurs, stderr is displayed in the test failure message

**assertError**: Used for error scenarios (invalid flags, unknown commands, etc.)

*   Calls `require.Error(t, err, stdout.String())` to verify an error occurred
*   Calls `golden.RequireEqual(t, stderr.Bytes())` to compare stderr against golden file
*   If no error occurs, stdout is displayed in the test failure message

The `golden.RequireEqual` function from `github.com/charmbracelet/x/exp/golden`:

*   Automatically determines the golden file path from `t.Name()`
*   Compares the provided bytes against the golden file content
*   Fails the test with a diff if content doesn't match
*   Can regenerate golden files when run with `-update` flag

**Sources:** [fang\_test.go361-369](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L361-L369)
 [fang\_test.go10](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L10-L10)

Golden File Organization
------------------------

### Directory Structure

Golden files are organized in a hierarchical structure under the `testdata/` directory that mirrors the test structure:

    testdata/
    └── TestSetup/
        ├── simple/
        │   ├── help.golden
        │   ├── version.golden
        │   └── error.golden
        ├── complete/
        │   ├── help.golden
        │   ├── version.golden
        │   ├── error.golden
        │   └── man/
        │       └── help.golden
        ├── use_with_args/
        │   └── help.golden
        ├── with_flags/
        │   └── help.golden
        └── ...
    

### Naming Convention

Golden file names follow the pattern constructed by the `golden` package based on `t.Name()`:

    testdata/{TestFunctionName}/{SubtestLevel1}/{SubtestLevel2}/{scenario}.golden
    

For example:

*   `testdata/TestSetup/simple/help.golden` - Help output for the "simple" test
*   `testdata/TestSetup/with_flags/error.golden` - Error output for the "with\_flags" test
*   `testdata/TestSetup/complete/man/help.golden` - Help output for the "man" subcommand in "complete" test

The hierarchy is automatically derived from nested `t.Run()` calls. When `doExercise` runs a test named "help" under `TestSetup/simple`, the `golden.RequireEqual` call uses `t.Name()` to construct the path `testdata/TestSetup/simple/help.golden`.

**Sources:** [fang\_test.go361-369](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L361-L369)

### Updating Golden Files

When intentional changes are made to output formatting, golden files need to be updated. The `golden` package from `github.com/charmbracelet/x/exp/golden` supports automatic regeneration via the `-update` flag:

This command:

1.  Executes all tests normally
2.  Instead of comparing against existing golden files, writes current output to golden files
3.  Creates directories in `testdata/` as needed
4.  Overwrites existing `.golden` files with new content

After running with `-update`:

1.  Review the diffs carefully using `git diff testdata/`
2.  Verify that changes are intentional and correct
3.  Check for unintended formatting changes or regressions
4.  Commit the updated golden files along with the code changes

### Test Width Environment Variable

All tests call `t.Setenv("__FANG_TEST_WIDTH", "45")` in the `doExercise` function to set a fixed terminal width of 45 characters. This environment variable is read by fang's width calculation logic to override terminal width detection, ensuring:

*   Consistent wrapping and truncation across different terminal sizes
*   Reproducible output in CI/CD environments
*   Predictable golden file content

Without this fixed width, the same code would produce different output on terminals of different sizes, making golden file testing impractical.

**Sources:** [fang\_test.go341](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L341-L341)

Custom Error Handler Testing
----------------------------

Fang allows applications to provide custom error handlers via the `fang.WithErrorHandler(func(w io.Writer, styles fang.Styles, err error))` option. The test suite validates this integration in [fang\_test.go22-32](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L22-L32)
:

The test:

1.  Creates a simple command with `Use: "simple"`
2.  Provides a custom error handler that outputs `"Custom error handler: %v\n"`
3.  Executes the command with invalid args `[]string{"nope"}` to trigger an error
4.  Uses `assertError` to verify the custom error message appears in stderr
5.  Compares stderr against the golden file

This ensures:

*   The `WithErrorHandler` option correctly replaces the default error handler
*   Custom error handlers receive the error and can write to the error stream
*   The `styles` parameter is passed to custom handlers (though not used in this test)
*   Custom error output is correctly captured and validated

**Sources:** [fang\_test.go22-32](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go#L22-L32)
