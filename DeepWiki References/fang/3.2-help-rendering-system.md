Purpose and Scope
-----------------

The Help Rendering System is responsible for generating styled, formatted help output for CLI applications. It replaces Cobra's default help function with a custom implementation that provides syntax-highlighted usage examples, organized command groups, and adaptive terminal formatting. This system processes Cobra command structures and transforms them into visually appealing help text using the Lipgloss styling engine.

For information about how colors and styles are defined, see [Styling and Themes](https://deepwiki.com/charmbracelet/fang/3.3-styling-and-themes)
. For error message rendering, see [Error Handling](https://deepwiki.com/charmbracelet/fang/3.4-error-handling)
. For the overall Fang execution lifecycle, see [fang.Execute Function](https://deepwiki.com/charmbracelet/fang/3.1-fang.execute-function)
.

**Sources:** [help.go1-108](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L1-L108)

Architecture Overview
---------------------

**Diagram: Help Rendering Pipeline**

The help rendering system follows a three-stage pipeline: (1) content evaluation extracts data from Cobra commands, (2) styling functions apply visual formatting, and (3) rendering functions output to the terminal with appropriate color profile adaptation.

**Sources:** [help.go42-108](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L42-L108)
 [help.go152-483](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L152-L483)

Main Entry Point: helpFn
------------------------

The `helpFn` function is the primary entry point for help rendering, invoked when a user requests help for a command (via `--help` or the `help` subcommand). It orchestrates the entire help generation process.

**Diagram: helpFn Execution Flow**

**Sources:** [help.go42-108](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L42-L108)

### Function Signature

The function receives three parameters:

*   `c *cobra.Command` - The Cobra command to generate help for
*   `w *colorprofile.Writer` - A color-aware writer for terminal output
*   `styles Styles` - Pre-configured styling definitions

### Processing Steps

| Step | Function | Purpose |
| --- | --- | --- |
| 1   | `writeLongShort` | Renders command description (Long or Short text) |
| 2   | `styleUsage` | Generates styled usage line with program name, commands, args, flags |
| 3   | `styleExamples` | Processes and styles example code blocks |
| 4   | Block width calculation | Determines optimal width for code blocks based on terminal width |
| 5   | `evalGroups`, `evalCmds`, `evalFlags` | Extracts command groups, subcommands, and flags from Cobra structure |
| 6   | `calculateSpace` | Computes column spacing for aligned output |
| 7   | `renderGroup` | Renders each section (commands, flags) with proper formatting |

**Sources:** [help.go42-108](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L42-L108)

Terminal Width Adaptation
-------------------------

The help system adapts to terminal width dynamically:

**Diagram: Terminal Width Detection and Application**

The `width` function uses `sync.OnceValue` to cache the terminal width determination. For testing, the environment variable `__FANG_TEST_WIDTH` can override detection. The maximum width is capped at 120 columns to maintain readability on wide terminals.

**Sources:** [help.go30-40](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L30-L40)

Usage Section Rendering
-----------------------

The `styleUsage` function generates the usage line that appears at the top of help output.

### Usage Line Components

**Diagram: Usage Line Construction**

### Processing Logic

1.  **Placeholder Detection**: The function searches for standard placeholders (`[args]`, `[flags]`, `[command]`) in the usage string
2.  **Regex Extraction**: Custom argument patterns (e.g., `[filename]`, `[url]`) are extracted using `otherArgsRe`
3.  **Conditional Assembly**: Components are added only if relevant:
    *   `[command]` appears if subcommands exist
    *   `[args]` appears if explicitly declared or detected
    *   `[--flags]` appears if flags are available
4.  **Differential Styling**: The `complete` parameter determines whether to style the full command path (for root commands) or just the command name (for subcommands)

**Sources:** [help.go160-222](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L160-L222)

* * *

Examples Section Rendering
--------------------------

The examples section provides the most sophisticated syntax highlighting in the help system, supporting shell syntax, environment variables, pipes, redirects, and multi-line continuations.

### Example Line Parsing

**Diagram: Example Line Syntax Highlighting**

### Supported Syntax Elements

| Element | Detection | Style Applied |
| --- | --- | --- |
| Comments | Starts with `#` | `styles.Codeblock.Comment` |
| Program Name | Matches root command name or aliases | `styles.Program.Name` |
| Subcommands | Validated via `isSubCommand()` using Cobra's `Traverse()` | `styles.Program.Command` |
| Flags | Starts with `-` | `styles.Program.Flag` |
| Quoted Strings | Enclosed in `"` or `'` | `styles.Program.QuotedString` |
| Environment Variables | Pattern `KEY=value` before program name | `styles.Program.Flag` + `styles.Program.Argument` |
| Pipes & Operators | `\|`, `\|`, `&&`, `&` | `styles.Program.DimmedArgument` |
| Redirects | `>`, `<`, `&>`, `2>`, `1>`, `>>`, `2>>` | `styles.Program.DimmedArgument` |
| Arguments | Unquoted non-flag words | `styles.Program.Argument` |
| Line Continuations | Trailing `\` | Adds indent to next line |

Flags Section Evaluation
------------------------

The `evalFlags` function processes flag definitions, handling multiline descriptions and default values.

### Flag Formatting

| Component | Rendering |
| --- | --- |
| Shorthand + Long | `-s --long-flag` |
| Long only | `--long-flag` |
| Description (first line) | Styled with `styles.FlagDescription` (includes transform) |
| Description (continuation) | Styled without transform to maintain alignment |
| Default value | Appended as `(value)` with `styles.FlagDefault` |

### Multiline Description Handling

The function processes multiline flag descriptions while preserving text transforms only on the first line:

**Diagram: Multiline Flag Description Processing**

This approach ensures that the capitalization transform (if configured) only applies to the first line, preventing unintended text transformation of detailed descriptions.

**Sources:** [help.go366-412](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L366-L412)

Rendering and Layout

### Group Rendering

The `renderGroup` function outputs a titled section with aligned key-value pairs:

**Diagram: Group Rendering Layout**

**Sources:** [help.go447-457](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L447-L457)

### Space Calculation

The `calculateSpace` function determines the column width needed for aligned output:

*   **Inputs**: Arrays of styled command keys and flag keys
*   **Algorithm**: Find maximum width among all keys, add 2 spaces for separation
*   **Minimum**: 10 characters (defined by `minSpace` constant)
*   **Purpose**: Ensures value column aligns vertically across all rows

**Sources:** [help.go459-466](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L459-L466)

Color Profile Adaptation
------------------------

The help system adapts rendering based on terminal capabilities:

### Block Style Adaptation

**Diagram: Terminal Capability Adaptation**

When the terminal doesn't support colors or has no background color configured, vertical padding is removed from code blocks to improve readability in plain-text environments.

**Sources:** [help.go55-59](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L55-L59)

### Width Constraints

Code blocks respect terminal width constraints:

1.  **Maximum Width**: `min(terminalWidth - padding, contentWidth + padding)`
2.  **Truncation**: If individual example lines exceed block width, they are truncated with ellipsis using `ansi.Truncate()`
3.  **Capping**: Terminal width is capped at 120 columns maximum

**Sources:** [help.go47-72](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L47-L72)

Integration Points
------------------

### Cobra Hook Installation

The help function is installed as a custom help handler in `fang.Execute`:

    cmd.SetHelpFunc(func(c *cobra.Command, args []string) {
        helpFn(c, w, styles)
    })
    

This replaces Cobra's default help rendering entirely.

**Sources:** Referenced from fang.go context

### Error Handler Connection

The `DefaultErrorHandler` uses a similar styling approach but for error messages:

| Aspect | Help Rendering | Error Rendering |
| --- | --- | --- |
| Output Stream | stdout (via `colorprofile.Writer`) | stderr (checked for TTY) |
| TTY Detection | Via `colorprofile.Writer.Profile` | Via `term.IsTerminal()` |
| Fallback | Adapt styling to profile | Plain text if not TTY |
| Content | Usage, examples, commands, flags | Error message + usage suggestion |

**Sources:** [help.go110-132](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L110-L132)

Summary
-------

The Help Rendering System provides a complete replacement for Cobra's default help output with:

1.  **Syntax-highlighted examples** supporting shell syntax, quotes, pipes, redirects, and continuations
2.  **Adaptive terminal rendering** that detects width and color capabilities
3.  **Organized command groups** preserving definition order and group hierarchy
4.  **Aligned flag listings** with multiline description support and default value display
5.  **Styled usage lines** differentiating program names, commands, flags, and arguments
6.  **Terminal-aware formatting** with width constraints, truncation, and padding adaptation

The system integrates tightly with the Styles system (page 3.3) to apply consistent visual theming across all help output components.

**Sources:** [help.go1-483](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L1-L483)
