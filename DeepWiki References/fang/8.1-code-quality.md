Code Quality
============

Relevant source files

*   [.golangci.yml](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml)
    

Purpose and Scope
-----------------

This document covers the code quality standards enforced in the fang repository through automated linting and formatting. The configuration is defined in `.golangci.yml` and uses golangci-lint v2 to maintain consistent code style, catch potential bugs, and enforce best practices. This page is relevant for contributors developing fang itself. For information about using fang in your own applications, see page 2.

Configuration Structure
-----------------------

The fang repository uses golangci-lint v2 with a configuration file that defines three main categories: linters, formatters, and issues handling. The configuration is designed to maintain high code quality while avoiding false positives.

**.golangci.yml Configuration Structure**

Sources: [.golangci.yml1-48](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml#L1-L48)

Run Configuration
-----------------

The `run` section controls which files golangci-lint processes. Test files are excluded from linting to avoid enforcing production code standards on test helpers and fixtures.

| Setting | Value | Description |
| --- | --- | --- |
| `tests` | `false` | Excludes `*_test.go` files from linting |

This configuration means linters run only against production code in [fang.go](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go)
 [help.go](https://github.com/charmbracelet/fang/blob/d89b30af/help.go)
 [theme.go](https://github.com/charmbracelet/fang/blob/d89b30af/theme.go)
 and [error.go](https://github.com/charmbracelet/fang/blob/d89b30af/error.go)
 but not against [fang\_test.go](https://github.com/charmbracelet/fang/blob/d89b30af/fang_test.go)
 [error\_test.go](https://github.com/charmbracelet/fang/blob/d89b30af/error_test.go)
 or [theme\_test.go](https://github.com/charmbracelet/fang/blob/d89b30af/theme_test.go)

Sources: [.golangci.yml2-3](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml#L2-L3)

Linter Exclusions
-----------------

The configuration includes targeted exclusions to suppress false positives without disabling entire linters.

### Context Exclusion Rule

The `noctx` linter requires `context.Context` parameters in functions that perform I/O operations. However, this rule is suppressed for logging statements, which are synchronous and don't require context propagation.

**noctx Exclusion Rule**

| Setting | Value | Description |
| --- | --- | --- |
| `text` | `'(slog\|log)\.\w+'` | Regex pattern matching logging function calls |
| `linters` | `[noctx]` | Applies only to the `noctx` linter |

Sources: [.golangci.yml28-32](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml#L28-L32)

### Generated Code Handling

The configuration uses the `lax` preset for generated code, which reduces strictness for files marked with generation comments like `// Code generated ... DO NOT EDIT.`

| Setting | Value | Files Affected |
| --- | --- | --- |
| `generated` | `lax` | Auto-generated files (e.g., from `go generate`) |
| `presets` | `common-false-positives` | Common patterns that produce false warnings |

Sources: [.golangci.yml33-35](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml#L33-L35)

Linter Settings
---------------

The `exhaustive` linter is configured with custom behavior through its settings section.

**exhaustive Linter Configuration**

When `default-signifies-exhaustive` is `true`, a switch statement with a `default` case is considered exhaustive even if not all enum values are explicitly handled. This setting is relevant for code like the color scheme selection in `theme.go`.

Sources: [.golangci.yml36-38](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml#L36-L38)

Formatters
----------

Two formatters automatically fix code style issues during the linting process:

| Formatter | Purpose | Impact |
| --- | --- | --- |
| `gofumpt` | Stricter version of `gofmt` with additional formatting rules | Enforces consistent formatting beyond standard Go conventions |
| `goimports` | Organizes import statements and removes unused imports | Maintains clean import blocks with standard library first, then third-party packages |

The formatters are enabled in [.golangci.yml42-45](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml#L42-L45)
 and, like linters, use the `lax` exclusion for generated files.

**Formatter Pipeline**

Both formatters run automatically when golangci-lint is executed, ensuring all code in the repository maintains consistent style without manual intervention.

Sources: [.golangci.yml42-47](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml#L42-L47)

Issues Configuration
--------------------

The `issues` section controls how golangci-lint reports problems. Both settings are configured to `0`, meaning no limits:

| Setting | Value | Effect |
| --- | --- | --- |
| `max-issues-per-linter` | `0` | Reports all issues found by each linter (no cap) |
| `max-same-issues` | `0` | Reports all occurrences of the same issue (no deduplication) |

This strict configuration ensures that all code quality issues are surfaced rather than being hidden by reporting limits. In practice, this means:

*   If `bodyclose` finds 5 unclosed response bodies, all 5 are reported
*   If the same `goconst` violation appears in 10 places, all 10 are reported
*   Developers must fix all issues; the tool won't hide problems to make output cleaner

Sources: [.golangci.yml39-41](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml#L39-L41)

Summary
-------

The fang repository enforces code quality through:

1.  **22 enabled linters** covering security, correctness, clarity, and best practices
2.  **2 formatters** (`gofumpt`, `goimports`) for automatic style fixes
3.  **Targeted exclusions** for logging calls and generated code
4.  **Zero-tolerance reporting** with no issue limits
5.  **Production-only linting** with test files excluded

This configuration maintains high code quality standards while minimizing false positives through carefully chosen exclusions and linter settings. Contributors must pass all linter checks before their code can be merged.

Sources: [.golangci.yml1-48](https://github.com/charmbracelet/fang/blob/d89b30af/.golangci.yml#L1-L48)
