Signal Handling
===============

Relevant source files

*   [example/main.go](https://github.com/charmbracelet/fang/blob/d89b30af/example/main.go)
    
*   [fang.go](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go)
    

This document explains how Fang enables context-aware signal handling in Cobra applications, allowing commands to respond gracefully to OS signals like `SIGINT` and `SIGTERM`. Signal handling is configured through the `WithNotifySignal` option and integrates with Go's standard `context` and `signal` packages to provide automatic context cancellation when signals are received.

For information about error handling and display, see [Error Handling](https://deepwiki.com/charmbracelet/fang/3.4-error-handling)
. For configuration options in general, see [Configuration Options](https://deepwiki.com/charmbracelet/fang/4-configuration-options)
.

Purpose and Scope
-----------------

The signal handling feature in Fang provides:

1.  **Automatic Context Cancellation**: Converts OS signals into context cancellation events
2.  **Graceful Shutdown**: Enables commands to detect interruption and clean up resources
3.  **Standard Signal Support**: Works with any `os.Signal` including `os.Interrupt`, `os.Kill`, `syscall.SIGTERM`, etc.
4.  **Zero Boilerplate**: Eliminates the need for manual signal channel setup in application code

This feature is optional and only activates when `WithNotifySignal` is provided to `fang.Execute`.

Configuration
-------------

Signal handling is configured using the `WithNotifySignal` option function, which accepts one or more `os.Signal` values.

### WithNotifySignal Function

The `WithNotifySignal` function is defined at [fang.go101-107](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L101-L107)
:

This function accepts a variadic list of signals to monitor. When any of these signals are received, the context passed to the command execution will be canceled.

**Common Signals:**

| Signal | Constant | Description |
| --- | --- | --- |
| SIGINT | `os.Interrupt` | User pressed Ctrl+C |
| SIGKILL | `os.Kill` | Force termination (cannot be caught on all platforms) |
| SIGTERM | `syscall.SIGTERM` | Graceful termination request |

Internal Implementation
-----------------------

### Settings Storage

Signal configuration is stored in the `settings` struct at [fang.go27-36](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L27-L36)
:

| Field | Type | Purpose |
| --- | --- | --- |
| `signals` | `[]os.Signal` | List of signals to monitor |

When `WithNotifySignal` is called, it populates the `signals` field which is later used during execution.

### Context Creation

The actual signal handling occurs in the `Execute` function at [fang.go167-171](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L167-L171)
:

**Signal Handling Flow with Code References**

The implementation follows this sequence:

1.  **Check Configuration** [fang.go167](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L167-L167)
    : If `len(opts.signals) > 0`, signal handling is enabled
2.  **Create Notify Context** [fang.go169](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L169-L169)
    : Call `signal.NotifyContext(ctx, opts.signals...)` to wrap the original context
3.  **Defer Cleanup** [fang.go170](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L170-L170)
    : Register `defer cancel()` to ensure cleanup
4.  **Execute with Context** [fang.go173](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L173-L173)
    : Pass the signal-aware context to `root.ExecuteContext(ctx)`

The `signal.NotifyContext` function from Go's standard library creates a context that will be canceled when any of the specified signals are received. This is a clean abstraction that eliminates manual signal channel management.

Sources: [fang.go167-171](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L167-L171)

| Component | Code Reference | Purpose |
| --- | --- | --- |

| Context Check | `<-c.Context().Done()` | Detects signal/cancellation |
| Error Return | `c.Context().Err()` | Returns appropriate error |

**Execution Flow:**

**Graceful Shutdown Pattern in example/main.go**

When a user presses Ctrl+C during execution:

1.  The `signal.NotifyContext` receives the signal
2.  The context is canceled
3.  `c.Context().Done()` channel closes
4.  The `select` statement returns from the context case
5.  The function returns `c.Context().Err()` which is typically `context.Canceled`

Sources: [example/main.go124-133](https://github.com/charmbracelet/fang/blob/d89b30af/example/main.go#L124-L133)

Context Error Handling
----------------------

### Error Types

When a context is canceled due to a signal, `c.Context().Err()` returns one of:

| Error | Constant | Meaning |
| --- | --- | --- |
| Canceled | `context.Canceled` | Context was explicitly canceled |
| DeadlineExceeded | `context.DeadlineExceeded` | Context timeout (if deadline was set) |

### Error Propagation

**Error Flow from Command to Error Handler**

When `c.Context().Err()` is returned from a command:

1.  Cobra's `ExecuteContext` receives the error [fang.go173](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L173-L173)
    
2.  `fang.Execute` checks `if err != nil` [fang.go173](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L173-L173)
    
3.  The error handler is invoked [fang.go175](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L175-L175)
    
4.  The styled error is written to stderr
5.  The error is returned to the caller [fang.go176](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L176-L176)

Platform Considerations
-----------------------

### Signal Availability

Not all signals are available on all platforms:

*   **Unix/Linux/macOS**: Full signal support including `SIGINT`, `SIGTERM`, `SIGHUP`, etc.
*   **Windows**: Limited signal support; `os.Interrupt` works with Ctrl+C when console control handlers are registered
*   **Cross-Platform**: `os.Interrupt` is the most portable signal

### Context Cleanup

The `defer cancel()` at [fang.go170](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L170-L170)
 ensures proper cleanup:

*   Stops the signal goroutine created by `signal.NotifyContext`
*   Releases resources even if execution completes normally
*   Prevents goroutine leaks in long-running applications

Sources: [fang.go167-171](https://github.com/charmbracelet/fang/blob/d89b30af/fang.go#L167-L171)
