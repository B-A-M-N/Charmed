Error Handling
==============

Purpose and Scope
-----------------

This document explains fang's error handling system, which provides styled error messages with intelligent detection of usage errors. The system enhances Cobra's default error output with theming, TTY detection, and contextual help suggestions.

For information about styling and themes used in error rendering, see [Styling and Themes](https://deepwiki.com/charmbracelet/fang/3.3-styling-and-themes)
. For the overall command execution flow, see [fang.Execute Function](https://deepwiki.com/charmbracelet/fang/3.1-fang.execute-function)
.

* * *

Overview
--------

Fang's error handling system intercepts errors from Cobra command execution and formats them with styled output. The system consists of three main components:

| Component | Purpose | Location |
| --- | --- | --- |
| `DefaultErrorHandler` | Main error formatting function | [help.go110-132](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L110-L132) |
| `isUsageError` | Detects flag/command syntax errors | [help.go136-150](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L136-L150) |
| TTY Detection | Determines whether to apply styling | [help.go112-119](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L112-L119) |

The error handler is designed to provide clear, styled feedback to users while gracefully degrading to plain text output when stderr is not a terminal.

**Sources:** [help.go110-150](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L110-L150)

DefaultErrorHandler Function
----------------------------

### Function Signature

    func DefaultErrorHandler(w io.Writer, styles Styles, err error)
    

The `DefaultErrorHandler` is the default `ErrorHandler` implementation that formats and outputs error messages with optional styling.

### Parameters

| Parameter | Type | Description |
| --- | --- | --- |
| `w` | `io.Writer` | Output destination (typically stderr) |
| `styles` | `Styles` | Styling configuration from theme system |
| `err` | `error` | The error to be displayed |

**Sources:** [help.go110-132](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L110-L132)

TTY Detection and Output Mode
-----------------------------

The error handler adapts its output based on whether stderr is connected to a terminal:

| Condition | Check | Output Format | Code Location |
| --- | --- | --- | --- |
| Type assertion fails | `w.(term.File)` returns `ok == false` | Styled (assumes TTY) | [help.go112](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L112-L112) |
| Not a terminal | `term.IsTerminal(w.Fd())` returns `false` | Plain text only | [help.go115-117](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L115-L117) |
| Is a terminal | `term.IsTerminal(w.Fd())` returns `true` | Fully styled | [help.go120-131](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L120-L131) |

### Implementation Details

The TTY detection at [help.go112-119](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L112-L119)
 follows this sequence:

1.  **Type assertion**: `if w, ok := w.(term.File); ok`
    
    *   If assertion fails, skip TTY check and proceed with styling
    *   If assertion succeeds, writer has a file descriptor
2.  **Terminal check**: `if !term.IsTerminal(w.Fd())`
    
    *   Calls `term.IsTerminal()` to check if fd is a TTY
    *   Uses the `github.com/charmbracelet/x/term` package
3.  **Plain output**: When not a TTY
    
    *   Single line: `fmt.Fprintln(w, err.Error())`
    *   Returns immediately without styling
4.  **Styled output**: When a TTY or type assertion failed
    
    *   Proceeds to full error formatting with `styles.ErrorHeader` and `styles.ErrorText`

This ensures error messages display correctly when:

*   Redirected to files (`mycmd 2> errors.log`)
*   Piped to other programs (`mycmd 2>&1 | grep error`)
*   Shown in CI/CD logs without ANSI support
*   Displayed in interactive terminals with full styling

**Sources:** [help.go112-119](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L112-L119)

Usage Error Detection
---------------------

### Pattern Matching

The `isUsageError` function at [help.go136-150](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L136-L150)
 detects common command-line syntax errors by checking error message prefixes:

**Diagram: isUsageError pattern matching logic using strings.HasPrefix**

### Detected Error Patterns

| Error Prefix | Cause | Example |
| --- | --- | --- |
| `flag needs an argument:` | Flag missing required value | `--output` without filename |
| `unknown flag:` | Unrecognized long flag | `--nope-nope-nope` |
| `unknown shorthand flag:` | Unrecognized short flag | `-z` |
| `unknown command` | Invalid subcommand | `mycmd invalidcmd` |
| `invalid argument` | Argument validation failure | Wrong type or format |

### Implementation Note

The implementation at [help.go136-150](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L136-L150)
 includes a comment noting this is a workaround pending Cobra PR #2266, suggesting this pattern matching approach may be temporary pending upstream improvements.

**Sources:** [help.go136-150](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L136-L150)

Styled Error Output
-------------------

When stderr is a terminal, errors are rendered with three distinct components:

### Output Structure

The styled error output consists of multiple components written sequentially to the writer:

**Diagram: Sequence of output operations in DefaultErrorHandler**

### Output Components

| Line | Component | Function Call | Purpose |
| --- | --- | --- | --- |
| 1   | Error Header | `fmt.Fprintln(w, styles.ErrorHeader.String())` | Display "ERROR" banner or header |
| 2   | Error Message | `fmt.Fprintln(w, styles.ErrorText.Render(err.Error()+"."))` | Show actual error text with period |
| 3   | Blank Line | `fmt.Fprintln(w)` | Visual spacing |
| 4   | Help Hint (conditional) | `lipgloss.JoinHorizontal(...)` | Usage help suggestion |
| 5   | Blank Line (conditional) | `fmt.Fprintln(w)` | Final spacing |

#### Help Hint Components

When `isUsageError(err)` returns `true`, the help hint is assembled using `lipgloss.JoinHorizontal` with three parts:

1.  **"Try" text**: `styles.ErrorText.UnsetWidth().Render("Try")`
    
    *   Uses error text style with width unset for inline rendering
2.  **"--help" flag**: `styles.Program.Flag.Render(" --help ")`
    
    *   Styled as a flag with spaces for readability
    *   Uses `styles.Program.Flag` from the Program style set
3.  **"for usage." text**: `styles.ErrorText.UnsetWidth().UnsetMargins().UnsetTransform().Render("for usage.")`
    
    *   Strips width, margins, and transform for plain inline text
    *   Ensures consistent spacing in the joined output

**Sources:** [help.go120-131](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L120-L131)

Integration with fang.Execute
-----------------------------

The error handler is configured and invoked as part of the `fang.Execute` lifecycle. The default handler can be customized through the `WithErrorHandler` option.

### Error Handler Configuration

| Component | Default Value | Customization |
| --- | --- | --- |
| Handler Function | `DefaultErrorHandler` | `WithErrorHandler(ErrorHandler)` option |
| Output Writer | `os.Stderr` | Passed to handler by fang.Execute |
| Styles | From `ColorScheme` | `WithColorSchemeFunc` option |

### Handler Invocation Flow

**Diagram: Error handler invocation in the fang.Execute lifecycle**

The error handler is invoked automatically when `cmd.ExecuteContext(ctx)` returns a non-nil error. The handler receives the configured `io.Writer` (typically stderr), the `Styles` derived from the `ColorScheme`, and the error itself.

**Sources:** [help.go110-132](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L110-L132)

* * *

Customization
-------------

While `DefaultErrorHandler` provides sensible defaults, applications can provide custom error handlers through the configuration system. Custom error handlers receive the same parameters (`io.Writer`, `Styles`, `error`) and can implement alternative formatting logic.

Key customization points:

*   Custom error message formatting
*   Alternative usage error detection
*   Different styling approaches
*   Additional context or debugging information

See [Configuration Options](https://deepwiki.com/charmbracelet/fang/4-configuration-options)
 for details on using `WithErrorHandler` option.

**Sources:** [help.go110-132](https://github.com/charmbracelet/fang/blob/d89b30af/help.go#L110-L132)
