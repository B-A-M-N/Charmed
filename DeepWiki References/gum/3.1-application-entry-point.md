Purpose and Scope
-----------------

This document details how the Gum application bootstraps and initializes itself through the `main.go` entry point. It covers the Kong CLI parser configuration, command registration via the `Gum` struct, version management, lipgloss color profile initialization, and error handling patterns that translate runtime errors into appropriate exit codes.

For information about individual command implementations, see [Interactive Commands](https://deepwiki.com/charmbracelet/gum/4-interactive-commands)
, [Progress and Background Commands](https://deepwiki.com/charmbracelet/gum/5-progress-and-background-commands)
, and [Formatting Commands](https://deepwiki.com/charmbracelet/gum/6-formatting-commands)
. For details on the Charmbracelet framework dependencies, see [Framework Components](https://deepwiki.com/charmbracelet/gum/3.2-framework-components)
.

Main Function Structure
-----------------------

The `main()` function in [main.go31-91](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L31-L91)
 orchestrates the application bootstrap through five distinct phases:

| Phase | Lines | Purpose |
| --- | --- | --- |
| Lipgloss Setup | [main.go32](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L32-L32) | Initialize color profile detection for terminal styling |
| Version Resolution | [main.go34-44](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L34-L44) | Determine version from ldflags or build info |
| Kong Configuration | [main.go46-75](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L46-L75) | Parse CLI arguments and configure help output |
| Command Execution | [main.go76](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L76-L76) | Invoke the matched command's `Run()` method |
| Error Handling | [main.go77-90](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L77-L90) | Translate errors to appropriate exit codes |

### Version Management

The application supports two version sources with fallback behavior:

1.  **Ldflags injection**: Variables `Version` and `CommitSHA` [main.go19-27](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L19-L27)
     are populated at build time via `-ldflags "-X main.Version=..."` by GoReleaser.
2.  **Build info fallback**: When built with `go install`, `debug.ReadBuildInfo()` [main.go35-36](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L35-L36)
     retrieves the module version.
3.  **Source build fallback**: Defaults to `"unknown (built from source)"` [main.go38](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L38-L38)
    

The version string format includes an optional commit SHA prefix. The `shaLen` constant [main.go17](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L17-L17)
 truncates the commit SHA to 7 characters [main.go43](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L43-L43)

**Sources:** [main.go17-44](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L17-L44)

### Help Configuration

The `kong.ConfigureHelp()` options [main.go51-55](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L51-L55)
 customize the help output format:

*   `Compact: true` - Minimizes whitespace in help text.
*   `Summary: false` - Disables command summary section.
*   `NoExpandSubcommands: true` - Prevents automatic expansion of all subcommands.

Combined with `kong.UsageOnError()` [main.go50](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L50-L50)
 invalid invocations display usage information before exiting.

### Command Categories

The struct fields organize commands into logical categories:

| Category | Commands | Struct Tags |
| --- | --- | --- |
| Meta | `Version`, `Completion`, `Man` | `Version`: `short:"v"`, Others: `hidden:""` |
| Interactive Selection | `Choose`, `Filter`, `File` | `cmd:""` |
| User Input | `Input`, `Write`, `Confirm` | `cmd:""` |
| Data Display | `Table`, `Pager` | `cmd:""` |
| Progress | `Spin` | `cmd:""` |
| Formatting | `Style`, `Format`, `Join`, `Log` | `cmd:""` |

The `cmd:""` tag [gum.go46-228](https://github.com/charmbracelet/gum/blob/30cc5169/gum.go#L46-L228)
 marks fields as Kong subcommands. The `hidden:""` tag [gum.go30-33](https://github.com/charmbracelet/gum/blob/30cc5169/gum.go#L30-L33)
 prevents commands from appearing in default help output.

**Sources:** [gum.go25-228](https://github.com/charmbracelet/gum/blob/30cc5169/gum.go#L25-L228)

### Error Type Mapping

| Error Type | Detection Method | Exit Code | Use Case |
| --- | --- | --- | --- |
| `exit.ErrExit` | `errors.As()` [main.go78](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L78-L78) | Custom (cast to int) | Command-specific exit codes (e.g., `confirm` returns 0/1) |
| `tea.ErrInterrupted` | `errors.Is()` [main.go81](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L81-L81) | 130 (`exit.StatusAborted`) | User pressed Ctrl+C |
| `tea.ErrProgramKilled` | `errors.Is()` [main.go84](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L84-L84) | 124 (`exit.StatusTimeout`) | Timeout exceeded via background execution |
| Other errors | Default case [main.go88-89](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L88-L89) | 1   | Generic failures (validation, I/O errors) |

### Exit Code Constants

The `internal/exit` package defines semantic exit code constants:

*   `exit.StatusAborted` = 130 [internal/exit/exit.go10](https://github.com/charmbracelet/gum/blob/30cc5169/internal/exit/exit.go#L10-L10)
    
*   `exit.StatusTimeout` = 124 [internal/exit/exit.go7](https://github.com/charmbracelet/gum/blob/30cc5169/internal/exit/exit.go#L7-L7)
    
*   `exit.ErrExit` is a custom type allowing integer casting to `os.Exit` [internal/exit/exit.go13-16](https://github.com/charmbracelet/gum/blob/30cc5169/internal/exit/exit.go#L13-L16)

Command Execution Pattern
-------------------------

The `ctx.Run()` call [main.go76](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L76-L76)
 performs reflection-based method invocation on the matched subcommand's `Options` struct. For example, the `version` command implementation in [version/command.go12-26](https://github.com/charmbracelet/gum/blob/30cc5169/version/command.go#L12-L26)
 demonstrates how the `Run` method accesses the Kong context and variables:

This pattern ensures that while `main.go` bootstraps the application, the actual business logic remains decoupled within specific command packages.
