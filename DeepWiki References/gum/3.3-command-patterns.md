This document describes the common architectural patterns shared across gum's 15+ commands. Understanding these patterns is essential for comprehending how individual commands are implemented and how they maintain consistent behavior.

For details on how commands are registered and routed through the application entry point, see [Application Entry Point](https://deepwiki.com/charmbracelet/gum/3.1-application-entry-point)
. For information about the underlying Charmbracelet framework components that these patterns leverage, see [Framework Components](https://deepwiki.com/charmbracelet/gum/3.2-framework-components)
.

Pattern Overview
----------------

All gum commands follow consistent structural patterns that enable maintainability and predictable behavior. Commands are organized into three categories, each with distinct execution patterns:

1.  **Interactive Commands**: Present TUI interfaces using Bubbletea (choose, filter, input, write, file, pager, confirm, table).
2.  **Formatting Commands**: Transform text without user interaction (style, format, join, log).
3.  **Background Commands**: Execute processes while displaying progress (spin).

Despite these differences, all commands share core patterns for configuration, execution, and I/O handling.

Options Struct Pattern
----------------------

Every command defines an `Options` struct that serves as the single source of configuration. This struct is populated by the `kong` library through a cascade of configuration sources.

### Configuration Cascade

The configuration system prioritizes sources in this order (highest to lowest):

| Priority | Source | Implementation | Example |
| --- | --- | --- | --- |
| 1   | CLI Arguments | Kong struct tags `arg:""` or named flags | `gum format --type code` |
| 2   | Environment Variables | Kong `env:""` tag | `GUM_FORMAT_TYPE=code` |
| 3   | Default Values | Kong `default:""` tag | `default:"markdown"` |

**Sources**: [format/options.go5-13](https://github.com/charmbracelet/gum/blob/30cc5169/format/options.go#L5-L13)
 [main.go47-75](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L47-L75)

Run Method Pattern
------------------

Every command implements a `Run() error` method on its `Options` struct. This method orchestrates the command's logic, whether it's launching a TUI or processing text.

### Non-Interactive Command Run Pattern

Formatting commands like `format` process input and print directly to stdout without a full Bubbletea loop.

Stdin/Stdout Handling
---------------------

Gum maintains a strict separation between user interface (TUI) and data output.

### Stdin Reading

The `internal/stdin` package provides a standard way to read piped input. It includes options to strip ANSI sequences or read a single line.

[internal/stdin/stdin.go37-77](https://github.com/charmbracelet/gum/blob/30cc5169/internal/stdin/stdin.go#L37-L77)
:

*   `stdin.Read(opts ...Option)`: Primary function for reading input.
*   `stdin.IsEmpty()`: Checks if data is available on the pipe [internal/stdin/stdin.go80-91](https://github.com/charmbracelet/gum/blob/30cc5169/internal/stdin/stdin.go#L80-L91)
    

### TTY-Aware Output

The `internal/tty` package handles printing based on whether the output is a terminal. If output is redirected (e.g., to a file), it can strip ANSI sequences to keep the data clean.

Bubbletea Model Lifecycle
-------------------------

Interactive commands implement the Bubbletea Model-View-Update (MVU) pattern.

1.  **Init**: Returns initial commands (e.g., cursor blinking).
2.  **Update**: Processes messages (key presses, window resizes) and updates state.
3.  **View**: Returns a string representation of the UI using `lipgloss`.

For interactive commands, the `Run()` method typically:

1.  Creates the `model`.
2.  Sets up a `timeout.Context` [internal/timeout/context.go10-16](https://github.com/charmbracelet/gum/blob/30cc5169/internal/timeout/context.go#L10-L16)
    
3.  Initializes `tea.NewProgram` with `tea.WithOutput(os.Stderr)` to ensure the UI does not interfere with `stdout`.
4.  Calls `p.Run()`.

Exit Code Pattern
-----------------

Gum uses specific exit codes to signal the result of an operation to the calling shell.

*   **0**: Success / Affirmative (e.g., `gum confirm` "Yes").
*   **1**: Rejection / Negative (e.g., `gum confirm` "No").
*   **130**: Aborted (Ctrl+C).

Styling Configuration Pattern
-----------------------------

Styling is managed through the `lipgloss` library and standard Kong variables. `main.go` defines a set of default style variables that are available to all commands via Kong's `Vars` map.

[main.go56-74](https://github.com/charmbracelet/gum/blob/30cc5169/main.go#L56-L74)

Commands use these variables in their `Options` struct tags to maintain a consistent look and feel across the entire application.
