This document covers the validation system in the charmbracelet/huh library, which enables form fields to validate user input before submission. Validation functions ensure that user input meets specific criteria before a form can be completed, providing immediate feedback when input is invalid.

Overview of Validation in huh
-----------------------------

The validation system in huh is designed to be flexible and easy to use. It allows you to add validation rules to form fields, with error messages that are displayed when validation fails. When a field fails validation, the user cannot proceed to the next field until valid input is provided.

Built-in Validation Functions
-----------------------------

The library provides several built-in validation functions in the `validate.go` file:

| Function | Description |
| --- | --- |
| `ValidateNotEmpty()` | Ensures input is not empty |
| `ValidateMinLength(n)` | Ensures input has at least n characters |
| `ValidateMaxLength(n)` | Ensures input has at most n characters |
| `ValidateLength(min, max)` | Ensures input length is between min and max |
| `ValidateOneOf(options...)` | Ensures input is one of the specified options |

These functions return a validation function with signature `func(string) error` which can be passed to a field's `Validate` method.

Sources: [validate.go9-61](https://github.com/charmbracelet/huh/blob/151ba059/validate.go#L9-L61)

Validation Workflow
-------------------

The validation process in huh follows a specific workflow:

1.  **Setup**: Define validation functions for each field
2.  **User Interaction**: User enters data into the field
3.  **Trigger**: Validation is triggered when:
    *   User attempts to navigate to the next field
    *   User attempts to submit the form
4.  **Result**:
    *   If validation passes, navigation proceeds
    *   If validation fails, an error is displayed and focus remains on the field

Error Handling and Display
--------------------------

When validation fails, the error is:

1.  Stored in the field's `err` property
2.  Displayed visually next to the field title
3.  In accessible mode, presented as text output

Each field type renders errors according to the active theme's `ErrorIndicator` style. The error indicator is typically displayed adjacent to the field's title.

Accessibility Considerations
----------------------------

huh provides special handling for validation in accessible mode:

1.  Errors are presented as plain text
2.  Users are re-prompted until valid input is provided
3.  Validation logic is shared between accessible and visual modes

Best Practices
--------------

Here are some recommended practices for implementing validation in huh:

1.  **Clear Error Messages**: Provide specific, actionable error messages
2.  **Immediate Feedback**: Validate on field exit rather than waiting for form submission
3.  **Consistent Rules**: Apply consistent validation rules across similar fields
4.  **Accessible Descriptions**: Ensure error messages make sense when read by screen readers
5.  **Composable Validation**: Build complex validation from simpler functions

Conclusion
----------

The validation system in huh provides a flexible and user-friendly way to ensure form data meets your requirements. By using the built-in validation functions and creating custom validation logic, you can create robust forms that guide users toward providing valid input.
