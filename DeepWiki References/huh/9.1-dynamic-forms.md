Purpose and Scope
-----------------

This document explains how to create forms with content that responds dynamically to user input in the charmbracelet/huh library. While standard forms have static content, dynamic forms allow fields to change their titles, options, and other properties based on the values of other fields. For information about form validation techniques, see [Validation](https://deepwiki.com/charmbracelet/huh/9.2-validation)
.

Dynamic Form Concepts
---------------------

Dynamic forms in charmbracelet/huh allow you to create interactive experiences where form elements update in response to user input. This enables complex workflows and conditional logic within your terminal forms.

Sources: [README.md283-354](https://github.com/charmbracelet/huh/blob/151ba059/README.md?plain=1#L283-L354)

### Core Pattern: The `*Func` Method System

Dynamic functionality is implemented through a consistent pattern where static methods have corresponding `*Func` versions that accept a function and binding values:

| Static Method | Dynamic Method | Purpose |
| --- | --- | --- |
| `Title()` | `TitleFunc()` | Sets a field's title dynamically |
| `Description()` | `DescriptionFunc()` | Sets a field's description dynamically |
| `Options()` | `OptionsFunc()` | Provides dynamically changing options |
| `Validate()` | `ValidateFunc()` | Dynamic validation logic |

The dynamic methods follow a common signature pattern:

1.  First parameter is a function that returns the appropriate value type
2.  Second parameter(s) are bindings - references to values that trigger recomputation when changed

Sources: [README.md327-342](https://github.com/charmbracelet/huh/blob/151ba059/README.md?plain=1#L327-L342)

### Dynamic Field Example Code Structure

When implementing a dynamic form, you typically:

1.  Define variables to store user selections
2.  Create fields with static properties for non-changing elements
3.  Use `*Func` methods for dynamic elements, passing:
    *   A function that computes the new value
    *   Reference(s) to variables whose changes should trigger recomputation

Performance Considerations
--------------------------

### Caching

Dynamic functions in charmbracelet/huh are only recomputed when their bindings change, not on every render cycle. The library internally caches computed values to avoid unnecessary recalculations.

This caching mechanism ensures that expensive operations (like API calls in the country/state example) are only performed when necessary.

Sources: [README.md316-321](https://github.com/charmbracelet/huh/blob/151ba059/README.md?plain=1#L316-L321)

Best Practices
--------------

1.  **Appropriate Bindings**: Only bind to values that directly influence the dynamic property
2.  **Performance**: Keep dynamic functions lightweight when possible
3.  **Error Handling**: Consider edge cases in dynamic functions, especially when they depend on external data
4.  **User Experience**: Ensure dynamic changes are intuitive and don't disrupt the user's flow
5.  **Update Limits**: When using dynamic options that could yield many values, consider using the `Limit()` method to constrain list size

Summary
-------

Dynamic forms in charmbracelet/huh provide a powerful way to create responsive, interactive terminal user interfaces. By using the `*Func` family of methods with appropriate bindings, you can create forms that adapt to user input, show contextually relevant options, and guide users through complex decision trees.
